## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пустой).

## Описание команды

Определить несогласованности, дубликаты, двусмысленности и недостаточно определенные элементы в трех основных артефактах (`spec.md`, `plan.md`, `tasks.md`) перед реализацией. Эта команда ДОЛЖНА запускаться только после успешного завершения `/speckit.tasks`, которое создало полный `tasks.md`.

## Ограничения выполнения

**СТРОГО ТОЛЬКО ЧТЕНИЕ**: НЕ изменяйте никакие файлы. Выведите структурированный отчет об анализе. Предложите план устранения (пользователь должен явно одобрить перед любыми последующими командами редактирования).

**Авторитет Конституции**: Конституция проекта (`.qwen/specify/memory/constitution.md`) является **непреложной** в рамках этого анализа. Конфликты с конституцией автоматически являются КРИТИЧЕСКИМИ и требуют корректировки спецификации, плана или задач — а не ослабления, переинтерпретации или молчаливого игнорирования принципа. Если сам принцип нуждается в изменении, это должно происходить в отдельном, явном обновлении конституции вне `/speckit.analyze`.

## Процесс выполнения

### 1. Инициализировать контекст анализа

Запустить `.qwen/specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` один раз из корня репозитория и распарсить JSON для FEATURE_DIR и AVAILABLE_DOCS. Вывести абсолютные пути:

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

Прервать с сообщением об ошибке, если какой-либо необходимый файл отсутствует (проинформировать пользователя о необходимости запустить отсутствующую команду-предварительное условие).
Для одинарных кавычек в аргументах, таких как "I'm Groot", использовать экранирующий синтаксис: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

### 2. Загрузить артефакты (Постепенное раскрытие)

Загрузить только минимально необходимый контекст из каждого артефакта:

**Из spec.md:**

- Обзор/Контекст
- Функциональные требования
- Нефункциональные требования
- Истории пользователей
- Граничные случаи (если присутствуют)

**Из plan.md:**

- Архитектура/выбор технологий
- Ссылки на модели данных
- Фазы
- Технические ограничения

**Из tasks.md:**

- Идентификаторы задач
- Описания
- Группировка по фазам
- Маркеры параллельности [P]
- Указанные пути к файлам

**Из конституции:**

- Загрузить `.qwen/specify/memory/constitution.md` для проверки принципов

### 3. Создать семантические модели

Создать внутренние представления (не включать исходные артефакты в вывод):

- **Инвентарь требований**: Каждое функциональное + нефункциональное требование с постоянным ключом (вывести слаг на основе императивного выражения; например, "Пользователь может загружать файл" → `user-can-upload-file`)
- **Инвентарь историй/действий пользователя**: Отдельные действия пользователя с критериями приемки
- **Сопоставление покрытия задачами**: Сопоставить каждую задачу с одним или несколькими требованиями или историями (вывод по ключевым словам / явным ссылочным шаблонам, таким как идентификаторы или ключевые фразы)
- **Набор правил конституции**: Извлечь названия принципов и нормативные утверждения MUST/SHOULD

### 4. Проходы обнаружения (Эффективный анализ токенов)

Сосредоточиться на высокозначимых находках. Ограничиться 50 находками всего; объединить остальные в сводку переполнения.

#### A. Обнаружение дубликатов

- Определить почти дублирующиеся требования
- Отметить менее качественную формулировку для объединения

#### B. Обнаружение двусмысленности

- Отметить расплывчатые прилагательные (быстрый, масштабируемый, безопасный, интуитивный, надежный), не имеющие измеримых критериев
- Отметить нерешенные заполнители (TODO, TKTK, ???, `<placeholder>`, и т.д.)

#### C. Недостаточная спецификация

- Требования с глаголами, но без объекта или измеримого результата
- Истории пользователей без критериев приемки
- Задачи, ссылающиеся на файлы или компоненты, не определенные в спецификации/плане

#### D. Согласование с конституцией

- Любое требование или элемент плана, противоречащее принципу MUST
- Отсутствующие обязательные разделы или контрольные точки качества из конституции

#### E. Пробелы в покрытии

- Требования без связанных задач
- Задачи без сопоставленного требования/истории
- Нефункциональные требования, не отраженные в задачах (например, производительность, безопасность)

#### F. Несоответствие

- Изменение терминологии (одно и то же понятие называется по-разному в разных файлах)
- Сущности данных, упомянутые в плане, но отсутствующие в спецификации (и наоборот)
- Противоречия в порядке задач (например, задачи интеграции до базовой настройки без пометки зависимости)
- Противоречивые требования (например, одно требует Next.js, другое указывает Vue)

### 5. Назначение серьезности

Использовать эту эвристику для приоритизации находок:

- **КРИТИЧЕСКИЙ**: Нарушает MUST конституции, отсутствует основной артефакт спецификации или требование без покрытия, блокирующее базовую функциональность
- **ВЫСОКИЙ**: Дублирующее или противоречивое требование, двусмысленный атрибут безопасности/производительности, нетестируемый критерий приемки
- **СРЕДНИЙ**: Изменение терминологии, отсутствующее покрытие задачами нефункциональных требований, недостаточно определенный граничный случай
- **НИЗКИЙ**: Улучшения стиля/формулировки, небольшая избыточность, не влияющая на порядок выполнения

### 6. Создать компактный отчет анализа

Вывести отчет в формате Markdown (без записи файлов) со следующей структурой:

## Отчет анализа спецификации

| ID | Категория | Серьезность | Расположение | Резюме | Рекомендация |
|----|----------|----------|-------------|---------|----------------|
| A1 | Дублирование | ВЫСОКИЙ | spec.md:L120-134 | Два похожих требования ... | Объединить формулировку; оставить более ясную версию |

(Добавить одну строку на каждую находку; генерировать постоянные ID с префиксом начальной буквы категории.)

**Таблица покрытия:**

| Ключ требования | Есть задача? | ID задач | Примечания |
|-----------------|-----------|----------|-------|

**Проблемы соответствия конституции:** (если есть)

**Несопоставленные задачи:** (если есть)

**Метрики:**

- Всего требований
- Всего задач
- Процент покрытия (требования с >=1 задачей)
- Количество двусмысленностей
- Количество дубликатов
- Количество критических проблем

### 7. Предоставить следующие действия

В конце отчета вывести краткий блок "Следующие действия":

- Если существуют КРИТИЧЕСКИЕ проблемы: Рекомендовать устранение перед `/speckit.implement`
- Если только НИЗКИЙ/СРЕДНИЙ: Пользователь может продолжить, но предоставить предложения по улучшению
- Предоставить конкретные предложения команд: например, "Запустить /speckit.specify с уточнением", "Запустить /speckit.plan для корректировки архитектуры", "Вручную отредактировать tasks.md для добавления покрытия для 'performance-metrics'"

### 8. Предложить устранение

Спросить пользователя: "Хотели бы вы, чтобы я предложил конкретные исправления для первых N проблем?" (НЕ применять их автоматически.)

## Принципы работы

### Эффективность контекста

- **Минимальное количество информативных токенов**: Сосредоточиться на действенных находках, а не на исчерпывающей документации
- **Постепенное раскрытие**: Загружать артефакты постепенно; не выгружать весь контент в анализ
- **Эффективный вывод токенов**: Ограничить таблицу находок 50 строками; суммировать переполнение
- **Детерминистические результаты**: Повторный запуск без изменений должен давать постоянные ID и подсчеты

### Руководящие принципы анализа

- **НИКОГДА не модифицировать файлы** (это анализ только для чтения)
- **НИКОГДА не выдумывать отсутствующие разделы** (если отсутствуют, сообщить о них точно)
- **Приоритизировать нарушения конституции** (они всегда КРИТИЧЕСКИЕ)
- **Использовать примеры, а не исчерпывающие правила** (цитировать конкретные случаи, а не общие шаблоны)
- **Грациозно сообщать об отсутствии проблем** (выдать отчет об успехе с метриками покрытия)
