## Назначение контрольного списка: "Модульные тесты для английского языка"

**КРИТИЧЕСКОЕ ПОНЯТИЕ**: Контрольные списки - это **МОДУЛЬНЫЕ ТЕСТЫ ДЛЯ НАПИСАНИЯ ТРЕБОВАНИЙ** - они проверяют качество, ясность и полноту требований в заданной области.

**НЕ для верификации/тестирования**:

- ❌ НЕ "Проверить, что кнопка корректно реагирует на клики"
- ❌ НЕ "Протестировать работу обработки ошибок"
- ❌ НЕ "Подтвердить, что API возвращает 200"
- ❌ НЕ проверка соответствия кода/реализации спецификации

**ДЛЯ проверки качества требований**:

- ✅ "Определены ли требования к визуальной иерархии для всех типов карточек?" (полнота)
- ✅ "Квантовано ли 'выдающееся отображение' с конкретными размерами/позиционированием?" (ясность)
- ✅ "Согласованы ли требования к состоянию при наведении для всех интерактивных элементов?" (последовательность)
- ✅ "Определены ли требования доступности для навигации с клавиатуры?" (покрытие)
- ✅ "Определяет ли спецификация, что происходит, когда изображение логотипа не загружается?" (краевые случаи)

**Метафора**: Если ваша спецификация - это код, написанный на английском языке, то контрольный список - это его набор модульных тестов. Вы тестируете, насколько хорошо написаны требования, полны ли они, однозначны и готовы ли к реализации - А НЕ работает ли реализация.

## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пустой).

## Шаги выполнения

1. **Настройка**: Запустить `.qwen/specify/scripts/bash/check-prerequisites.sh --json` из корня репозитория и разобрать JSON для получения FEATURE_DIR и списка AVAILABLE_DOCS.
   - Все пути к файлам должны быть абсолютными.
   - Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте синтаксис экранирования: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

2. **Уточнение цели (динамическое)**: Вывести до ТРЕХ начальных контекстных уточняющих вопросов (без предварительно подготовленного каталога). Они ДОЛЖНЫ:
   - Быть сгенерированы из формулировки пользователя + сигналов, извлеченных из спецификации/плана/задач
   - Задаваться только о информации, которая существенно изменяет содержание контрольного списка
   - Пропускаться индивидуально, если уже однозначно в `$ARGUMENTS`
   - Предпочтительно точность перед широтой

   Алгоритм генерации:
   1. Извлечь сигналы: ключевые слова области функциональности (например, аутентификация, задержка, UX, API), индикаторы риска ("критично", "обязательно", "соответствие"), подсказки заинтересованных сторон ("QA", "обзор", "команда безопасности") и явные результаты ("a11y", "откат", "контракты").
   2. Кластеризовать сигналы в кандидатские области фокуса (максимум 4) с ранжированием по актуальности.
   3. Определить вероятную аудиторию и сроки (автор, рецензент, QA, релиз), если не указаны явно.
   4. Обнаружить недостающие измерения: широту охвата, глубину/строгость, акцент на риски, границы исключения, измеримые критерии приемки.
   5. Сформулировать вопросы, выбранные из этих архетипов:
      - Уточнение охвата (например, "Следует ли включать точки интеграции с X и Y или ограничиться корректностью локального модуля?")
      - Приоритизация рисков (например, "Какие из этих потенциальных областей риска должны получить обязательные проверки на прохождение?")
      - Калибровка глубины (например, "Это легкий список проверки перед коммитом или формальные ворота релиза?")
      - Фрейминг аудитории (например, "Будет ли это использоваться только автором или коллегами во время проверки PR?")
      - Исключение границ (например, "Следует ли явно исключить элементы настройки производительности на этот раз?")
      - Пробел в классе сценариев (например, "Не обнаружено потоков восстановления - находятся ли в охвате пути отката / частичного отказа?")

   Правила форматирования вопросов:
   - При представлении вариантов генерировать компактную таблицу с колонками: Вариант | Кандидат | Почему это важно
   - Ограничить максимум вариантами A–E; опустить таблицу, если свободная форма ответа яснее
   - Никогда не просить пользователя повторить то, что он уже сказал
   - Избегать спекулятивных категорий (без галлюцинаций). При неуверенности спрашивать прямо: "Подтвердите, принадлежит ли X к охвату."

   Значения по умолчанию, когда взаимодействие невозможно:
   - Глубина: Стандартная
   - Аудитория: Рецензент (PR), если связано с кодом; Автор в противном случае
   - Фокус: Топ 2 кластера актуальности

   Вывести вопросы (с метками Q1/Q2/Q3). После ответов: если ≥2 класса сценариев (Альтернативный / Исключительный / Восстановление / Нефункциональная область) остаются неясными, вы МОЖЕТЕ задать до ДВУХ дополнительных целевых уточнений (Q4/Q5) с однострочным обоснованием каждое (например, "Нерешенный риск путей восстановления"). Не превышать пять общих вопросов. Пропустить эскалацию, если пользователь явно отказывается от большего количества.

3. **Понять запрос пользователя**: Объединить `$ARGUMENTS` + уточняющие ответы:
   - Вывести тему контрольного списка (например, безопасность, обзор, развертывание, ux)
   - Объединить явные обязательные элементы, упомянутые пользователем
   - Отобразить выборы фокуса на каркас категории
   - Вывести любой недостающий контекст из спецификации/плана/задач (НЕ галлюцинировать)

4. **Загрузить контекст функции**: Читать из FEATURE_DIR:
   - spec.md: Требования к функции и охват
   - plan.md (если существует): Технические детали, зависимости
   - tasks.md (если существует): Задачи реализации

   **Стратегия загрузки контекста**:
   - Загружать только необходимые части, относящиеся к активным областям фокуса (избегать полной выгрузки файлов)
   - Предпочитать суммирование длинных разделов в виде кратких пунктов сценария/требований
   - Использовать прогрессивное раскрытие: добавлять последующее извлечение только при обнаружении пробелов
   - Если исходные документы большие, генерировать промежуточные элементы резюме вместо встраивания необработанного текста

5. **Сгенерировать контрольный список** - Создать "Модульные тесты для требований":
   - Создать директорию `FEATURE_DIR/checklists/`, если она не существует
   - Сгенерировать уникальное имя файла контрольного списка:
     - Использовать короткое, описательное имя, основанное на области (например, `ux.md`, `api.md`, `security.md`)
     - Формат: `[domain].md`
     - Если файл существует, добавить к существующему файлу
   - Нумеровать элементы последовательно, начиная с CHK001
   - Каждый запуск `/speckit.checklist` создает НОВЫЙ файл (никогда не перезаписывает существующие контрольные списки)

   **ОСНОВНОЙ ПРИНЦИП - Тестировать требования, а не реализацию**:
   Каждый элемент контрольного списка ДОЛЖЕН оценивать САМИ ТРЕБОВАНИЯ на:
   - **Полноту**: Все ли необходимые требования присутствуют?
   - **Ясность**: Являются ли требования однозначными и конкретными?
   - **Последовательность**: Согласуются ли требования друг с другом?
   - **Измеримость**: Можно ли объективно проверить требования?
   - **Покрытие**: Рассмотрены ли все сценарии/краевые случаи?

   **Структура категории** - Группировать элементы по измерениям качества требований:
   - **Полнота требований** (Все ли необходимые требования документированы?)
   - **Ясность требований** (Являются ли требования конкретными и однозначными?)
   - **Последовательность требований** (Согласуются ли требования без конфликтов?)
   - **Качество критериев приемки** (Измеримы ли критерии успеха?)
   - **Покрытие сценариев** (Рассмотрены ли все потоки/случаи?)
   - **Покрытие краевых случаев** (Определены ли условия границы?)
   - **Нефункциональные требования** (Производительность, Безопасность, Доступность и т.д. - указаны ли они?)
   - **Зависимости и допущения** (Документированы и проверены ли они?)
   - **Неоднозначности и конфликты** (Что требует уточнения?)

   **КАК ПИСАТЬ ЭЛЕМЕНТЫ КОНТРОЛЬНОГО СПИСКА - "Модульные тесты для английского языка"**:

   ❌ **НЕПРАВИЛЬНО** (Тестирование реализации):
   - "Проверить, что главная страница отображает 3 карточки эпизодов"
   - "Протестировать работу состояний при наведении на рабочем столе"
   - "Подтвердить, что клик по логотипу ведет на главную"

   ✅ **ПРАВИЛЬНО** (Тестирование качества требований):
   - "Определены ли точное количество и макет рекомендованных эпизодов?" [Полнота]
   - "Квантовано ли 'выдающееся отображение' с конкретными размерами/позиционированием?" [Ясность]
   - "Согласованы ли требования к состоянию при наведении для всех интерактивных элементов?" [Последовательность]
   - "Определены ли требования навигации с клавиатуры для всего интерактивного UI?" [Покрытие]
   - "Определено ли поведение при сбое, когда изображение логотипа не загружается?" [Краевые случаи]
   - "Определены ли состояния загрузки для асинхронных данных эпизодов?" [Полнота]
   - "Определяет ли спецификация визуальную иерархию для конкурирующих элементов UI?" [Ясность]

   **СТРУКТУРА ЭЛЕМЕНТА**:
   Каждый элемент должен следовать этому шаблону:
   - Формат вопроса, спрашивающего о качестве требований
   - Сосредоточиться на том, что НАПИСАНО (или не написано) в спецификации/плане
   - Включить измерение качества в скобках [Полнота/Ясность/Последовательность/и т.д.]
   - Ссылаться на раздел спецификации `[Spec §X.Y]` при проверке существующих требований
   - Использовать маркер `[Gap]` при проверке отсутствующих требований

   **ПРИМЕРЫ ПО ИЗМЕРЕНИЮ КАЧЕСТВА**:

   Полнота:
   - "Определены ли требования обработки ошибок для всех режимов отказа API? [Gap]"
   - "Указаны ли требования доступности для всех интерактивных элементов? [Полнота]"
   - "Определены ли требования мобильных брейкпоинтов для адаптивных макетов? [Gap]"

   Ясность:
   - "Квантовано ли 'быстрое загрузка' с конкретными временными порогами? [Ясность, Spec §NFR-2]"
   - "Явно ли определены критерии отбора 'связанных эпизодов'? [Ясность, Spec §FR-5]"
   - "Определено ли 'выдающееся' с измеримыми визуальными свойствами? [Неоднозначность, Spec §FR-4]"

   Последовательность:
   - "Согласуются ли требования навигации на всех страницах? [Последовательность, Spec §FR-10]"
   - "Согласованы ли требования компонента карточки между главной и детальной страницами? [Последовательность]"

   Покрытие:
   - "Определены ли требования для сценариев нулевого состояния (нет эпизодов)? [Покрытие, Краевой случай]"
   - "Рассмотрены ли сценарии параллельного взаимодействия пользователей? [Покрытие, Gap]"
   - "Определены ли требования для частичных сбоев загрузки данных? [Покрытие, Исключительный поток]"

   Измеримость:
   - "Измеримы/тестируемы ли требования визуальной иерархии? [Критерии приемки, Spec §FR-1]"
   - "Можно ли объективно проверить 'сбалансированный визуальный вес'? [Измеримость, Spec §FR-2]"

   **Классификация сценариев и покрытие** (Фокус на качество требований):
   - Проверить, существуют ли требования для: Основных, Альтернативных, Исключительных/Ошибочных, Восстановительных, Нефункциональных сценариев
   - Для каждого класса сценариев спрашивать: "Полны, ясны и согласованы ли требования [тип сценария]?"
   - Если класс сценариев отсутствует: "Преднамеренно ли исключены или отсутствуют требования [тип сценария]? [Gap]"
   - Включить устойчивость/откат, когда происходит изменение состояния: "Определены ли требования отката для сбоев миграции? [Gap]"

   **Требования трассируемости**:
   - МИНИМУМ: ≥80% элементов ДОЛЖНЫ включать хотя бы одну ссылку трассируемости
   - Каждый элемент должен ссылаться на: раздел спецификации `[Spec §X.Y]` или использовать маркеры: `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]`
   - Если система ID не существует: "Установлена ли схема ID требований и критериев приемки? [Трассируемость]"

   **Поверхность и разрешение проблем** (Проблемы качества требований):
   Спрашивать о самих требованиях:
   - Неоднозначности: "Квантован ли термин 'быстро' с конкретными метриками? [Неоднозначность, Spec §NFR-1]"
   - Конфликты: "Конфликтуют ли требования навигации между §FR-10 и §FR-10a? [Conflict]"
   - Допущения: "Проверено ли допущение о 'всегда доступном API подкастов'? [Assumption]"
   - Зависимости: "Документированы ли требования внешнего API подкастов? [Dependency, Gap]"
   - Отсутствующие определения: "Определена ли 'визуальная иерархия' с измеримыми критериями? [Gap]"

   **Консолидация содержимого**:
   - Мягкий лимит: Если кандидатов элементов > 40, приоритизировать по риску/влиянию
   - Объединить близкие дубликаты, проверяющие тот же аспект требований
   - Если >5 маловажных краевых случаев, создать один элемент: "Рассмотрены ли в требованиях краевые случаи X, Y, Z? [Покрытие]"

   **🚫 СТРОГО ЗАПРЕЩЕНО** - Это делает его тестом реализации, а не тестом требований:
   - ❌ Любой элемент, начинающийся с "Проверить", "Тестировать", "Подтвердить", "Проверить" + поведение реализации
   - ❌ Ссылки на выполнение кода, действия пользователя, поведение системы
   - ❌ "Отображается правильно", "работает должным образом", "функционирует как ожидается"
   - ❌ "Клик", "переход", "рендер", "загрузка", "выполнение"
   - ❌ Тестовые сценарии, планы тестирования, процедуры QA
   - ❌ Детали реализации (фреймворки, API, алгоритмы)

   **✅ ТРЕБУЕМЫЕ ШАБЛОНЫ** - Эти проверяют качество требований:
   - ✅ "Определены/указаны/документированы ли [тип требования] для [сценария]?"
   - ✅ "Квантовано/уточнено ли [неясный термин] с конкретными критериями?"
   - ✅ "Согласованы ли требования между [раздел A] и [раздел B]?"
   - ✅ "Можно ли объективно измерить/проверить [требование]?"
   - ✅ "Рассмотрены ли [краевые случаи/сценарии] в требованиях?"
   - ✅ "Определяет ли спецификация [отсутствующий аспект]?"

6. **Справочная структура**: Сгенерировать контрольный список, следуя каноническому шаблону в `.qwen/specify/templates/checklist-template.md` для заголовка, мета-раздела, заголовков категорий и форматирования ID. Если шаблон недоступен, использовать: заголовок H1, строки мета о назначении/создании, разделы `##` категории, содержащие строки `- [ ] CHK### <элемент требования>` с глобально увеличивающимися ID, начиная с CHK001.

7. **Отчет**: Вывести полный путь к созданному контрольному списку, количество элементов и напомнить пользователю, что каждый запуск создает новый файл. Резюмировать:
   - Выбранные области фокуса
   - Уровень глубины
   - Актер/время
   - Любые явно указанные пользователем обязательные элементы, включенные

**Важно**: Каждый вызов команды `/speckit.checklist` создает файл контрольного списка с использованием коротких, описательных имен, если файл еще не существует. Это позволяет:

- Несколько контрольных списков разных типов (например, `ux.md`, `test.md`, `security.md`)
- Простые, запоминающиеся имена файлов, указывающие цель контрольного списка
- Легкое определение и навигация по папке `checklists/`

Чтобы избежать беспорядка, используйте описательные типы и очищайте устаревшие контрольные списки по завершении.

## Примеры типов контрольных списков и образцы элементов

**Качество требований UX:** `ux.md`

## Антипримеры: Что НЕ делать

**❌ НЕПРАВИЛЬНО - Эти тесты проверяют реализацию, а не требования:**

```markdown
- [ ] CHK001 - Проверить, что главная страница отображает 3 карточки эпизодов [Spec §FR-001]
- [ ] CHK002 - Протестировать, что состояния при наведении работают корректно на рабочем столе [Spec §FR-003]
- [ ] CHK003 - Подтвердить, что клик по логотипу ведет на главную страницу [Spec §FR-010]
- [ ] CHK004 - Проверить, что раздел связанных эпизодов показывает 3-5 элементов [Spec §FR-005]
```

**✅ ПРАВИЛЬНО - Эти тесты проверяют качество требований:**

```markdown
- [ ] CHK001 - Являются ли количество и макет рекомендованных эпизодов явно указаны? [Полнота, Spec §FR-001]
- [ ] CHK002 - Последовательно ли определены требования к состоянию при наведении для всех интерактивных элементов? [Последовательность, Spec §FR-003]
- [ ] CHK003 - Ясны ли требования навигации для всех кликабельных брендовых элементов? [Ясность, Spec §FR-010]
- [ ] CHK004 - Документированы ли критерии отбора для связанных эпизодов? [Gap, Spec §FR-005]
- [ ] CHK005 - Определены ли требования состояния загрузки для асинхронных данных эпизодов? [Gap]
- [ ] CHK006 - Можно ли объективно измерить требования "визуальной иерархии"? [Измеримость, Spec §FR-001]
```

**Ключевые различия:**

- Неправильно: Проверяет, работает ли система корректно
- Правильно: Проверяет, корректно ли написаны требования
- Неправильно: Верификация поведения
- Правильно: Проверка качества требований
- Неправильно: "Делает ли она X?"
- Правильно: "Является ли X четко указанным?"
