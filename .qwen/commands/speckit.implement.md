## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пустой).

## Структура

1. Запустить `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` из корня репозитория и разобрать список FEATURE_DIR и AVAILABLE_DOCS. Все пути должны быть абсолютными. Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте экранирующий синтаксис: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

2. **Проверить статус контрольных списков** (если существует FEATURE_DIR/checklists/):
   - Просканировать все файлы контрольных списков в каталоге checklists/
   - Для каждого контрольного списка подсчитать:
     - Всего элементов: Все строки, соответствующие `- [ ]` или `- [X]` или `- [x]`
     - Выполненные элементы: Строки, соответствующие `- [X]` или `- [x]`
     - Невыполненные элементы: Строки, соответствующие `- [ ]`
   - Создать таблицу статуса:

     ```text
     | Контрольный список | Всего | Выполнено | Невыполнено | Статус |
     |--------------------|-------|-----------|-------------|--------|
     | ux.md              | 12    | 12        | 0           | ✓ ПРОЙДЕН |
     | test.md            | 8     | 5         | 3           | ✗ НЕ ПРОЙДЕН |
     | security.md        | 6     | 6         | 0           | ✓ ПРОЙДЕН |
     ```

   - Рассчитать общий статус:
     - **ПРОЙДЕН**: Все контрольные списки имеют 0 невыполненных элементов
     - **НЕ ПРОЙДЕН**: Один или несколько контрольных списков имеют невыполненные элементы

   - **Если какой-либо контрольный список неполный**:
     - Отобразить таблицу с количеством невыполненных элементов
     - **ОСТАНОВИТЬСЯ** и спросить: "Некоторые контрольные списки неполные. Хотите продолжить реализацию в любом случае? (да/нет)"
     - Дождаться ответа пользователя перед продолжением
     - Если пользователь говорит "нет" или "подождать" или "остановить", прекратить выполнение
     - Если пользователь говорит "да" или "продолжить" или "вперед", перейти к шагу 3

   - **Если все контрольные списки завершены**:
     - Отобразить таблицу, показывающую, что все контрольные списки пройдены
     - Автоматически перейти к шагу 3

3. Загрузить и проанализировать контекст реализации:
   - **ОБЯЗАТЕЛЬНО**: Прочитать tasks.md для полного списка задач и плана выполнения
   - **ОБЯЗАТЕЛЬНО**: Прочитать plan.md для технологического стека, архитектуры и структуры файлов
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитать data-model.md для сущностей и отношений
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитать contracts/ для спецификаций API и требований к тестированию
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитать research.md для технических решений и ограничений
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитать quickstart.md для сценариев интеграции

4. **Проверка настройки проекта**:
   - **ОБЯЗАТЕЛЬНО**: Создать/проверить файлы игнорирования на основе фактической настройки проекта:

   **Логика обнаружения и создания**:
   - Проверить, успешно ли выполняется следующая команда, чтобы определить, является ли репозиторий git-репозиторием (создать/проверить .gitignore, если да):

     ```sh
     git rev-parse --git-dir 2>/dev/null
     ```

   - Проверить, существует ли Dockerfile* или Docker в plan.md → создать/проверить .dockerignore
   - Проверить, существует ли .eslintrc* → создать/проверить .eslintignore
   - Проверить, существует ли eslint.config.* → убедиться, что записи `ignores` в конфигурации охватывают необходимые шаблоны
   - Проверить, существует ли .prettierrc* → создать/проверить .prettierignore
   - Проверить, существует ли .npmrc или package.json → создать/проверить .npmignore (при публикации)
   - Проверить, существуют ли файлы terraform (*.tf) → создать/проверить .terraformignore
   - Проверить, нужен ли .helmignore (присутствуют helm-чарты) → создать/проверить .helmignore

   **Если файл игнорирования уже существует**: Проверить, содержит ли он основные шаблоны, добавить только отсутствующие критические шаблоны
   **Если файл игнорирования отсутствует**: Создать с полным набором шаблонов для обнаруженной технологии

   **Общие шаблоны по технологии** (из технологического стека в plan.md):
   - **Node.js/JavaScript/TypeScript**: `node_modules/`, `dist/`, `build/`, `*.log`, `.env*`
   - **Python**: `__pycache__/`, `*.pyc`, `.venv/`, `venv/`, `dist/`, `*.egg-info/`
   - **Java**: `target/`, `*.class`, `*.jar`, `.gradle/`, `build/`
   - **C#/.NET**: `bin/`, `obj/`, `*.user`, `*.suo`, `packages/`
   - **Go**: `*.exe`, `*.test`, `vendor/`, `*.out`
   - **Ruby**: `.bundle/`, `log/`, `tmp/`, `*.gem`, `vendor/bundle/`
   - **PHP**: `vendor/`, `*.log`, `*.cache`, `*.env`
   - **Rust**: `target/`, `debug/`, `release/`, `*.rs.bk`, `*.rlib`, `*.prof*`, `.idea/`, `*.log`, `.env*`
   - **Kotlin**: `build/`, `out/`, `.gradle/`, `.idea/`, `*.class`, `*.jar`, `*.iml`, `*.log`, `.env*`
   - **C++**: `build/`, `bin/`, `obj/`, `out/`, `*.o`, `*.so`, `*.a`, `*.exe`, `*.dll`, `.idea/`, `*.log`, `.env*`
   - **C**: `build/`, `bin/`, `obj/`, `out/`, `*.o`, `*.a`, `*.so`, `*.exe`, `Makefile`, `config.log`, `.idea/`, `*.log`, `.env*`
   - **Swift**: `.build/`, `DerivedData/`, `*.swiftpm/`, `Packages/`
   - **R**: `.Rproj.user/`, `.Rhistory`, `.RData`, `.Ruserdata`, `*.Rproj`, `packrat/`, `renv/`
   - **Универсальные**: `.DS_Store`, `Thumbs.db`, `*.tmp`, `*.swp`, `.vscode/`, `.idea/`

   **Шаблоны для конкретных инструментов**:
   - **Docker**: `node_modules/`, `.git/`, `Dockerfile*`, `.dockerignore`, `*.log*`, `.env*`, `coverage/`
   - **ESLint**: `node_modules/`, `dist/`, `build/`, `coverage/`, `*.min.js`
   - **Prettier**: `node_modules/`, `dist/`, `build/`, `coverage/`, `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`
   - **Terraform**: `.terraform/`, `*.tfstate*`, `*.tfvars`, `.terraform.lock.hcl`
   - **Kubernetes/k8s**: `*.secret.yaml`, `secrets/`, `.kube/`, `kubeconfig*`, `*.key`, `*.crt`

5. Разобрать структуру tasks.md и извлечь:
   - **Фазы задач**: Настройка, Тесты, Ядро, Интеграция, Полировка
   - **Зависимости задач**: Правила последовательного и параллельного выполнения
   - **Детали задач**: ID, описание, пути к файлам, маркеры параллельности [P]
   - **Поток выполнения**: Порядок и требования зависимости

6. Выполнить реализацию в соответствии с планом задач:
   - **Последовательное выполнение по фазам**: Завершить каждую фазу перед переходом к следующей
   - **Уважать зависимости**: Выполнять последовательные задачи по порядку, параллельные задачи [P] могут выполняться вместе
   - **Следовать подходу TDD**: Выполнять задачи тестирования до соответствующих задач реализации
   - **Координация на основе файлов**: Задачи, влияющие на одни и те же файлы, должны выполняться последовательно
   - **Контрольные точки проверки**: Проверять завершение каждой фазы перед продолжением

7. Правила выполнения реализации:
   - **Сначала настройка**: Инициализировать структуру проекта, зависимости, конфигурацию
   - **Тесты перед кодом**: Если нужно написать тесты для контрактов, сущностей и сценариев интеграции
   - **Основная разработка**: Реализовать модели, сервисы, команды CLI, конечные точки
   - **Интеграционная работа**: Подключения к базе данных, промежуточное ПО, ведение журналов, внешние службы
   - **Полировка и проверка**: Модульные тесты, оптимизация производительности, документация

8. Отслеживание прогресса и обработка ошибок:
   - Сообщать о прогрессе после каждой завершенной задачи
   - Остановить выполнение, если какая-либо не параллельная задача завершается неудачно
   - Для параллельных задач [P], продолжать с успешными задачами, сообщать о неудавшихся
   - Предоставлять четкие сообщения об ошибках с контекстом для отладки
   - Предлагать следующие шаги, если реализация не может продолжаться
   - **ВАЖНО** Для завершенных задач обязательно отметить задачу как [X] в файле задач.

9. Проверка завершения:
   - Проверить, что все обязательные задачи выполнены
   - Проверить, соответствуют ли реализованные функции первоначальной спецификации
   - Проверить, проходят ли тесты и соответствует ли покрытие требованиям
   - Подтвердить, что реализация следует техническому плану
   - Сообщить о конечном статусе с резюме выполненной работы

Примечание: Эта команда предполагает, что полная декомпозиция задач существует в tasks.md. Если задачи неполные или отсутствуют, предложить сначала запустить `/speckit.tasks` для повторного создания списка задач.
