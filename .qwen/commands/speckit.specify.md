```text
$ARGUMENTS
```

## Структура

Текст, который пользователь ввел после `/speckit.specify` в вызывающем сообщении, **является** описанием функции. Предполагайте, что он всегда доступен в этом разговоре, даже если `{{args}}` появляется буквально ниже. Не просите пользователя повторить его, если он предоставил пустую команду.

Учитывая это описание функции, сделайте следующее:

1. **Создайте краткое короткое имя** (2-4 слова) для ветки:
   - Проанализируйте описание функции и извлеките наиболее значимые ключевые слова
   - Создайте короткое имя из 2-4 слов, которое отражает суть функции
   - Используйте формат действие-существительное, когда это возможно (например, "add-user-auth", "fix-payment-bug")
   - Сохраняйте технические термины и аббревиатуры (OAuth2, API, JWT и т.д.)
   - Делайте его кратким, но достаточно описательным, чтобы понять функцию с первого взгляда
   - Примеры:
     - "Я хочу добавить аутентификацию пользователя" → "user-auth"
     - "Реализовать интеграцию OAuth2 для API" → "oauth2-api-integration"
     - "Создать панель управления для аналитики" → "analytics-dashboard"
     - "Исправить ошибку тайм-аута обработки платежей" → "fix-payment-timeout"

2. **Проверьте наличие существующих веток перед созданием новой**:

   a. Сначала получите все удаленные ветки, чтобы убедиться, что у нас есть самая последняя информация:

      ```bash
      git fetch --all --prune
      ```

   b. Найдите самый высокий номер функции во всех источниках для короткого имени:
      - Удаленные ветки: `git ls-remote --heads origin | grep -E 'refs/heads/[0-9]+-<short-name>$'`
      - Локальные ветки: `git branch | grep -E '^[* ]*[0-9]+-<short-name>$'`
      - Каталоги спецификаций: Проверьте наличие каталогов, соответствующих `specs/[0-9]+-<short-name>`

   c. Определите следующий доступный номер:
      - Извлеките все числа из всех трех источников
      - Найдите самое высокое число N
      - Используйте N+1 для нового номера ветки

   d. Запустите скрипт `.specify/scripts/bash/create-new-feature.sh --json "{{args}}"` с вычисленным номером и коротким именем:
      - Передайте `--number N+1` и `--short-name "your-short-name"` вместе с описанием функции
      - Пример bash: `.specify/scripts/bash/create-new-feature.sh --json "{{args}}" --json --number 5 --short-name "user-auth" "Add user authentication"`
      - Пример PowerShell: `.specify/scripts/bash/create-new-feature.sh --json "{{args}}" -Json -Number 5 -ShortName "user-auth" "Add user authentication"`

   **ВАЖНО**:
   - Проверьте все три источника (удаленные ветки, локальные ветки, каталоги спецификаций), чтобы найти самое высокое число
   - Сопоставляйте только ветки/каталоги с точным шаблоном короткого имени
   - Если не найдено существующих веток/каталогов с этим коротким именем, начните с номера 1
   - Вы должны запускать этот скрипт только один раз на функцию
   - JSON предоставляется в терминале в качестве вывода - всегда обращайтесь к нему, чтобы получить фактическое содержимое, которое вы ищете
   - Вывод JSON будет содержать пути BRANCH_NAME и SPEC_FILE
   - Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте синтаксис экранирования: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot")

3. Загрузите `.specify/templates/spec-template.md`, чтобы понять требуемые разделы.

4. Следуйте этому порядку выполнения:

    1. Разбор описания пользователя из ввода
       Если пусто: ОШИБКА "Не предоставлено описание функции"
    2. Извлечение ключевых концепций из описания
       Определите: актеры, действия, данные, ограничения
    3. Для неясных аспектов:
       - Сделайте обоснованные предположения на основе контекста и отраслевых стандартов
       - Отмечайте только с помощью [НЕОБХОДИМО УТОЧНИТЬ: конкретный вопрос], если:
         - Выбор значительно влияет на объем функций или пользовательский опыт
         - Существует несколько разумных интерпретаций с различными последствиями
         - Не существует разумного умолчания
       - **ОГРАНИЧЕНИЕ: Максимум 3 маркера [НЕОБХОДИМО УТОЧНИТЬ] всего**
       - Приоритетизация уточнений по влиянию: объем > безопасность/конфиденциальность > пользовательский опыт > технические детали
    4. Заполните раздел пользовательских сценариев и тестирования
       Если нет четкого пользовательского потока: ОШИБКА "Невозможно определить пользовательские сценарии"
    5. Генерация функциональных требований
       Каждое требование должно быть тестируемым
       Используйте разумные значения по умолчанию для неуказанных деталей (документируйте предположения в разделе Предположения)
    6. Определите критерии успеха
       Создайте измеримые, технологически независимые результаты
       Включите как количественные показатели (время, производительность, объем), так и качественные меры (удовлетворенность пользователей, завершение задач)
       Каждый критерий должен быть проверяемым без деталей реализации
    7. Определите ключевые сущности (если задействованы данные)
    8. Возврат: УСПЕХ (спецификация готова к планированию)

5. Запишите спецификацию в SPEC_FILE, используя структуру шаблона, заменяя заполнители конкретными деталями, полученными из описания функции (аргументов), при этом сохраняя порядок и заголовки разделов.

6. **Проверка качества спецификации**: После написания первоначальной спецификации проверьте ее по критериям качества:

   a. **Создание контрольного списка качества спецификации**: Создайте файл контрольного списка в `FEATURE_DIR/checklists/requirements.md`, используя структуру шаблона контрольного списка со следующими элементами проверки:

      ```markdown
      # Контрольный список качества спецификации: [НАЗВАНИЕ ФУНКЦИИ]

      **Назначение**: Проверка полноты и качества спецификации перед переходом к планированию
      **Создано**: [ДАТА]
      **Функция**: [Ссылка на spec.md]

      ## Качество содержания

      - [ ] Нет деталей реализации (языки, фреймворки, API)
      - [ ] Сосредоточено на ценности для пользователя и бизнес-потребностях
      - [ ] Написано для заинтересованных сторон, не являющихся техническими специалистами
      - [ ] Все обязательные разделы заполнены

      ## Полнота требований

      - [ ] Не осталось маркеров [НЕОБХОДИМО УТОЧНИТЬ]
      - [ ] Требования поддаются тестированию и однозначны
      - [ ] Критерии успеха измеримы
      - [ ] Критерии успеха технологически независимы (без деталей реализации)
      - [ ] Все сценарии приемки определены
      - [ ] Граничные случаи идентифицированы
      - [ ] Объем четко ограничен
      - [ ] Зависимости и предположения идентифицированы

      ## Готовность функции

      - [ ] Все функциональные требования имеют четкие критерии приемки
      - [ ] Пользовательские сценарии охватывают основные потоки
      - [ ] Функция соответствует измеримым результатам, определенным в критериях успеха
      - [ ] В спецификации не просачиваются детали реализации

      ## Примечания

      - Элементы, помеченные как неполные, требуют обновления спецификации перед `/speckit.clarify` или `/speckit.plan`
      ```

   b. **Выполнить проверку валидации**: Проверьте спецификацию по каждому элементу контрольного списка:
      - Для каждого элемента определите, проходит ли он или нет
      - Документируйте конкретные найденные проблемы (цитируйте соответствующие разделы спецификации)

   c. **Обработка результатов проверки**:

      - **Если все элементы проходят**: Отметьте контрольный список как завершенный и переходите к шагу 6

      - **Если элементы не проходят (кроме [НЕОБХОДИМО УТОЧНИТЬ])**:
        1. Перечислите не прошедшие элементы и конкретные проблемы
        2. Обновите спецификацию, чтобы устранить каждую проблему
        3. Повторно выполните проверку до тех пор, пока все элементы не будут проходить (максимум 3 итерации)
        4. Если по-прежнему не проходит после 3 итераций, задокументируйте оставшиеся проблемы в заметках контрольного списка и предупредите пользователя

      - **Если остаются маркеры [НЕОБХОДИМО УТОЧНИТЬ]**:
        1. Извлеките все маркеры [НЕОБХОДИМО УТОЧНИТЬ: ...] из спецификации
        2. **ПРОВЕРКА ОГРАНИЧЕНИЯ**: Если существует более 3 маркеров, оставьте только 3 самых критических (по влиянию на объем/безопасность/UX) и сделайте обоснованные предположения для остальных
        3. Для каждого необходимого уточнения (максимум 3) представьте пользователю варианты в следующем формате:

           ```markdown
           ## Вопрос [N]: [Тема]

           **Контекст**: [Цитата соответствующего раздела спецификации]

           **Что нам нужно знать**: [Конкретный вопрос из маркера NEEDS CLARIFICATION]

           **Предлагаемые ответы**:

           | Вариант | Ответ | Последствия |
           |--------|--------|--------------|
           | A      | [Первый предлагаемый ответ] | [Что это значит для функции] |
           | B      | [Второй предлагаемый ответ] | [Что это значит для функции] |
           | C      | [Третий предлагаемый ответ] | [Что это значит для функции] |
           | Пользовательский | Предоставьте свой собственный ответ | [Объясните, как предоставить пользовательский ввод] |

           **Ваш выбор**: _[Ждать ответа пользователя]_
           ```

        4. **КРИТИЧЕСКОЕ - Форматирование таблиц**: Убедитесь, что таблицы markdown правильно отформатированы:
           - Используйте согласованный интервал с выровненными трубами
           - В каждой ячейке должны быть пробелы вокруг содержимого: `| Content |`, а не `|Content|`
           - Разделитель заголовков должен иметь не менее 3 тире: `|--------|`
           - Проверьте, что таблица правильно отображается в предварительном просмотре markdown
        5. Нумеруйте вопросы последовательно (Q1, Q2, Q3 - максимум 3 всего)
        6. Представьте все вопросы вместе перед ожиданием ответов
        7. Ждите, пока пользователь не ответит своими выборами для всех вопросов (например, "Q1: A, Q2: Custom - [детали], Q3: B")
        8. Обновите спецификацию, заменив каждый маркер [НЕОБХОДИМО УТОЧНИТЬ] выбранным или предоставленным пользователем ответом
        9. Повторно выполните проверку после решения всех уточнений

   d. **Обновление контрольного списка**: После каждой итерации проверки обновите файл контрольного списка с текущим статусом прохождения/непрохождения

7. Сообщите о завершении с именем ветки, путем к файлу спецификации, результатами контрольного списка и готовностью к следующему этапу (`/speckit.clarify` или `/speckit.plan`).

**ПРИМЕЧАНИЕ:** Скрипт создает и переключается на новую ветку и инициализирует файл спецификации перед записью.

## Общие рекомендации

## Быстрые рекомендации

- Сосредоточьтесь на **ЧЕМ** нуждаются пользователи и **ПОЧЕМУ**.
- Избегайте описания КАК реализовать (без стека технологий, API, структуры кода).
- Написано для бизнес-заинтересованных сторон, а не для разработчиков.
- НЕ создавайте никаких контрольных списков, которые встроены в спецификацию. Это будет отдельная команда.

### Требования к разделам

- **Обязательные разделы**: Должны быть заполнены для каждой функции
- **Необязательные разделы**: Включать только тогда, когда они актуальны для функции
- Когда раздел не применяется, полностью удалите его (не оставляйте как "N/A")

### Для генерации ИИ

При создании этой спецификации из пользовательского запроса:

1. **Делайте обоснованные предположения**: Используйте контекст, отраслевые стандарты и общие шаблоны для заполнения пробелов
2. **Документируйте предположения**: Записывайте разумные значения по умолчанию в раздел предположений
3. **Ограничьте уточнения**: Максимум 3 маркера [НЕОБХОДИМО УТОЧНИТЬ] - используйте только для критических решений, которые:
   - Значительно влияют на объем функций или пользовательский опыт
   - Имеют несколько разумных интерпретаций с различными последствиями
   - Не имеют никакого разумного умолчания
4. **Приоритезируйте уточнения**: объем > безопасность/конфиденциальность > пользовательский опыт > технические детали
5. **Думайте как тестировщик**: Каждое расплывчатое требование должно не пройти элемент проверки "тестируемое и однозначное"
6. **Распространенные области, требующие уточнения** (только если не существует разумного умолчания):
   - Объем функций и границы (включать/исключать конкретные варианты использования)
   - Типы пользователей и разрешения (если возможны несколько противоречащих интерпретаций)
   - Требования безопасности/соответствия (когда юридически/финансово значимо)

**Примеры разумных значений по умолчанию** (не спрашивайте об этих):

- Хранение данных: Отраслевые стандарты для домена
- Целевые показатели производительности: Стандартные ожидания для веб/мобильных приложений, если не указано иное
- Обработка ошибок: Понятные сообщения для пользователей с соответствующими резервными вариантами
- Метод аутентификации: Стандартная сессия или OAuth2 для веб-приложений
- Шаблоны интеграции: RESTful API, если не указано иное

### Руководящие принципы критериев успеха

Критерии успеха должны быть:

1. **Измеримыми**: Включать конкретные метрики (время, процент, количество, частота)
2. **Технологически независимыми**: Никаких упоминаний фреймворков, языков, баз данных или инструментов
3. **Ориентированными на пользователя**: Описывать результаты с точки зрения пользователя/бизнеса, а не внутренних систем
4. **Проверяемыми**: Можно протестировать/проверить без знания деталей реализации

**Хорошие примеры**:

- "Пользователи могут завершить оформление заказа менее чем за 3 минуты"
- "Система поддерживает 10 000 одновременных пользователей"
- "95% поисков возвращают результаты менее чем за 1 секунду"
- "Коэффициент завершения задач увеличивается на 40%"

**Плохие примеры** (ориентированные на реализацию):

- "Время отклика API менее 200 мс" (слишком технически, используйте "Пользователи видят результаты мгновенно")
- "База данных может обрабатывать 1000 TPS" (деталь реализации, используйте пользовательскую метрику)
- "Компоненты React отображаются эффективно" (специфично для фреймворка)
- "Коэффициент попадания в кеш Redis выше 80%" (специфично для технологии)
