## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пустой).

## Структура

1. **Настройка**: Запустить `.specify/scripts/bash/check-prerequisites.sh --json` из корня репозитория и проанализировать список FEATURE_DIR и AVAILABLE_DOCS. Все пути должны быть абсолютными. Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте экранирующий синтаксис: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

2. **Загрузить документы дизайна**: Читать из FEATURE_DIR:
   - **Обязательно**: plan.md (технический стек, библиотеки, структура), spec.md (пользовательские истории с приоритетами)
   - **Опционально**: data-model.md (сущности), contracts/ (конечные точки API), research.md (решения), quickstart.md (сценарии тестирования)
   - Примечание: Не все проекты имеют все документы. Генерировать задачи на основе доступного.

3. **Выполнить рабочий процесс генерации задач**:
   - Загрузить plan.md и извлечь технический стек, библиотеки, структуру проекта
   - Загрузить spec.md и извлечь пользовательские истории с их приоритетами (P1, P2, P3 и т.д.)
   - Если существует data-model.md: Извлечь сущности и сопоставить с пользовательскими историями
   - Если существует contracts/: Сопоставить конечные точки с пользовательскими историями
   - Если существует research.md: Извлечь решения для задач настройки
   - Генерировать задачи, организованные по пользовательским историям (см. правила генерации задач ниже)
   - Генерировать граф зависимостей, показывающий порядок завершения пользовательских историй
   - Создать примеры параллельного выполнения для каждой пользовательской истории
   - Проверить полноту задач (каждая пользовательская история имеет все необходимые задачи, независимо тестируемые)

4. **Сгенерировать tasks.md**: Использовать `.specify/templates/tasks-template.md` как структуру, заполнить:
   - Правильное имя функции из plan.md
   - Фаза 1: Задачи настройки (инициализация проекта)
   - Фаза 2: Фундаментальные задачи (блокирующие предварительные условия для всех пользовательских историй)
   - Фаза 3+: По одной фазе на каждую пользовательскую историю (в порядке приоритета из spec.md)
   - Каждая фаза включает: цель истории, критерии независимого тестирования, тесты (если запрошены), задачи реализации
   - Последняя фаза: Полировка и сквозные вопросы
   - Все задачи должны следовать строгому формату контрольного списка (см. правила генерации задач ниже)
   - Четкие пути к файлам для каждой задачи
   - Раздел зависимостей, показывающий порядок завершения историй
   - Примеры параллельного выполнения для каждой истории
   - Раздел стратегии реализации (MVP первым, поэтапная доставка)

5. **Отчет**: Выходной путь к сгенерированному tasks.md и резюме:
   - Общее количество задач
   - Количество задач на пользовательскую историю
   - Идентифицированные возможности параллелизма
   - Критерии независимого тестирования для каждой истории
   - Предлагаемый объем MVP (обычно только пользовательская история 1)
   - Проверка формата: Подтвердить, что ВСЕ задачи следуют формату контрольного списка (флажок, ID, метки, пути к файлам)

Контекст для генерации задач: {{args}}

Файл tasks.md должен быть немедленно исполняемым - каждая задача должна быть достаточно конкретной, чтобы ИИ мог выполнить ее без дополнительного контекста.

## Правила генерации задач

**КРИТИЧЕСКИ ВАЖНО**: Задачи ДОЛЖНЫ быть организованы по пользовательским историям, чтобы обеспечить независимую реализацию и тестирование.

**Тесты являются ОПЦИОНАЛЬНЫМИ**: Генерировать задачи тестирования только если они явно запрошены в спецификации функции или если пользователь запрашивает подход TDD.

### Формат контрольного списка (ОБЯЗАТЕЛЬНО)

Каждая задача ДОЛЖНА строго следовать этому формату:

```text
- [ ] [ID задачи] [P?] [История?] Описание с путем к файлу
```

**Компоненты формата**:

1. **Флажок**: ВСЕГДА начинать с `- [ ]` (флажок markdown)
2. **ID задачи**: Последовательный номер (T001, T002, T003...) в порядке выполнения
3. **Маркер [P]**: Включать ТОЛЬКО если задача допускает параллелизм (разные файлы, нет зависимостей от незавершенных задач)
4. **Метка [История]**: ОБЯЗАТЕЛЬНО только для задач фаз пользовательских историй
   - Формат: [US1], [US2], [US3] и т.д. (сопоставляется с пользовательскими историями из spec.md)
   - Фаза настройки: БЕЗ метки истории
   - Фундаментальная фаза: БЕЗ метки истории
   - Фазы пользовательских историй: ДОЛЖНЫ иметь метку истории
   - Фаза полировки: БЕЗ метки истории
5. **Описание**: Четкое действие с точным путем к файлу

**Примеры**:

- ✅ ПРАВИЛЬНО: `- [ ] T001 Создать структуру проекта в соответствии с планом реализации`
- ✅ ПРАВИЛЬНО: `- [ ] T005 [P] Реализовать промежуточное программное обеспечение аутентификации в src/middleware/auth.py`
- ✅ ПРАВИЛЬНО: `- [ ] T012 [P] [US1] Создать модель пользователя в src/models/user.py`
- ✅ ПРАВИЛЬНО: `- [ ] T014 [US1] Реализовать UserService в src/services/user_service.py`
- ❌ НЕПРАВИЛЬНО: `- [ ] Создать модель пользователя` (отсутствует ID и метка истории)
- ❌ НЕПРАВИЛЬНО: `T001 [US1] Создать модель` (отсутствует флажок)
- ❌ НЕПРАВИЛЬНО: `- [ ] [US1] Создать модель пользователя` (отсутствует ID задачи)
- ❌ НЕПРАВИЛЬНО: `- [ ] T001 [US1] Создать модель` (отсутствует путь к файлу)

### Организация задач

1. **Из пользовательских историй (spec.md)** - ОСНОВНАЯ ОРГАНИЗАЦИЯ:
   - Каждая пользовательская история (P1, P2, P3...) получает свою собственную фазу
   - Сопоставить все связанные компоненты с их историей:
     - Модели, необходимые для этой истории
     - Сервисы, необходимые для этой истории
     - Конечные точки/интерфейс, необходимые для этой истории
     - Если запрошены тесты: Тесты, специфичные для этой истории
   - Отметить зависимости историй (большинство историй должны быть независимыми)

2. **Из контрактов**:
   - Сопоставить каждый контракт/конечную точку → с пользовательской историей, которую он обслуживает
   - Если запрошены тесты: Каждый контракт → задача тестирования контракта [P] перед реализацией в фазе этой истории

3. **Из модели данных**:
   - Сопоставить каждую сущность с пользовательской историей(ями), которая в ней нуждается
   - Если сущность обслуживает несколько историй: Поместить в самую раннюю историю или в фазу настройки
   - Отношения → задачи уровня сервиса в соответствующей фазе истории

4. **Из настройки/инфраструктуры**:
   - Общая инфраструктура → Фаза настройки (Фаза 1)
   - Фундаментальные/блокирующие задачи → Фундаментальная фаза (Фаза 2)
   - Настройка, специфичная для истории → внутри фазы этой истории

### Структура фаз

- **Фаза 1**: Настройка (инициализация проекта)
- **Фаза 2**: Фундаментальная (блокирующие предварительные условия - ДОЛЖНЫ быть завершены до пользовательских историй)
- **Фаза 3+**: Пользовательские истории в порядке приоритета (P1, P2, P3...)
  - Внутри каждой истории: Тесты (если запрошены) → Модели → Сервисы → Конечные точки → Интеграция
  - Каждая фаза должна быть завершенным, независимо тестируемым приращением
- **Последняя фаза**: Полировка и сквозные вопросы
