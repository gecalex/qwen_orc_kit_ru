# Система контрольных точек качества (Quality Gates)

## Обзор

Система контрольных точек качества (Quality Gates) представляет собой набор проверок, которые должны быть пройдены на различных этапах разработки для обеспечения соответствия проекта установленным стандартам. Они обеспечивают соблюдение стандартов и помогают выявлять проблемы на ранних стадиях.

## Назначение

- Обеспечение соответствия кода стандартам качества
- Предотвращение попадания дефектов в основную ветку
- Автоматизация проверок качества
- Упрощение процесса ревью кода

## Типы контрольных точек

### 1. Блокирующие (Blocking)

Блокирующие контрольные точки останавливают выполнение процесса при неудачном результате. Используются для критических проверок, таких как безопасность или работоспособность.

### 2. Неблокирующие (Non-blocking)

Неблокирующие контрольные точки генерируют предупреждения при неудачном результате, но не останавливают выполнение процесса. Используются для проверок стиля, покрытия тестами и других аспектов, не влияющих на работоспособность.

## Категории контрольных точек

### 1. Проверки безопасности

- **Назначение**: Проверка кода на наличие уязвимостей
- **Скрипт**: `scripts/quality-gates/check-security.sh`
- **Тип**: Блокирующая
- **Частота**: При каждом коммите в защищенные ветки

### 2. Проверки покрытия тестами

- **Назначение**: Проверка уровня покрытия кода тестами
- **Скрипт**: `scripts/quality-gates/check-coverage.sh`
- **Тип**: Неблокирующая
- **Частота**: При каждом пул-реквесте

### 3. Проверки размера бандла

- **Назначение**: Проверка размера собранного приложения
- **Скрипт**: `scripts/quality-gates/check-bundle-size.sh`
- **Тип**: Неблокирующая
- **Частота**: При каждом билде

### 4. Проверки типизации

- **Назначение**: Проверка корректности типов в коде
- **Скрипт**: `scripts/quality-gates/check-typing.sh`
- **Тип**: Блокирующая
- **Частота**: При каждом коммите

### 5. Проверки линтинга

- **Назначение**: Проверка соответствия кода стилю
- **Скрипт**: `scripts/quality-gates/check-linting.sh`
- **Тип**: Неблокирующая
- **Частота**: При каждом коммите

## Архитектура системы

### Навыки

- `run-quality-gate` - основной навык для выполнения контрольных точек
- `validate-quality-report` - навык для проверки отчетов о качестве

### Скрипты

Все скрипты контрольных точек находятся в директории `scripts/quality-gates/` и следуют соглашению об именовании `check-{название}.sh`.

## Интеграция с оркестраторами

Оркестраторы могут использовать контрольные точки качества следующим образом:

```markdown
## Quality Gate N: Validate Phase N
- Check worker report exists
- Run quality gates (run-quality-gate Skill)
- If blocking fails: STOP, rollback, exit
- If passes: proceed to next phase
```

## Навык run-quality-gate

Навык `run-quality-gate` предоставляет унифицированный интерфейс для выполнения различных проверок качества. Он принимает команду проверки, тип (блокирующая/неблокирующая) и ожидаемый результат.

### Входной формат:
```json
{
  "command": "string (команда для выполнения)",
  "isBlocking": "boolean (является ли проверка блокирующей)",
  "expectedResult": "string (ожидаемый результат)",
  "timeout": "number (таймаут в секундах, опционально)"
}
```

### Выходной формат:
```json
{
  "gateId": "string",
  "command": "string (выполненная команда)",
  "isBlocking": "boolean",
  "result": "enum (PASSED, FAILED, TIMEOUT)",
  "output": "string (вывод команды)",
  "passed": "boolean (пройдена ли проверка)",
  "blockingFailure": "boolean (является ли ошибка блокирующей)"
}
```

## Создание новых контрольных точек

Для создания новой контрольной точки:

1. Создайте скрипт в `scripts/quality-gates/check-{название}.sh`
2. Следуйте шаблону скрипта с корректным кодом возврата
3. Определите, является ли проверка блокирующей или нет
4. Документируйте новую проверку в этом файле

## Примеры использования

### В оркестраторе:
```
task '{
  "subagent_type": "skill",
  "description": "Run quality gate for security check",
  "prompt": "Execute run-quality-gate skill with parameters: {command: \\"scripts/quality-gates/check-security.sh\\", isBlocking: true, expectedResult: \\"No critical vulnerabilities found\\"}"
}'
```

### В процессе CI/CD:
Контрольные точки качества могут быть интегрированы в процесс CI/CD для автоматического выполнения при пул-реквестах или коммитах.

## Мониторинг и отчетность

Результаты контрольных точек качества сохраняются в формате, подходящем для дальнейшего анализа. Это позволяет отслеживать тенденции качества кода и эффективность процессов разработки.

## Лучшие практики

1. **Регулярное обновление**: Обновляйте контрольные точки по мере изменения стандартов качества
2. **Баланс**: Поддерживайте баланс между строгостью и производительностью команды
3. **Ясность**: Обеспечьте ясность сообщений об ошибках для быстрого устранения проблем
4. **Производительность**: Оптимизируйте скрипты для быстрого выполнения
5. **Документация**: Документируйте все контрольные точки и их назначение
