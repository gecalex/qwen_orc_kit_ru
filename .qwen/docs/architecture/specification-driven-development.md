# Разработка на основе спецификаций

## Обзор

Разработка на основе спецификаций (Specification-Driven Development, SDD) - это подход, при котором функциональность сначала полностью описывается в виде спецификации, а затем реализуется в соответствии с этим описанием.

**Преимущества:**
- Убедиться, что требования к функциональности четко определены
- Снизить риск недопонимания между заинтересованными сторонами
- Обеспечить согласованность реализации с ожиданиями
- Упростить тестирование и проверку качества
- Интеграция с Фазой 0 планирования
- Quality Gates для проверки спецификаций

## Процесс

### 0. Инициализация Фазы 0

**Перед созданием спецификации:**

```bash
# Запустить анализ Фазы 0
.qwen/specify/scripts/phase0-analyzer.sh .qwen/specify/specs/{ID}-{feature}

# Проверить результаты
cat .qwen/specify/specs/{ID}/plans/phase0-plan.json
```

### 1. Создание спецификации

Для создания спецификации используйте команду:

```bash
speckit.specify "Описание требуемой функциональности"
```

**Расположение:** `.qwen/specify/specs/{ID}-{feature}/spec.md`

Эта команда:
- Создает новую ветку для функции
- Генерирует спецификацию на основе шаблона
- Присваивает номер и имя ветке в формате `номер-описание`
- Создает контрольный список качества

### 2. Уточнение спецификации

После создания базовой спецификации используйте команду:

```bash
speckit.clarify
```

Эта команда:
- Идентифицирует неоднозначные или недостающие элементы
- Задает уточняющие вопросы
- Интегрирует ответы в спецификацию

### 3. Планирование реализации (Фаза 0)

Для создания плана реализации используйте:

```bash
speckit.plan
```

**Результаты Фазы 0:**
- `.qwen/specify/specs/{ID}/plans/phase0-plan.json`
- `.qwen/specify/specs/{ID}/plans/phase0-agents.json`
- `.qwen/specify/specs/{ID}/plans/phase0-assignments.json`

Эта команда:
- Создает технический план реализации
- Определяет архитектурные решения
- Генерирует артефакты проектирования
- Назначает агентов на задачи

### 4. Генерация задач

Для создания задач реализации используйте:

```bash
speckit.tasks
```

**Расположение:** `.qwen/specify/specs/{ID}/tasks.md`

Эта команда:
- Разбивает реализацию на конкретные задачи
- Определяет зависимости между задачами
- Создает файл `tasks.md` с планом выполнения
- Использует назначения из Фазы 0

### 5. Проверка спецификации (Quality Gate 5)

**Перед реализацией:**

```bash
# Проверка качества спецификации
.qwen/scripts/quality-gates/check-specifications.sh .qwen/specify/specs/{ID}
```

**Проверки:**
- ✅ Все разделы спецификации заполнены
- ✅ Требования тестируемые
- ✅ Критерии успеха измеримы
- ✅ Нет деталей реализации

### 6. Реализация

Для выполнения задач используйте:

```bash
speckit.implement
```

Эта команда:
- Выполняет задачи в соответствии с планом
- Следит за прогрессом
- Проверяет выполнение условий завершения
- Проходит Quality Gates

## Структура спецификации

**Расположение:** `.qwen/specify/specs/{ID}-{feature}/spec.md`

Каждая спецификация должна содержать:

1. **Краткое описание** - краткое изложение сути функциональности
2. **Контекст** - описание ситуации, в которой будет использоваться функциональность
3. **Акторы** - пользователи или системы, взаимодействующие с функциональностью
4. **Требования** - функциональные и нефункциональные требования
5. **Сценарии использования** - описания типичных сценариев использования
6. **Условия успеха** - критерии, по которым можно определить успешность реализации
7. **Ограничения** - ограничения, которые должны быть учтены при реализации
8. **Предположения** - предположения, на которые опирается спецификация
9. **Возможные риски** - потенциальные риски при реализации
10. **Дополнительная информация** - любая другая полезная информация

## Структура директории спецификаций

```
.qwen/specify/specs/{ID}-{feature}/
├── spec.md                      # Спецификация функции
├── plan.md                      # План реализации
├── tasks.md                     # Задачи реализации
├── research.md                  # Исследования
├── data-model.md                # Модель данных
├── quickstart.md                # Быстрый старт
├── contracts/                   # API контракты
│   ├── api-spec.yaml           # OpenAPI спецификация
│   └── mcp-config.json         # MCP конфигурация
└── plans/
    ├── phase0-plan.json        # План Фазы 0
    ├── phase0-agents.json      # Требуемые агенты
    └── phase0-assignments.json # Назначения
```

## Качество спецификаций

Хорошая спецификация должна быть:

- **Полной** - содержать все необходимые разделы
- **Ясной** - использовать понятный и однозначный язык
- **Последовательной** - не содержать противоречий
- **Тестируемой** - требования должны быть проверяемыми
- **Измеримой** - критерии успеха должны быть количественными
- **Независимой от реализации** - описывать ЧТО, а не КАК

## Quality Gate 5: Pre-Implementation Gate

**Скрипт:** `.qwen/scripts/quality-gates/check-specifications.sh`

**Проверки:**
1. ✅ Все разделы спецификации заполнены
2. ✅ Требования тестируемые и однозначные
3. ✅ Критерии успеха измеримы
4. ✅ Нет деталей реализации
5. ✅ Пользовательские сценарии определены
6. ✅ Риски идентифицированы

**Блокирующая:** true

**При неудаче:**
- ОСТАНОВКА процесса
- Возврат к уточнению спецификации
- Повторная проверка после исправлений

## Интеграция с Фазой 0

**Процесс:**

1. **speckit.specify** создает спецификацию
2. **Фаза 0** анализирует задачи и назначает агентов
3. **speckit.plan** создает план реализации
4. **speckit.tasks** генерирует задачи с назначениями
5. **Quality Gate 5** проверяет спецификацию
6. **speckit.implement** выполняет реализацию

## Примеры использования

### Пример 1: Создание новой функции

```bash
# 1. Создать спецификацию
speckit.specify "Добавить аутентификацию пользователя через OAuth2"

# 2. Инициализировать Фазу 0
.qwen/specify/scripts/phase0-analyzer.sh .qwen/specify/specs/001-oauth2-auth

# 3. Создать план
speckit.plan

# 4. Генерировать задачи
speckit.tasks

# 5. Проверить спецификацию
.qwen/scripts/quality-gates/check-specifications.sh .qwen/specify/specs/001-oauth2-auth

# 6. Реализовать
speckit.implement
```

### Пример 2: Уточнение существующей спецификации

```bash
# Уточнить требования
speckit.clarify

# Обновить план
speckit.plan

# Перегенерировать задачи
speckit.tasks
```
- **Тестопригодной** - иметь измеримые критерии успеха
- **Реалистичной** - учитывать ограничения и риски

## Интеграция с навыками

Спецификации могут использовать навыки агентов для:

- Анализа качества спецификаций (навык `specification-analyzer`)
- Проверки соответствия требованиям
- Идентификации потенциальных проблем
- Генерации дополнительных артефактов

## Интеграция с системой агентов

### Специализированные агенты для работы со спецификациями

- `specification-analyst`: Анализ спецификаций на полноту, ясность и соответствие требованиям
- `specification-compliance-checker`: Проверка соответствия реализации требованиям спецификаций

### Автоматическое определение агентов

При генерации задач через `speckit.tasks` система автоматически определяет, какие агенты наиболее подходят для выполнения каждой задачи:

- Задачи тестирования → `code-quality-checker`
- Задачи безопасности → `security-orchestrator`
- Задачи исправления ошибок → `bug-fixer`
- Задачи анализа кода → `bug-hunter`
- Задачи перевода документации → `tech-translator-ru`

Для задач, для которых нет подходящего агента, используется метка `[futures:agent-name]`, что указывает на необходимость создания специализированного агента.

## Контрольные точки качества для спецификаций

### Gate 5: Pre-Implementation Checks

Перед началом реализации спецификации проходят проверку качества:

1. Проверка наличия всех обязательных разделов
2. Проверка соответствия конституции проекта
3. Проверка тестопригодности требований
4. Проверка измеримости условий успеха

Запуск проверки:
```bash
./scripts/run_quality_gate.sh 5 specs/номер-название-функции/
```

## Поддержка TDD (Test-Driven Development)

### Автоматическая генерация тестов

Для поддержки TDD подхода доступен скрипт генерации тестов на основе спецификаций:

```bash
scripts/specification-tools/generate-tests-from-spec.sh specs/номер-название-функции/spec.md
```

Скрипт генерирует:
- Модульные тесты на основе функциональных требований
- Интеграционные тесты на основе сценариев использования
- Контрактные тесты на основе API спецификаций
- Тест-план с критериями завершения

### Принципы TDD в процессе

1. **Сначала тесты**: Тесты создаются до реализации
2. **Красный-зеленый-рефакторинг**:
   - Сначала тесты должны падать (красный)
   - Затем реализация, чтобы тесты проходили (зеленый)
   - Затем рефакторинг с сохранением прохождения тестов
3. **Тесты как спецификация**: Тесты документируют ожидаемое поведение

## Лучшие практики

- Создавайте спецификации до начала реализации
- Вовлекайте заинтересованные стороны в процесс уточнения
- Используйте шаблоны для обеспечения согласованности
- Регулярно пересматривайте и обновляйте спецификации
- Связывайте спецификации с задачами и тестами
- Используйте контрольные точки качества для проверки спецификаций
- Применяйте TDD подход: сначала тесты, потом реализация
- Назначайте подходящие агенты для выполнения задач
- Используйте метки futures для обозначения необходимости новых агентов
