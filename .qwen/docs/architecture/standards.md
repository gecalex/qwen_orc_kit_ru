# Архитектурные стандарты проекта Qwen Code CLI

## Общие принципы

### 1. Модульность
- Каждый компонент должен иметь одну четко определенную ответственность
- Компоненты должны быть слабо связаны между собой
- Использовать принципы SOLID при разработке

### 2. Документация
- Все компоненты должны быть задокументированы
- Документация должна быть актуальной и отражать текущее состояние
- Использовать шаблоны для обеспечения консистентности

### 3. Безопасность
- Все компоненты должны учитывать безопасность при проектировании
- Проверять входные данные
- Не хранить чувствительную информацию в открытом виде

## Структура проекта

### .qwen/ - Компоненты системы
- `agents/` - Определения агентов
- `commands/` - Определения команд
- `skills/` - Определения навыков

### specs/ - Спецификации функций
- Каждая функция в отдельной поддиректории
- Следовать шаблону спецификации

### docs/ - Документация
- `architecture/` - Архитектурная документация
- `guides/` - Руководства пользователя и разработчика

### scripts/ - Скрипты автоматизации
- `orchestration-tools/` - Скрипты оркестрации
- `quality-gates/` - Скрипты контрольных точек качества

## Стандарты кодирования

### Именование
- Использовать snake_case для имен файлов и директорий
- Использовать PascalCase для имен агентов
- Использовать kebab-case для имен команд

### Форматирование
- Использовать 2 пробела для отступов в YAML и MD файлах
- Использовать 4 пробела для отступов в Bash скриптах
- Соблюдать единообразие в форматировании

## Тестирование

### Типы тестов
- Модульные тесты для отдельных компонентов
- Интеграционные тесты для взаимодействия компонентов
- Сквозные тесты для проверки функциональности

### Покрытие
- Минимальное покрытие 80% для новых компонентов
- Покрытие должно включать как позитивные, так и негативные сценарии

## Контроль качества

### Проверки
- Все изменения должны проходить базовую проверку качества
- Использовать скрипты из `scripts/orchestration-tools/` для проверок
- Проверять соответствие архитектурным стандартам

### Рецензирование
- Все изменения должны проходить код-ревью
- Использовать систему задач для отслеживания изменений
- Обеспечивать прозрачность процесса разработки
