# Архитектура TDD в Qwen Code Orchestrator Kit

**Версия:** 1.0.0  
**Дата:** 2026-03-25  
**Статус:** RELEASE

---

## 🎯 Цель документа

Этот документ описывает архитектуру TDD (Test-Driven Development) в проекте Qwen Code Orchestrator Kit.

---

## 📊 Проблема

**До внедрения TDD:**

```
❌ Агенты разработки (backend_dev, frontend_dev) писали код БЕЗ тестов
❌ Тесты создавались ПОСЛЕ (если вообще создавались)
❌ TDD требование в описании агента игнорировалось
❌ Качество кода было низким
```

**Почему это происходило:**

```
❌ Один агент отвечал и за тесты, и за код
❌ Не было явного разделения задач
❌ Не было зависимостей между тестами и кодом
❌ backend_dev мог написать код без тестов
```

---

## 🏗️ Решение: TDD Architecture

### **Ключевая идея:**

```
✅ Разделение задач: TEST → CODE
✅ Разные агенты: test_engineer → backend_dev
✅ Явные зависимости: CODE зависит от TEST
✅ TDD гарантирован на архитектурном уровне
```

---

## 📋 Архитектура

### **1. work_planning_test_assigner**

**Назначение:** Анализ задач и создание TEST/CODE подзадач

**Workflow:**
```
tasks.md (N задач)
    ↓
work_planning_test_assigner
    ↓
tasks-with-test-assignments.json:
  - T-004-001-T (TEST, work_testing_tdd_specialist, 2h)
  - T-004-001-C (CODE, work_backend_api_validator, 4h, depends_on: T-004-001-T)
```

**Результат:**
- Для КАЖДОЙ задачи создаются ДВЕ подзадачи
- TEST подзадача выполняется ПЕРЕД CODE подзадачей
- CODE подзадача имеет явную зависимость от TEST

---

### **2. work_testing_tdd_specialist**

**Назначение:** Создание тестов ПЕРЕД кодом (TDD First)

**Workflow:**
```
1. ✅ Получить задачу T-004-001-T
2. ✅ Прочитать acceptance criteria
3. ✅ Создать тесты (tests/test_004_001.py)
4. ✅ Запустить тесты → RED (падение)
5. ✅ Закоммитить тесты
6. ✅ Передать разработчику
```

**Почему это важно:**
- ✅ backend_dev НИКОГДА не напишет тесты перед кодом
- ✅ Тесты без явного указания будут пропущены
- ✅ Разделение задач улучшает качество кода

---

### **3. work_backend_api_validator / work_frontend_component_generator**

**Назначение:** Написание кода под СУЩЕСТВУЮЩИЕ тесты

**Workflow:**
```
1. ✅ Получить задачу T-004-001-C
2. ✅ Проверить что тесты СУЩЕСТВУЮТ
3. ✅ Прочитать acceptance criteria из тестов
4. ✅ Написать код под тесты
5. ✅ Запустить тесты → GREEN (прохождение)
6. ✅ Закоммитить код с тестами
```

**Запрещено:**
- ❌ НЕ пиши тесты (это делает test_engineer)
- ❌ НЕ пиши код БЕЗ существующих тестов
- ❌ НЕ коммить код без прохождения тестов

---

### **4. orc_testing_quality_assurer**

**Назначение:** Оркестратор тестирования

**Workflow:**
```
1. ✅ Прочитать tasks-with-test-assignments.json
2. ✅ Назначить TEST задачи → work_testing_tdd_specialist
3. ✅ Дождаться завершения тестов (RED)
4. ✅ Назначить CODE задачи → work_backend_api_validator
5. ✅ Проверить что тесты прошли (GREEN)
6. ✅ Quality Gate → коммит
```

## 📊 Метрики

### **До внедрения TDD:**

```
❌ Покрытие тестами: 0%
❌ Тестов написано: 0
❌ Качество кода: низкое
```

### **После внедрения TDD:**

```
✅ Покрытие тестами: ≥ 80%
✅ Тестов написано: 106 (по одному на задачу)
✅ Качество кода: высокое
✅ Все тесты проходят (GREEN)
```

## 📋 Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Гарантированный TDD** | work_planning_test_assigner создаёт TEST задачи автоматически |
| **Независимость** | test_engineer НЕ зависит от backend_dev |
| **Качество** | Код пишется под тесты = высокое качество |
| **Покрытие** | ≥ 80% покрытие гарантировано |
| **Взаимодействие** | test_engineer → backend_dev → test_engineer |
| **Масштабируемость** | Можно добавить новые типы тестов |

---

## 📋 Недостатки

| Недостаток | Описание | Митигация |
|------------|----------|-----------|
| **Увеличение задач** | 106 → 2N задач | Автоматическое создание work_planning_test_assigner |
| **Увеличение времени** | +2 часа на задачу | Выше качество, меньше багов |
| **Сложность координации** | TEST → CODE зависимости | orc_testing_quality_assurer координирует |

## 📋 Best Practices

### **1. TDD First**

```
✅ ВСЕГДА пиши тесты ПЕРЕД кодом
✅ Запускай тесты → RED (перед кодом)
✅ Пиши код → GREEN (после кода)
✅ Рефакторинг → тесты должны пройти
```

### **2. Разделение ответственности**

```
✅ test_engineer пишет тесты
✅ backend_dev пишет код
✅ НЕ смешивай ответственность
```

### **3. Зависимости**

```
✅ CODE задача зависит от TEST задачи
✅ CODE НЕ может быть написана БЕЗ TEST
✅ orc_testing_quality_assurer проверяет зависимости
```

### **4. Quality Gate**

```
✅ ВСЕГДА проходи Quality Gate перед коммитом
✅ Проверяй покрытие (≥ 80%)
✅ Проверяй что ВСЕ тесты прошли (GREEN)
```
