# Механизм вызова подагентов в системе Claude Code Orchestrator Kit

## Введение

Одним из ключевых аспектов архитектуры Claude Code Orchestrator Kit является механизм взаимодействия между компонентами системы. Важно понимать, что существуют два разных паттерна оркестрации, которые используются в зависимости от типа задачи:

1. **Основной паттерн (95% случаев)** - используется при ручной разработке, исправлении багов, рефакторинге
2. **Паттерн на основе агентов (5% случаев)** - используется для автоматизированных рабочих процессов проверки состояния

## Основной паттерн оркестрации (95% случаев)

В этом паттерне, который описан в файле `CLAUDE.md`, **оркестратором является сама основная сессия Claude Code**. Это означает, что:

- Claude Code напрямую взаимодействует с пользователем
- Claude Code может использовать инструмент `Task` для вызова подагентов
- Claude Code сам анализирует задачи, делегирует их подагентам и проверяет результаты

### Как это работает:
1. Пользователь запускает команду (например, `/speckit.implement`)
2. Claude Code (в роли оркестратора) анализирует задачу
3. Claude Code использует инструмент `Task` для вызова нужного подагента
4. После выполнения подагентом задачи, Claude Code проверяет результат
5. Процесс продолжается до завершения

В этом паттерне **оркестратор (Claude Code) имеет доступ к инструменту `Task`** и может напрямую вызывать подагентов.

## Паттерн на основе агентов (5% случаев)

Этот паттерн используется для автоматизированных рабочих процессов проверки состояния, таких как:
- `/health-bugs` - Обнаружение и исправление ошибок
- `/health-security` - Сканирование уязвимостей безопасности
- `/health-cleanup` - Удаление мертвого кода
- `/health-deps` - Управление зависимостями

### Ключевое отличие

В этом паттерне **оркестраторы НЕ могут использовать инструмент `Task`**. Вместо этого используется **шаблон возврата управления** с файлами плана.

### Архитектура взаимодействия:

```
Пользователь запускает /health-bugs
  ↓
Команда с косой чертой (главный оркестратор)
  ↓
Инструмент Task → bug-orchestrator (подоркестратор)
  ↓
Создает файл плана: .tmp/current/plans/bug-detection.json
Возвращает управление
  ↓
Команда читает файл плана
  ↓
Инструмент Task → bug-hunter (работник)
  ↓
Работник выполняет, генерирует отчет, возвращает управление
  ↓
Команда → инструмент Task → bug-orchestrator (проверка)
  ↓
Оркестратор проверяет на контрольной точке качества
Создает следующий файл плана
Возвращает управление
  ↓
...цикл продолжается
```

### Почему агенты-оркестраторы не могут использовать инструмент Task?

Согласно документации:

> "Агенты-оркестраторы работают в изолированном контексте без доступа к инструменту Task. Они координируются через файловую коммуникацию (файлы плана)."

### Шаблон возврата управления

Поскольку Claude Code НЕ поддерживает автоматический вызов агентов, используется следующий шаблон:

1. **Оркестратор создает файл плана** с полем `nextAgent`
2. **Оркестратор сообщает о готовности и возвращает управление**
3. **Основная сессия (команда) читает файл плана**
4. **Основная сессия вызывает работника с помощью инструмента Task** с `subagent_type` из плана
5. **Работник выполняет задачу и возвращает управление**
6. **Основная сессия вызывает оркестратора для проверки**
7. **Цикл повторяется**

### Протокол сигнала готовности

Оркестраторы должны выводить:
```
Phase preparation complete.

Plan: .tmp/current/plans/.bug-detection-plan.json
Next Agent: bug-hunter

Returning control to main session.
```

## Файлы плана

Файлы плана являются основным средством коммуникации между оркестраторами и работниками:

```json
{
  "workflow": "bug-management",
  "phase": "detection",
  "config": {
    "priority": "all",
    "severity": "critical"
  },
  "validation": {
    "required": ["report-exists", "type-check", "build"],
    "optional": ["tests-pass"]
  },
  "nextAgent": "bug-hunter"
}
```

## Заключение

Таким образом, в системе Claude Code Orchestrator Kit реализованы два разных подхода к оркестрации:

1. **В основном паттерне (95% случаев)**: Claude Code сам является оркестратором и может использовать инструмент `Task` для вызова подагентов напрямую.

2. **В паттерне на основе агентов (5% случаев)**: Используется шаблон возврата управления, где команды (с косой чертой) выступают в роли главных оркестраторов, а агенты-оркестраторы не имеют доступа к инструменту `Task` и взаимодействуют через файлы плана.

Это объясняет, почему в документации говорится, что "агенты-оркестраторы не могут использовать инструмент Task" - это относится только к 5% случаев автоматизированных рабочих процессов проверки состояния, а не к основному паттерну разработки.
