## Полный детальный отчет об алгоритме и логике, реализованных в CLAUDE.md

### 1. Общая архитектура и назначение CLAUDE.md

Файл `CLAUDE.md` служит "поведенческой операционной системой" для проекта Claude Code Orchestrator Kit. Он переопределяет поведение Claude Code по умолчанию и определяет основные правила оркестрации агентов. В отличие от традиционных подходов, где Claude Code выполняет все задачи напрямую, этот файл устанавливает паттерн оркестратора, где Claude Code координирует работу специализированных подагентов.

### 2. Основной паттерн: Claude Code как оркестратор

#### 2.1. Правила поведения

CLAUDE.md определяет 8 основных правил поведения Claude Code:

1. **Сбор полного контекста впервые (обязательно)**:
   - Прочитать существующий код в связанных файлах
   - Выполнить поиск по кодовой базе на предмет похожих паттернов
   - Изучить релевантную документацию
   - Проверить последние коммиты в связанных модулях
   - Понять зависимости и точки интеграции

2. **Делегирование подагентам**:
   - Предоставить полный контекст (фрагменты кода, пути к файлам, паттерны, документация)
   - Четко указать ожидаемый результат и критерии валидации
   - После делегирования всегда верифицировать результат
   - Если результат неверен: повторно делегировать с исправлениями
   - Если ошибки TypeScript: повторно делегировать тому же агенту или `typescript-types-specialist`

3. **Прямая реализация (только для тривиальных случаев)**:
   - Установка одного пакета
   - Исправления в одну строку
   - Простые импорты
   - Минимальные изменения конфигурации

4. **Отслеживание прогресса**:
   - Создавать todo при старте задачи
   - Помечать как `in_progress` до начала работы
   - Помечать как `completed` только после верификации

5. **Стратегия коммитов**:
   - Выполнять `/push patch` после каждой завершенной задачи
   - Отмечать задачу как `[X]` в `tasks.md`
   - Добавлять артефакты
   - Обновлять TodoWrite до статуса `completed`

6. **Паттерн выполнения**:
   - СОБРАТЬ ПОЛНЫЙ КОНТЕКСТ
   - Делегировать подагенту ИЛИ реализовать напрямую (только если тривиально)
   - ВЕРИФИЦИРОВАТЬ результат
   - Цикл accept/reject (повторная делегация при необходимости)
   - Обновить TodoWrite до статуса completed
   - Отметить задачу как [X] в tasks.md + добавить артефакты
   - Выполнить /push patch

7. **Обработка противоречий**:
   - Собрать контекст, проанализировать паттерны проекта
   - Если действительно неоднозначно: задать уточняющий вопрос пользователю
   - Задавать вопрос только если невозможно определить лучшую практику

8. **Подход "библиотека вперёд"**:
   - Перед написанием нового кода (>20 строк) всегда искать существующие библиотеки
   - Проверять недельные загрузки >1000, коммиты за последние 6 месяцев, поддержку TypeScript/типов

#### 2.2. Фаза планирования

Перед реализацией задач:

- Анализ модели выполнения (параллельная/последовательная)
- Назначение исполнителей: `MAIN` для тривиальных, `existing` при 100% совпадении, `FUTURE` иначе
- Создание `FUTURE`-агентов: запуск N вызовов `meta-agent-v3` в одном сообщении
- Разрешение исследований (простые: решать сейчас, сложные: промпт `deepsresearch`)
- Атомарность: 1 задача = 1 вызов агента
- Параллелизм: запуск N вызовов в одном сообщении

### 3. Паттерн health-workflow (5% случаев)

Для автоматизированных рабочих процессов проверки состояния (`/health-bugs`, `/health-security`, `/health-cleanup`, `/health-deps`) используется другой подход, описанный в `docs/Agents Ecosystem/AGENT-ORCHESTRATION.md`.

### 4. Архитектура оркестрации агентов

Согласно документации в `AGENT-ORCHESTRATION.md`, архитектура основана на двухуровневой иерархии:

#### 4.1. Оркестраторы (Уровень 1)

**Назначение**: Координировать многофазные рабочие процессы

**Обязанности**:
- Создавать файлы плана для каждой фазы
- Сигнализировать готовность пользователю (Возврат управления)
- Проверять выводы работников на контрольных точках качества
- Отслеживать прогресс через TodoWrite
- Обрабатывать ошибки с инструкциями отката
- Генерировать итоговые сводки отчетов

**Критические правила**:
- НЕ использовать инструмент Task для вызова подагентов
- НЕ выполнять реализацию (делегировать работникам)
- НЕ пропускать проверки контрольных точек качества
- СОЗДАВАТЬ файлы плана перед сигнализацией
- ПРОВЕРЯТЬ выводы работников на контрольных точках качества
- СООБЩАТЬ статус пользователю

#### 4.2. Работники (Уровень 2)

**Назначение**: Выполнять работу, специфичную для области

**Обязанности**:
- Сначала прочитать файл плана
- Выполнить работу области (обнаружение, исправление, проверка)
- Проверить работу внутри себя
- Сгенерировать структурированный отчет
- Вернуться в главную сессию

**Критические правила**:
- НЕ вызывать других агентов
- НЕ пропускать генерацию отчета
- НЕ сообщать об успехе без проверки
- ПРОЧИТАТЬ файл плана первым
- СГЕНЕРИРОВАТЬ журналы изменений (для отката)
- ПРОВЕРИТЬ СВОЮ работу

#### 4.3. Навыки (Утилиты)

**Назначение**: Повторно используемые утилитарные функции

**Характеристики**:
- Без состояния (контекст не нужен)
- <100 строк логики
- Одна ответственность
- No agent invocation

### 5. Шаблон возврата управления

Ключевая особенность архитектуры - шаблон возврата управления, поскольку Claude Code не поддерживает автоматический вызов агентов. Процесс выглядит следующим образом:

1. Оркестратор создает файл плана (например, `.bug-detection-plan.json`) с полем `nextAgent`
2. Оркестратор проверяет файл плана с помощью навыка `validate-plan-file`
3. Оркестратор обновляет TodoWrite
4. Оркестратор сигнализирует пользователю и возвращает управление основной сессии
5. Основная сессия (slash command или пользователь) читает файл плана
6. Основная сессия явно вызывает работника с помощью инструмента Task
7. Работник читает план файла, выполняет работу, генерирует отчет, возвращает управление
8. Основная сессия возобновляет оркестратора для проверки

### 6. Файлы плана

Файлы плана являются основным средством коммуникации между оркестраторами и работниками. Они следуют стандартной структуре:

```json
{
  "workflow": "bug-management",
  "phase": "detection",
  "config": {
    "priority": "all",
    "severity": "critical"
  },
  "validation": {
    "required": ["report-exists", "type-check", "build"],
    "optional": ["tests-pass"]
  },
  "nextAgent": "bug-hunter"
}
```

Все файлы плана должны соответствовать JSON-схемам, определенным в `.claude/schemas/`:
- `base-plan.schema.json` - базовая схема (для всех планов)
- `bug-plan.schema.json` - схема для управления багами
- `security-plan.schema.json` - схема для аудита безопасности
- `dead-code-plan.schema.json` - схема для очистки мертвого кода
- `dependency-plan.schema.json` - схема для управления зависимостями

### 7. Контрольные точки качества

Согласно `QUALITY-GATES-SPECIFICATION.md`, контрольные точки качества - это точки проверки валидации, которые обеспечивают соответствие работы стандартам качества перед переходом к следующей фазе. Они бывают двух типов:

#### 7.1. Блокирующие точки

- Останавливают прогресс рабочего процесса, если критерии не пройдены
- Используются для критических стандартов качества
- Требуют вмешательства пользователя (исправить или пропустить)
- Записываются с высокой степенью важности

#### 7.2. Неблокирующие точки

- Предупреждают, но позволяют продолжение
- Используются для лучших практик и рекомендаций
- Записываются в сводный отчет
- Пользователь может устранить позже

### 8. Навыки (Skills)

Навыки реализованы как переиспользуемые утилиты в формате SKILL.md файлов. Основные навыки:

- `validate-plan-file` - проверка файлов плана по JSON-схемам
- `run-quality-gate` - выполнение команд проверки с настраиваемым поведением блокировки
- `rollback-changes` - откат изменений при сбое проверки
- `generate-report-header` - создание стандартизированных заголовков отчетов
- `validate-report-file` - проверка полноты отчетов

### 9. Структура отчетов

Согласно `REPORT-TEMPLATE-STANDARD.md`, все рабочие агенты должны генерировать отчеты, следуя стандартизированному шаблону, который включает:

- YAML frontmatter с метаданными
- Заголовок с временной меткой, статусом, версией
- Итоги с ключевыми метриками
- Подробные находки (специфичные для типа отчета)
- Результаты валидации
- Следующие шаги

### 10. Интеграция с MCP-серверами

Архитектура предусматривает использование различных MCP-серверов (Model Context Protocol) для расширения возможностей Claude Code. В зависимости от задачи могут использоваться:

- Context7 - документация по библиотекам и лучшие практики
- Sequential Thinking - усиленные возможности рассуждения
- Supabase - доступ и управление базой данных
- Playwright - автоматизация браузера
- ShadCN - система UI-компонентов
- n8n - интеграция платформы автоматизации рабочих процессов

### 11. Ведение журнала изменений и откат

Для обеспечения возможности восстановления при сбоях валидации все изменения логируются в формате JSON-файлов. При сбое валидации используется навык `rollback-changes` для отката всех изменений.

### 12. Соглашения проекта

CLAUDE.md также определяет соглашения проекта:
- Организация файлов: агенты в `.claude/agents/{domain}/{orchestrators|workers}/`
- Команды в `.claude/commands/`
- Навыки в `.claude/skills/{skill-name}/SKILL.md`
- Временные файлы в `.tmp/current/` (игнорируются Git)
- Отчеты в `docs/reports/{domain}/{YYYY-MM}/`

### 13. Заключение

CLAUDE.md реализует сложную архитектуру оркестрации агентов, которая позволяет Claude Code эффективно координировать работу специализированных подагентов. Основные принципы:

1. **Сохранение контекста** - Claude Code остается легким (~10-15K токенов) за счет делегирования сложных задач подагентам
2. **Специализация** - каждый подагент является экспертом в своей области
3. **Контроль качества** - обязательная верификация после каждой делегации
4. **Изоляция контекста** - каждый агент работает в своем изолированном контексте
5. **Стандартизация** - все компоненты следуют строго определенным шаблонам и форматам

Эта архитектура позволяет эффективно масштабировать сложные рабочие процессы, такие как обнаружение и исправление багов, аудит безопасности, управление зависимостями и очистка мертвого кода, при этом сохраняя высокое качество и надежность результатов.
