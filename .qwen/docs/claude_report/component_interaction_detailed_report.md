# Подробный отчет о взаимодействии компонентов системы Claude Code Orchestrator Kit

## Введение

В этом отчете подробно описаны архитектурные аспекты системы Claude Code Orchestrator Kit, включая взаимодействие между центральным оркестратором (поведение которого регулируется файлом CLAUDE.md), оркестраторами и работниками, а также ключевые аспекты реализации системы.

## Архитектура системы

### Двухуровневая иерархия агентов

Система использует двухуровневую иерархию агентов:

1. **Оркестраторы (Уровень 1)** - координируют многофазные рабочие процессы
2. **Работники (Уровень 2)** - выполняют работу, специфичную для области

### Центральный оркестратор (CLAUDE.md)

Центральный оркестратор - это Claude Code, поведение которого регулируется файлом `CLAUDE.md`. Он реализует два основных паттерна:

1. **Основной паттерн (95% случаев)** - используется при ручной разработке, исправлении багов, рефакторинге
2. **Паттерн health-workflow (5% случаев)** - используется для автоматизированных рабочих процессов проверки состояния

## Взаимодействие между компонентами

### 1. Взаимодействие между центральным оркестратором и пользователем

Центральный оркестратор (Claude Code) напрямую взаимодействует с пользователем:

- Принимает команды от пользователя (например, `/speckit.implement`, `/health-bugs`)
- Обрабатывает ввод пользователя и применяет правила из `CLAUDE.md`
- Отображает результаты пользователю
- Запрашивает уточнения при необходимости
- Управляет процессом выполнения задач

### 2. Взаимодействие между центральным оркестратором и обычным оркестратором

#### В основном паттерне (95% случаев):
- Центральный оркестратор сам является оркестратором и может напрямую вызывать подагентов через инструмент `Task`
- Нет отдельных "обычных оркестраторов" - Claude Code сам координирует работу

#### В паттерне health-workflow (5% случаев):
- Пользователь запускает команду (например, `/health-bugs`)
- Центральный оркестратор вызывает команду с косой чертой
- Команда вызывает подоркестратор через инструмент `Task`
- Подоркестратор создает файл плана и возвращает управление
- Команда читает файл плана и вызывает соответствующего работника
- После выполнения работника команда возобновляет подоркестратор для проверки

### 3. Взаимодействие между центральным оркестратором и работником

#### В основном паттерне (95% случаев):
- Центральный оркестратор (Claude Code) напрямую вызывает работников через инструмент `Task`
- Передает полный контекст (фрагменты кода, пути к файлам, паттерны, критерии проверки)
- После выполнения проверяет результаты

#### В паттерне health-workflow (5% случаев):
- Центральный оркестратор не вызывает работников напрямую
- Взаимодействие осуществляется через команды и файлы плана
- Команда читает файл плана и вызывает работника через инструмент `Task`

### 4. Взаимодействие между обычным оркестратором и работником

В паттерне health-workflow взаимодействие между оркестратором и работником осуществляется через шаблон возврата управления:

1. **Оркестратор создает файл плана** с информацией о следующем агенте:
   ```json
   {
     "workflow": "bug-management",
     "phase": "detection",
     "config": {
       "priority": "all",
       "severity": "critical"
     },
     "validation": {
       "required": ["report-exists", "type-check", "build"],
       "optional": ["tests-pass"]
     },
     "nextAgent": "bug-hunter"
   }
   ```

2. **Оркестратор проверяет файл плана** с помощью навыка `validate-plan-file`

3. **Оркестратор обновляет TodoWrite** и сигнализирует готовность

4. **Оркестратор возвращает управление** основной сессии (НЕ вызывает работника напрямую)

5. **Команда читает файл плана** и узнает, какого работника вызвать

6. **Команда вызывает работника** через инструмент `Task` с типом агента из поля `nextAgent`

7. **Работник читает файл плана**, выполняет задачу, генерирует отчет и возвращает управление

8. **Команда возобновляет оркестратора** для проверки результатов

## Шаблон возврата управления

Шаблон возврата управления - ключевой аспект архитектуры, позволяющий оркестраторам координировать работу без прямого вызова работников:

```
Пользователь запускает /health-bugs
  ↓
Команда с косой чертой (главный оркестратор)
  ↓
Инструмент Task → bug-orchestrator (подоркестратор)
  ↓
Создает файл плана: .tmp/current/plans/bug-detection.json
Возвращает управление
  ↓
Команда читает файл плана
  ↓
Инструмент Task → bug-hunter (работник)
  ↓
Работник выполняет, генерирует отчет, возвращает управление
  ↓
Команда → инструмент Task → bug-orchestrator (проверка)
  ↓
Оркестратор проверяет на контрольной точке качества
Создает следующий файл плана
Возвращает управление
  ↓
...цикл продолжается
```

## Ключевые аспекты реализации

### 1. Файлы плана

**Расположение**: `.tmp/current/plans/{workflow}-{phase}.json`

**Функция**: Коммуникация между оркестраторами и работниками

**Схемы валидации**: Все файлы плана должны соответствовать JSON-схемам в `.claude/schemas/`:
- `base-plan.schema.json` - базовая схема
- `bug-plan.schema.json` - схема для управления багами
- `security-plan.schema.json` - схема для аудита безопасности
- `dead-code-plan.schema.json` - схема для очистки мертвого кода
- `dependency-plan.schema.json` - схема для управления зависимостями

**Навык валидации**: `validate-plan-file` - проверяет соответствие файлов плана ожидаемой схеме

### 2. Файлы отчетов

**Временные файлы**: `.tmp/current/reports/{report-name}.md`
**Постоянные файлы**: `docs/reports/{domain}/{YYYY-MM}/{date}-{report-name}.md`

**Структура**: Следует стандарту, определенному в `REPORT-TEMPLATE-STANDARD.md`:
- YAML frontmatter с метаданными
- Заголовок с временной меткой, статусом, версией
- Итоги с ключевыми метриками
- Подробные находки (специфичные для типа отчета)
- Результаты валидации
- Следующие шаги

**Навык генерации заголовка**: `generate-report-header` - создает стандартизованные заголовки отчетов

### 3. Контрольные точки качества

**Типы**:
- **Блокирующие точки** - останавливают прогресс, если критерии не пройдены
- **Неблокирующие точки** - предупреждают, но позволяют продолжение

**Навык проверки**: `run-quality-gate` - выполнение команд проверки с настраиваемым поведением блокировки

**Проверки**:
- Проверка типов (type-check)
- Сборка (build)
- Тесты (tests)
- Линтинг (lint)
- Пользовательские команды

### 4. Ведение журнала изменений и откат

**Журналы изменений**: `.tmp/current/changes/.{domain}-changes.json`
**Резервные копии**: `.tmp/current/backups/`

**Навык отката**: `rollback-changes` - откат изменений при сбое проверки

### 5. Навыки (Skills)

**Расположение**: `.claude/skills/{skill-name}/SKILL.md`

**Назначение**: Повторно используемые утилитарные функции

**Примеры**:
- `validate-plan-file` - проверка файлов плана
- `run-quality-gate` - выполнение проверок качества
- `rollback-changes` - откат изменений
- `generate-report-header` - создание заголовков отчетов
- `validate-report-file` - проверка полноты отчетов
- `load-project-context` - загрузка контекста проекта
- `save-session-context` - сохранение контекста сессии
- `resume-session` - возобновление сессии

### 6. Временные файлы и каталоги

**Расположение**: `.tmp/`

**Структура**:
```
.tmp/
├── current/              # Активный запуск оркестрации
│   ├── plans/           # Файлы плана для работников
│   ├── reports/         # Временные отчеты от работников
│   ├── changes/         # Журналы изменений для отката
│   ├── backups/         # Резервные копии файлов перед модификацией
│   └── locks/           # Файлы блокировки для предотвращения конфликта
└── archive/             # Исторические запуски (автоочистка > 7 дней)
    └── YYYY-MM-DD-HHMMSS/
        ├── plans/
        ├── changes/
        └── reports/
```

### 7. Организация файлов агентов

**Оркестраторы**: `.claude/agents/{domain}/orchestrators/`
**Работники**: `.claude/agents/{domain}/workers/`

**Примеры**:
- `.claude/agents/health/orchestrators/bug-orchestrator.md`
- `.claude/agents/health/workers/bug-hunter.md`
- `.claude/agents/health/workers/bug-fixer.md`

### 8. Команды с косой чертой

**Расположение**: `.claude/commands/`

**Примеры**:
- `.claude/commands/health-bugs.md`
- `.claude/commands/health-security.md`
- `.claude/commands/speckit.*.md`

## Заключение

Система Claude Code Orchestrator Kit реализует сложную архитектуру оркестрации, которая позволяет эффективно координировать работу специализированных подагентов. Ключевыми аспектами являются:

1. **Шаблон возврата управления** - позволяет оркестраторам координировать работу без прямого вызова работников
2. **Файлы плана** - обеспечивают структурированную коммуникацию между компонентами
3. **Контрольные точки качества** - обеспечивают соблюдение стандартов качества
4. **Навыки** - обеспечивают переиспользуемую функциональность
5. **Ведение журнала изменений** - позволяет откатывать изменения при сбоях

Эта архитектура позволяет масштабировать сложные рабочие процессы, такие как обнаружение и исправление багов, аудит безопасности, управление зависимостями и очистка мертвого кода, при этом сохраняя высокое качество и надежность результатов.
