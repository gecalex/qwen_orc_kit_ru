## Реализация git workflow и логики проверок и коммитов

### 1. Git Workflow

Git workflow в системе Claude Code Orchestrator Kit реализован через команду `/push`, которая запускает скрипт `.claude/scripts/release.sh`. Этот скрипт:

- Автоматически фиксирует незафиксированные изменения перед релизом
- Определяет тип коммита на основе изменённых файлов:
  - Новые агенты → `feat(agents)`
  - Новые навыки → `feat(skills)`
  - Новые команды → `feat(commands)`
  - Модифицированные навыки → `feat(skills)`
  - Модифицированные команды → `feat(commands)`
  - Модифицированные скрипты → `chore(scripts)`
  - Модифицированные агенты → `chore(agents)`
  - Модифицированные MCP-конфиги → `chore(mcp)`
  - Изменения документации → `docs`

- Создаёт коммит с автоматически сгенерированным сообщением
- Обновляет версию в package.json файлах
- Генерирует запись в CHANGELOG.md
- Создаёт git-тег и отправляет изменения в GitHub

### 2. Логика проверок после выполнения задачи

Проверки реализованы через навык `run-quality-gate`, который выполняет:

- **Проверка типов**: `pnpm type-check`
- **Проверка сборки**: `pnpm build`
- **Проверка тестов**: `pnpm test`
- **Проверка линтинга**: `pnpm lint`
- **Пользовательские проверки**: настраиваемые команды

Каждая проверка может быть:
- **Блокирующей** (`blocking: true`) - останавливает процесс при сбое
- **Неблокирующей** (`blocking: false`) - предупреждает при сбое, но продолжает процесс

### 3. Механизм запуска проверок и тестов

В команде `speckit.implement.md` на шаге 8 описан процесс выполнения задач:

1. После выполнения задачи (шаг 4) - **ПРОВЕРИТЬ**:
   - Прочитать ВСЕ измененные файлы
   - Запустить проверку типов
   - Проверить наличие регрессий

2. На шаге 5 - **ПРИНЯТЬ/ОТКЛОНИТЬ**:
   - Если проверки прошли - продолжить
   - Если проверки не прошли - переделегировать с исправлениями

3. На шаге 7 - **ОБНОВИТЬ TODO**: Отметить задачу `completed` в TodoWrite
4. На шаге 8 - **ОБНОВИТЬ tasks.md**: Отметить задачу `[X]`
5. На шаге 9 - **ЗАФИКСИРОВАТЬ**: Запустить `/push patch`
6. На шаге 10 - **СЛЕДУЮЩАЯ ЗАДАЧА**: Перейти к следующей незавершенной задаче

### 4. Механизм git commit

Git commit реализован через команду `/push patch`, которая:
- Автоматически фиксирует все изменения
- Создаёт коммит с автоматически сгенерированным сообщением
- Обновляет версию и CHANGELOG
- Создаёт тег и отправляет изменения

### 5. Что происходит, если задача выполнена с ошибками

Если задача выполнена с ошибками:

1. **Проверки не проходят** (шаг 5 в speckit.implement.md):
   - Если проверки не прошли, система **переделегирует** задачу с исправлениями и сообщениями об ошибках
   - Возвращается к шагу 4 для повторной попытки

2. **Блокирующие проверки**:
   - Если блокирующая проверка не проходит (`action: "stop"`), процесс останавливается
   - Система сообщает об ошибке и не переходит к следующей задаче

3. **Неблокирующие проверки**:
   - Если неблокирующая проверка не проходит (`action: "warn"`), процесс продолжается, но регистрируется предупреждение

### 6. Механизм строгого контроля и предотвращения перехода к следующей задаче

Система реализует строгий контроль через:

1. **Критические правила** (раздел 10 в speckit.implement.md):
   - **НИКОГДА не пропускать проверку**
   - **НИКОГДА не продолжать, если задача не удалась**
   - **НИКОГДА не объединять фиксации (1 задача = 1 фиксация)**

2. **Атомарное выполнение**:
   - Только одна задача может быть `in_progress` одновременно
   - Система не переходит к следующей задаче, пока текущая не будет завершена успешно

3. **Проверки зависимостей**:
   - Система уважает зависимости задач: последовательные задачи выполняются по порядку
   - Параллельные задачи [P] могут выполняться вместе, но только если они не влияют на одни и те же файлы

4. **Остановка выполнения**:
   - Если какая-либо непараллельная задача не удается, выполнение останавливается
   - Для параллельных задач [P], система продолжает с успешными задачами, но сообщает о неудавшихся

5. **Фиксация после каждой задачи**:
   - Система запускает `/push patch` перед переходом к следующей задаче
   - Если фиксация не удается, процесс останавливается

Таким образом, система обеспечивает строгий контроль качества, гарантируя, что каждая задача из tasks.md проходит проверки и успешно фиксируется в git перед переходом к следующей задаче.
