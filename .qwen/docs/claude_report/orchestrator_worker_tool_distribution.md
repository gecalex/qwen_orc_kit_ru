# Отчет о распределении инструментов между оркестратором и работником

На основе анализа файлов `claudemd_algorithm_logic_report.md`, `orchestration_spec_agents_mcp_report.md` и `qwen_code_tools_reference.md`, представлено распределение инструментов Qwen Code CLI между оркестратором и работником.

## Инструменты для оркестратора

Оркестратор (уровень 1) отвечает за координацию многофазных рабочих процессов, создание файлов плана, проверку результатов работников и отслеживание прогресса. Согласно архитектуре, оркестратор НЕ должен использовать инструмент Task для вызова подагентов, вместо этого создавая файлы плана и возвращая управление основной сессии.

### Обязательные инструменты для оркестратора:

1. **TodoWrite** - для отслеживания прогресса выполнения фаз (обязательно согласно архитектуре)
2. **ReadFile** - для чтения файлов плана, созданных другими компонентами
3. **WriteFile** - для создания файлов плана (обязательно согласно архитектуре)
4. **ReadManyFiles** - для чтения нескольких файлов при необходимости анализа
5. **Glob** - для поиска файлов при необходимости анализа структуры проекта
6. **Grep** - для поиска по кодовой базе при анализе (обязательно согласно CLAUDE.md)
7. **ListFiles** - для перечисления файлов и структуры каталогов
8. **Shell** - для выполнения команд проверки (type-check, build и т.д.)
9. **Skill** - для вызова навыков валидации и качества (обязательно согласно архитектуре)
10. **WebSearch** - для поиска информации при необходимости
11. **WebFetch** - для получения информации с веб-сайтов при необходимости
12. **SaveMemory** - для сохранения информации о прогрессе между сессиями
13. **Edit** - для редактирования файлов при необходимости (редко)
14. **ExitPlanMode** - при работе в режиме планирования (если применимо)

## Инструменты для работника

Работник (уровень 2) отвечает за выполнение конкретных задач, специфичных для области, на основе файлов плана, генерацию отчетов и возврат управления в основную сессию.

### Обязательные инструменты для работника:

1. **ReadFile** - для чтения файла плана в начале работы (обязательно согласно архитектуре)
2. **ReadManyFiles** - для чтения нескольких файлов при выполнении задачи
3. **WriteFile** - для создания отчетов и файлов результатов (обязательно согласно архитектуре)
4. **Glob** - для поиска файлов при необходимости анализа
5. **Grep** - для поиска по кодовой базе при выполнении задачи
6. **ListFiles** - для перечисления файлов и структуры каталогов
7. **Shell** - для выполнения команд проверки и других операций (обязательно согласно архитектуре)
8. **Skill** - для вызова навыков валидации и отчетности (обязательно согласно архитектуре)
9. **TodoWrite** - для отслеживания прогресса выполнения задачи
10. **Edit** - для редактирования файлов при выполнении задачи
11. **WebSearch** - для поиска информации при необходимости
12. **WebFetch** - для получения информации с веб-сайтов при необходимости
13. **SaveMemory** - для сохранения информации о выполнении задачи

## Отличия в использовании инструментов

### Оркестратор:
- Использует **WriteFile** в основном для создания файлов плана
- Использует **Skill** для вызова навыков валидации планов (`validate-plan-file`)
- Не должен использовать **Task** для вызова подагентов (критическое правило)
- Основной фокус на координации и проверке результатов

### Работник:
- Использует **ReadFile** в первую очередь для чтения файла плана
- Использует **Skill** для вызова навыков валидации результатов и генерации отчетов
- Может использовать **Shell** для выполнения задач, специфичных для области
- Основной фокус на выполнении конкретных задач

## Инструменты, которые могут использоваться обоими:

- **ReadFile**, **ReadManyFiles**, **Glob**, **Grep**, **ListFiles**, **Shell**, **Skill**, **WebSearch**, **WebFetch**, **SaveMemory**, **Edit**, **TodoWrite** - могут использоваться как оркестратором, так и работником в зависимости от потребностей конкретной фазы или задачи.

Инструмент **Task** специально исключен из использования оркестратором согласно архитектуре "возврата управления", где оркестратор создает файлы плана вместо прямого вызова подагентов.
