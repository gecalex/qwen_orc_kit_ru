# Логика и алгоритм адаптации Speckit в парадигме Claude Code Orchestrator Kit

## Введение

Проект Claude Code Orchestrator Kit реализует сложную систему оркестрации, в которой основная идея заключается в использовании паттерна "оркестратор-воркер". В рамках этой архитектуры интегрированы команды Speckit, адаптированные из проекта GitHub's SpecKit, для обеспечения структурированного подхода к разработке функций.

## 1. Архитектура интеграции Speckit

### 1.1. Основные компоненты

- **Команды Speckit** - набор команд с префиксом `/speckit.*` для спецификационно-ориентированной разработки
- **Фаза 0 планирования** - критическая фаза перед реализацией, включающая анализ задач, создание агентов и назначение исполнителей
- **Шаблоны и спецификации** - структурированные документы для определения требований, архитектуры и задач

### 1.2. Команды Speckit

| Команда | Назначение |
|---------|------------|
| `/speckit.analyze` | Анализ требований и извлечение ключевых пунктов |
| `/speckit.specify` | Генерация детальных спецификаций |
| `/speckit.clarify` | Задание уточняющих вопросов |
| `/speckit.plan` | Создание плана реализации |
| `/speckit.implement` | Выполнение реализации |
| `/speckit.checklist` | Генерация контрольного списка QA |
| `/speckit.tasks` | Разбиение на выполнимые задачи |
| `/speckit.constitution` | Определение конституции проекта |
| `/speckit.taskstoissues` | Экспорт в GitHub Issues |

## 2. Алгоритм адаптации спецификаций и шаблонов

### 2.1. Интеграция спецификаций

Спецификации в рамках проекта организованы следующим образом:

1. **spec.md** - содержит пользовательские истории с приоритетами
2. **plan.md** - определяет технологический стек, архитектуру и структуру файлов
3. **data-model.md** - описывает сущности и отношения
4. **contracts/** - содержит спецификации API и требования к тестированию
5. **research.md** - документирует технические решения и ограничения

### 2.2. Адаптация шаблонов

Шаблоны адаптированы для соответствия архитектуре оркестратора:

1. **tasks-template.md** - шаблон для генерации исполняемых задач
2. **Формат задач** - строгое следование формату чек-листа с ID, метками параллельности и пользовательскими историями
3. **Фазовая организация** - задачи группируются по фазам (настройка, фундаментальная, пользовательские истории, полировка)

## 3. Подробная логика реализации фазы 0

Фаза 0 планирования является критическим элементом архитектуры проекта. Она выполняется перед реализацией и включает в себя следующие шаги:

### 3.1. Шаг 1: Анализ задач (P001)

**Цель**: Проанализировать все задачи и определить требуемые типы агентов и возможности.

**Алгоритм**:
1. Прочитать файл `tasks.md` и извлечь все задачи
2. Для каждой задачи определить:
   - Тип требуемого агента (например, fullstack-developer, security-analyst, db-expert)
   - Необходимые возможности (например, TypeScript, Supabase, React)
   - Сложность задачи (тривиальная, средняя, сложная)
3. Создать список отсутствующих типов агентов с техническими характеристиками
4. Определить, какие задачи могут выполняться параллельно

**Результат**: Документированный список требуемых агентов и их характеристик

### 3.2. Шаг 2: Создание агентов (P002)

**Цель**: Создать отсутствующих агентов с использованием meta-agent-v3.

**Алгоритм**:
1. Проверить существование всех требуемых агентов в `.claude/agents/`
2. Для каждого отсутствующего агента:
   - Запустить вызов `meta-agent-v3` с полным описанием требований
   - Передать технические характеристики и обязанности агента
   - Создать файл агента в соответствующей директории
3. Запустить N вызовов `meta-agent-v3` в одном сообщении (для параллельного выполнения)
4. После создания агентов: попросить пользователя перезапустить Claude Code
5. После перезапуска: проверить существование новых агентов перед продолжением

**Результат**: Созданные агенты, готовые к назначению задач

### 3.3. Шаг 3: Назначение исполнителей (P003)

**Цель**: Назначить исполнителей для всех задач на основе критериев.

**Алгоритм**:
1. Для каждой задачи в `tasks.md` определить подходящего исполнителя:
   - **[EXECUTOR: MAIN]** - ТОЛЬКО для тривиальных задач (исправления в 1-2 строки, простые импорты, установка одной зависимости)
   - **[EXECUTOR: existing-agent]** - ТОЛЬКО если 100% совпадение возможностей после тщательного изучения
   - **[EXECUTOR: specific-agent-name]** - Для всех остальных задач с использованием существующих или вновь созданных агентов
2. Аннотировать все задачи с `[EXECUTOR: name]` и `[SEQUENTIAL]`/`[PARALLEL-GROUP-X]`
3. Убедиться, что каждая задача имеет назначенного исполнителя
4. Проверить, что параллельные задачи не конфликтуют по ресурсам

**Результат**: Обновленный файл `tasks.md` с аннотациями исполнителей

### 3.4. Шаг 4: Решение исследований (P004)

**Цель**: Решить вопросы исследований, возникающие в процессе планирования.

**Алгоритм**:
1. Идентифицировать задачи, требующие исследований:
   - Простые исследования: решать с помощью инструментов агента (Grep, Read, WebSearch, Context7, документация Supabase)
   - Сложные исследования: создать промпт исследования в `research/`, дождаться глубокого исследования пользователя, включить результаты
2. Для простых исследований:
   - Использовать WebSearch для поиска информации
   - Использовать Context7 для получения документации по библиотекам
   - Принять решение на основе полученной информации
3. Для сложных исследований:
   - Создать файл запроса исследования в `FEATURE_DIR/research/`
   - Использовать инструмент deepresearch для глубокого анализа
   - Включить результаты исследования в `research.md`
4. Обновить задачи с учетом результатов исследований

**Результат**: Решенные вопросы исследований, документированные в `research.md`

## 4. Интеграция с архитектурой оркестратора

### 4.1. Правила атомарности

- **1 Задача = 1 Вызов агента** - никогда не давать несколько задач одному агенту за один запуск
- **Параллельное выполнение** - запуск N вызовов агента в одном сообщении (не последовательно)
- **Пример**: 3 параллельные задачи для meta-agent → 3 вызова meta-agent в одном сообщении
- **Пример**: 5 параллельных задач для fullstack → 5 вызовов fullstack в одном сообщении
- **Последовательные задачи** - 1 запуск агента, дождаться завершения, затем следующий запуск агента

### 4.2. Взаимодействие с оркестратором

- Оркестратор (уровень 1) создает файлы плана и проверяет результаты работников
- Работник (уровень 2) выполняет конкретные задачи на основе файлов плана
- В фазе 0 оркестратор анализирует задачи и создает планы для назначения исполнителей
- Назначение исполнителей происходит до начала реализации

### 4.3. Интеграция с системой контрольных точек качества

- Фаза 0 включает проверку соответствия требованиям конституции проекта
- Проверка наличия необходимых агентов и их соответствия требованиям
- Проверка полноты назначения исполнителей для всех задач

## 5. Пример реализации фазы 0

### 5.1. Входные данные

Файл `tasks.md` с задачами для реализации новой функции:

```text
# Задачи: Аутентификация через OAuth2

## Фаза 0: Планирование (Назначение исполнителей)

- [ ] P001 Проанализировать все задачи и определить требуемые типы агентов и возможности
- [ ] P002 Создать отсутствующих агентов с использованием meta-agent-v3 (запустить N вызовов в одном сообщении, по 1 на агента), затем попросить пользователя перезапустить
- [ ] P003 Назначить исполнителей для всех задач: MAIN (только для тривиальных), существующие агенты (100% совпадение), или конкретные имена агентов
- [ ] P004 Решить задачи исследования: простые (решить с помощью инструментов сейчас), сложные (создать запросы в research/)

## Фаза 1: Настройка

- [ ] T001 Создать структуру проекта в соответствии с планом реализации
- [ ] T002 Инициализировать проект Node.js с зависимостями Express и Passport.js
- [ ] T003 [P] Настроить инструменты проверки стиля кода и форматирования

## Фаза 2: Фундаментальная

- [ ] T004 Настроить схему базы данных и фреймворк миграций
- [ ] T005 [P] Реализовать фреймворк аутентификации/авторизации
- [ ] T006 [P] Настроить маршрутизацию API и структуру middleware
```

### 5.2. Выполнение фазы 0

#### Шаг P001: Анализ задач
- Задача T002 требует агента с возможностями Node.js, Express, Passport.js
- Задача T004 требует агента с возможностями баз данных (PostgreSQL/MySQL)
- Задача T005 требует агента с опытом в аутентификации и OAuth2

#### Шаг P002: Создание агентов
- Создать агента `oauth2-specialist` с опытом в OAuth2 и Passport.js
- Создать агента `db-auth-expert` с опытом в аутентификации и базах данных

#### Шаг P003: Назначение исполнителей
- T001: [EXECUTOR: setup-agent] [SEQUENTIAL]
- T002: [EXECUTOR: oauth2-specialist] [SEQUENTIAL]
- T003: [EXECUTOR: code-quality-agent] [PARALLEL-GROUP-1]
- T004: [EXECUTOR: db-auth-expert] [SEQUENTIAL]
- T005: [EXECUTOR: oauth2-specialist] [PARALLEL-GROUP-2]
- T006: [EXECUTOR: backend-architect] [PARALLEL-GROUP-2]

#### Шаг P004: Решение исследований
- Простое исследование: сравнение провайдеров OAuth2 (Google, Facebook, GitHub)
- Результат: выбрать Google и GitHub как основные провайдеры

## 6. Заключение

Фаза 0 планирования в системе Claude Code Orchestrator Kit представляет собой критическую стадию подготовки к реализации. Она обеспечивает:

1. **Анализ требований** - определение необходимых агентов и их возможностей
2. **Создание специализированных агентов** - генерация агентов с требуемыми характеристиками
3. **Назначение исполнителей** - распределение задач между подходящими агентами
4. **Решение исследований** - разрешение неопределенности перед реализацией

Эта фаза интегрирована с общей архитектурой оркестратора, обеспечивая эффективное делегирование задач специализированным подагентам и соблюдение принципов атомарности и параллельности выполнения задач.
