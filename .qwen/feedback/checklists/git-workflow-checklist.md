# Git Workflow Checklist

Чек-лист для проверки соблюдения git workflow в проекте.

---

## 1. Ветвление

- [ ] **1.1** Работа ведется в feature-ветках, а не напрямую в main/master
- [ ] **1.2** Названия веток следуют конвенции: `feature/xxx`, `bugfix/xxx`, `hotfix/xxx`
- [ ] **1.3** Ветка dev/develop используется для интеграции перед merge в main
- [ ] **1.4** Отсутствуют stale ветки (старше 14 дней без изменений)
- [ ] **1.5** Ветки удаляются после merge

## 2. Коммиты

- [ ] **2.1** Сообщения коммитов следуют Conventional Commits (feat:, fix:, docs:, etc.)
- [ ] **2.2** Коммиты атомарные (один коммит = одна логическая change)
- [ ] **2.3** Размер коммита не превышает 10 измененных файлов
- [ ] **2.4** Отсутствуют коммиты с сообщениями типа "fix", "update", "changes"
- [ ] **2.5** Коммиты подписаны (git commit -s) при необходимости

## 3. Code Review

- [ ] **3.1** Все изменения проходят через Pull/Merge Request
- [ ] **3.2** PR имеют описание изменений и ссылку на задачу
- [ ] **3.3** Минимум один approve требуется для merge
- [ ] **3.4** CI проверки проходят успешно перед merge
- [ ] **3.5** Отсутствуют прямые push в защищенные ветки

## 4. Теги и версии

- [ ] **4.1** Теги версий следуют SemVer (vMAJOR.MINOR.PATCH)
- [ ] **4.2** Теги аннотированные (git tag -a)
- [ ] **4.3** Каждый релиз имеет тег
- [ ] **4.4** CHANGELOG обновляется при создании тега
- [ ] **4.5** Не более 20 коммитов между тегами

## 5. Чистота репозитория

- [ ] **5.1** Отсутствуют незакоммиченные изменения в рабочей директории
- [ ] **5.2** .gitignore настроен корректно
- [ ] **5.3** Отсутствуют закоммиченные секреты/ключи
- [ ] **5.4** Отсутствуют закоммиченные бинарные файлы (кроме необходимых)
- [ ] **5.5** История git чистая (нет чувствительных данных)

## 6. Pre-commit хуки

- [ ] **6.1** Pre-commit хуки настроены
- [ ] **6.2** Линтеры запускаются перед коммитом
- [ ] **6.3** Форматтеры применяются автоматически
- [ ] **6.4** Тесты запускаются при необходимости
- [ ] **6.5** Хуки задокументированы для новых разработчиков

## 7. Интеграция

- [ ] **7.1** Feature ветки регулярно обновляются из dev
- [ ] **7.2** Конфликты разрешаются до создания PR
- [ ] **7.3** Merge commit или squash используются консистентно
- [ ] **7.4** История main линейная (при использовании rebase)
- [ ] **7.5** Отсутствуют merge коммиты с конфликтами

## 8. Безопасность

- [ ] **8.1** Защищенные ветки настроены (main, master, dev)
- [ ] **8.2** Force push запрещен в защищенные ветки
- [ ] **8.3** Delete protected для основных веток
- [ ] **8.4** Required status checks настроены
- [ ] **8.5** CODEOWNERS определен для критических файлов

## 9. Документация workflow

- [ ] **9.1** CONTRIBUTING.md описывает git workflow
- [ ] **9.2** README.md содержит информацию о ветках
- [ ] **9.3** Шаблон PR настроен
- [ ] **9.4** Шаблон issue настроен
- [ ] **9.5** Workflow документирован для команды

## 10. Автоматизация

- [ ] **10.1** CI/CD пайплайн настроен
- [ ] **10.2** Автоматические тесты при PR
- [ ] **10.3** Автоматический deploy при merge в main
- [ ] **10.4** Автоматическое создание релизов
- [ ] **10.5** Уведомления о статусе сборок

---

## Использование

Проверяйте этот чек-лист регулярно (рекомендуется еженедельно) или перед релизом.

**Автоматическая проверка:** Запустите `git-workflow-analyzer.sh` для автоматической проверки большинства пунктов.

**Оценка:**
- ✅ 90-100%: Отлично
- ⚠️ 70-89%: Требуется улучшение
- ❌ <70%: Критические проблемы
