## 1. Принципы разработки

### Принцип 1: Specification-Driven Development

**Название:** SDD  
**Описание:** Разработка ведется на основе спецификаций. Спецификация является первичным артефактом, из которого генерируется код.

**Непреложные правила:**
- Каждая функция должна иметь спецификацию перед реализацией
- Спецификация должна быть проверена (Gate 5: Pre-Implementation Gate)
- Изменения в коде должны соответствовать спецификации

**Мотивировка:** Обеспечивает согласованность, тестируемость и поддерживаемость кода.

---

### Принцип 2: Library-First Approach

**Название:** LFA  
**Описание:** Перед написанием нового кода разработчики должны искать и оценивать существующие решения.

**Непреложные правила:**
- Всегда искать готовые решения перед реализацией
- Оценивать решения по критериям: функциональность, лицензирование, поддержка, безопасность
- Документировать принятое решение и причины выбора

**Мотивировка:** Снижает дублирование, ускоряет разработку, повышает качество.

---

### Принцип 3: Quality Gates

**Название:** QG  
**Описание:** Все изменения должны проходить контрольные точки качества.

**Непреложные правила:**
- Gate 1: Pre-Execution Gate — проверка корректности задачи
- Gate 2: Post-Execution Gate — верификация результата
- Gate 3: Pre-Commit Gate — проверки перед коммитом
- Gate 4: Pre-Merge Gate — интеграционные проверки
- Gate 5: Pre-Implementation Gate — проверка спецификаций

**Мотивировка:** Обеспечивает соблюдение стандартов и выявляет проблемы на ранних стадиях.

---

### Принцип 4: Git Workflow

**Название:** GWF  
**Описание:** Строгое следование Git workflow с использованием Conventional Commits.

**Непреложные правила:**
- Ветки: main (production), develop (development), feature/*, bugfix/*, hotfix/*
- Коммиты только после подтверждения пользователя
- Мерж только через Pull Request с проверками
- Теги версий только на main

**Мотивировка:** Поддерживаемая кодовая база, четкая история изменений.

---

### Принцип 5: Минимальные Привилегии

**Название:** MLP  
**Описание:** Агентам предоставляется только минимально необходимый набор инструментов.

**Непреложные правила:**
- Каждый агент получает только необходимые инструменты
- Запрещен доступ к чувствительным данным без необходимости
- Логирование всех операций

**Мотивировка:** Безопасность и предотвращение несанкционированных изменений.

## 3. Стандарты Кода

### 3.1. Conventional Commits

**Формат:**
```
<type>: <краткое описание>

[опционально: подробное описание]

[опционально: закрываемые issues]
```

**Типы:**
- `feat` — новая функция
- `fix` — исправление бага
- `docs` — документация
- `style` — форматирование
- `refactor` — рефакторинг
- `test` — тесты
- `chore` — служебные

---

### 3.2. Документирование

**Правила:**
- Комментарии только для сложной логики
- Фокус на ПОЧЕМУ, а не на ЧТО
- Высокоценные комментарии для API
- Документация на русском языке

---

## 4. Требования к Безопасности

### 4.1. Защита Данных

**Правила:**
- Никогда не коммитить API ключи, пароли, токены
- Использовать .gitignore и secrets managers
- Валидация всех входных данных
- Логирование всех операций

---

### 4.2. Контроль Доступа

**Правила:**
- Принцип минимальных привилегий
- Разделение обязанностей
- Аудит всех изменений

---

## 5. Процессы Code Review

### 5.1. Pre-Commit Gate

**Проверки:**
- Линтеры проходят
- Тесты проходят
- Типы корректны
- CHANGELOG обновлен

---

### 5.2. Pre-Merge Gate

**Проверки:**
- Интеграционные тесты проходят
- Нет конфликтов
- Покрытие кода на уровне требований
- Документация обновлена

---

## 7. Процедура Внесения Поправок

### 7.1. Типы Поправок

**MAJOR:** Обратно несовместимые удаления или переопределения  
**MINOR:** Добавлен новый принцип/раздел или существенно расширено руководство  
**PATCH:** Уточнения, переформулировки, исправления опечаток

---

### 7.2. Процесс Внесения Поправок

1. Создать issue с описанием поправки
2. Обсуждение (минимум 3 участника)
3. Голосование (квалифицированное большинство 2/3)
4. Реализация поправки
5. Обновление версии
6. Документирование в CHANGELOG.md

---

## 8. Ожидания Проверки Соответствия

### 8.1. Автоматические Проверки
