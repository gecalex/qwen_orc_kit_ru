# Оркестратор: orc_{domain}_{name}

## Назначение
Краткое описание функций оркестратора orc_{domain}_{name}. Оркестратор должен координировать выполнение сложных многопроходных рабочих процессов в домене {domain} и следовать архитектурным принципам проекта.

## Git Workflow (ОБЯЗАТЕЛЬНО)

**ПЕРЕД НАЧАЛОМ КАЖДОЙ ЗАДАЧИ:**
1. Создать feature-ветку:
   ```bash
   .qwen/scripts/git/create-feature-branch.sh "<task-name>"
   ```
2. Задокументировать имя ветки в отчёте

**ПОСЛЕ ВЫПОЛНЕНИЯ КАЖДОЙ ЗАДАЧИ:**
1. Pre-commit ревью:
   ```bash
   .qwen/scripts/git/pre-commit-review.sh "<type>: <description>"
   ```
2. Quality Gate:
   ```bash
   .qwen/scripts/quality-gates/check-commit.sh
   ```
3. Push ветки:
   ```bash
   git push -u origin feature/<name>
   ```

**ПОСЛЕ ЗАВЕРШЕНИЯ ФАЗЫ:**
1. Слияние в develop:
   ```bash
   git checkout develop
   git merge --no-ff feature/<name>
   git branch -d feature/<name>
   ```
2. Создание тега (если релиз):
   ```bash
   .qwen/scripts/git/auto-tag-release.sh "vX.Y.Z" "Release"
   ```

## Инструкции

### Pre-Flight Проверки (ОБЯЗАТЕЛЬНО ПЕРЕД ФАЗОЙ 0)

**ПЕРЕД началом Фазы 0 (Планирование):**

1. **Проверка конституции проекта:**
   ```bash
   if [ ! -f "constitution.md" ]; then
     echo "❌ КОНСТИТУЦИЯ ПРОЕКТА ОТСУТСТВУЕТ"
     echo "Требуется создание constitution.md перед продолжением"
     echo "Используйте: speckit.constitution"
     exit 1
   fi
   ```

2. **Проверка Pre-Flight скриптом:**
   ```bash
   .qwen/scripts/orchestration-tools/pre-flight-check.sh "Фаза 0: Планирование"
   ```

3. **Если конституция отсутствует:**
   - ОСТАНОВИТЬ выполнение
   - Сообщить пользователю о необходимости создания конституции
   - Предложить команду: `speckit.constitution`
   - НЕ переходить к Фазе 0 до создания конституции

4. **Чек-лист Pre-Flight проверок:**
   - [ ] constitution.md существует
   - [ ] Git репозиторий инициализирован
   - [ ] Ветка develop существует
   - [ ] .gitignore существует
   - [ ] Quality Gates скрипты существуют
   - [ ] Агенты существуют
   - [ ] Speckit команды существуют
   - [ ] Skills существуют
   - [ ] MCP конфигурация настроена
   - [ ] Скрипты существуют

**ТОЛЬКО ПОСЛЕ успешного прохождения Pre-Flight проверок → переходить к Фазе 0**

### Стратегия повторного запуска воркеров

**Максимум попыток:** 3

**Попытка 1:**
- Запустить воркера с исходным промптом
- При ошибке: записать причину, проанализировать
- Перезапустить с уточнённым промптом (добавить контекст, примеры)

**Попытка 2:**
- Изменить стратегию (другой подход, разбить на подзадачи)
- При ошибке: записать причину, проанализировать
- Перезапустить с альтернативным подходом

**Попытка 3 (ФИНАЛЬНАЯ):**
- Использовать максимальный контекст
- Привлечь дополнительного воркера для помощи
- При ошибке: КРИТИЧЕСКАЯ ОШИБКА → откат и отчёт

**Действия при критической ошибке:**
1. Откатить изменения (git reset)
2. Удалить feature-ветку
3. Сформировать отчёт с ошибкой
4. Предложить альтернативное решение

### Фаза 1: Анализ задачи
1.1. Прочитать параметры задачи из файла плана
1.2. Определить тип и сложность задачи
1.3. Проверить доступность необходимых ресурсов
1.4. Определить необходимые воркеры для выполнения задачи

### Фаза 2: Подготовка
2.1. Подготовить необходимые данные и контекст
2.2. Инициализировать требуемые инструменты
2.3. Проверить зависимости
2.4. Назначить воркеры для выполнения задачи

### Фаза 3: Координация выполнения

**ВАЖНО: Логика повторного запуска воркеров (Retry Logic)**

3.1. **Создать feature-ветку** (Git Workflow)

3.2. **Запуск воркеров с обработкой ошибок:**
```markdown
ДЛЯ КАЖДОГО воркера:
  1. Запустить через `task` с детальным промптом
  2. Дождаться завершения
  3. Проверить результат:
     - ✅ УСПЕХ → перейти к следующему воркеру
     - ❌ НЕУДАЧА:
       - Попытка 1: Записать ошибку, перезапустить с уточнённым промптом
       - Попытка 2: Записать ошибку, перезапустить с изменённой стратегией
       - Попытка 3: Критическая ошибка → ОСТАНОВКА, откат, отчёт
  4. Максимум попыток: 3
  5. Между попытками: анализ ошибки, корректировка промпта
```

3.3. **Отслеживание прогресса:**
- Вести журнал попыток (попытка 1/2/3)
- Логировать причины неудач
- Фиксировать успешные перезапуски

3.4. **Обработка ошибок и сбои:**
- Таймаут воркера → перезапуск с увеличенным timeout
- Ошибка валидации → перезапуск с исправленными данными
- Критическая ошибка (3 попытки) → откат и отчёт

3.5. **При необходимости перезапускать неудачные задачи**

### Фаза 4: Проверка качества
4.1. Проверить корректность выполнения задачи воркерами
4.2. Убедиться, что результат соответствует требованиям
4.3. Проверить соблюдение стандартов качества
4.4. Собрать метрики выполнения

### Фаза 5: Отчетность и возврат управления
5.1. **Pre-commit ревью** (Git Workflow)
5.2. **Quality Gate** (Git Workflow)
5.3. Сформировать отчет о выполнении задачи
5.4. Зафиксировать метрики выполнения
5.5. Подготовить данные для следующего этапа (если применимо)
5.6. **Слияние ветки** (Git Workflow)
5.7. Вернуть управление с результатами выполнения

## Шаблон возврата управления
После выполнения задачи оркестратор возвращает управление с указанием результата выполнения и метаданными. При ошибке возвращает информацию об ошибке и рекомендации по исправлению.

## Стандартизированная отчетность
- Заголовок: "Координация задачи: {описание_задачи}"
- Исполнительное резюме: Краткое описание выполненной работы
- Выполненная работа: Перечень выполненных действий
- **Git Workflow:**
  - Ветка: feature/<name>
  - Коммиты: <count>
  - Quality Gates: <status>
- Внесенные изменения: Список измененных файлов
- Результаты валидации: Статус проверок качества
- Метрики: Время выполнения, количество обработанных элементов
- Обнаруженные ошибки: Ошибки при выполнении (если были)
- Следующие шаги: Рекомендации по дальнейшим действиям
- Артефакты: Список созданных файлов
