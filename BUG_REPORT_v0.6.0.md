## 🔴 P0 - КРИТИЧЕСКИЕ БАГИ (Блокируют релиз)

### **БАГ #12: Неполная интеграция Speckit ↔ Phase 0 ↔ Агенты**

**Критичность:** 🔴 P0  
**Компонент:** Speckit, Phase 0, Planning  
**Влияние:** Вся архитектура оркестрации нарушена

**Описание:**
Speckit создаёт спецификации, Phase 0 создаёт задачи, но назначение агентов некорректное. Оркестраторы доменов выполняют код вместо воркеров.

**Ожидаемое поведение:**
```
Speckit → Phase 0 → Агенты:
  1. speckit.specify → spec.md
  2. speckit.plan → plan.md + data-model.md + contracts/ + quickstart.md
  3. speckit.tasks → tasks.md с [agent:work_*] метками
  4. work_* агенты выполняют, orc_* координируют
```

**Фактическое поведение:**
- ❌ Отсутствуют артефакты: research.md, data-model.md, contracts/, quickstart.md
- ❌ Отсутствуют Pre-Implementation Gates
- ❌ orc_* назначаются на выполнение вместо work_*
- ❌ Нет порядка: contracts → тесты → код

**Причина:**
- `work_planning_executor_assigner` не различает orc_ и work_
- `phase0-analyzer.sh` только анализирует, не создаёт артефакты
- Speckit и Phase 0 работают изолированно

**Влияние:**
- Нарушается архитектура оркестрации
- Оркестраторы пишут код (БАГ #7)
- Нет трассировки требований
- TDD не работает

**Приоритет:** #1 (самый критичный)

### **БАГ #11: Подагенты не соблюдают Git Workflow**

**Критичность:** 🔴 P0  
**Компонент:** Domain Orchestrators, Git  
**Влияние:** История проекта, возможность отката

**Описание:**
Центральный оркестратор соблюдает Git Workflow, но доменные оркестраторы пишут код без коммитов, без создания feature-веток.

**Ожидаемое поведение:**
```
Оркестратор домена:
  1. Получить задачу
  2. Создать feature-ветку (.qwen/scripts/git/create-feature-branch.sh)
  3. Делегировать work_* воркеру
  4. Воркер выполняет → коммит (.qwen/scripts/git/pre-commit-review.sh)
  5. Quality Gate
  6. Следующая задача → новая ветка
```

**Фактическое поведение:**
- ❌ orc_backend_api_coordinator пишет 33 файла без коммитов
- ❌ Нет создания feature-веток
- ❌ Нет Pre-commit ревью
- ❌ Нет Quality Gates после задач

**Причина:**
- В `orc_*_coordinator.md` не прописан Git Workflow
- Нет интеграции со скриптами `.qwen/scripts/git/`
- Оркестраторы считают себя "выше" коммитов

**Влияние:**
- Невозможно откатить отдельную задачу
- Страдает история проекта
- Нарушается принцип атомарных коммитов

**Приоритет:** #2

### **БАГ #7: Оркестратор пишет код вместо делегирования**

**Критичность:** 🔴 P0  
**Компонент:** Domain Orchestrators  
**Влияние:** Архитектура, масштабируемость

**Описание:**
`orc_backend_api_coordinator` выполнил всю работу сам (33 файла, 5 моделей, 4 сервиса, 6 роутеров), не делегировав `work_backend_*` воркерам.

**Ожидаемое поведение:**
```
orc_backend_api_coordinator (координация)
  ↓ task
work_backend_api_validator (валидация)
  ↓ task
work_dev_code_analyzer (анализ)
  ↓ task
work_backend_api_developer (разработка)
```

**Фактическое поведение:**
- ❌ Оркестратор выполняет всю работу сам
- ❌ Нет делегирования через `task`
- ❌ Нарушен принцип QWEN.md раздел 1.1, 1.4

**Причина:**
- Интеграция Speckit ↔ Domain Agents не работает
- `tasks.md` назначает задачи оркестраторам, а не воркерам
- Оркестраторы не имеют инструкции делегировать воркерам

**Влияние:**
- Нарушается паттерн Command→Agent→Skill
- Теряется гранулярность выполнения
- Сложнее отслеживать прогресс
- При 43 задачах оркестратор станет узким местом

**Приоритет:** #3

### **БАГ #4: Оркестратор не перезапускает подагента после сбоя**

**Критичность:** 🔴 P0  
**Компонент:** Central Orchestrator  
**Влияние:** Надёжность, отказоустойчивость

**Описание:**
`orc_planning_task_analyzer` завершился с ошибкой. Агент сообщил об ошибке и пытался продолжить сам. Не был перезапущен подагент. Пользователю пришлось вручную просить продолжить.

**Ожидаемое поведение:**
```
1. Зафиксировать ошибку
2. Проанализировать причину сбоя
3. Перезапустить подагента с уточнённым промптом
4. При повторной ошибке → запросить у пользователя
```

**Фактическое поведение:**
- ❌ Нет retry механизма
- ❌ Нет анализа ошибок
- ❌ Пользователь вмешивается вручную

**Причина:**
- В центральном оркестраторе не реализован паттерн обработки ошибок
- Отсутствует логика retry
- Нет интеграции с `bug-orchestrator`

**Влияние:**
- Критическая проблема надёжности
- Пользователь должен контролировать процесс
- Теряется преимущество автономности

**Приоритет:** #5

### **БАГ #1: Инициализация проекта — файлы шаблона**

**Критичность:** 🔴 P0  
**Компонент:** Project Initialization  
**Влияние:** UX при старте проекта

**Описание:**
Пользователь копирует шаблон в новый проект. В проекте появляются файлы шаблона (CHANGELOG.md, README.md, QWEN.md и др.), которые нужно удалять вручную.

**Ожидаемое поведение:**
```
1. Пользователь копирует шаблон
2. Запускает .qwen/scripts/init/init-project.sh
3. Скрипт удаляет файлы шаблона
4. Создаёт пустые файлы проекта
```

**Фактическое поведение:**
- ❌ Пользователь удаляет вручную
- ❌ Нет скрипта инициализации
- ❌ CHANGELOG.md шаблона в проекте пользователя

**Причина:**
- Нет скрипта `init-project.sh`
- Нет `.qwenignore` для исключения файлов
- Шаблон включает документацию шаблона

**Влияние:**
- Плохой UX
- Путаница между документацией шаблона и проекта
- Риск удаления нужных файлов
