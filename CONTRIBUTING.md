# Руководство по участию в проекте (Contributing Guide)

## Структура ветвления

Мы используем Git Flow для управления версиями кода:

- `main` — стабильная ветка, содержащая релизный код
- `develop` — основная ветка разработки, в которую мержатся все новые фичи
- `feature/*` — ветки для разработки новых фич
- `bugfix/*` — ветки для исправления багов в develop
- `hotfix/*` — срочные исправления для production (ветки от main)

## Создание новой фичи

```bash
# Обновляем develop
git checkout develop && git pull origin develop

# Создаем ветку для новой фичи
git checkout -b feature/short-description

# Делаем изменения, коммитим
git add .
git commit -m "feat(scope): описание изменений"

# Пушим ветку
git push origin feature/short-description
```

## Исправление багов

```bash
# Обновляем develop
git checkout develop && git pull origin develop

# Создаем ветку для багфикса
git checkout -b bugfix/describe-issue

# Делаем изменения, коммитим
git add .
git commit -m "fix(scope): описание исправления"

# Пушим ветку
git push origin bugfix/describe-issue
```

## Соглашения о коммитах

Мы используем Conventional Commits:

```
<type>(<scope>): <subject>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```

Типы коммитов:
- `feat`: Новая функциональность
- `fix`: Исправление бага
- `docs`: Изменения в документации
- `style`: Изменения форматирования без изменения логики
- `refactor`: Рефакторинг без изменения поведения
- `test`: Добавление или изменение тестов
- `chore`: Обновление зависимостей, настроек и т.п.
- `perf`: Улучшения производительности
- `revert`: Откат изменений

Примеры:
```
feat(orchestrator): добавить команду task для делегирования

- Реализован парсинг JSON-конфига для команды task
- Добавлена валидация полей subagent_type и prompt
- Интегрирован вызов под-агентов через единый интерфейс

Closes #15
```

```
fix(auth): исправить утечку памяти в валидаторе JWT

Использование глобального кэша без TTL приводило к линейному росту
потребления памяти. Заменён на LRU-кэш с лимитом в 1000 записей.

Fixes #42
```

## Процесс код-ревью и мержа

1. Создайте Pull Request из вашей ветки в `develop`
2. Укажите в описании:
   - Что было сделано и зачем
   - Как тестировать изменения
   - Ссылки на связанные задачи
   - Какие контрольные точки качества были пройдены
3. Дождитесь код-ревью от других участников команды
4. После получения одобрения выполните мерж (предпочтительно Squash and Merge)

## Требования к качеству кода

- Код должен соответствовать стилю проекта (проверяется линтерами)
- Все изменения должны быть покрыты тестами (если применимо)
- Необходимо обновлять документацию при необходимости
- Все CI-проверки должны проходить успешно
- Перед коммитом должны быть пройдены контрольные точки качества (quality gates)

## Использование агентов

В проекте используются ИИ-агенты для автоматизации задач:

- `bug-fixer`: Исправление ошибок в коде
- `bug-hunter`: Поиск ошибок и уязвимостей
- `code-quality-checker`: Проверка качества кода
- `security-orchestrator`: Обнаружение и исправление уязвимостей
- `tech-translator-ru`: Перевод технической документации на русский

Для запуска агента используйте команду:

```bash
task '{
  "subagent_type": "agent-type",
  "description": "Краткое описание задачи",
  "prompt": "Подробное описание задачи с контекстом"
}'
```

## Контрольные точки качества

Перед каждым коммитом должны быть пройдены контрольные точки качества:

1. **Pre-Execution Gate**: Проверка корректности задачи перед выполнением агентом
2. **Post-Execution Gate**: Проверка результата работы агента
3. **Pre-Commit Gate**: Проверка кода перед коммитом
4. **Pre-Merge Gate**: Проверка перед мержем в develop

Для ручной проверки используйте скрипты из `scripts/`:

```bash
# Проверка перед выполнением задачи
./scripts/run_quality_gate.sh 1

# Проверка после выполнения задачи
./scripts/run_quality_gate.sh 2

# Проверка перед коммитом
./scripts/run_quality_gate.sh 3
```

## Безопасность

- Никогда не коммитьте пароли, API-ключи, токены доступа
- Не коммитьте приватные ключи или сертификаты
- Не коммитьте файлы .env с продакшен-переменными
- При работе с агентами не передавайте чувствительные данные в промптах
