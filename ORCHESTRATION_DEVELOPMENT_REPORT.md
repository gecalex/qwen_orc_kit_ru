## 1. ОБЗОР ПРОЕКТА

### 1.1. Цель проекта

Qwen Code Orchestrator Kit — это система для оркестрации задач разработки с использованием ИИ-агентов. Проект предназначен для использования как шаблон для новых проектов.

### 1.2. Текущее состояние (на 18.03.2026)

**Версия:** v0.3.0  
**Ветки:**
- `main` — релизная ветка (172 файла, ~2.2 MB)
- `develop` — ветка разработки (200+ файлов, ~3.5 MB)

**Компоненты:**
- 25 агентов (7 оркестраторов, 18 воркеров)
- 31 навык (skills)
- 22 команды (commands)
- ~20 скриптов автоматизации
- 4 Quality Gates

## 5. ОШИБКИ В РЕАЛИЗАЦИИ ОРКЕСТРАЦИИ

### 5.1. Ошибка: Отсутствие итеративной логики

**Описание:**
Оркестраторы не реализуют итеративную логику с контролем числа итераций.

**Проблема:**
```markdown
# В текущей документации:
5. Финальная фаза: Резюме планирования
   - Собирайте все отчеты
   - Рассчитывайте метрики
```

**Недостатки:**
- Нет ограничения на число итераций
- Нет обработки бесконечных циклов
- Нет явного шаблона возвращения управления

**Решение:**
```markdown
## Итеративная логика

1. Установите максимальное число итераций (MAX_ITERATIONS=5)
2. Инициализируйте счетчик (iteration=0)
3. Цикл:
   - increment iteration
   - если iteration > MAX_ITERATIONS:
     - log_error("Превышено максимальное число итераций")
     - сохранить состояние
     - вернуть управление с ошибкой
   - выполнить задачу
   - если успех: выйти из цикла
   - если ошибка: retry (max 3 раза)
4. Вернуть управление
```

```markdown
   - Выведите: "✅ Фаза N завершена"

   - Используйте навык save-session-context
   - Укажите: phase, status, artifacts

   - Удалите .tmp/current/
   - Архивируйте в .tmp/archive/{timestamp}/

5. Выйдите из системы (exit)
```

```markdown
## Выполненная работа

- Задача 2: Статус

## Результаты проверки
- Quality Gate: ПРОЙДЕН/НЕ ПРОЙДЕН

## Метрики
- Продолжительность: {time}
- Задачи: {count}

## Обнаруженные ошибки
- Ошибка 1: Описание

## Артефакты
- Файл 1: {path}

```

### 5.4. Ошибка: Неправильная работа с Git

**Описание:**
Агенты создают коммиты без подтверждения пользователя.

**Проблема:**
Из GIT_WORKFLOW.md:
```markdown
⚠️ ПРАВИЛО: Коммит только после тестирования пользователем

ЗАПРЕЩЕНО: Создавать коммит без подтверждения пользователя
```

**Выявленные нарушения:**
1. ❌ Коммиты создавались без явного подтверждения
2. ❌ Мерж выполнялся без запроса
3. ❌ Удаление веток без объяснения причины

**Решение:**
```markdown
## Алгоритм коммита

1. Агент вносит изменения
2. Агент показывает изменения (git diff)
3. **Агент запрашивает подтверждение**: "Создать коммит?"
4. Пользователь отвечает: "да", "коммить", "подтверждаю"
5. **Только после подтверждения** агент создаёт коммит

## Пример запроса

Агент: Изменения готовы. Создать коммит с сообщением:
       "docs: добавлен отчёт"
       
       Файлы:
       - REPORT.md (+500 строк)
       
       Создать коммит?
```

## 6. ПРОБЛЕМЫ С ДОКУМЕНТАЦИЕЙ

### 6.1. Проблема: Дублирование документации

**Описание:**
Документация дублируется в разных местах.

**Примеры:**
- `GIT_WORKFLOW.md` в корне
- `.qwen/docs/architecture/GIT_WORKFLOW.md`
- `QWEN.md` в корне
- `.qwen/QWEN.md` (интегрирован)

**Последствия:**
- Путаница для разработчиков
- Рассинхронизация версий
- Большой размер репозитория

**Решение:**
```
Для main:
- GIT_WORKFLOW.md только в .qwen/docs/architecture/

Для develop:
- GIT_WORKFLOW.md в корне (для разработки)
- GIT_WORKFLOW.md в .qwen/docs/architecture/ (для релиза)
```

---

### 6.2. Проблема: Устаревшая документация

**Описание:**
Документация не обновляется после изменений.

**Примеры:**
- CHANGELOG.md не обновлялся с января 2026
- specification-driven-development.md не упоминает .qwen/specify/
- planning-phase.md использует старые пути

**Последствия:**
- Разработчики используют устаревшие пути
- Невозможность интеграции новых компонентов

**Решение:**
- Регулярное обновление CHANGELOG (автоматизировано)
- Проверка ссылок при каждом изменении
- Интеграция проверки документации в Quality Gates

---
