# Адаптивная логика оркестратора проекта

## Обзор

Центральный оркестратор проекта адаптирует свое поведение в зависимости от состояния проекта. Это позволяет эффективно работать как с новыми, пустыми проектами, так и с существующими проектами, которые уже содержат код, спецификации или другие артефакты.

## Принципы адаптации

### 1. Анализ состояния
Оркестратор начинает с анализа текущего состояния проекта:
- Наличие файлов кода
- Наличие структуры проекта
- Наличие спецификаций
- Наличие конституции проекта
- Наличие файлов конфигурации

### 2. Классификация состояния
На основе анализа проект классифицируется как:
- **Пустой проект**: только системные файлы (git, readme, gitignore)
- **Существующий код, но без спецификаций**: есть код, но нет спецификаций
- **Частичные спецификации**: есть спецификации, но не полные
- **Полные спецификации**: есть все необходимые артефакты Speckit

### 3. Адаптация поведения
Оркестратор адаптирует свое поведение в зависимости от классификации:
- Для пустых проектов: фокус на создание конституции и спецификаций
- Для проектов с кодом, но без спецификаций: анализ кода и реверс-инжиниринг спецификаций
- Для проектов с частичными спецификациями: доработка и проверка соответствия
- Для проектов с полными спецификациями: стандартный процесс Speckit

## Процесс инициализации

### 1. Определение состояния проекта
Оркестратор запускает скрипт `scripts/orchestration-tools/analyze-project-state.sh`, который:
- Подсчитывает файлы кода
- Проверяет наличие директорий проекта
- Проверяет наличие файлов конфигурации
- Проверяет наличие спецификаций и конституции
- Возвращает код состояния (10-40) для дальнейшей обработки

### 2. Адаптация поведения на основе состояния

#### Состояние 1: Пустой проект (код 10)
Когда проект пустой, оркестратор:
- Предлагает создать конституцию проекта
- Рекомендует определить основные принципы и стандарты
- Предлагает создать первую спецификацию
- Обеспечивает пошаговое руководство пользователя

#### Состояние 2: Существующий код, но без спецификаций (код 20)
Когда проект содержит код, но не содержит спецификаций, оркестратор:
- Проводит анализ существующего кода
- Определяет технологический стек
- Предлагает создать реверс-инжиниринг спецификаций
- Использует агентов для анализа кода и выявления компонентов

#### Состояние 3: Частичные спецификации (код 30)
Когда проект имеет частичные спецификации, оркестратор:
- Определяет, какие спецификации существуют
- Предлагает доработать и заполнить пробелы
- Проверяет соответствие реализации существующим спецификациям
- Рекомендует пути улучшения

#### Состояние 4: Полные спецификации (код 40)
Когда проект имеет полные спецификации, оркестратор:
- Проверяет соответствие реализации спецификациям
- Продолжает стандартный процесс Speckit
- Использует контрольные точки качества
- Обеспечивает соблюдение архитектурных принципов

## Интеграция с процессом Speckit

### 1. Автоматическое определение этапа
Оркестратор автоматически определяет, на каком этапе процесса Speckit находится проект:
- Если нет конституции → начинает с `speckit.constitution`
- Если есть конституция, но нет спецификаций → переходит к `speckit.specify`
- Если есть спецификации, но нет плана → переходит к `speckit.plan`
- Если есть план, но нет задач → переходит к `speckit.tasks`
- Если есть задачи → переходит к `speckit.implement`

### 2. Проверка соответствия
Оркестратор использует агента `specification-compliance-checker` для проверки соответствия реализации спецификациям:
- Сравнивает реализованный код с требованиями спецификаций
- Проверяет покрытие тестами функциональных требований
- Идентифицирует расхождения между спецификацией и реализацией

## Безопасность и контрольные точки

### 1. Принцип минимальных привилегий
Для каждого агента оркестратор обеспечивает предоставление только минимально необходимых инструментов:
- `read_file`, `write_file` для безопасной работы с файлами
- `web_search` только для получения информации из доверенных источников
- Ограничение на использование `run_shell_command` и других потенциально опасных инструментов

### 2. Контрольные точки качества
Оркестратор запускает соответствующие контрольные точки в зависимости от состояния:
- Gate 5: Проверка качества спецификаций перед реализацией
- Gate 1-4: Стандартные проверки в процессе разработки
- Автоматическая проверка соответствия реализации спецификациям

## Примеры использования

### Пример 1: Пустой проект
```
Пользователь: "Начать новый проект"
Оркестратор: 
1. Анализирует проект → определяет как пустой (код 10)
2. Предлагает создать конституцию: "speckit.constitution"
3. После создания конституции предлагает первую спецификацию: "speckit.specify 'описание функции'"
```

### Пример 2: Существующий проект с кодом
```
Пользователь: "Работать с существующим проектом"
Оркестратор:
1. Анализирует проект → определяет как "existing_code_no_specs" (код 20)
2. Проводит анализ кода с помощью bug-hunter
3. Предлагает создать реверс-инжиниринг спецификаций
4. Направляет к процессу Speckit с учетом существующего кода
```

### Пример 3: Проект с частичными спецификациями
```
Пользователь: "Продолжить работу над проектом"
Оркестратор:
1. Анализирует проект → определяет как "partial_specification" (код 30)
2. Проверяет существующие спецификации на полноту
3. Предлагает доработать недостающие разделы
4. Проверяет соответствие реализации спецификациям
```

## Интеграция с системой агентов

### 1. Автоматическое назначение агентов
Оркестратор использует скрипт `scripts/specification-tools/assign-agents-to-tasks.sh` для автоматического назначения агентов задачам:
- Анализирует описание задачи
- Определяет наиболее подходящего агента
- Добавляет метку `[agent:AGENT_TYPE]` или `[futures:AGENT_TYPE]`

### 2. Поддержка TDD
Оркестратор интегрирован с процессом TDD:
- Использует скрипт `scripts/specification-tools/generate-tests-from-spec.sh` для генерации тестов
- Обеспечивает создание тестов до реализации
- Проверяет, что тесты сначала падают (TDD принцип)

## Сценарии использования

### Сценарий 1: Новый проект
1. Оркестратор определяет проект как пустой
2. Предлагает создать конституцию
3. После создания конституции предлагает первую спецификацию
4. Продолжает стандартный процесс Speckit

### Сценарий 2: Существующий проект
1. Оркестратор анализирует существующий код
2. Проверяет наличие спецификаций
3. Предлагает соответствующий путь интеграции с процессом Speckit
4. Обеспечивает плавный переход к разработке на основе спецификаций

### Сценарий 3: Проект с частичной автоматизацией
1. Оркестратор определяет текущий уровень автоматизации
2. Предлагает улучшения и дополнения
3. Обеспечивает постепенную интеграцию недостающих компонентов

## Лучшие практики

### 1. Для пустых проектов
- Начинайте с создания конституции проекта
- Определите основные принципы и стандарты
- Создайте первую спецификацию с четким описанием
- Используйте шаблоны для ускорения процесса

### 2. Для существующих проектов
- Проведите анализ существующего кода перед созданием спецификаций
- Создайте "базовую линию" спецификаций на основе существующего кода
- Постепенно интегрируйте процесс Speckit
- Обеспечьте минимальное влияние на существующую разработку

### 3. Для проектов с частичными спецификациями
- Доработайте существующие спецификации до полноты
- Проверьте соответствие реализации спецификациям
- Заполните пробелы в спецификациях
- Обновите артефакты в соответствии с текущим состоянием

### 4. Для проектов с полными спецификациями
- Следуйте стандартному процессу Speckit
- Регулярно проверяйте соответствие реализации спецификациям
- Используйте контрольные точки качества
- Поддерживайте актуальность спецификаций
