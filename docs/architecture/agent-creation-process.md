# Процесс создания агентов

## Обзор

Этот документ описывает процесс создания новых агентов в системе Qwen Code CLI с использованием автоматизированных инструментов и шаблонов.

## Назначение

Процесс создания агентов предназначен для:
- Упрощения создания новых агентов
- Обеспечения согласованности структуры агентов
- Автоматизации рутинных задач при создании агентов
- Поддержания стандартов именования и структуры

## Компоненты процесса создания агентов

### 1. Meta-Agent Creator (`work_meta_agent_creator`)

Центральный компонент для автоматического создания новых агентов по шаблонам. Может создавать оркестраторы, воркеры и простые агенты.

### 2. Скрипт создания агентов (`scripts/agent-creation/create-agent.sh`)

Командный скрипт для быстрого создания агентов с использованием стандартных шаблонов.

### 3. Шаблоны агентов (`templates/agent-generation/`)

Шаблоны для различных типов агентов, обеспечивающие согласованную структуру.

### 4. Навык проверки уникальности имени (`check-agent-name-uniqueness`)

Проверяет, что имя агента уникально в системе перед созданием.

## Процесс создания агента

### Вариант 1: Использование скрипта

1. Определите тип агента (оркестратор `orc` или воркер `work`)
2. Определите домен и имя агента (например, `dev_task_coordinator`)
3. Вызовите скрипт:
   ```bash
   ./scripts/agent-creation/create-agent.sh orc dev_task_coordinator
   ```

### Вариант 2: Использование meta-agent

1. Определите требования к новому агенту
2. Вызовите meta-agent с описанием:
   ```bash
   task '{
     "subagent_type": "work_meta_agent_creator",
     "description": "Create new agent",
     "prompt": "Create orchestrator for coordinating deployment tasks in the dev domain. Name: orc_dev_deployment_coordinator"
   }'
   ```

## Стандарты именования

Агенты именуются по следующему шаблону:
```
{тип}_{домен}_{имя}.md
```

Где:
- `{тип}` - `orc` для оркестраторов, `work` для воркеров
- `{домен}` - область применения (dev, frontend, backend, testing, research и т.д.)
- `{имя}` - конкретное имя агента в формате kebab-case

Примеры:
- `orc_dev_task_coordinator.md` - оркестратор для координации задач разработки
- `work_backend_api_validator.md` - воркер для валидации API в бэкенде

## Структура агента

Все агенты должны содержать:

### YAML заголовок
```yaml
---
name: {имя_агента}
description: {описание_назначения}
model: sonnet
color: {цвет_домена}
---
```

### Обязательные разделы
1. **Purpose** - описание назначения агента
2. **MCP Server Usage** - использование MCP серверов
3. **Instructions** - пошаговые инструкции для выполнения задач
4. **Return of Control Pattern** - описание возврата управления
5. **Standardized Reporting** - формат отчетов
6. **Skills Integration** - интеграция с навыками

## Проверка качества

После создания агента:
1. Проверьте соответствие стандартам именования
2. Убедитесь, что все обязательные разделы присутствуют
3. Проверьте корректность YAML заголовка
4. Убедитесь, что агент соответствует своей области применения
5. Проверьте интеграцию с навыками и MCP серверами

## Интеграция с MCP

Агенты должны использовать MCP серверы при необходимости:
- Для получения документации по библиотекам используйте `mcp__context7__*`
- Для работы с базами данных используйте соответствующие MCP серверы
- Для выполнения специфичных задач используйте рекомендованные MCP серверы

## Стандартизированные отчеты

Все агенты должны генерировать отчеты в стандартизированном формате, включающем:
- Заголовок с метаданными
- Исполнительное резюме
- Выполненная работа
- Внесенные изменения
- Результаты валидации
- Метрики выполнения
- Обнаруженные ошибки
- Следующие шаги
- Артефакты выполнения
