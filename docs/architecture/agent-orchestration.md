# Расширенные возможности оркестрации

## Обзор

Этот документ описывает расширенные возможности оркестрации, включая поддержку итеративной логики и возвращения управления, реализованные в системе Qwen Code CLI.

## Назначение

Расширенные возможности оркестрации позволяют:
- Управлять сложными многопроходными рабочими процессами
- Реализовывать итеративную логику для постепенного улучшения результатов
- Обеспечивать возврат управления между агентами для координации
- Поддерживать согласованность состояния между сессиями

## Компоненты расширенной оркестрации

### 1. Паттерн возвращения управления (Return of Control Pattern)

Ключевой паттерн, позволяющий агентам возвращать управление основной сессии после выполнения своих задач:

1. Генерация стандартизированного отчета с использованием навыка `generate-report-header`
2. Сохранение отчета в назначенное место
3. Сигнализирование о завершении через корректный выход
4. Возобновление основной сессии для продолжения работы

### 2. Итеративная логика

Поддержка многопроходных процессов с возможностью повторного выполнения до достижения цели:

- **Предварительная проверка итерации**: Проверка счетчика итераций и достижения цели
- **Цикл выполнения фазы**: Выполнение рабочих процессов с контролем качества
- **Поститерационная проверка**: Оценка прогресса и определение необходимости повторения

### 3. Контрольные точки качества

Интеграция проверок качества на каждом этапе для обеспечения соответствия стандартам:

- Блокирующие проверки (останавливают выполнение при критических ошибках)
- Неблокирующие проверки (предупреждают о проблемах)
- Автоматическая интеграция с рабочими процессами

### 4. Управление состоянием сессии

Система сохранения и восстановления состояния сессии для обеспечения непрерывности:

- Навык `save-session-context` для сохранения состояния
- Навык `resume-session` для восстановления состояния
- Файловое хранилище состояний в `.tmp/current/sessions/`

## Архитектура оркестрации

### Оркестраторы

Оркестраторы реализуют следующую структуру выполнения:

1. **Фаза 0: Планирование** (если не завершена)
   - Анализ задач и определение типов агентов
   - Назначение исполнителей
   - Создание плана выполнения

2. **Фазы 1-N: Выполнение**
   - Обновление прогресса
   - Создание файлов плана
   - Включение рекомендаций MCP
   - Проверка плана
   - Сигнализирование готовности и возврат управления

3. **Контрольные точки качества N: Проверка фазы N**
   - Проверка существования отчета воркера
   - Запуск контрольных точек качества
   - Обработка результатов (остановка при блокирующих ошибках или продолжение при успехе)

4. **Финальная фаза: Резюме**
   - Сбор всех отчетов
   - Расчет метрик
   - Генерация итогового резюме
   - Архивирование выполнения
   - Очистка временных файлов

### Итеративная логика

Для оркестраторов с итеративными рабочими процессами:

1. **Предитерационная проверка**
   - Проверка счетчика итераций < максимума (по умолчанию 3)
   - Если максимум достигнут: генерация резюме, выход

2. **Цикл выполнения фазы**
   - Фаза 1: Открытие (воркер генерирует план)
   - Контрольная точка 1: Проверка плана
   - Фаза 2: Реализация (воркер выполняет)
   - Контрольная точка 2: Проверка реализации

3. **Поститерационная проверка**
   - Если вся работа выполнена: архивирование, выход
   - Если осталась работа: итерация++, повтор
   - Если максимум итераций: генерация частичного резюме, выход

## Интеграция с MCP

Оркестраторы включают рекомендации по использованию MCP серверов в файлы плана:

```json
{
  "phase": 2,
  "config": {
    "priority": "critical",
    "scope": ["src/", "lib/"]
  },
  "validation": {
    "required": ["type-check", "build"],
    "optional": ["tests"]
  },
  "mcpGuidance": {
    "recommended": ["mcp__context7__*"],
    "library": "security",
    "reason": "Check current security patterns before implementing fixes"
  },
  "nextAgent": "work_security_specialist"
}
```

## Навыки оркестрации

### 1. `save-session-context`

Сохраняет текущее состояние сессии для последующего возобновления:

- Используется при завершении фаз, перед окончанием сессии или при переключении контекста
- Сохраняет состояние выполнения, прогресс и контекст
- Создает файл состояния, который можно возобновить позже

### 2. `resume-session`

Возобновляет предыдущее сохраненное состояние сессии:

- Используется при продолжении работы из предыдущих сессий
- Загружает состояние выполнения, прогресс и контекст
- Обеспечивает непрерывность между сессиями

## Примеры использования

### Пример 1: Итеративное исправление ошибок

Оркестратор ошибок может использовать итеративную логику для постепенного устранения всех обнаруженных ошибок:

1. Фаза обнаружения: `bug-hunter` генерирует план с ошибками
2. Контрольная точка: Проверка полноты плана
3. Фаза исправления: `bug-fixer` устраняет ошибки
4. Контрольная точка: Проверка качества исправлений
5. Повтор до тех пор, пока не останется ошибок или не будет достигнут максимум итераций

### Пример 2: Постепенное улучшение безопасности

Оркестратор безопасности может использовать итеративную логику для постепенного устранения уязвимостей:

1. Фаза сканирования: `security-scanner` обнаруживает уязвимости
2. Контрольная точка: Проверка критических уязвимостей
3. Фаза исправления: `vulnerability-fixer` устраняет уязвимости
4. Контрольная точка: Проверка эффективности исправлений
5. Повтор до тех пор, пока не будет достигнут приемлемый уровень безопасности

## Лучшие практики

1. **Ясное определение условий завершения**: Убедитесь, что итеративные процессы имеют четкие условия завершения
2. **Ограничение числа итераций**: Установите максимальное количество итераций для предотвращения бесконечных циклов
3. **Сохранение прогресса**: Используйте навыки сохранения состояния для обеспечения восстанавливаемости
4. **Валидация на каждом этапе**: Используйте контрольные точки качества для проверки результатов
5. **Четкая сигнализация завершения**: Обеспечьте корректный возврат управления после выполнения задач
6. **Стандартизированные отчеты**: Используйте стандартизированный формат отчетов для согласованности

## Интеграция с агентами

- Оркестраторы должны использовать паттерн возвращения управления
- Воркеры должны генерировать стандартизированные отчеты
- Все агенты должны интегрироваться с системой контрольных точек качества
- Агенты должны использовать рекомендованные MCP серверы при необходимости
