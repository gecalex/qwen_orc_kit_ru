# Архитектура оркестрации агентов

## 1. Обзор системы
Диаграмма взаимодействия:
```
[Пользователь/Задача] 
    → [Qwen Code CLI с QWEN.md] 
    → [Команда `task`] 
    → [Под-агент (bug-fixer, code-quality-checker, кастомный)] 
    → [Результат] 
    → [Верификация оркестратором]
```

## 2. Роли и ответственности

### 2.1. Центральный оркестратор (главный агент)
- **Файл**: `QWEN.md` (поведенческая парадигма)
- **Ответственность**:
  - Прием и анализ высокоуровневых задач
  - Планирование последовательности выполнения
  - Сбор контекста перед делегированием
  - Формирование корректного JSON для команды `task`
  - Верификация результатов работы под-агентов
  - Управление состоянием системы (файлы в `state/`)
  - Обеспечение прохождения контрольных точек качества

### 2.2. Под-агенты (SubAgent)
- **Встроенные типы**:
  - `bug-fixer` — исправление ошибок
  - `bug-hunter` — поиск ошибок и проблем
  - `bug-orchestrator` — автоматическое обнаружение и исправление ошибок
  - `code-quality-checker` — проверка качества кода, стиля
  - `security-orchestrator` — обнаружение и исправление уязвимостей
  - `tech-translator-ru` — перевод технической документации на русский
- **Кастомные агенты**: Создаются в `.qwen/agents/` для специфических задач
- **Ответственность**:
  - Выполнение конкретной, узкой задачи в рамках своего expertise
  - Возврат структурированного результата
  - Сохранение артефактов в `workspace/` или `state/`
  - Четкое следование полученному `prompt`

## 3. Механизм делегирования через `task`

### 3.1. Формат команды `task`
```json
{
  "subagent_type": "code-quality-checker",
  "description": "Краткое описание для логов и отслеживания",
  "prompt": "ДЕТАЛЬНОЕ ТЕХНИЧЕСКОЕ ЗАДАНИЕ. Обязательно укажи: 1) Контекст проблемы или задачи 2) Конкретные файлы/директории для анализа 3) Четкие критерии успешного выполнения 4) Формат ожидаемого отчета или выходных данных"
}
```

### 3.2. Жизненный цикл задачи
1. **Планирование**: Оркестратор анализирует задачу, определяет тип под-агента и разбивает на подзадачи.
2. **Формирование промпта**: Создание детального ТЗ с полным контекстом, ссылками на файлы, примерами кода.
3. **Выполнение**: Запуск `task` с JSON-конфигом. Под-агент работает автономно.
4. **Верификация**: Оркестратор проверяет результат (код, тесты, линтинг) согласно контрольным точкам качества.
5. **Итерация**: При неудаче — анализ ошибки, создание нового, уточненного `task`.

### 3.3. Пример: процесс разработки новой функции
```
1. planner (кастомный) → Создает технический план в `state/feature_plan.md`
2. backend_dev (кастомный) → Реализует серверную часть в `workspace/src/`
3. frontend_dev (кастомный) → Реализует клиентскую часть
4. code-quality-checker (встроенный) → Проверяет код на соответствие стандартам
5. security-orchestrator (встроенный) → Проверяет безопасность реализации
6. Оркестратор → Запускает интеграционные тесты, фиксирует изменения
```

## 4. Управление состоянием и контекстом

### 4.1. Директория `state/` — единый источник истины
```
state/
├── current_phase              # Текущая фаза (planning, development, testing, review)
├── task_queue.json           # Очередь задач для выполнения [{id, type, status, agent}]
├── agent_context/            # Контекст по каждому выполнению агента
│   ├── last_bugfix.json     # Промпт и результат последнего bug-fixer
│   └── quality_report.md    # Последний отчет code-quality-checker
├── audit_reports/            # Отчеты аудиторов (security, bugs)
│   ├── security_scan_20240501.md
│   └── code_quality_module_auth.md
└── system_metrics.json      # Метрики выполнения (время, успешность)
```

### 4.2. Передача контекста между агентами
- **Через файлы состояния**: Агент A пишет отчет → оркестратор читает → включает выдержку в `prompt` для агента B.
- **Прямая передача в `prompt`**: Явное описание что было сделано ранее, с цитатами кода.
- **Важно**: Контекст должен быть релевантным и сжатым, чтобы не переполнять токены.

**Пример передачи контекста**:
```bash
# Оркестратор формирует task для bug-fixer, используя отчет bug-hunter
task '{
  "subagent_type": "bug-fixer",
  "description": "Исправить SQL-инъекцию в модуле аутентификации",
  "prompt": "Контекст из отчета bug-hunter (state/audit_reports/security_scan.md): 'Уязвимость: SQL-инъекция в функции authenticate_user, файл workspace/src/auth.py, строка 42, вызов raw_query().' Файл с уязвимостью: workspace/src/auth.py. Исправь, используя параметризованные запросы или ORM. Предоставь исправленный код и краткое описание изменений."
}'
```

## 5. Обработка ошибок и исключительных ситуаций

### 5.1. Стратегия реагирования на проблемы
- **Под-агент не выполнил задачу**: Проанализировать вывод, создать уточненный `task` с исправленным `prompt`.
- **Под-агент сломал сборку**: Откатить изменения (через `git`), делегировать `bug-fixer` на анализ и исправление.
- **Под-агент завис или превысил время**: Прервать выполнение (если возможно), запустить `bug-hunter` для диагностики системы.
- **Некорректный результат**: Запустить `code-quality-checker` для анализа проблем, затем `bug-fixer` для исправления.

### 5.2. Стратегия повторных попыток (Retry Policy)
1. **Попытка 1**: Детализировать `prompt`, добавить больше контекста и примеров.
2. **Попытка 2**: Сменить тип под-агента (например, с `bug-fixer` на `bug-orchestrator`).
3. **Попытка 3**: Разбить задачу на более мелкие подзадачи и делегировать по очереди.
4. **Попытка 4**: Выполнить задачу напрямую (если оркестратор обладает нужной экспертизой).

## 6. Интеграция с MCP-инструментами

### 6.1. Расширение возможностей через MCP
- **Конфигурация**: MCP-серверы добавляются в `~/.qwen/settings.json` или `.qwen/settings.json` проекта.
- **Примеры инструментов**:
  - `MCP PostgreSQL` — запросы к базе данных
  - `MCP Docker` — управление контейнерами
  - `MCP Browser` — автоматизация веб-взаимодействий
  - `MCP File System` — расширенная работа с файлами

### 6.2. Безопасность при работе с MCP
- **Никогда не передавать секреты** в `prompt` (пароли, ключи API).
- **Использовать переменные окружения** или secure storage MCP-сервера.
- **Валидация входных данных** при работе с внешними системами.

## 7. Шаблоны для типовых сценариев

### 7.1. Разработка новой фичи (Feature Development)
```
1. planner (кастомный) → Анализ требований, создание плана (state/feature_plan.md)
2. architect (кастомный) → Проектирование архитектуры, API дизайн
3. backend_dev (кастомный) → Реализация бэкенд-логики (workspace/src/)
4. frontend_dev (кастомный) → Реализация интерфейса (workspace/frontend/)
5. code-quality-checker → Проверка кода на стандарты и стиль
6. security-orchestrator → Проверка безопасности
7. tester (кастомный) → Написание и запуск тестов
8. Оркестратор → Интеграция, финальная проверка, коммит
```

### 7.2. Экстренный хотфикс (Hotfix)
```
1. bug-hunter → Срочная диагностика проблемы в main/production
2. hotfix-агент (кастомный) → Минимальное необходимое исправление
3. security-orchestrator → Валидация исправления на безопасность
4. Оркестратор → Непосредственный мерж в main, деплой
5. bug-fixer → Последующее полное исправление в develop
```

### 7.3. Рефакторинг модуля (Refactoring)
```
1. code-quality-checker → Аудит текущего кода, выявление проблем
2. architect → Проектирование новой структуры
3. refactor-агент (кастомный) → Постепенное переписывание кода
4. bug-hunter → Поиск регрессий после рефакторинга
5. tester → Обновление тестов под новую архитектуру
```

## 8. Мониторинг и логирование

### 8.1. Логи выполнения
- **Каждый вызов `task`** логируется в `state/agent_logs/YYYY-MM-DD/`
- **Формат лога**: timestamp, subagent_type, description, статус (success/error), хэш промпта
- **Анализ логов**: Для поиска паттернов, проблемных агентов, частых ошибок

### 8.2. Метрики эффективности

# 9. СОЗДАНИЕ И УПРАВЛЕНИЕ КАСТОМНЫМИ АГЕНТАМИ

## 9.1. Когда создавать кастомного агента?

**Создавайте кастомного агента, когда:**

- **Специфичная экспертиза**: Задача требует знаний или контекста, отсутствующих у встроенных агентов (например, работа с внутренней бизнес-логикой).
- **Часто повторяющаяся задача**: Автоматизация регулярных действий (генерация API-клиентов, миграции БД, создание шаблонов кода).
- **Интеграция со специфичным инструментом**: Работа с внутренними API, проприетарными системами или особыми форматами данных.
- **Специализированный workflow**: Последовательность действий, характерная для конкретного проекта или команды.

**Не создавайте кастомного агента для:**

- Задач, которые уже хорошо решают встроенные агенты (`bug-fixer`, `code-quality-checker`).
- Одноразовых задач, которые вряд ли повторятся.
- Простых задач, решаемых 5-10 строками кода.

## 9.2. Формат файла агента (YAML frontmatter + Markdown)

Кастомные агенты создаются как Markdown-файлы с YAML frontmatter в директории `.qwen/agents/`.

### Базовая структура:
```markdown
---
name: unique-agent-name        # Уникальное имя агента (латинские буквы, дефисы)
description: Краткое описание того, когда и как использовать этого агента
tools:                         # Необязательно: список доступных инструментов/MCP
  - tool1
  - tool2
  - tool3
variables:                     # Необязательно: переменные для шаблонизации
  project_name: "Мой проект"
  default_language: "python"
---
```

```markdown
- Создание SQL-миграций для изменения схемы БД

5. Проверяю SQL на отсутствие блокирующих операций в продакшене

## Требования к входным данным
Для эффективной работы предоставьте:
1. Четкое описание изменений (что добавляем/удаляем/изменяем)
2. Имена таблиц и колонок, которые затрагиваются
3. Ограничения по времени выполнения (для продакшена)

## Формат выходных данных

1. Файлы миграций в `workspace/migrations/YYYYMMDD_HHMMSS_description/`
2. README.md с описанием миграции и потенциальных рисков
```

## 9.4. Обнаружение агентов системой и действия при их отсутствии

### Алгоритм обнаружения агентов:
```bash
# Оркестратор проверяет доступность агентов при старте
echo "=== ПРОВЕРКА ДОСТУПНЫХ АГЕНТОВ ==="

# Проверка встроенных агентов (известные типы)
BUILTIN_AGENTS=("bug-fixer" "bug-hunter" "code-quality-checker" "security-orchestrator" "tech-translator-ru")
echo "Встроенные агенты: ${BUILTIN_AGENTS[*]}"

# Проверка кастомных агентов
if [ -d ".qwen/agents" ]; then
    CUSTOM_AGENTS=$(ls .qwen/agents/*.md 2>/dev/null | xargs -I {} basename {} .md)
    echo "Кастомные агенты: $CUSTOM_AGENTS"
    
    AGENT_COUNT=$(echo "$CUSTOM_AGENTS" | wc -w)
    if [ "$AGENT_COUNT" -eq 0 ]; then
        echo "ВНИМАНИЕ: Кастомные агенты не найдены"
    fi
else
    echo "ВНИМАНИЕ: Директория .qwen/agents не существует"
fi

# Проверка конкретного агента
check_agent() {
    AGENT_NAME=$1
    if [[ " ${BUILTIN_AGENTS[@]} " =~ " ${AGENT_NAME} " ]]; then
        echo "✓ Встроенный агент '$AGENT_NAME' доступен"
        return 0
    elif [ -f ".qwen/agents/${AGENT_NAME}.md" ]; then
        echo "✓ Кастомный агент '$AGENT_NAME' доступен"
        return 0
    else
        echo "✗ Агент '$AGENT_NAME' НЕ НАЙДЕН"
        return 1
    fi
}

# Проверка агентов, необходимых для текущего проекта
REQUIRED_AGENTS=("project-planner" "code-quality-checker" "bug-fixer")
echo ""
echo "=== ПРОВЕРКА НЕОБХОДИМЫХ АГЕНТОВ ==="
MISSING_AGENTS=()

for agent in "${REQUIRED_AGENTS[@]}"; do
    if ! check_agent "$agent"; then
        MISSING_AGENTS+=("$agent")
    fi
done
```

### Действия при отсутствии необходимых агентов:

**Если агенты не обнаружены после создания:**

```bash
if [ ${#MISSING_AGENTS[@]} -gt 0 ]; then

    echo "   !head -20 .qwen/agents/${MISSING_AGENTS[0]}.md 2>/dev/null || echo 'Не удалось прочитать файл'"
    echo ""
    echo "3. [ВАЖНО] Если файлы существуют и имеют правильный формат,"

    echo "Если проблема persists, проверьте правильность структуры файлов агентов."

else
```

## 9.6. Best practices для создания агентов

1. **Именование файлов**:
   - Используйте kebab-case: `database-migration-specialist.md`
   - Избегайте пробелов и специальных символов
   - Имя файла должно соответствовать `name` в frontmatter

2. **Структура контента**:
   - Frontmatter обязателен (между `---`)
   - В описании укажите конкретные сценарии использования
   - Включите реальные примеры промптов

3. **Тестирование агента**:
   - Создайте тестовый промпт перед использованием в продакшене
   - Проверьте, что агент возвращает ожидаемый формат
   - Убедитесь, что агент не пересекается с существующими

4. **Документирование**:
   - Для каждого агента создайте запись в `docs/agents/`
   - Включите примеры использования и ограничения
   - Обновляйте документацию при изменении агента

5. **Версионирование** (опционально):
   ```markdown
   ---
   name: my-agent
   description: Мой агент
   version: 1.2.0
   changelog:
     - "1.2.0: Добавлена поддержка Python 3.11"
     - "1.1.0: Добавлен шаблон для отчетов"
   ---
   ```

**ВАЖНО**: После создания или изменения файлов агентов в `.qwen/agents/`, Qwen Code CLI требуется перезапуск для обнаружения новых агентов. Оркестратор должен явно сообщать об этом пользователю и приостанавливать работу до перезапуска системы.
