# Builder навыков V2 - Специализированный генератор навыков

Создает готовые к производству навыки, следуя проектным шаблонам. Навыки - это многоразовые утилиты, вызываемые агентами.

## Быстрый старт

**Навыки против Агентов:**
- Навыки = Чистые функции (<100 строк, без состояния)
- Агенты = Сложные рабочие процессы (чтение файлов, координация, отслеживание состояния)

**Когда создавать Навык:**
- Утилиты парсинга/форматирования
- Логика проверки
- Рендеринг шаблонов
- Преобразования данных
- Выполнение контрольных точек качества

**Когда создавать Агента вместо этого:**
- Многошаговые рабочие процессы
- Чтение/запись файлов
- Вызовы внешних API
- Управление состоянием

## Сбор требований

Спросить пользователя:
1. **Имя навыка** (kebab-case, например, `parse-error-logs`)
2. **Назначение** (какую утилиту предоставляет?)
3. **Формат ввода** (какие данные поступают?)
4. **Формат вывода** (какие данные выходят?)
5. **Необходимые инструменты** (Bash, Read или NONE для чистой логики?)

## Категории навыков

### **Навыки парсинга** (Извлечение структурированных данных)
- Ввод: Необработанный текст/логи/вывод
- Вывод: Структурированный JSON
- Инструменты: Обычно NONE (чистый парсинг)
- Пример: `parse-error-logs`, `parse-git-status`

### **Навыки форматирования** (Генерация форматированного вывода)
- Ввод: Объект данных
- Вывод: Форматированная строка (markdown, JSON и т.д.)
- Инструменты: Обычно NONE (чистое форматирование)
- Пример: `format-markdown-table`, `format-commit-message`

### **Навыки проверки** (Проверка соответствия)
- Ввод: Путь к файлу или объект данных
- Вывод: `{valid: boolean, errors: []}`
- Инструменты: Read (если проверяются файлы)
- Пример: `validate-plan-file`, `validate-report-file`

### **Навыки выполнения** (Запуск команд с логикой)
- Ввод: Объект конфигурации
- Вывод: Результат выполнения с состоянием
- Инструменты: Bash (выполнение команд)
- Пример: `run-quality-gate`

### **Навыки преобразования** (Конвертация данных)
- Ввод: Данные в формате A
- Вывод: Данные в формате B
- Инструменты: Обычно NONE
- Пример: `extract-version`, `calculate-priority-score`

## Руководящие принципы allowed-tools

**NONE (Чистые логические навыки):**
- Парсинг (parse-error-logs)
- Форматирование (format-markdown-table)
- Вычисления (calculate-priority-score)
- Преобразования (extract-version)

**Read (Чтение файлов):**
- Проверка, требующая содержимого файла (validate-plan-file)
- Парсинг из файлов (parse-package-json)

**Bash (Выполнение команд):**
- Запуск команд проверки (run-quality-gate)
- Выполнение системных операций

**Read, Bash (Комбинированные):**
- Сложная проверка с выполнением команд
- Анализ файлов + системные проверки

**Руководящий принцип:** Предпочтение NONE. Добавлять инструменты только при строгой необходимости.

## Соглашения об именовании

**Формат:** `{глагол}-{существительное}` или `{действие}-{объект}`

**Хорошие имена:**
- `parse-error-logs` (действие-объект)
- `validate-plan-file` (действие-объект)
- `format-commit-message` (действие-объект)
- `calculate-priority-score` (действие-существительное)
- `run-quality-gate` (действие-объект)

**Плохие имена:**
- `error-parser` (существительное-первое)
- `validation` (слишком расплывчато)
- `helper` (бессмысленно)

## YAML Frontmatter

```yaml
---
name: {название-навыка}
description: {Однострочное описание}. Использовать при {триггер}. {Что он делает}.
allowed-tools: Bash, Read  # Опционально, пропустить если чистая логика
---
```

**Формула описания:**
```
{Действие} {объект} с {функцией}. Использовать при {триггер}. {Результат}.
```

**Примеры:**
- `parse-error-logs`: "Парсит логи ошибок сборки/тестирования в структурированный JSON. Использовать при обработке вывода команд проверки. Извлекает ошибки, предупреждения, пути к файлам и номера строк."
- `run-quality-gate`: "Выполняет проверку контрольной точки качества с настраиваемым блокированием. Использовать при запуске проверки типов, сборки, тестов в оркестраторах или работниках. Возвращает структурированный результат прохождения/провала."

## Стандарты ввода/вывода

### Формат ввода
Всегда документировать ожидаемую структуру ввода:

```markdown
**Ожидаемый ввод**:
```json
{
  "field1": "string",
  "field2": 123,
  "field3": ["array"]
}
```

**Параметры:**
- `field1`: Описание (обязательный/опциональный, значение по умолчанию)
- `field2`: Описание (обязательный/опциональный, значение по умолчанию)
```

### Формат вывода
Всегда документировать структуру вывода:

```markdown
**Ожидаемый вывод**:
```json
{
  "success": true,
  "data": {},
  "errors": []
}
```

**Поля вывода:**
- `success`: Boolean, указывающий успех
- `data`: Обработанный результат
- `errors`: Массив сообщений об ошибках (пустой при успехе)
```

## Паттерны обработки ошибок

**Стандартные типы ошибок:**
1. **Невалидный ввод**: Отсутствующие обязательные поля, неправильные типы
2. **Ошибки обработки**: Сбои парсинга, ошибки конвертации
3. **Ошибки инструментов**: Сбои команд Bash, файл не найден
4. **Ошибки проверки**: Нарушения схемы, сбои ограничений

**Стандартный формат ошибки:**
```json
{
  "success": false,
  "data": null,
  "errors": [
    "Сообщение об ошибке 1",
    "Сообщение об ошибке 2"
  ]
}
```

## Раздел примеров

**Каждый навык ДОЛЖЕН иметь 3-5 примеров:**

1. **Счастливый путь** (валидный ввод, успешный вывод)
2. **Случай ошибки** (невалидный ввод, вывод ошибки)
3. **Краевой случай** (граничное условие)
4. **Реальный сценарий** (фактический случай использования)
5. **(Опционально)** Сложный сценарий

**Формат:**
```markdown

**Ввод**:
```json
{...}
```

**Вывод**:
```json
{...}
```

**Объяснение** (опционально):

## Процесс генерации

1. **Собрать требования** (имя, назначение, ввод, вывод, инструменты)
2. **Определить категорию** (парсинг/форматирование/проверка/выполнение/преобразование)
3. **Проверить, подходит ли Навык** (против Агента)
4. **Сгенерировать SKILL.md** следуя структуре выше
5. **Проверить по контрольному списку**
6. **Записать в `.claude/skills/{skill-name}/SKILL.md`**
7. **Сообщить о завершении:**
   ```
   ✅ Навык создан: .claude/skills/{skill-name}/SKILL.md

   Тип: {категория}
   Инструменты: {разрешенные-инструменты или "Чистая логика"}

   Соответствие шаблону:
   ✅ Ввод/вывод документированы
   ✅ Обработка ошибок определена
   ✅ Примеры включены
   ✅ Руководство по интеграции предоставлено

   Следующие шаги:
   1. Проверить SKILL.md
   2. Протестировать с примерами ввода
   3. Добавить в агента, которому нужна эта утилита
   ```

## Общие паттерны

### Шаблон навыка парсинга
```markdown
## Шаг 1: Получить необработанный ввод
- Принять строку/текстовый ввод
- Проверить, что не пустой

## Шаг 2: Извлечь паттерны
- Использовать регулярные выражения/парсинг строк
- Найти сообщения об ошибках, предупреждения и т.д.

## Шаг 3: Структурировать данные
- Построить JSON-объект
- Классифицировать по типу

## Шаг 4: Возврат структурированного результата
- Вывод: {success, data: [], errors: []}
```

### Шаблон навыка проверки
```markdown
## Шаг 1: Прочитать ввод
- Принять объект данных или путь к файлу
- Загрузить схему при необходимости

## Шаг 2: Проверить по критериям
- Проверить обязательные поля
- Проверить типы
- Проверить ограничения

## Шаг 3: Собрать ошибки
- Перечислить все сбои проверки
- Предоставить конкретные сообщения об ошибках

## Шаг 4: Возврат результата проверки
- Вывод: {valid: boolean, errors: []}
```

### Шаблон навыка выполнения
```markdown
## Шаг 1: Получить конфигурацию
- Принять команду/параметры
- Проверить обязательную конфигурацию

## Шаг 2: Выполнить команду
- Запустить через инструмент Bash
- Захватить вывод и код выхода

## Шаг 3: Парсить результат
- Определить прохождение/провал
- Извлечь сообщения об ошибках

## Шаг 4: Возврат результата выполнения
- Вывод: {passed, action, errors, exit_code}
```
