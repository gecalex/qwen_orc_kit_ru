# Проверка здоровья багов

> **ПАТТЕРН**: Оркестрация на основе агентов (см. `docs/Agents Ecosystem/AGENT-ORCHESTRATION.md` для подробностей)

Полный рабочий процесс обнаружения и исправления багов с координацией оркестратор-воркер.

**Что делает:**
- Полное обнаружение багов (все приоритеты, все категории)
- Поэтапное исправление (критические → высокие → средние → низкие)
- Контрольные точки качества после каждого этапа
- Проверочное сканирование
- До 3 итераций, если проблемы остаются
- Комплексный финальный отчет

**Конфигурация не требуется** - всегда запускает комплексные проверки.

---

## Ваша задача

### Шаг 1: Фаза 0 - Вызов оркестратора (Предварительная проверка)

Используйте инструмент Task для вызова bug-orchestrator для предварительной проверки:

```
subagent_type: "bug-orchestrator"
description: "Предварительная проверка баг-оркестратора"
prompt: "Выполнить Фазу 0: Предварительная проверка

Задачи:
1. Проверить окружение (package.json, скрипты, статус git)
2. Инициализировать отслеживание прогресса через TodoWrite
3. Инициализировать отслеживание итераций (итерация=1, макс=3)
4. Создать .tmp/current/plans/bug-detection.json для отслеживания отката
5. Сообщить о статусе предварительной проверки

ВАЖНО: После завершения предварительной проверки создать .tmp/current/plans/bug-detection.json и вернуть управление основной сессии.

Вернуть следующую информацию:
- Статус предварительной проверки (✅/⛔)
- Результаты проверки окружения
- Созданный путь к файлу плана
- Готовность к следующей фазе: true/false
"
```

**Затем**: Дождитесь возврата оркестратора.

---

### Шаг 2: Фаза 1 - Вызов bug-hunter (Обнаружение)

После возврата оркестратора:

1. **Прочитайте файл плана**, чтобы подтвердить его создание:
   ```
   Используйте инструмент Read: .tmp/current/plans/bug-detection.json
   Проверьте nextAgent === "bug-hunter"
   ```

2. **Вызовите bug-hunter** с помощью инструмента Task:
   ```
   subagent_type: "bug-hunter"
   description: "Фаза обнаружения багов"
   prompt: "Выполнить обнаружение багов на основе файла плана: .tmp/current/plans/bug-detection.json

Прочитайте файл плана и выполните комплексное обнаружение багов:
- Сканировать всю кодовую базу
- Классифицировать по приоритету (критические → высокие → средние → низкие)
- Сгенерировать bug-hunting-report.md

Вернуться в основную сессию по завершении."
   ```

**Затем**: Дождитесь возврата bug-hunter с отчетом.

---

### Шаг 3: Контрольная точка качества 1 - Возобновить оркестратор (Проверить обнаружение)

После возврата bug-hunter:

1. **Возобновить оркестратор** для проверки с помощью инструмента Task:
   ```
   subagent_type: "bug-orchestrator"
   description: "Проверить обнаружение багов"
   prompt: "Выполнить Контрольную точку качества 1: Проверка обнаружения

Фаза: Проверить вывод bug-hunter

Задачи:
1. Проверить существование bug-hunting-report.md
2. Проверить структуру отчета (требуемые разделы)
3. Разобрать количество багов по приоритету
4. Запустить проверку типов (предупреждение без блокировки)
5. Сообщить результаты контрольной точки

ВАЖНО: После проверки, если баги найдены:
- Создать .tmp/current/plans/bug-fixing-{priority}.json для критического приоритета (или самого высокого доступного)
- Вернуть управление основной сессии

Если баги не найдены или все контрольные точки не пройдены:
- Перейти к финальному резюме
- Вернуть управление

Вернуть следующее:
- Статус контрольной точки (✅ ПРОЙДЕНО / ⛔ НЕ ПРОЙДЕНО / ⚠️ ПРЕДУПРЕЖДЕНИЯ)
- Количество багов по приоритету
- Следующая фаза: fixing-critical / fixing-high / final-summary
- Созданный файл плана (если применимо)
"
   ```

**Затем**: Дождитесь результатов проверки оркестратора.

---

### Шаг 4: Фаза 2-5 - Этапы исправления (Итеративно)

После возврата оркестратора с планом исправления:

**Для каждого уровня приоритета** (критические → высокие → средние → низкие):

1. **Проверьте, есть ли у этого приоритета баги**:
   - Прочитайте ответ оркестратора
   - Если оркестратор говорит "перейти к следующему приоритету" → продолжить цикл
   - Если оркестратор говорит "финальное резюме" → перейти к Шагу 5

2. **Прочитайте план исправления**:
   ```
   Используйте инструмент Read: .tmp/current/plans/bug-fixing-{priority}.json
   Проверьте nextAgent === "bug-fixer"
   Проверьте config.priority === "{current-priority}"
   ```

3. **Вызовите bug-fixer** с помощью инструмента Task:
   ```
   subagent_type: "bug-fixer"
   description: "Исправить баги {priority}"
   prompt: "Выполнить исправление багов на основе файла плана: .tmp/current/plans/bug-fixing-{priority}.json

Прочитайте файл плана и исправьте баги для приоритета: {priority}
- Прочитайте bug-hunting-report.md для списка багов
- Исправляйте баги по одному
- Логируйте изменения в .bug-changes.json
- Обновите bug-fixes-implemented.md (обобщенный отчет)

Вернуться в основную сессию по завершении."
   ```

4. **Возобновите оркестратор** для проверки:
   ```
   subagent_type: "bug-orchestrator"
   description: "Проверить исправления {priority}"
   prompt: "Выполнить Контрольную точку качества 2: Проверка исправлений для приоритета={priority}

Задачи:
1. Проверить существование bug-fixes-implemented.md
2. Запустить проверку типов (БЛОКИРУЮЩАЯ)
3. Запустить сборку (БЛОКИРУЮЩАЯ)
4. Разобрать успешность исправлений
5. Проверить, нужна ли повторная попытка (если < 80% успеха)

Если проверка ПРОЙДЕНА и остаются приоритеты:
- Создать следующий .tmp/current/plans/bug-fixing-{priority}.json
- Вернуть управление

Если проверка НЕ ПРОЙДЕНА:
- Предоставить инструкции по откату
- Вернуть управление с ошибкой

Если все приоритеты завершены:
- Перейти к фазе проверки
- Вернуть управление

Вернуть:
- Статус контрольной точки
- Успешность исправлений
- Следующая фаза: fixing-{next-priority} / verification / final-summary
"
   ```

5. **Повторить** для следующего уровня приоритета.

---

### Шаг 5: Фаза 6 - Проверка

После завершения всех этапов исправления:

1. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "bug-orchestrator"
   description: "Создать план проверки"
   prompt: "Выполнить Фазу 6: Подготовка проверки

Создать .tmp/current/plans/bug-verification.json для повторного сканирования кодовой базы.

Вернуть управление с путем к файлу плана."
   ```

2. **Вызовите bug-hunter** для проверки:
   ```
   subagent_type: "bug-hunter"
   description: "Сканирование проверки"
   prompt: "Выполнить сканирование проверки на основе: .tmp/current/plans/bug-verification.json

Повторно просканировать кодовую базу для проверки исправлений. Перезаписывает bug-hunting-report.md.

Вернуться по завершении."
   ```

3. **Возобновите оркестратор** для проверки проверки:
   ```
   subagent_type: "bug-orchestrator"
   description: "Проверить проверку"
   prompt: "Выполнить Контрольную точку качества 3: Проверка проверки

Сравнить оригинальный bug-hunting-report.md (базовый) с новым сканированием:
- Подсчитать исправленные баги
- Проверить, введены ли новые баги
- Определить, нужна ли итерация

Вернуть:
- Статус проверки
- Оставшиеся баги
- Решение об итерации: итерация / завершить
"
   ```

---

### Шаг 6: Финальное резюме

После завершения всех фаз:

1. **Возобновите оркестратор** для финального резюме:
   ```
   subagent_type: "bug-orchestrator"
   description: "Сгенерировать финальное резюме"
   prompt: "Выполнить Фазу 8: Финальное резюме

Сгенерировать комплексный bug-fix-orchestration-summary.md:
- Все обнаруженные баги
- Все исправленные баги
- Успешность по приоритетам
- Результаты проверки
- Резюме итерации
- Инструкции по очистке

Вернуть финальное резюме."
   ```

2. **Отобразить результаты** пользователю:
   ```
   Прочитать bug-fix-orchestration-summary.md
   Отобразить ключевые метрики
   Показать статус проверки
   Перечислить следующие шаги
   ```

## Архитектурные заметки

**Роль оркестратора**:
- Создает файлы плана
- Проверяет вывод воркеров
- Возвращает управление основной сессии
- НЕТ прямого вызова воркеров

**Роль основной сессии** (эта команда):
- Читает файлы плана
- Вызывает воркеров через инструмент Task
- Возобновляет оркестратор для проверки
- Управляет полным циклом

**Роль воркера**:
- Читает файл плана
- Выполняет работу
- Генерирует отчет
- Возвращается в основную сессию

Этот паттерн следует реальным возможностям Claude Code (без автоматического вызова).
