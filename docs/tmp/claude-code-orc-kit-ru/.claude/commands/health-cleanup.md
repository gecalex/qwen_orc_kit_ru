# Проверка состояния очистки

> **ПАТТЕРН**: Оркестрация на основе агентов (см. `docs/Agents Ecosystem/AGENT-ORCHESTRATION.md` для получения подробной информации)

Полный рабочий процесс обнаружения и удаления мертвого кода с координацией оркестратора и работника.

**Что он делает:**
- Полное сканирование мертвого кода (неиспользуемые импорты, закомментированный код, недостижимый код, отладочные артефакты, неиспользуемые переменные)
- Поэтапное удаление (critical → high → medium → low)
- Контрольные точки качества после каждого этапа
- Проверочное сканирование
- До 3 итераций, если проблемы остаются
- Комплексный итоговый отчет

**Конфигурация не требуется** - всегда запускает комплексное обнаружение мертвого кода.

---

## Ваша задача

### Шаг 1: Фаза 0 - Вызов оркестратора (Предварительная проверка)

Используйте инструмент Task для вызова dead-code-orchestrator для предварительной проверки:

```
subagent_type: "dead-code-orchestrator"
description: "Предварительная проверка оркестратора очистки"
prompt: "Выполнить фазу 0: Предварительная проверка

Задачи:
1. Проверить окружение (package.json, скрипты, статус git)
2. Инициализировать отслеживание прогресса через TodoWrite
3. Инициализировать отслеживание итераций (итерация=1, макс=3)
4. Создать .tmp/current/plans/dead-code-detection.json для отслеживания отката
5. Сообщить статус предварительной проверки

ВАЖНО: После завершения предварительной проверки создать .tmp/current/plans/dead-code-detection.json и вернуть управление в основную сессию.

Вернуть следующую информацию:
- Статус предварительной проверки (✅/⛔)
- Результаты проверки окружения
- Созданный путь к файлу плана
- Готовность к следующей фазе: true/false
"
```

**Затем**: Дождаться возврата оркестратора.

---

### Шаг 2: Фаза 1 - Вызов dead-code-hunter (Обнаружение)

После возврата оркестратора:

1. **Прочитать файл плана** для подтверждения его создания:
   ```
   Использовать инструмент Read: .tmp/current/plans/dead-code-detection.json
   Проверить nextAgent === "dead-code-hunter"
   ```

2. **Вызвать dead-code-hunter** с помощью инструмента Task:
   ```
   subagent_type: "dead-code-hunter"
   description: "Фаза обнаружения мертвого кода"
   prompt: "Выполнить обнаружение мертвого кода на основе файла плана: .tmp/current/plans/dead-code-detection.json

Прочитать файл плана и выполнить комплексное обнаружение мертвого кода:
- Просканировать всю кодовую базу
- Классифицировать по приоритету (critical → high → medium → low)
- Сгенерировать dead-code-report.md

Вернуться в основную сессию по завершении."
   ```

**Затем**: Дождаться возврата dead-code-hunter с отчетом.

---

### Шаг 3: Контрольная точка 1 - Возобновить оркестратор (Проверка обнаружения)

После возврата dead-code-hunter:

1. **Возобновить оркестратор** для проверки с помощью инструмента Task:
   ```
   subagent_type: "dead-code-orchestrator"
   description: "Проверить обнаружение мертвого кода"
   prompt: "Выполнить контрольную точку 1: Проверка обнаружения

Фаза: Проверить вывод dead-code-hunter

Задачи:
1. Проверить существование dead-code-report.md
2. Проверить структуру отчета (требуемые разделы)
3. Разобрать количество мертвого кода по приоритету
4. Запустить проверку типов (предупреждение без блокировки)
5. Сообщить результаты контрольной точки

ВАЖНО: После проверки, если мертвый код найден:
- Создать .tmp/current/plans/cleanup-removal-{priority}.json для критического приоритета (или самого высокого доступного)
- Вернуть управление в основную сессию

Если мертвый код не найден или все контрольные точки не пройдены:
- Перейти к итоговому отчету
- Вернуть управление

Вернуть следующее:
- Статус контрольной точки (✅ PASSED / ⛔ FAILED / ⚠️ WARNINGS)
- Количество мертвого кода по приоритету
- Следующая фаза: removal-critical / removal-high / final-summary
- Созданный файл плана (если применимо)
"
   ```

**Затем**: Дождаться результатов проверки оркестратора.

---

### Шаг 4: Фаза 2-5 - Этапы удаления (Итеративно)

После возврата оркестратора с планом исправления:

**Для каждого уровня приоритета** (critical → high → medium → low):

1. **Проверить, есть ли мертвый код у этого приоритета**:
   - Прочитать ответ оркестратора
   - Если оркестратор говорит "перейти к следующему приоритету" → продолжить цикл
   - Если оркестратор говорит "итоговый отчет" → перейти к шагу 5

2. **Прочитать план исправления**:
   ```
   Использовать инструмент Read: .tmp/current/plans/cleanup-removal-{priority}.json
   Проверить nextAgent === "dead-code-remover"
   Проверить config.priority === "{current-priority}"
   ```

3. **Вызвать dead-code-remover** с помощью инструмента Task:
   ```
   subagent_type: "dead-code-remover"
   description: "Удалить мертвый код {priority}"
   prompt: "Выполнить удаление мертвого кода на основе файла плана: .tmp/current/plans/cleanup-removal-{priority}.json

Прочитать файл плана и удалить мертвый код для приоритета: {priority}
- Прочитать dead-code-report.md для списка мертвого кода
- Удалять мертвый код по одному
- Записывать изменения в .dead-code-changes.json
- Обновить dead-code-cleanup-summary.md (объединенный отчет)

Вернуться в основную сессию по завершении."
   ```

4. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "dead-code-orchestrator"
   description: "Проверить исправления {priority}"
   prompt: "Выполнить контрольную точку 2: Проверка удаления для приоритета={priority}

Задачи:
1. Проверить существование dead-code-cleanup-summary.md
2. Запустить проверку типов (БЛОКИРУЕТСЯ)
3. Запустить сборку (БЛОКИРУЕТСЯ)
4. Разобрать процент успеха удаления
5. Проверить, нужна ли повторная попытка (если < 80% успеха)

Если проверка ПРОЙДЕНА и остались приоритеты:
- Создать следующий .tmp/current/plans/cleanup-removal-{priority}.json
- Вернуть управление

Если проверка НЕ ПРОЙДЕНА:
- Предоставить инструкции по откату
- Вернуть управление с ошибкой

Если все приоритеты выполнены:
- Перейти к фазе проверки
- Вернуть управление

Вернуть:
- Статус контрольной точки
- Процент успеха исправлений
- Следующая фаза: removal-{next-priority} / verification / final-summary
"
   ```

5. **Повторить** для следующего уровня приоритета.

---

### Шаг 5: Фаза 6 - Проверка

После завершения всех этапов исправления:

1. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "dead-code-orchestrator"
   description: "Создать план проверки"
   prompt: "Выполнить фазу 6: Подготовка проверки

Создать .tmp/current/plans/dead-code-verification.json для повторного сканирования кодовой базы.

Вернуть управление с путем к файлу плана."
   ```

2. **Вызвать dead-code-hunter** для проверки:
   ```
   subagent_type: "dead-code-hunter"
   description: "Проверочное сканирование"
   prompt: "Выполнить проверочное сканирование на основе: .tmp/current/plans/dead-code-verification.json

Повторно просканировать кодовую базу для проверки удалений. Перезаписывает dead-code-report.md.

Вернуться по завершении."
   ```

3. **Возобновить оркестратор** для проверки проверки:
   ```
   subagent_type: "dead-code-orchestrator"
   description: "Проверить проверку"
   prompt: "Выполнить контрольную точку 3: Проверка проверки

Сравнить исходный dead-code-report.md (базовый) с новым сканированием:
- Подсчитать удаленный мертвый код
- Проверить, введен ли новый мертвый код
- Определить, нужна ли итерация

Вернуть:
- Статус проверки
- Оставшийся мертвый код
- Решение об итерации: итерировать / завершить
"
   ```

---

### Шаг 6: Итоговый отчет

После завершения всех фаз:

1. **Возобновить оркестратор** для итогового отчета:
   ```
   subagent_type: "dead-code-orchestrator"
   description: "Сгенерировать итоговый отчет"
   prompt: "Выполнить фазу 8: Итоговый отчет

Сгенерировать комплексный dead-code-orchestration-summary.md:
- Весь обнаруженный мертвый код
- Весь удаленный мертвый код
- Проценты успеха по приоритету
- Результаты проверки
- Итог итерации
- Инструкции по архивированию и очистке

Вернуть итоговый отчет."
   ```

2. **Отобразить результаты** пользователю:
   ```
   Прочитать dead-code-orchestration-summary.md
   Отобразить ключевые метрики
   Показать статус проверки
   Список следующих шагов
   ```

## Примечания по архитектуре

**Роль оркестратора**:
- Создает файлы плана
- Проверяет вывод работников
- Возвращает управление в основную сессию
- НЕТ прямого вызова работника

**Роль основной сессии** (эта команда):
- Читает файлы плана
- Вызывает работников через инструмент Task
- Возобновляет оркестратор для проверки
- Управляет полным циклом

**Роль работника**:
- Читает файл плана
- Выполняет работу
- Генерирует отчет
- Возвращается в основную сессию

Этот паттерн следует реальным возможностям Claude Code (без автовызова).
