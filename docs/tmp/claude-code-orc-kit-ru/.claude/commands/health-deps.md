# Проверка состояния зависимостей

> **ПАТТЕРН**: Оркестрация на основе агентов (см. `docs/Agents Ecosystem/AGENT-ORCHESTRATION.md` для получения подробной информации)

Полный рабочий процесс аудита и обновления зависимостей с координацией оркестратора и работника.

**Что он делает:**
- Полный аудит зависимостей (устаревшие пакеты, проблемы безопасности зависимостей, неиспользуемые зависимости)
- Поэтапные обновления (critical → high → medium → low)
- Контрольные точки качества после каждого этапа
- Проверочное сканирование
- До 3 итераций, если проблемы остаются
- Комплексный итоговый отчет

**Конфигурация не требуется** - всегда запускает комплексный аудит зависимостей.

---

## Ваша задача

### Шаг 1: Фаза 0 - Вызов оркестратора (Предварительная проверка)

Используйте инструмент Task для вызова dependency-orchestrator для предварительной проверки:

```
subagent_type: "dependency-orchestrator"
description: "Предварительная проверка оркестратора зависимостей"
prompt: "Выполнить фазу 0: Предварительная проверка

Задачи:
1. Проверить окружение (package.json, скрипты, статус git)
2. Инициализировать отслеживание прогресса через TodoWrite
3. Инициализировать отслеживание итераций (итерация=1, макс=3)
4. Создать .tmp/current/plans/dependency-detection.json для отслеживания отката
5. Сообщить статус предварительной проверки

ВАЖНО: После завершения предварительной проверки создать .tmp/current/plans/dependency-detection.json и вернуть управление в основную сессию.

Вернуть следующую информацию:
- Статус предварительной проверки (✅/⛔)
- Результаты проверки окружения
- Созданный путь к файлу плана
- Готовность к следующей фазе: true/false
"
```

**Затем**: Дождаться возврата оркестратора.

---

### Шаг 2: Фаза 1 - Вызов dependency-auditor (Обнаружение)

После возврата оркестратора:

1. **Прочитать файл плана** для подтверждения его создания:
   ```
   Использовать инструмент Read: .tmp/current/plans/dependency-detection.json
   Проверить nextAgent === "dependency-auditor"
   ```

2. **Вызвать dependency-auditor** с помощью инструмента Task:
   ```
   subagent_type: "dependency-auditor"
   description: "Фаза аудита зависимостей"
   prompt: "Выполнить аудит зависимостей на основе файла плана: .tmp/current/plans/dependency-detection.json

Прочитать файл плана и выполнить комплексный аудит зависимостей:
- Просканировать всю кодовую базу
- Классифицировать по приоритету (critical → high → medium → low)
- Сгенерировать dependency-scan-report.md

Вернуться в основную сессию по завершении."
   ```

**Затем**: Дождаться возврата dependency-auditor с отчетом.

---

### Шаг 3: Контрольная точка 1 - Возобновить оркестратор (Проверка обнаружения)

После возврата dependency-auditor:

1. **Возобновить оркестратор** для проверки с помощью инструмента Task:
   ```
   subagent_type: "dependency-orchestrator"
   description: "Проверить аудит зависимостей"
   prompt: "Выполнить контрольную точку 1: Проверка обнаружения

Фаза: Проверить вывод dependency-auditor

Задачи:
1. Проверить существование dependency-scan-report.md
2. Проверить структуру отчета (требуемые разделы)
3. Разобрать количество проблем зависимостей по приоритету
4. Запустить проверку типов (предупреждение без блокировки)
5. Сообщить результаты контрольной точки

ВАЖНО: После проверки, если проблемы зависимостей найдены:
- Создать .tmp/current/plans/dependency-update-{priority}.json для критического приоритета (или самого высокого доступного)
- Вернуть управление в основную сессию

Если проблемы зависимостей не найдены или все контрольные точки не пройдены:
- Перейти к итоговому отчету
- Вернуть управление

Вернуть следующее:
- Статус контрольной точки (✅ PASSED / ⛔ FAILED / ⚠️ WARNINGS)
- Количество проблем безопасности по приоритету
- Следующая фаза: update-critical / update-high / final-summary
- Созданный файл плана (если применимо)
"
   ```

**Затем**: Дождаться результатов проверки оркестратора.

---

### Шаг 4: Фаза 2-5 - Этапы обновления (Итеративно)

После возврата оркестратора с планом обновления:

**Для каждого уровня приоритета** (critical → high → medium → low):

1. **Проверить, есть ли у этого приоритета проблемы зависимостей**:
   - Прочитать ответ оркестратора
   - Если оркестратор говорит "перейти к следующему приоритету" → продолжить цикл
   - Если оркестратор говорит "итоговый отчет" → перейти к шагу 5

2. **Прочитать план обновления**:
   ```
   Использовать инструмент Read: .tmp/current/plans/dependency-update-{priority}.json
   Проверить nextAgent === "dependency issue-fixer"
   Проверить config.priority === "{current-priority}"
   ```

3. **Вызвать dependency issue-fixer** с помощью инструмента Task:
   ```
   subagent_type: "dependency issue-fixer"
   description: "Обновить зависимости {priority}"
   prompt: "Выполнить обновление проблем зависимостей на основе файла плана: .tmp/current/plans/dependency-update-{priority}.json

Прочитать файл плана и исправить проблемы зависимостей для приоритета: {priority}
- Прочитать dependency-scan-report.md для списка проблем зависимостей
- Исправлять проблемы зависимостей по одной
- Записывать изменения в .dependency issue-changes.json
- Обновить dependency-updates-implemented.md (объединенный отчет)

Вернуться в основную сессию по завершении."
   ```

4. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "dependency-orchestrator"
   description: "Проверить обновления {priority}"
   prompt: "Выполнить контрольную точку 2: Проверка обновлений для приоритета={priority}

Задачи:
1. Проверить существование dependency-updates-implemented.md
2. Запустить проверку типов (БЛОКИРУЕТСЯ)
3. Запустить сборку (БЛОКИРУЕТСЯ)
4. Разобрать процент успеха исправления
5. Проверить, нужна ли повторная попытка (если < 80% успеха)

Если проверка ПРОЙДЕНА и остались приоритеты:
- Создать следующий .tmp/current/plans/dependency-update-{priority}.json
- Вернуть управление

Если проверка НЕ ПРОЙДЕНА:
- Предоставить инструкции по откату
- Вернуть управление с ошибкой

Если все приоритеты выполнены:
- Перейти к фазе проверки
- Вернуть управление

Вернуть:
- Статус контрольной точки
- Процент успеха исправления
- Следующая фаза: update-{next-priority} / verification / final-summary
"
   ```

5. **Повторить** для следующего уровня приоритета.

---

### Шаг 5: Фаза 6 - Проверка

После завершения всех этапов обновления:

1. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "dependency-orchestrator"
   description: "Создать план проверки"
   prompt: "Выполнить фазу 6: Подготовка проверки

Создать .tmp/current/plans/dependency-verification.json для повторного сканирования кодовой базы.

Вернуть управление с путем к файлу плана."
   ```

2. **Вызвать dependency-auditor** для проверки:
   ```
   subagent_type: "dependency-auditor"
   description: "Проверочное сканирование"
   prompt: "Выполнить проверочное сканирование на основе: .tmp/current/plans/dependency-verification.json

Повторно просканировать кодовую базу для проверки обновлений. Перезаписывает dependency-scan-report.md.

Вернуться по завершении."
   ```

3. **Возобновить оркестратор** для проверки проверки:
   ```
   subagent_type: "dependency-orchestrator"
   description: "Проверить проверку"
   prompt: "Выполнить контрольную точку 3: Проверка проверки

Сравнить исходный dependency-scan-report.md (базовый) с новым сканированием:
- Подсчитать исправленные проблемы зависимостей
- Проверить, введены ли новые проблемы зависимостей
- Определить, нужна ли итерация

Вернуть:
- Статус проверки
- Оставшиеся проблемы безопасности
- Решение об итерации: итерировать / завершить
"
   ```

---

### Шаг 6: Итоговый отчет

После завершения всех фаз:

1. **Возобновить оркестратор** для итогового отчета:
   ```
   subagent_type: "dependency-orchestrator"
   description: "Сгенерировать итоговый отчет"
   prompt: "Выполнить фазу 8: Итоговый отчет

Сгенерировать комплексный dependency-orchestration-summary.md:
- Все обнаруженные проблемы зависимостей
- Все исправленные проблемы зависимостей
- Проценты успеха по приоритету
- Результаты проверки
- Итог итерации
- Инструкции по очистке

Вернуть итоговый отчет."
   ```

2. **Отобразить результаты** пользователю:
   ```
   Прочитать dependency-orchestration-summary.md
   Отобразить ключевые метрики
   Показать статус проверки
   Список следующих шагов
   ```

## Примечания по архитектуре

**Роль оркестратора**:
- Создает файлы плана
- Проверяет вывод работников
- Возвращает управление в основную сессию
- НЕТ прямого вызова работника

**Роль основной сессии** (эта команда):
- Читает файлы плана
- Вызывает работников через инструмент Task
- Возобновляет оркестратор для проверки
- Управляет полным циклом

**Роль работника**:
- Читает файл плана
- Выполняет работу
- Генерирует отчет
- Возвращается в основную сессию

Этот паттерн следует реальным возможностям Claude Code (без автовызова).
