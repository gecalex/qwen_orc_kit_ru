# Команда: /health-metrics

Вы анализируете метрики состояния экосистемы агентов для генерации ежемесячных отчетов о производительности.

## Назначение

Генерировать комплексный ежемесячный отчет по метрикам, показывающий производительность агентов, контрольные точки качества, распределение сложности, использование Context7 и состояние системы для принятия решений по оптимизации на основе данных.

## Процесс

### Шаг 1: Определение отчетного периода

**По умолчанию**: Текущий месяц (формат YYYY-MM)

**Пользователь может указать**: Предыдущий месяц или конкретный месяц

**Примеры**:
- `/health-metrics` → Текущий месяц (2025-10)
- `/health-metrics 2025-09` → Сентябрь 2025
- `/health-metrics last-month` → Предыдущий месяц

### Шаг 2: Чтение файлов метрик

**Текущий месяц**:
```
Файл: .tmp/metrics/YYYY-MM.json
```

**Предыдущий месяц** (для сравнения):
```
Файл: .tmp/metrics/YYYY-(MM-1).json
```

**Если файл не существует**:
- Сообщить: "Нет данных метрик для {месяца}"
- Предложить: "Запустите рабочие процессы агента для генерации метрик сначала"
- Завершить работу корректно

**Если файл поврежден**:
- Попытаться прочитать резервную копию: `.tmp/metrics/YYYY-MM.json.backup`
- Если резервная копия также повреждена: Сообщить об ошибке и выйти
- Зарегистрировать инцидент повреждения

### Шаг 3: Анализ метрик

Рассчитать ключевые показатели из данных метрик.

#### Анализ производительности агента

**Для каждого агента**:
- **Процент успеха**: `(успешные / вызовы) * 100`
- **Тенденция производительности**: Сравнить avg_duration с предыдущим месяцем
- **Доставленная ценность**: Количество найденных багов/уязвимостей/проблем

**Классификация агентов**:
- **Лучшие исполнители**: Процент успеха >= 90%
- **Требуют улучшения**: Процент успеха < 70%
- **Недоиспользуются**: Вызовы < 5 в месяц

#### Анализ производительности оркестратора

**Для каждого оркестратора**:
- **Процент успеха**: `(завершено / запуски) * 100`
- **Процент прерываний**: `(прервано / запуски) * 100`
- **Эффективность итераций**: avg_iterations (меньше - лучше)
- **Достигнуто максимальное количество итераций**: Количество (указывает на проблемы со сложностью)

#### Анализ контрольных точек качества

**Для каждой точки (type-check, сборка, тесты)**:
- **Процент прохождения**: Уже рассчитан в метриках
- **Тенденция**: Сравнить процент прохождения с предыдущим месяцем
- **Отметить проблемы**: Процент прохождения < 85%

#### Распределение сложности

**Анализировать уровни сложности**:
- **Процент по уровню**: Рассчитать % для тривиальных/умеренных/высоких/критических
- **Использование исследований**: % высоких/критических, которые вызвали исследование
- **Средняя оценка**: Указывает общую сложность задач

#### Интеграция Context7

**Метрики**:
- **Процент успеха**: `(успешные_запросы / всего_запросов) * 100`
- **Доступность**: `100 - (события_недоступности / всего_вызовов * 100)`
- **Производительность**: avg_response_time_ms
- **Топ библиотек**: Сортировка по количеству запросов

#### Состояние поведенческой ОС

**Частота резервных стратегий**:
- Подсчитать общее количество резервных стратегий
- Рассчитать % вызовов, которые вызвали резервные стратегии
- Отметить, если любая резервная стратегия > 5% частоты

**Аварийные инциденты**:
- Подсчитать каждый тип аварийного режима
- Отметить, если произошел любой аварийный режим (должно быть редко)

**Нарушения основного директивного принципа**:
- Должно быть 0 (самодиагностика предотвращает нарушения)
- Если > 0: Критическая проблема, требует расследования

### Шаг 4: Генерация анализа тенденций

Сравнить текущий месяц с предыдущим (если доступно).

**Метрики для сравнения**:
- Общее количество вызовов (индикатор роста)
- Общий процент успеха (тенденция надежности)
- Проценты прохождения контрольных точек (тенденция состояния кода)
- Использование Context7 (тенденция внедрения)
- Запущенные фазы исследований (тенденция сложности)

**Индикаторы изменений**:
- ⬆️ Увеличение
- ⬇️ Уменьшение
- ➡️ Без изменений (в пределах 2%)

### Шаг 5: Генерация рекомендаций

На основе анализа данных предоставить рекомендации, подлежащие реализации.

**Высокий приоритет** (Процент успеха < 70%, аварийные инциденты):
- Конкретные улучшения агента, необходимые
- Требуется расследование контрольных точек
- Проблемы надежности системы

**Средний приоритет** (Процент прохождения < 85%, недоиспользование):
- Улучшения обнаружения агентов
- Обновления документации
- Оптимизация рабочих процессов

**Низкий приоритет** (Желательные оптимизации):
- Улучшения эффективности токенов
- Оптимизация производительности
- Запросы функций

## Обработка ошибок

### Нет данных метрик

**Проблема**: Файл метрик не существует для запрашиваемого месяца

**Действие**:
```
❌ Нет данных метрик для {месяца}

Файл метрик не найден: .tmp/metrics/{YYYY-MM}.json

Это означает, что рабочие процессы агента не запускались в течение {месяца}.

Для генерации метрик:
1. Запустить рабочие процессы агента: /health-bugs, /health-security и т.д.
2. Агенты автоматически записывают метрики через навык record-metrics
3. Повторно запустить /health-metrics после завершения рабочих процессов

Примечание: Метрики хранятся ежемесячно в каталоге .tmp/metrics/.
```

### Поврежденный файл метрик

**Проблема**: Файл метрик существует, но имеет недействительный JSON

**Действие**:
```
⛔ Файл метрик поврежден

Файл: .tmp/metrics/{YYYY-MM}.json
Ошибка: Недействительный JSON (SyntaxError: ...)

Попытка чтения резервной копии...
```

Если резервная копия существует и действительна:
```
✅ Восстановлено из резервной копии

Резервная копия: .tmp/metrics/{YYYY-MM}.json.backup
Статус: Действительна

Генерация отчета из данных резервной копии...
```

Если резервная копия также повреждена:
```
❌ Резервная копия также повреждена или не найдена

Невозможно сгенерировать отчет из поврежденных данных.

Варианты:
1. Проверить .tmp/metrics/{YYYY-MM}.json вручную
2. Удалить поврежденный файл и начать заново в следующем месяце
3. Сообщить об этой проблеме (повреждение файла не должно происходить)

Повреждение файла может указывать на:
- Конфликт одновременной записи
- Ошибку ввода-вывода диска
- Сбой системы во время записи метрик
```

### Сбой генерации отчета

**Проблема**: Ошибка при создании файла отчета

**Действие**:
```
⛔ Сбой генерации отчета

Цель: docs/reports/metrics/{YYYY-MM}-ecosystem-health.md
Ошибка: {сообщение об ошибке}

Возможные причины:
- Недостаточно места на диске
- Отказ в доступе (проверить разрешения docs/reports/metrics/)
- Ошибка ввода-вывода диска

Рекомендация:
1. Проверить место на диске: df -h
2. Проверить разрешения: ls -la docs/reports/
3. Повторить после устранения проблемы
```

## Примечания по реализации

### Расчет месяца

**Текущий месяц**:
```
Сегодня: 2025-10-21
Текущий месяц: 2025-10
```

**Предыдущий месяц**:
```
Текущий: 2025-10
Предыдущий: 2025-09

Крайний случай (январь):
Текущий: 2025-01
Предыдущий: 2024-12
```

### Определение состояния здоровья

**Процент успеха агента**:
- >= 90%: Лучший исполнитель ✅
- 70-89%: Приемлемо
- < 70%: Требует улучшения ⚠️

**Общее состояние системы**:
- >= 90% процент успеха + нет аварийных режимов + положительные тенденции: ЗДОРОВОЕ И УЛУЧШАЕТСЯ ✅
- >= 80% процент успеха + < 2 аварийных режима: ЗДОРОВОЕ ✅
- 70-79% процент успеха ИЛИ 2-5 аварийных режимов: ТРЕБУЕТ ВНИМАНИЯ ⚠️
- < 70% процент успеха ИЛИ > 5 аварийных режимов: КРИТИЧНОЕ ❌

### Именование файлов отчетов

**Формат**: `{YYYY-MM}-ecosystem-health.md`

**Примеры**:
- `2025-10-ecosystem-health.md`
- `2025-09-ecosystem-health.md`

### Индикаторы тенденций

**Расчет**:
```
Изменение % = ((Новое - Старое) / Старое) * 100

Если изменение > 2%: ⬆️ (увеличение)
Если изменение < -2%: ⬇️ (уменьшение)
Если изменение между -2% и 2%: ➡️ (стабильно)
```
