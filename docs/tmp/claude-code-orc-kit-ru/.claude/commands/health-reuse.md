# Проверка состояния повторного использования кода

> **ПАТТЕРН**: Оркестрация на основе агентов (см. `docs/Agents Ecosystem/AGENT-ORCHESTRATION.md` для получения подробной информации)

Полный рабочий процесс обнаружения и объединения дублирования кода с координацией оркестратора и работника.

**Что он делает:**
- Полное обнаружение дублирования (типы, схемы, константы, утилиты)
- Поэтапное объединение (HIGH → MEDIUM → LOW приоритет)
- Контрольные точки качества после каждого этапа
- Проверочное сканирование
- До 3 итераций, если проблемы остаются
- Комплексный итоговый отчет

**Конфигурация не требуется** - всегда запускает комплексные проверки.

---

## Ваша задача

### Шаг 1: Фаза 0 - Вызов оркестратора (Предварительная проверка)

Используйте инструмент Task для вызова reuse-orchestrator для предварительной проверки:

```
subagent_type: "reuse-orchestrator"
description: "Предварительная проверка оркестратора повторного использования"
prompt: "Выполнить фазу 0: Предварительная проверка

Задачи:
1. Проверить окружение (package.json, скрипты, статус git)
2. Инициализировать отслеживание прогресса через TodoWrite
3. Инициализировать отслеживание итераций (итерация=1, макс=3)
4. Создать .tmp/current/plans/reuse-detection.json для отслеживания отката
5. Сообщить статус предварительной проверки

ВАЖНО: После завершения предварительной проверки создать .tmp/current/plans/reuse-detection.json и вернуть управление в основную сессию.

Вернуть следующую информацию:
- Статус предварительной проверки (✅/⛔)
- Результаты проверки окружения
- Созданный путь к файлу плана
- Готовность к следующей фазе: true/false
"
```

**Затем**: Дождаться возврата оркестратора.

---

### Шаг 2: Фаза 1 - Вызов reuse-hunter (Обнаружение)

После возврата оркестратора:

1. **Прочитать файл плана** для подтверждения его создания:
   ```
   Использовать инструмент Read: .tmp/current/plans/reuse-detection.json
   Проверить nextAgent === "reuse-hunter"
   ```

2. **Вызвать reuse-hunter** с помощью инструмента Task:
   ```
   subagent_type: "reuse-hunter"
   description: "Фаза обнаружения дублирования кода"
   prompt: "Выполнить обнаружение дублирования кода на основе файла плана: .tmp/current/plans/reuse-detection.json

Прочитать файл плана и выполнить комплексное обнаружение дублирования:
- Просканировать всю кодовую базу на дублированные типы, интерфейсы, схемы Zod
- Найти дублированные константы и конфигурационные объекты
- Определить утилитарные функции, которые копируются вместо импорта
- Классифицировать по приоритету (HIGH → MEDIUM → LOW)
- Сгенерировать reuse-hunting-report.md

Вернуться в основную сессию по завершении."
   ```

**Затем**: Дождаться возврата reuse-hunter с отчетом.

---

### Шаг 3: Контрольная точка 1 - Возобновить оркестратор (Проверка обнаружения)

После возврата reuse-hunter:

1. **Возобновить оркестратор** для проверки с помощью инструмента Task:
   ```
   subagent_type: "reuse-orchestrator"
   description: "Проверить обнаружение дублирования"
   prompt: "Выполнить контрольную точку 1: Проверка обнаружения

Фаза: Проверить вывод reuse-hunter

Задачи:
1. Проверить существование reuse-hunting-report.md
2. Проверить структуру отчета (требуемые разделы)
3. Разобрать количество дублирований по приоритету
4. Запустить проверку типов (предупреждение без блокировки)
5. Сообщить результаты контрольной точки

ВАЖНО: После проверки, если дублирования найдены:
- Создать .tmp/current/plans/reuse-consolidation-{priority}.json для самого высокого приоритета
- Вернуть управление в основную сессию

Если дублирования не найдены или все контрольные точки не пройдены:
- Перейти к итоговому отчету
- Вернуть управление

Вернуть следующее:
- Статус контрольной точки (✅ PASSED / ⛔ FAILED / ⚠️ WARNINGS)
- Количество дублирований по приоритету
- Следующая фаза: consolidation-high / consolidation-medium / final-summary
- Созданный файл плана (если применимо)
"
   ```

**Затем**: Дождаться результатов проверки оркестратора.

---

### Шаг 4: Фаза 2-4 - Этапы объединения (Итеративно)

После возврата оркестратора с планом объединения:

**Для каждого уровня приоритета** (HIGH → MEDIUM → LOW):

1. **Проверить, есть ли дублирования у этого приоритета**:
   - Прочитать ответ оркестратора
   - Если оркестратор говорит "перейти к следующему приоритету" → продолжить цикл
   - Если оркестратор говорит "итоговый отчет" → перейти к шагу 5

2. **Прочитать план объединения**:
   ```
   Использовать инструмент Read: .tmp/current/plans/reuse-consolidation-{priority}.json
   Проверить nextAgent === "reuse-fixer"
   Проверить config.priority === "{current-priority}"
   ```

3. **Вызвать reuse-fixer** с помощью инструмента Task:
   ```
   subagent_type: "reuse-fixer"
   description: "Объединить дублирования {priority}"
   prompt: "Выполнить объединение кода на основе файла плана: .tmp/current/plans/reuse-consolidation-{priority}.json

Прочитать файл плана и объединить дублирования для приоритета: {priority}
- Прочитать reuse-hunting-report.md для списка дублирований
- Для каждого дублирования:
  - Определить каноническое местоположение (обычно shared-types)
  - Создать/обновить канонический файл
  - Заменить дубликаты ре-экспортами
- Записать изменения в .reuse-changes.json
- Обновить reuse-consolidation-implemented.md (объединенный отчет)

Вернуться в основную сессию по завершении."
   ```

4. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "reuse-orchestrator"
   description: "Проверить объединение {priority}"
   prompt: "Выполнить контрольную точку 2: Проверка объединения для приоритета={priority}

Задачи:
1. Проверить существование reuse-consolidation-implemented.md
2. Запустить проверку типов (БЛОКИРУЕТСЯ)
3. Запустить сборку (БЛОКИРУЕТСЯ)
4. Разобрать процент успеха объединения
5. Проверить, нужна ли повторная попытка (если < 80% успеха)

Если проверка ПРОЙДЕНА и остались приоритеты:
- Создать следующий .tmp/current/plans/reuse-consolidation-{priority}.json
- Вернуть управление

Если проверка НЕ ПРОЙДЕНА:
- Предоставить инструкции по откату
- Вернуть управление с ошибкой

Если все приоритеты выполнены:
- Перейти к фазе проверки
- Вернуть управление

Вернуть:
- Статус контрольной точки
- Процент успеха объединения
- Следующая фаза: consolidation-{next-priority} / verification / final-summary
"
   ```

5. **Повторить** для следующего уровня приоритета.

---

### Шаг 5: Фаза 5 - Проверка

После завершения всех этапов объединения:

1. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "reuse-orchestrator"
   description: "Создать план проверки"
   prompt: "Выполнить фазу 5: Подготовка проверки

Создать .tmp/current/plans/reuse-verification.json для повторного сканирования кодовой базы.

Вернуть управление с путем к файлу плана."
   ```

2. **Вызвать reuse-hunter** для проверки:
   ```
   subagent_type: "reuse-hunter"
   description: "Проверочное сканирование"
   prompt: "Выполнить проверочное сканирование на основе: .tmp/current/plans/reuse-verification.json

Повторно просканировать кодовую базу для проверки объединений. Перезаписывает reuse-hunting-report.md.

Вернуться по завершении."
   ```

3. **Возобновить оркестратор** для проверки проверки:
   ```
   subagent_type: "reuse-orchestrator"
   description: "Проверить проверку"
   prompt: "Выполнить контрольную точку 3: Проверка проверки

Сравнить исходный reuse-hunting-report.md (базовый) с новым сканированием:
- Подсчитать решенные дублирования
- Проверить, введены ли новые дублирования
- Определить, нужна ли итерация

Вернуть:
- Статус проверки
- Оставшиеся дублирования
- Решение об итерации: итерировать / завершить
"
   ```

---

### Шаг 6: Итоговый отчет

После завершения всех фаз:

1. **Возобновить оркестратор** для итогового отчета:
   ```
   subagent_type: "reuse-orchestrator"
   description: "Сгенерировать итоговый отчет"
   prompt: "Выполнить фазу 7: Итоговый отчет

Сгенерировать комплексный reuse-orchestration-summary.md:
- Все обнаруженные дублирования
- Все выполненные объединения
- Проценты успеха по приоритету
- Результаты проверки
- Итог итерации
- Инструкции по очистке

Вернуть итоговый отчет."
   ```

2. **Отобразить результаты** пользователю:
   ```
   Прочитать reuse-orchestration-summary.md
   Отобразить ключевые метрики
   Показать статус проверки
   Список следующих шагов
   ```

## Примечания по архитектуре

**Роль оркестратора**:
- Создает файлы плана
- Проверяет вывод работников
- Возвращает управление в основную сессию
- НЕТ прямого вызова работника

**Роль основной сессии** (эта команда):
- Читает файлы плана
- Вызывает работников через инструмент Task
- Возобновляет оркестратор для проверки
- Управляет полным циклом

**Роль работника**:
- Читает файл плана
- Выполняет работу
- Генерирует отчет
- Возвращается в основную сессию

Этот паттерн следует реальным возможностям Claude Code (без автовызова).

## Категории дублирования

**Типы/Интерфейсы** (shared-types):
- Типы базы данных
- Типы API
- Схемы Zod
- Общие перечисления

**Константы** (shared-types):
- Конфигурационные объекты
- MIME типы, ограничения файлов
- Флаги функций

**Утилиты** (общий пакет или ре-экспорт):
- Вспомогательные функции
- Утилиты проверки
- Форматтеры

**Паттерн одного источника истины**:
1. Каноническое местоположение: `packages/shared-types/src/`
2. Другие пакеты: `export * from '@megacampus/shared-types/{module}'`
3. НИКОГДА не копировать код между пакетами
