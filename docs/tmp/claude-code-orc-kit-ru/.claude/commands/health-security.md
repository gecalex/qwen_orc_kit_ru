# Проверка состояния безопасности

> **ПАТТЕРН**: Оркестрация на основе агентов (см. `docs/Agents Ecosystem/AGENT-ORCHESTRATION.md` для получения подробной информации)

Полный рабочий процесс сканирования безопасности и исправления уязвимостей с координацией оркестратора и работника.

**Что он делает:**
- Полное сканирование безопасности (SQL-инъекции, XSS, проблемы с аутентификацией, политики RLS, секреты)
- Поэтапное исправление (critical → high → medium → low)
- Контрольные точки качества после каждого этапа
- Проверочное сканирование
- До 3 итераций, если проблемы остаются
- Комплексный итоговый отчет

**Конфигурация не требуется** - всегда запускает комплексный аудит безопасности.

---

## Ваша задача

### Шаг 1: Фаза 0 - Вызов оркестратора (Предварительная проверка)

Используйте инструмент Task для вызова security-orchestrator для предварительной проверки:

```
subagent_type: "security-orchestrator"
description: "Предварительная проверка оркестратора безопасности"
prompt: "Выполнить фазу 0: Предварительная проверка

Задачи:
1. Проверить окружение (package.json, скрипты, статус git)
2. Инициализировать отслеживание прогресса через TodoWrite
3. Инициализировать отслеживание итераций (итерация=1, макс=3)
4. Создать .tmp/current/plans/security-detection.json для отслеживания отката
5. Сообщить статус предварительной проверки

ВАЖНО: После завершения предварительной проверки создать .tmp/current/plans/security-detection.json и вернуть управление в основную сессию.

Вернуть следующую информацию:
- Статус предварительной проверки (✅/⛔)
- Результаты проверки окружения
- Созданный путь к файлу плана
- Готовность к следующей фазе: true/false
"
```

**Затем**: Дождаться возврата оркестратора.

---

### Шаг 2: Фаза 1 - Вызов security-scanner (Обнаружение)

После возврата оркестратора:

1. **Прочитать файл плана** для подтверждения его создания:
   ```
   Использовать инструмент Read: .tmp/current/plans/security-detection.json
   Проверить nextAgent === "security-scanner"
   ```

2. **Вызвать security-scanner** с помощью инструмента Task:
   ```
   subagent_type: "security-scanner"
   description: "Фаза обнаружения безопасности"
   prompt: "Выполнить обнаружение уязвимостей на основе файла плана: .tmp/current/plans/security-detection.json

Прочитать файл плана и выполнить комплексное обнаружение уязвимостей:
- Просканировать всю кодовую базу
- Классифицировать по приоритету (critical → high → medium → low)
- Сгенерировать security-scan-report.md

Вернуться в основную сессию по завершении."
   ```

**Затем**: Дождаться возврата security-scanner с отчетом.

---

### Шаг 3: Контрольная точка 1 - Возобновить оркестратор (Проверка обнаружения)

После возврата security-scanner:

1. **Возобновить оркестратор** для проверки с помощью инструмента Task:
   ```
   subagent_type: "security-orchestrator"
   description: "Проверить обнаружение уязвимостей"
   prompt: "Выполнить контрольную точку 1: Проверка обнаружения

Фаза: Проверить вывод security-scanner

Задачи:
1. Проверить существование security-scan-report.md
2. Проверить структуру отчета (требуемые разделы)
3. Разобрать количество уязвимостей по приоритету
4. Запустить проверку типов (предупреждение без блокировки)
5. Сообщить результаты контрольной точки

ВАЖНО: После проверки, если уязвимости найдены:
- Создать .tmp/current/plans/security-fixing-{priority}.json для критического приоритета (или самого высокого доступного)
- Вернуть управление в основную сессию

Если уязвимости не найдены или все контрольные точки не пройдены:
- Перейти к итоговому отчету
- Вернуть управление

Вернуть следующее:
- Статус контрольной точки (✅ PASSED / ⛔ FAILED / ⚠️ WARNINGS)
- Количество безопасности по приоритету
- Следующая фаза: fixing-critical / fixing-high / final-summary
- Созданный файл плана (если применимо)
"
   ```

**Затем**: Дождаться результатов проверки оркестратора.

---

### Шаг 4: Фаза 2-5 - Этапы исправления (Итеративно)

После возврата оркестратора с планом исправления:

**Для каждого уровня приоритета** (critical → high → medium → low):

1. **Проверить, есть ли у этого приоритета уязвимости**:
   - Прочитать ответ оркестратора
   - Если оркестратор говорит "перейти к следующему приоритету" → продолжить цикл
   - Если оркестратор говорит "итоговый отчет" → перейти к шагу 5

2. **Прочитать план исправления**:
   ```
   Использовать инструмент Read: .tmp/current/plans/security-fixing-{priority}.json
   Проверить nextAgent === "vulnerability-fixer"
   Проверить config.priority === "{current-priority}"
   ```

3. **Вызвать vulnerability-fixer** с помощью инструмента Task:
   ```
   subagent_type: "vulnerability-fixer"
   description: "Исправить уязвимости {priority}"
   prompt: "Выполнить исправление уязвимостей на основе файла плана: .tmp/current/plans/security-fixing-{priority}.json

Прочитать файл плана и исправить уязвимости для приоритета: {priority}
- Прочитать security-scan-report.md для списка уязвимостей
- Исправлять уязвимости по одной
- Записывать изменения в .vulnerability-changes.json
- Обновить security-fixes-implemented.md (объединенный отчет)

Вернуться в основную сессию по завершении."
   ```

4. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "security-orchestrator"
   description: "Проверить исправления {priority}"
   prompt: "Выполнить контрольную точку 2: Проверка исправлений для приоритета={priority}

Задачи:
1. Проверить существование security-fixes-implemented.md
2. Запустить проверку типов (БЛОКИРУЕТСЯ)
3. Запустить сборку (БЛОКИРУЕТСЯ)
4. Разобрать процент успеха исправления
5. Проверить, нужна ли повторная попытка (если < 80% успеха)

Если проверка ПРОЙДЕНА и остались приоритеты:
- Создать следующий .tmp/current/plans/security-fixing-{priority}.json
- Вернуть управление

Если проверка НЕ ПРОЙДЕНА:
- Предоставить инструкции по откату
- Вернуть управление с ошибкой

Если все приоритеты выполнены:
- Перейти к фазе проверки
- Вернуть управление

Вернуть:
- Статус контрольной точки
- Процент успеха исправления
- Следующая фаза: fixing-{next-priority} / verification / final-summary
"
   ```

5. **Повторить** для следующего уровня приоритета.

---

### Шаг 5: Фаза 6 - Проверка

После завершения всех этапов исправления:

1. **Возобновить оркестратор** для проверки:
   ```
   subagent_type: "security-orchestrator"
   description: "Создать план проверки"
   prompt: "Выполнить фазу 6: Подготовка проверки

Создать .tmp/current/plans/security-verification.json для повторного сканирования кодовой базы.

Вернуть управление с путем к файлу плана."
   ```

2. **Вызвать security-scanner** для проверки:
   ```
   subagent_type: "security-scanner"
   description: "Проверочное сканирование"
   prompt: "Выполнить проверочное сканирование на основе: .tmp/current/plans/security-verification.json

Повторно просканировать кодовую базу для проверки исправлений. Перезаписывает security-scan-report.md.

Вернуться по завершении."
   ```

3. **Возобновить оркестратор** для проверки проверки:
   ```
   subagent_type: "security-orchestrator"
   description: "Проверить проверку"
   prompt: "Выполнить контрольную точку 3: Проверка проверки

Сравнить исходный security-scan-report.md (базовый) с новым сканированием:
- Подсчитать исправленные уязвимости
- Проверить, введены ли новые уязвимости
- Определить, нужна ли итерация

Вернуть:
- Статус проверки
- Оставшиеся безопасности
- Решение об итерации: итерировать / завершить
"
   ```

---

### Шаг 6: Итоговый отчет

После завершения всех фаз:

1. **Возобновить оркестратор** для итогового отчета:
   ```
   subagent_type: "security-orchestrator"
   description: "Сгенерировать итоговый отчет"
   prompt: "Выполнить фазу 8: Итоговый отчет

Сгенерировать комплексный security-orchestration-summary.md:
- Все обнаруженные уязвимости
- Все исправленные уязвимости
- Проценты успеха по приоритету
- Результаты проверки
- Итог итерации
- Инструкции по очистке

Вернуть итоговый отчет."
   ```

2. **Отобразить результаты** пользователю:
   ```
   Прочитать security-orchestration-summary.md
   Отобразить ключевые метрики
   Показать статус проверки
   Список следующих шагов
   ```

## Примечания по архитектуре

**Роль оркестратора**:
- Создает файлы плана
- Проверяет вывод работников
- Возвращает управление в основную сессию
- НЕТ прямого вызова работника

**Роль основной сессии** (эта команда):
- Читает файлы плана
- Вызывает работников через инструмент Task
- Возобновляет оркестратор для проверки
- Управляет полным циклом

**Роль работника**:
- Читает файл плана
- Выполняет работу
- Генерирует отчет
- Возвращается в основную сессию

Этот паттерн следует реальным возможностям Claude Code (без автовызова).
