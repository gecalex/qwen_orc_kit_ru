## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пусто).

## Цель

Определить несогласованности, дубликаты, двусмысленности и недостаточно определенные элементы в трех основных артефактах (`spec.md`, `plan.md`, `tasks.md`) перед реализацией. Эта команда ДОЛЖНА запускаться только после того, как `/speckit.tasks` успешно создал полный `tasks.md`.

## Ограничения эксплуатации

**СТРОГО ТОЛЬКО ЧТЕНИЕ**: НЕ изменять никакие файлы. Вывести структурированный отчет анализа. Предложить план устранения по желанию (пользователь должен явно одобрить, прежде чем какие-либо последующие команды редактирования будут вызваны вручную).

**Авторитет конституции**: Конституция проекта (`.specify/memory/constitution.md`) является **непреложной** в рамках этого анализа. Конфликты с конституцией автоматически являются КРИТИЧЕСКИМИ и требуют корректировки спецификации, плана или задач - не ослабления, переинтерпретации или молчаливого игнорирования принципа. Если сам принцип требует изменения, это должно происходить в отдельном, явном обновлении конституции вне `/speckit.analyze`.

## Шаги выполнения

### 1. Инициализировать контекст анализа

Запустить `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` один раз из корня репозитория и разобрать JSON для FEATURE_DIR и AVAILABLE_DOCS. Получить абсолютные пути:

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

Прервать с сообщением об ошибке, если какой-либо необходимый файл отсутствует (указать пользователю запустить отсутствующую команду-предпосылку).
Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте синтаксис экранирования: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

### 2. Загрузить артефакты (Поэтапное раскрытие)

Загрузить только минимально необходимый контекст из каждого артефакта:

**Из spec.md:**

- Обзор/Контекст
- Функциональные требования
- Нефункциональные требования
- Пользовательские истории
- Граничные случаи (если присутствуют)

**Из plan.md:**

- Выбор архитектуры/стека
- Ссылки на модель данных
- Фазы
- Технические ограничения

**Из tasks.md:**

- ID задач
- Описания
- Группировка по фазам
- Маркеры параллельности [P]
- Указанные пути файлов

**Из конституции:**

- Загрузить `.specify/memory/constitution.md` для проверки принципов

### 3. Построить семантические модели

Создать внутренние представления (не включать необработанные артефакты в вывод):

- **Инвентарь требований**: Каждое функциональное + нефункциональное требование с постоянным ключом (вывести slug на основе императивной фразы; например, "User can upload file" → `user-can-upload-file`)
- **Инвентарь пользовательских историй/действий**: Дискретные действия пользователя с критериями приемки
- **Сопоставление покрытия задачами**: Сопоставить каждую задачу с одним или несколькими требованиями или историями (вывод по ключевому слову / явные шаблоны ссылок, такие как ID или ключевые фразы)
- **Набор правил конституции**: Извлечь названия принципов и нормативные высказывания MUST/SHOULD

### 4. Проходы обнаружения (Анализ с эффективным использованием токенов)

Сосредоточиться на находках с высоким сигналом. Ограничить до 50 находок всего; объединить остаток в сводке переполнения.

#### A. Обнаружение дубликатов

- Определить почти дублирующиеся требования
- Отметить формулировки более низкого качества для объединения

#### B. Обнаружение двусмысленности

- Отметить расплывчатые прилагательные (fast, scalable, secure, intuitive, robust), не имеющие измеримых критериев
- Отметить нерешенные заполнители (TODO, TKTK, ???, `<placeholder>`, и т.д.)

#### C. Недостаточная спецификация

- Требования с глаголами, но без объекта или измеримого результата
- Пользовательские истории без соответствия критериям приемки
- Задачи, ссылающиеся на файлы или компоненты, не определенные в spec/plan

#### D. Соответствие конституции

- Любой элемент требования или плана, противоречащий принципу MUST
- Отсутствующие обязательные разделы или контрольные точки качества из конституции

#### E. Пробелы в покрытии

- Требования без связанных задач
- Задачи без сопоставленного требования/истории
- Нефункциональные требования, не отраженные в задачах (например, производительность, безопасность)

#### F. Несоответствие

- Семантический дрейф (одно и то же понятие названо по-разному в разных файлах)
- Сущности данных, упомянутые в плане, но отсутствующие в спецификации (или наоборот)
- Противоречия в порядке задач (например, задачи интеграции до задач фундаментальной настройки без заметки о зависимости)
- Конфликтующие требования (например, одно требует Next.js, а другое указывает Vue)

### 5. Назначение серьезности

Использовать эту эвристику для приоритезации находок:

- **КРИТИЧЕСКОЕ**: Нарушает принцип MUST конституции, отсутствует основной артефакт спецификации или требование с нулевым покрытием, блокирующее базовую функциональность
- **ВЫСОКОЕ**: Дублирующееся или конфликтующее требование, двусмысленный атрибут безопасности/производительности, непроверяемый критерий приемки
- **СРЕДНЕЕ**: Семантический дрейф, отсутствующее покрытие задач нефункционального характера, недостаточно определенный граничный случай
- **НИЗКОЕ**: Улучшения стиля/формулировки, незначительная избыточность, не влияющая на порядок выполнения

### 6. Произвести компактный отчет анализа

Вывести отчет в формате Markdown (без записи файлов) со следующей структурой:

## Отчет анализа спецификации

| ID | Категория | Серьезность | Местоположение(-я) | Краткое изложение | Рекомендация |
|----|----------|----------|-------------|---------|----------------|
| A1 | Дублирование | ВЫСОКОЕ | spec.md:L120-134 | Два похожих требования ... | Объединить формулировку; оставить более ясную версию |

(Добавить одну строку на каждую находку; генерировать стабильные ID с префиксом по начальной букве категории.)

**Таблица сводки покрытия:**

| Ключ требования | Есть задача? | ID задач | Примечания |
|-----------------|-----------|----------|-------|

**Проблемы соответствия конституции:** (если есть)

**Несопоставленные задачи:** (если есть)

**Метрики:**

- Всего требований
- Всего задач
- Процент покрытия (требования с >=1 задачей)
- Количество двусмысленностей
- Количество дубликатов
- Количество критических проблем

### 7. Предоставить следующие действия

В конце отчета вывести краткий блок "Следующие действия":

- Если существуют КРИТИЧЕСКИЕ проблемы: Рекомендовать решение до `/speckit.implement`
- Если только НИЗКИЕ/СРЕДНИЕ: Пользователь может продолжить, но предоставить предложения по улучшению
- Предоставить конкретные предложения команд: например, "Запустить /speckit.specify с уточнением", "Запустить /speckit.plan для корректировки архитектуры", "Вручную отредактировать tasks.md для добавления покрытия для 'performance-metrics'"

### 8. Предложить устранение

Спросить пользователя: "Хотели бы вы, чтобы я предложил конкретные исправляющие правки для топ N проблем?" (НЕ применять их автоматически.)

## Принципы работы

### Эффективность контекста

- **Минимальные токены с высоким сигналом**: Сосредоточиться на действенных находках, а не на исчерпывающей документации
- **Поэтапное раскрытие**: Загружать артефакты постепенно; не сбрасывать весь контент в анализ
- **Эффективный вывод токенов**: Ограничить таблицу находок 50 строками; суммировать переполнение
- **Детерминистические результаты**: Повторный запуск без изменений должен давать согласованные ID и подсчеты

### Руководящие принципы анализа

- **НИКОГДА не изменять файлы** (это анализ только для чтения)
- **НИКОГДА не выдумывать отсутствующие разделы** (если отсутствуют, сообщить об этом точно)
- **Приоритизировать нарушения конституции** (они всегда КРИТИЧЕСКИЕ)
- **Использовать примеры вместо исчерпывающих правил** (цитировать конкретные случаи, а не общие шаблоны)
- **Корректно сообщать об отсутствии проблем** (выдать отчет об успехе со статистикой покрытия)
