## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пусто).

## План действий

Вы обновляете конституцию проекта в `.specify/memory/constitution.md`. Этот файл является ШАБЛОНОМ, содержащим токены-заполнители в квадратных скобках (например, `[PROJECT_NAME]`, `[PRINCIPLE_1_NAME]`). Ваша задача - (а) собрать/вывести конкретные значения, (б) точно заполнить шаблон и (в) распространить любые изменения на зависимые артефакты.

Следуйте этому рабочему процессу:

1. Загрузить существующий шаблон конституции в `.specify/memory/constitution.md`.
   - Определить каждый токен-заполнитель вида `[ALL_CAPS_IDENTIFIER]`.
   **ВАЖНО**: Пользователь может потребовать меньше или больше принципов, чем использовано в шаблоне. Если число указано, уважайте это - следуйте общему шаблону. Вы обновите документ соответственно.

2. Собрать/вывести значения для заполнителей:
   - Если ввод пользователя (беседа) предоставляет значение, используйте его.
   - В противном случае выведите из существующего контекста репозитория (README, документы, предыдущие версии конституции, если встроены).
   - Для дат управления: `RATIFICATION_DATE` - это первоначальная дата принятия (если неизвестна, спросите или пометьте TODO), `LAST_AMENDED_DATE` - сегодня, если вносятся изменения, иначе сохраните предыдущую.
   - `CONSTITUTION_VERSION` должен увеличиваться по правилам семантического версионирования:
     - MAJOR: Несовместимые изменения управления/удаления принципов или переопределения.
     - MINOR: Добавлен новый принцип/раздел или материально расширенные рекомендации.
     - PATCH: Уточнения, формулировки, исправления опечаток, несемантические уточнения.
   - Если тип увеличения версии неясен, предложите обоснование перед финализацией.

3. Подготовить обновленное содержание конституции:
   - Заменить каждый заполнитель конкретным текстом (без оставшихся токенов в скобках, кроме намеренно оставленных слотов шаблона, которые проект решил пока не определять - явно обоснуйте любые оставшиеся).
   - Сохранить иерархию заголовков, комментарии можно удалить после замены, если они не добавляют поясняющего руководства.
   - Обеспечить, чтобы каждый раздел Принципа: краткая строка названия, абзац (или список с маркерами) фиксировал непреложные правила, явное обоснование, если не очевидно.
   - Обеспечить, чтобы раздел Управления перечислял процедуру внесения изменений, политику версионирования и ожидания проверки соответствия.

4. Контрольный список распространения согласованности (преобразовать предыдущий контрольный список в активные проверки):
   - Прочитать `.specify/templates/plan-template.md` и убедиться, что любая "Проверка конституции" или правила соответствуют обновленным принципам.
   - Прочитать `.specify/templates/spec-template.md` для согласования объема/требований - обновить, если конституция добавляет/удаляет обязательные разделы или ограничения.
   - Прочитать `.specify/templates/tasks-template.md` и убедиться, что категоризация задач отражает новые или удаленные типы задач, основанные на принципах (например, наблюдаемость, версионирование, дисциплина тестирования).
   - Прочитать каждый файл команды в `.specify/templates/commands/*.md` (включая этот) для проверки отсутствия устаревших ссылок (только имена агентов, такие как CLAUDE), когда требуется общее руководство.
   - Прочитать любые документы руководства во время выполнения (например, `README.md`, `docs/quickstart.md` или файлы руководства конкретных агентов, если присутствуют). Обновить ссылки на измененные принципы.

5. Создать Отчет о синхронизации воздействия (добавить в виде HTML-комментария в начале файла конституции после обновления):
   - Изменение версии: старая → новая
   - Список измененных принципов (старое название → новое название, если переименовано)
   - Добавленные разделы
   - Удаленные разделы
   - Шаблоны, требующие обновления (✅ обновлено / ⚠ ожидание) с путями к файлам
   - Последующие задачи TODO, если какие-либо заполнители намеренно отложены.

6. Проверка перед окончательным выводом:
   - Нет оставшихся необъясненных токенов в скобках.
   - Строка версии соответствует отчету.
   - Даты в формате ISO YYYY-MM-DD.
   - Принципы декларативны, тестируемы и не содержат расплывчатого языка ("should" → заменить на обоснование MUST/SHOULD там, где это уместно).

7. Записать завершенную конституцию обратно в `.specify/memory/constitution.md` (перезаписать).

8. Вывести итоговый обзор пользователю с:
   - Новой версией и обоснованием увеличения.
   - Любыми файлами, помеченными для ручного последующего выполнения.
   - Предлагаемым сообщением коммита (например, `docs: amend constitution to vX.Y.Z (principle additions + governance update)`).

Требования к форматированию и стилю:

- Использовать заголовки Markdown точно как в шаблоне (не понижать/повышать уровни).
- Переносить длинные строки обоснования для сохранения читаемости (<100 символов идеально), но не применять жестко неудобные разрывы.
- Оставлять одну пустую строку между разделами.
- Избегать концевых пробелов.

Если пользователь предоставляет частичные обновления (например, только одну ревизию принципа), все равно выполняйте проверки и принимайте решения о версии.

Если критическая информация отсутствует (например, дата ратификации действительно неизвестна), вставьте `TODO(<FIELD_NAME>): explanation` и включите в Отчет о синхронизации воздействия в отложенные элементы.

Не создавайте новый шаблон; всегда работайте с существующим файлом `.specify/memory/constitution.md`.
