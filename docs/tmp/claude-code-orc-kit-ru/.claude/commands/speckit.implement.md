> **НАПОМИНАНИЕ ОРКЕСТРАЦИИ**: Вы оркестратор, а не исполнитель. Делегируйте все сложные задачи субагентам с полным контекстом. Собирайте полный контекст перед делегированием (читайте код, ищите паттерны, просматривайте документы, проверяйте коммиты). Тщательно проверяйте результаты (читайте файлы, запускайте проверку типов). Переделегируйте с исправлениями, если проверка не проходит. Выполняйте напрямую только тривиальные задачи (исправления в 1-2 строки, импорты, одиночный npm install).

## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пустой).

## Структура

0. **Проверка сессии** (DeksdenFlow):
   - Вызовите навык `resume-session`, чтобы проверить наличие существующей сессии
   - Если найдена действительная сессия: спросите пользователя "Возобновить предыдущую сессию или начать с нуля?"
   - Если возобновление: загрузите context.md и session-log.md, перейдите к сохраненной фазе
   - Если с нуля или нет сессии: переходите к шагу 1
   - **Также**: Вызовите навык `load-project-context`, чтобы загрузить карту структуры проекта

1. Запустите `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` из корня репозитория и разберите список FEATURE_DIR и AVAILABLE_DOCS. Все пути должны быть абсолютными. Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте синтаксис экранирования: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

2. **Проверка статуса чек-листов** (если существует FEATURE_DIR/checklists/):
   - Просканируйте все файлы чек-листов в директории checklists/
   - Для каждого чек-листа подсчитайте:
     - Всего элементов: Все строки, соответствующие `- [ ]` или `- [X]` или `- [x]`
     - Завершенных элементов: Строки, соответствующие `- [X]` или `- [x]`
     - Незавершенных элементов: Строки, соответствующие `- [ ]`
   - Создайте таблицу статуса:

     ```text
     | Чек-лист | Всего | Завершено | Незавершено | Статус |
     |----------|-------|-----------|-------------|--------|
     | ux.md    | 12    | 12        | 0           | ✓ PASS |
     | test.md  | 8     | 5         | 3           | ✗ FAIL |
     | security.md | 6  | 6         | 0           | ✓ PASS |
     ```

   - Рассчитайте общий статус:
     - **PASS**: Все чек-листы имеют 0 незавершенных элементов
     - **FAIL**: Один или несколько чек-листов имеют незавершенные элементы

   - **Если какой-либо чек-лист неполный**:
     - Отобразите таблицу с количеством незавершенных элементов
     - **ОСТАНОВИТЬСЯ** и спросить: "Некоторые чек-листы неполные. Хотите все равно продолжить реализацию? (yes/no)"
     - Дождитесь ответа пользователя перед продолжением
     - Если пользователь говорит "no" или "wait" или "stop", остановить выполнение
     - Если пользователь говорит "yes" или "proceed" или "continue", переходите к шагу 3

   - **Если все чек-листы завершены**:
     - Отобразите таблицу, показывающую, что все чек-листы пройдены
     - Автоматически переходите к шагу 3

3. Загрузите и проанализируйте контекст реализации:
   - **ОБЯЗАТЕЛЬНО**: Прочитайте `.specify/memory/constitution.md` для принципов проекта, требований безопасности и контрольных точек качества
   - **ОБЯЗАТЕЛЬНО**: Прочитайте tasks.md для полного списка задач и плана выполнения
   - **ОБЯЗАТЕЛЬНО**: Прочитайте plan.md для технологического стека, архитектуры и структуры файлов
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте data-model.md для сущностей и отношений
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте contracts/ для спецификаций API и требований к тестированию
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте research.md для технических решений и ограничений
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте research/ для сложных исследовательских находок и решений
   - **ЕСЛИ СУЩЕСТВУЕТ**: Прочитайте quickstart.md для сценариев интеграции

4. **ФАЗА ПЛАНИРОВАНИЯ** (Выполнить перед реализацией):
   - Просмотрите все задачи и классифицируйте модель выполнения (параллельная против последовательной)
   - **Шаг 1: Анализ задач**:
     * Проанализируйте все задачи и определите требуемые возможности агента
     * Определите, какие задачи требуют MAIN (только тривиальные), существующие агенты или новые агенты
     * Создайте список отсутствующих типов агентов с техническими характеристиками
   - **Шаг 2: Создание агента** (если необходимо):
     * Запустите N вызовов meta-agent-v3 в одном сообщении (1 вызов на отсутствующего агента)
     * После создания агента: попросите пользователя перезапустить claude-code
     * После перезапуска: проверьте существование новых агентов перед продолжением
   - **Шаг 3: Назначение исполнителя**:
     * [EXECUTOR: MAIN] - ТОЛЬКО для тривиальных задач (исправления в 1-2 строки, простые импорты, установка одной зависимости)
     * [EXECUTOR: existing-agent] - ТОЛЬКО если 100% совпадение возможностей после тщательного изучения
     * [EXECUTOR: specific-agent-name] - Для всех остальных задач с использованием существующих или вновь созданных агентов
     * Аннотируйте все задачи с `[EXECUTOR: name]` и `[SEQUENTIAL]`/`[PARALLEL-GROUP-X]`
   - **Шаг 4: Решение исследований**:
     * Простые исследования: решать с помощью инструментов агента (Grep, Read, WebSearch, Context7, документы Supabase)
     * Сложные исследования: создать промпт исследования в research/, дождаться глубокого исследования пользователя, включить результаты
   - Вывод: Обновленный tasks.md с аннотациями исполнителя
   - **Правило атомарности (КРИТИЧНО)**: 1 Задача = 1 Вызов агента
     * Никогда не давайте несколько задач одному агенту за один запуск
     * **Параллельное выполнение**: Запуск N вызовов агента в одном сообщении (не последовательно)
     * Пример: 3 параллельные задачи для meta-agent → 3 вызова meta-agent в одном сообщении
     * Пример: 5 параллельных задач для fullstack → 5 вызовов fullstack в одном сообщении
     * Последовательные задачи: 1 запуск агента, дождаться завершения, затем следующий запуск агента

5. **Проверка настройки проекта**:
   - **ОБЯЗАТЕЛЬНО**: Создать/проверить файлы игнорирования на основе фактической настройки проекта:

   **Логика обнаружения и создания**:
   - Проверьте, успешен ли следующий команды, чтобы определить, является ли репозиторий git-репозиторием (создать/проверить .gitignore если да):

     ```sh
     git rev-parse --git-dir 2>/dev/null
     ```

   - Проверьте, существует ли Dockerfile* или Docker в plan.md → создать/проверить .dockerignore
   - Проверьте, существует ли .eslintrc* или eslint.config.* → создать/проверить .eslintignore
   - Проверьте, существует ли .prettierrc* → создать/проверить .prettierignore
   - Проверьте, существует ли .npmrc или package.json → создать/проверить .npmignore (если публикация)
   - Проверьте, существуют ли файлы terraform (*.tf) → создать/проверить .terraformignore
   - Проверьте, нужен ли .helmignore (наличие helm-чартов) → создать/проверить .helmignore

   **Если файл игнорирования отсутствует**: Создайте с полным набором паттернов для обнаруженной технологии

6. Разберите структуру tasks.md и извлеките:
   - **Фазы задач**: Установка, Тесты, Ядро, Интеграция, Полировка
   - **Зависимости задач**: Правила последовательного против параллельного выполнения
   - **Детали задач**: ID, описание, пути к файлам, маркеры параллельности [P]
   - **Поток выполнения**: Порядок и требования зависимости

7. Выполните реализацию в соответствии с планом задач:
   - **Обнаружение задач**: Найти ПЕРВУЮ незавершенную задачу (соблюдая порядок фаз)
   - **Выполнение по фазам**: Завершите каждую фазу перед переходом к следующей
   - **Уважение зависимостей**: Выполняйте последовательные задачи по порядку, параллельные задачи [P] могут выполняться вместе
   - **Следование подходу TDD**: Выполняйте задачи тестирования перед соответствующими задачами реализации
   - **Координация на основе файлов**: Задачи, влияющие на одни и те же файлы, должны выполняться последовательно
   - **Контрольные точки проверки**: Проверяйте завершение каждой фазы перед продолжением

8. Атомарное выполнение задач (на задачу):
   1. ОБНОВИТЬ TODO: Отметить задачу `in_progress` в TodoWrite
   2. ПРОВЕРИТЬ ИСПОЛНИТЕЛЯ:
      - [EXECUTOR: MAIN]? → Выполнить напрямую, если тривиально, иначе делегировать
      - [EXECUTOR: subagent-name]? → Делегировать указанному субагенту
   3. СОБРАТЬ КОНТЕКСТ: Прочитать существующий код, искать паттерны, просматривать документы, проверять коммиты
   3.5. ПОИСК БИБЛИОТЕКИ: Перед написанием >20 строк нового кода, искать существующие npm/pypi пакеты
      - Использовать WebSearch + Context7 для поиска и оценки библиотек
      - Если подходящая библиотека найдена: установить и настроить вместо реализации с нуля
      - Проверить: еженедельные загрузки >1000, недавние коммиты, поддержка TypeScript, отсутствие критических уязвимостей
   3.6. ПОЛУЧИТЬ ДОКУМЕНТАЦИЮ БИБЛИОТЕКИ (ОБЯЗАТЕЛЬНО): Перед написанием кода, использующего ЛЮБУЮ библиотеку:
      - Вызвать `mcp__context7__resolve-library-id`, чтобы получить ID библиотеки
      - Вызвать `mcp__context7__get-library-docs` с соответствующей темой (например, "hooks", "routing", "auth")
      - Использовать полученные документы, чтобы обеспечить правильное использование API и избежать устаревших паттернов
      - Это применяется к React, Next.js, Supabase, Zod, tRPC и ВСЕМ другим библиотекам
   4. ВЫПОЛНИТЬ:
      - Напрямую: Использовать инструменты Edit/Write только для тривиальных задач
      - Делегировано: Запустить инструмент Task с полным контекстом (фрагменты кода, пути к файлам, паттерны, критерии проверки)
   5. ПРОВЕРИТЬ: Прочитать ВСЕ измененные файлы, запустить проверку типов, проверить наличие регрессий
   6. ПРИНЯТЬ/ОТКЛОНИТЬ:
      - Принять? → Продолжить до шага 7
      - Отклонить? → Переделегировать с исправлениями и сообщениями об ошибках, перейти к шагу 4
   7. ОБНОВИТЬ TODO: Отметить задачу `completed` в TodoWrite
   8. ОБНОВИТЬ tasks.md: Отметить задачу `[X]`, добавить артефакты: `→ Артефакты: [file1](path), [file2](path)`

   10. СЛЕДУЮЩАЯ ЗАДАЧА: Перейти к следующей незавершенной задаче, перейти к шагу 1

9. Правила выполнения реализации:
   - **Сначала настройка**: Инициализировать структуру проекта, зависимости, конфигурацию
   - **Тесты перед кодом**: Если вам нужно написать тесты для контрактов, сущностей и сценариев интеграции
   - **Основная разработка**: Реализовать модели, сервисы, команды CLI, конечные точки
   - **Интеграционная работа**: Подключения к базе данных, промежуточное ПО, ведение журналов, внешние сервисы
   - **Полировка и проверка**: Модульные тесты, оптимизация производительности, документация

10. Отслеживание прогресса и обработка ошибок:
   - Сообщать о прогрессе после каждой завершенной задачи
   - **Фиксировать после каждой задачи**: Запустить `/push patch` перед переходом к следующей
   - Остановить выполнение, если какая-либо непараллельная задача не удается
   - Для параллельных задач [P], продолжить с успешными задачами, сообщить о неудавшихся
   - Предоставлять четкие сообщения об ошибках с контекстом для отладки
   - Предлагать следующие шаги, если реализация не может продолжаться
   - **ВАЖНО** Для завершенных задач, обязательно отмечайте задачу как [X] в файле задач
   - **Контекст сессии** (DeksdenFlow):
     * После завершения каждой ФАЗЫ (не задачи): вызвать навык `save-session-context`
     * При ошибке или паузе: сохранить контекст перед остановкой
     * Записывать важные решения в session-log.md (архитектурные выборы, решения проблем)
   - **Критические правила**:
     * НИКОГДА не пропускать проверку
     * НИКОГДА не продолжать, если задача не удалась
     * НИКОГДА не объединять фиксации (1 задача = 1 фиксация)
     * ОДНА задача in_progress одновременно (атомарное выполнение)

11. Проверка завершения:
   - Проверить, что все необходимые задачи завершены
   - Проверить, что реализованные функции соответствуют исходной спецификации
   - Проверить, что тесты проходят и покрытие соответствует требованиям
   - Подтвердить, что реализация следует техническому плану
   - Сообщить о конечном статусе с резюме выполненной работы
