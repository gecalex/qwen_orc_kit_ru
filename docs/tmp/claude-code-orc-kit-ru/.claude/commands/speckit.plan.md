## План действий

1. **Настройка**: Запустить `.specify/scripts/bash/setup-plan.sh --json` из корня репозитория и разобрать JSON для FEATURE_SPEC, IMPL_PLAN, SPECS_DIR, BRANCH. Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте синтаксис экранирования: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

2. **Загрузить контекст**: Прочитать FEATURE_SPEC и `.specify/memory/constitution.md`. Загрузить шаблон IMPL_PLAN (уже скопирован).

3. **Выполнить рабочий процесс планирования**: Следуйте структуре в шаблоне IMPL_PLAN для:
   - Заполнить технический контекст (отметить неизвестные как "NEEDS CLARIFICATION")
   - Заполнить раздел проверки конституции из конституции
   - Оценить контрольные точки (ОШИБКА при необоснованных нарушениях)
   - Фаза 0: Сгенерировать research.md (разрешить все NEEDS CLARIFICATION)
   - Фаза 1: Сгенерировать data-model.md, contracts/, quickstart.md
   - Фаза 1: Обновить контекст агента, запустив сценарий агента
   - Переоценить проверку конституции после дизайна

4. **Остановиться и сообщить**: Команда завершается после планирования фазы 2. Сообщить о ветке, пути IMPL_PLAN и сгенерированных артефактах.

## Фазы

### Фаза 0: Обзор и исследование

1. **Извлечь неизвестные из технического контекста** выше:
   - Для каждого NEEDS CLARIFICATION → задача исследования
   - Для каждой зависимости → задача лучших практик
   - Для каждой интеграции → задача паттернов

2. **Поиск библиотек в первую очередь** (ОБЯЗАТЕЛЬНО перед исследованием):
   - Для каждого основного компонента (>20 строк ожидается), искать существующие библиотеки:
     - WebSearch: "npm {функциональность} library 2024" или "python {функциональность} package"
     - Context7: документация для кандидатских библиотек
     - Проверить: еженедельные загрузки >1000, коммиты за последние 6 месяцев, поддержка TypeScript/типов
   - Документировать решения по библиотекам в research.md

3. **Сгенерировать и направить агентов исследования**:

   ```text
   Для каждого неизвестного в техническом контексте:
     Задача: "Исследовать {неизвестное} для {контекста функции}"
   Для каждого выбора технологии:
     Задача: "Найти лучшие практики для {технологии} в {домене}"
   ```

4. **Рассмотрения исследования**:
   - **Простое исследование**: Вопросы, решаемые с помощью доступных инструментов (Grep, Read, WebSearch, Context7) - решить немедленно
   - **Сложное исследование**: Требует глубокого расследования → создать запрос исследования в `FEATURE_DIR/research/` для инструмента deepresearch

5. **Объединить результаты** в `research.md` с использованием формата:
   - Решение: [что было выбрано]
   - Обоснование: [почему выбрано]
   - Рассмотренные альтернативы: [что еще оценивалось]
   - Библиотека: [если применимо - название, версия, почему выбрана вместо альтернатив]

**Вывод**: research.md со всеми NEEDS CLARIFICATION решенными, research/ с запросами сложного исследования (если есть)

### Фаза 1: Дизайн и контракты

**Предварительные условия:** `research.md` завершен

1. **Извлечь сущности из спецификации функции** → `data-model.md`:
   - Имя сущности, поля, отношения
   - Правила валидации из требований
   - Переходы состояний, если применимо

2. **Сгенерировать контракты API** из функциональных требований:
   - Для каждого действия пользователя → конечная точка
   - Использовать стандартные паттерны REST/GraphQL
   - Вывести схему OpenAPI/GraphQL в `/contracts/`

3. **Обновление контекста агента**:
   - Запустить `.specify/scripts/bash/update-agent-context.sh claude`
   - Эти сценарии определяют, какой агент ИИ используется
   - Обновить соответствующий файл контекста агента
   - Добавить только новые технологии из текущего плана
   - Сохранить ручные добавления между маркерами

**Вывод**: data-model.md, /contracts/*, quickstart.md, файл, специфичный для агента

## Ключевые правила

- Использовать абсолютные пути
- ОШИБКА при сбоях контрольных точек или нерешенных уточнениях
- Библиотеки в первую очередь: Всегда проверять существующие библиотеки перед планированием пользовательских реализаций
