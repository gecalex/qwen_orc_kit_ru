## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пусто).

## План действий

Текст, который пользователь ввел после `/speckit.specify` в вызывающем сообщении, **является** описанием функции. Предполагайте, что он всегда доступен в этом разговоре, даже если `$ARGUMENTS` появляется буквально ниже. Не просите пользователя повторить его, если они предоставили пустую команду.

Учитывая это описание функции, сделайте следующее:

1. **Создать краткое короткое имя** (2-4 слова) для ветки:
   - Проанализировать описание функции и извлечь наиболее значимые ключевые слова
   - Создать 2-4 словесное короткое имя, которое отражает суть функции
   - Использовать формат действие-существительное, когда возможно (например, "add-user-auth", "fix-payment-bug")
   - Сохранить технические термины и аббревиатуры (OAuth2, API, JWT и т.д.)
   - Сделать его кратким, но достаточно описательным, чтобы понять функцию с первого взгляда
   - Примеры:
     - "I want to add user authentication" → "user-auth"
     - "Implement OAuth2 integration for the API" → "oauth2-api-integration"
     - "Create a dashboard for analytics" → "analytics-dashboard"
     - "Fix payment processing timeout bug" → "fix-payment-timeout"

2. **Проверить наличие существующих веток перед созданием новой**:

   a. Сначала получить все удаленные ветки, чтобы убедиться, что у нас есть самая последняя информация:

      ```bash
      git fetch --all --prune
      ```

   b. Найти самый высокий номер функции среди всех источников для короткого имени:
      - Удаленные ветки: `git ls-remote --heads origin | grep -E 'refs/heads/[0-9]+-<short-name>$'`
      - Локальные ветки: `git branch | grep -E '^[* ]*[0-9]+-<short-name>$'`
      - Каталоги спецификаций: Проверить каталоги, соответствующие `specs/[0-9]+-<short-name>`

   c. Определить следующий доступный номер:
      - Извлечь все номера из всех трех источников
      - Найти самый высокий номер N
      - Использовать N+1 для нового номера ветки

   d. Запустить скрипт `.specify/scripts/bash/create-new-feature.sh --json "$ARGUMENTS"` с вычисленным номером и коротким именем:
      - Передать `--number N+1` и `--short-name "your-short-name"` вместе с описанием функции
      - Пример bash: `.specify/scripts/bash/create-new-feature.sh --json --number 5 --short-name "user-auth" "Add user authentication"`
      - Пример PowerShell: `.specify/scripts/powershell/create-new-feature.ps1 -Json -Number 5 -ShortName "user-auth" "Add user authentication"`

   **ВАЖНО**:
   - Проверить все три источника (удаленные ветки, локальные ветки, каталоги спецификаций), чтобы найти самый высокий номер
   - Сопоставлять только ветки/каталоги с точным шаблоном короткого имени
   - Если не найдены существующие ветки/каталоги с этим коротким именем, начать с номера 1
   - Вы должны запускать этот скрипт только один раз на функцию
   - JSON предоставляется в терминале в качестве вывода - всегда обращайтесь к нему, чтобы получить фактическое содержимое, которое вы ищете
   - Вывод JSON будет содержать пути BRANCH_NAME и SPEC_FILE
   - Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте синтаксис экранирования: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot")

3. Загрузить `.specify/templates/spec-template.md`, чтобы понять требуемые разделы.

4. Следовать этому рабочему процессу:

    1. Разобрать описание пользователя из ввода
       Если пусто: ОШИБКА "Описание функции не предоставлено"
    2. Извлечь ключевые концепции из описания
       Определить: актеров, действия, данные, ограничения
    3. Для неясных аспектов:
       - Сделать обоснованные предположения на основе контекста и отраслевых стандартов
       - Отмечать только [NEEDS CLARIFICATION: конкретный вопрос], если:
         - Выбор значительно влияет на объем функции или опыт пользователя
         - Существует несколько разумных интерпретаций с разными последствиями
         - Не существует разумного умолчания
       - **ОГРАНИЧЕНИЕ: Максимум 3 маркера [NEEDS CLARIFICATION] всего**
       - Приоритизировать уточнения по влиянию: объем > безопасность/конфиденциальность > опыт пользователя > технические детали
       - **Примечание**: Если уточнения раскрывают сложные неизвестности, отметить для фазы исследования
    4. Заполнить раздел сценариев и тестирования пользователей
       Если нет ясного пользовательского потока: ОШИБКА "Невозможно определить сценарии пользователей"
    5. Сгенерировать функциональные требования
       Каждое требование должно быть тестируемым
       Использовать разумные умолчания для неуказанных деталей (документировать предположения в разделе предположений)
    6. Определить критерии успеха
       Создать измеримые, независимые от технологии результаты
       Включить как количественные метрики (время, производительность, объем), так и качественные меры (удовлетворенность пользователя, выполнение задач)
       Каждый критерий должен быть проверяемым без деталей реализации
    7. Определить ключевые сущности (если связаны с данными)
    8. Вернуть: УСПЕХ (спецификация готова к планированию)

5. Записать спецификацию в SPEC_FILE, используя структуру шаблона, заменяя заполнители конкретными деталями, полученными из описания функции (аргументы), при этом сохраняя порядок разделов и заголовки.

6. **Проверка качества спецификации**: После написания первоначальной спецификации проверить ее по критериям качества:

   a. **Создать контрольный список качества спецификации**: Сгенерировать файл контрольного списка в `FEATURE_DIR/checklists/requirements.md`, используя структуру шаблона контрольного списка с этими элементами проверки:

      ```markdown
      # Контрольный список качества спецификации: [НАЗВАНИЕ ФУНКЦИИ]

      **Назначение**: Проверить полноту и качество спецификации перед переходом к планированию
      **Создано**: [ДАТА]
      **Функция**: [Ссылка на spec.md]

      ## Качество содержимого

      - [ ] Нет деталей реализации (языки, фреймворки, API)
      - [ ] Сосредоточено на ценности для пользователя и бизнес-потребностях
      - [ ] Написано для нетехнических заинтересованных сторон
      - [ ] Все обязательные разделы завершены

      ## Полнота требований

      - [ ] Не осталось маркеров [NEEDS CLARIFICATION]
      - [ ] Требования тестируемы и однозначны
      - [ ] Критерии успеха измеримы
      - [ ] Критерии успеха не зависят от технологии (без деталей реализации)
      - [ ] Все сценарии приемки определены
      - [ ] Граничные случаи идентифицированы
      - [ ] Объем четко ограничен
      - [ ] Зависимости и предположения идентифицированы

      ## Готовность функции

      - [ ] Все функциональные требования имеют четкие критерии приемки
      - [ ] Сценарии пользователей охватывают основные потоки
      - [ ] Функция соответствует измеримым результатам, определенным в критериях успеха
      - [ ] Детали реализации не просачиваются в спецификацию

      ## Примечания

      - Элементы, отмеченные как неполные, требуют обновления спецификации перед `/speckit.clarify` или `/speckit.plan`
      ```

   b. **Выполнить проверку по контрольному списку**: Проверить спецификацию по каждому элементу контрольного списка:
      - Для каждого элемента определить, пройден ли он или нет
      - Документировать найденные конкретные проблемы (цитировать соответствующие разделы спецификации)

   c. **Обработать результаты проверки**:

      - **Если все элементы пройдены**: Отметить контрольный список как завершенный и перейти к шагу 6

      - **Если элементы не пройдены (исключая [NEEDS CLARIFICATION])**:
        1. Перечислить непройденные элементы и конкретные проблемы
        2. Обновить спецификацию, чтобы устранить каждую проблему
        3. Повторно запустить проверку до тех пор, пока все элементы не будут пройдены (максимум 3 итерации)
        4. Если все еще не проходят после 3 итераций, документировать оставшиеся проблемы в примечаниях контрольного списка и предупредить пользователя

      - **Если маркеры [NEEDS CLARIFICATION] остаются**:
        1. Извлечь все маркеры [NEEDS CLARIFICATION: ...] из спецификации
        2. **ПРОВЕРКА ОГРАНИЧЕНИЯ**: Если существует более 3 маркеров, оставить только 3 самых критических (по влиянию на объем/безопасность/UX) и сделать обоснованные предположения для остальных
        3. Для каждого необходимого уточнения (максимум 3) представить пользователю варианты в этом формате:

           ```markdown
           ## Вопрос [N]: [Тема]

           **Контекст**: [Цитата соответствующего раздела спецификации]

           **Что мы должны знать**: [Конкретный вопрос из маркера NEEDS CLARIFICATION]

           **Предлагаемые ответы**:

           | Вариант | Ответ | Последствия |
           |--------|--------|--------------|
           | A      | [Первый предлагаемый ответ] | [Что это значит для функции] |
           | B      | [Второй предлагаемый ответ] | [Что это значит для функции] |
           | C      | [Третий предлагаемый ответ] | [Что это значит для функции] |
           | Custom | Предоставьте свой собственный ответ | [Объясните, как предоставить пользовательский ввод] |

           **Ваш выбор**: _[Дождитесь ответа пользователя]_
           ```

        4. **КРИТИЧНО - Форматирование таблицы**: Убедиться, что таблицы markdown правильно отформатированы:
           - Использовать последовательные пробелы с выровненными вертикальными линиями
           - Каждая ячейка должна иметь пробелы вокруг содержимого: `| Content |`, а не `|Content|`
           - Разделитель заголовков должен иметь не менее 3 дефисов: `|--------|`
           - Проверить, что таблица правильно отображается в предварительном просмотре markdown
        5. Нумеровать вопросы последовательно (Q1, Q2, Q3 - максимум 3 всего)
        6. Представить все вопросы вместе перед ожиданием ответов
        7. Дождаться, пока пользователь не ответит своим выбором для всех вопросов (например, "Q1: A, Q2: Custom - [детали], Q3: B")
        8. Обновить спецификацию, заменив каждый маркер [NEEDS CLARIFICATION] выбранным или предоставленным пользователем ответом
        9. Повторно запустить проверку после разрешения всех уточнений

   d. **Обновить контрольный список**: После каждой итерации проверки обновить файл контрольного списка с текущим статусом пройдено/не пройдено

7. Сообщить о завершении с именем ветки, путем к файлу спецификации, результатами контрольного списка и готовностью к следующей фазе (`/speckit.clarify` или `/speckit.plan`).

**ПРИМЕЧАНИЕ:** Скрипт создает и переключается на новую ветку и инициализирует файл спецификации перед записью.

## Общие рекомендации

## Краткие рекомендации

- Сосредоточьтесь на **ЧЕМ** нуждаются пользователи и **ПОЧЕМУ**.
- Избегайте ОПИСАНИЯ реализации (нет стека технологий, API, структуры кода).
- Написано для бизнес-заинтересованных сторон, а не для разработчиков.
- НЕ создавайте какие-либо контрольные списки, встроенные в спецификацию. Это будет отдельной командой.

### Требования к разделам

- **Обязательные разделы**: Должны быть заполнены для каждой функции
- **Необязательные разделы**: Включать только когда релевантны для функции
- Когда раздел не применяется, полностью удалить его (не оставлять как "N/A")

### Для генерации ИИ

При создании этой спецификации из пользовательского запроса:

1. **Делать обоснованные предположения**: Использовать контекст, отраслевые стандарты и общие паттерны для заполнения пробелов
2. **Документировать предположения**: Записывать разумные умолчания в раздел предположений
3. **Ограничивать уточнения**: Максимум 3 маркера [NEEDS CLARIFICATION] - использовать только для критических решений, которые:
   - Существенно влияют на объем функции или опыт пользователя
   - Имеют несколько разумных интерпретаций с разными последствиями
   - Не имеют никакого разумного умолчания
4. **Приоритизировать уточнения**: объем > безопасность/конфиденциальность > опыт пользователя > технические детали
5. **Думать как тестировщик**: Каждое расплывчатое требование должно не пройти элемент проверки "тестируемое и однозначное"
6. **Общие области, требующие уточнения** (только если не существует разумного умолчания):
   - Объем функции и границы (включать/исключать конкретные варианты использования)
   - Типы пользователей и права доступа (если возможно несколько конфликтующих интерпретаций)
   - Требования безопасности/соответствия (когда юридически/финансово значимо)

**Примеры разумных умолчаний** (не спрашивать об этих):

- Хранение данных: Отраслевые стандартные практики для домена
- Цели производительности: Стандартные ожидания веб-/мобильного приложения, если не указано иное
- Обработка ошибок: Понятные сообщения для пользователя с соответствующими запасными вариантами
- Метод аутентификации: Стандартная сессия или OAuth2 для веб-приложений
- Паттерны интеграции: RESTful API, если не указано иное

### Руководящие принципы критериев успеха

Критерии успеха должны быть:

1. **Измеримыми**: Включать конкретные метрики (время, процент, количество, скорость)
2. **Независимыми от технологии**: Без упоминания фреймворков, языков, баз данных или инструментов
3. **Ориентированными на пользователя**: Описывать результаты с точки зрения пользователя/бизнеса, а не внутренних системных компонентов
4. **Проверяемыми**: Могут быть протестированы/проверены без знания деталей реализации

**Хорошие примеры**:

- "Пользователи могут завершить оформление заказа менее чем за 3 минуты"
- "Система поддерживает 10 000 одновременных пользователей"
- "95% поисков возвращают результаты менее чем за 1 секунду"
- "Ставка завершения задач повышается на 40%"

**Плохие примеры** (ориентированные на реализацию):

- "Время ответа API менее 200 мс" (слишком технически, используйте "Пользователи видят результаты мгновенно")
- "База данных может обрабатывать 1000 TPS" (деталь реализации, используйте метрику, ориентированную на пользователя)
- "Компоненты React отображаются эффективно" (специфично для фреймворка)
- "Ставка попадания в кэш Redis выше 80%" (специфично для технологии)
