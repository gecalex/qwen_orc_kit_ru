
Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пусто).

## План действий

1. **Настройка**: Запустить `.specify/scripts/bash/check-prerequisites.sh --json` из корня репозитория и разобрать список FEATURE_DIR и AVAILABLE_DOCS. Все пути должны быть абсолютными. Для одинарных кавычек в аргументах, таких как "I'm Groot", используйте синтаксис экранирования: например 'I'\''m Groot' (или двойные кавычки, если возможно: "I'm Groot").

2. **Загрузить проектные документы**: Прочитать из FEATURE_DIR:
   - **Обязательно**: plan.md (стек технологий, библиотеки, структура), spec.md (пользовательские истории с приоритетами)
   - **Опционально**: data-model.md (сущности), contracts/ (конечные точки API), research.md (решения), research/ (сложные исследования), quickstart.md (сценарии тестирования)
   - Примечание: Не все проекты имеют все документы. Создавать задачи на основе доступного.

3. **Выполнить рабочий процесс генерации задач**:
   - Загрузить plan.md и извлечь стек технологий, библиотеки, структуру проекта
   - Загрузить spec.md и извлечь пользовательские истории с их приоритетами (P1, P2, P3 и т.д.)
   - Если data-model.md существует: Извлечь сущности и сопоставить с пользовательскими историями
   - Если contracts/ существует: Сопоставить конечные точки с пользовательскими историями
   - Если research.md существует: Извлечь решения для задач настройки
   - Если research/ существует: Определить сложные исследовательские задачи, требующие решения
   - Сгенерировать задачи, организованные по пользовательским историям (см. Правила генерации задач ниже)
   - Сгенерировать граф зависимостей, показывающий порядок завершения пользовательских историй
   - Создать примеры параллельного выполнения для каждой пользовательской истории
   - Проверить полноту задач (каждая пользовательская история имеет все необходимые задачи, независимо тестируемые)

4. **Создать tasks.md**: Использовать `.specify/templates/tasks-template.md` как структуру, заполнить:
   - Правильное название функции из plan.md
   - Фаза 0: Планирование (назначение исполнителя) - ВСЕГДА включать
   - Фаза 1: Задачи настройки (инициализация проекта)
   - Фаза 2: Фундаментальные задачи (блокирующие предварительные условия для всех пользовательских историй)
   - Фаза 3+: По одной фазе на пользовательскую историю (в порядке приоритета из spec.md)
   - Каждая фаза включает: цель истории, критерии независимого тестирования, тесты (если запрошены), задачи реализации
   - Финальная фаза: Полировка и сквозные вопросы
   - Все задачи должны следовать строгому формату чек-листа (см. Правила генерации задач ниже)
   - Четкие пути к файлам для каждой задачи
   - Раздел зависимостей, показывающий порядок завершения историй
   - Примеры параллельного выполнения для каждой истории
   - Раздел стратегии реализации (MVP первым, инкрементная доставка)

5. **Отчет**: Вывести путь к сгенерированному файлу tasks.md и резюме:
   - Общее количество задач
   - Количество задач на пользовательскую историю
   - Идентифицированные возможности параллельного выполнения
   - Критерии независимого тестирования для каждой истории
   - Предлагаемый объем MVP (обычно только пользовательская история 1)
   - Проверка формата: Подтвердить, что ВСЕ задачи следуют формату чек-листа (флажок, ID, метки, пути к файлам)

Контекст для генерации задач: $ARGUMENTS

Файл tasks.md должен быть сразу исполняемым - каждая задача должна быть достаточно конкретной, чтобы LLM мог выполнить её без дополнительного контекста.

**Исследовательские задачи**: Вопросы без очевидных ответов
- **Простые**: Агент решает с помощью доступных инструментов (Grep, Read, WebSearch, Context7, документация Supabase)
- **Сложные**: Создать запрос исследования в research/ → дождаться deepresearch → включить

**Фаза планирования**: После генерации задач, фаза 0: Планирование будет добавлена автоматически шаблоном. Эта фаза включает:
- **P001**: Анализ задач (идентификация требуемых типов агентов и возможностей)
- **P002**: Создание агента через meta-agent-v3 в одном сообщении, затем запрос перезапуска
- **P003**: Назначение исполнителя (MAIN только для тривиальных, существующий при 100% совпадении, конкретные агенты в противном случае)
- **P004**: Решение исследований (простые: решить сейчас, сложные: создать запросы)

## Правила генерации задач

**КРИТИЧЕСКИЕ**: Задачи ДОЛЖНЫ быть организованы по пользовательским историям для обеспечения независимой реализации и тестирования.

**Тесты являются ОПЦИОНАЛЬНЫМИ**: Создавать задачи тестирования только если они явно запрошены в спецификации функции или пользователь запрашивает подход TDD.

**Оценка библиотеки (ПЕРЕД созданием задач реализации)**:
- Для каждого основного компонента (>20 строк), проверить, существует ли подходящая библиотека
- Использовать решения по библиотекам из research.md из фазы планирования
- Если подходящая библиотека найдена: задача становится "Установить и настроить {библиотеку} для {функциональности}"
- Если подходящей библиотеки нет: задача становится "Реализовать {компонент} в {путь_к_файлу}"
- Документировать обоснование выбора библиотеки в описании задачи, когда это уместно

### Формат чек-листа (ОБЯЗАТЕЛЬНЫЙ)

Каждая задача ДОЛЖНА строго следовать этому формату:

```text
- [ ] [ID задачи] [P?] [История?] Описание с путем к файлу
```

**Компоненты формата**:

1. **Флажок**: ВСЕГДА начинать с `- [ ]` (флажок markdown)
2. **ID задачи**: Последовательный номер (T001, T002, T003...) в порядке выполнения
3. **Маркер [P]**: Включать ТОЛЬКО если задача может выполняться параллельно (разные файлы, без зависимостей от незавершенных задач)
4. **Метка [История]**: ОБЯЗАТЕЛЬНА только для задач фазы пользовательской истории
   - Формат: [US1], [US2], [US3], и т.д. (сопоставляется с пользовательскими историями из spec.md)
   - Фаза настройки: НЕТ метки истории
   - Фундаментальная фаза: НЕТ метки истории
   - Фазы пользовательских историй: ДОЛЖНЫ иметь метку истории
   - Фаза полировки: НЕТ метки истории
5. **Описание**: Четкое действие с точным путем к файлу

**Примеры**:

- ✅ ПРАВИЛЬНО: `- [ ] T001 Создать структуру проекта в соответствии с планом реализации`
- ✅ ПРАВИЛЬНО: `- [ ] T005 [P] Реализовать middleware аутентификации в src/middleware/auth.py`
- ✅ ПРАВИЛЬНО: `- [ ] T012 [P] [US1] Создать модель User в src/models/user.py`
- ✅ ПРАВИЛЬНО: `- [ ] T014 [US1] Реализовать UserService в src/services/user_service.py`
- ❌ НЕПРАВИЛЬНО: `- [ ] Создать модель User` (отсутствует ID и метка истории)
- ❌ НЕПРАВИЛЬНО: `T001 [US1] Создать модель` (отсутствует флажок)
- ❌ НЕПРАВИЛЬНО: `- [ ] [US1] Создать модель User` (отсутствует ID задачи)
- ❌ НЕПРАВИЛЬНО: `- [ ] T001 [US1] Создать модель` (отсутствует путь к файлу)

### Организация задач

1. **Из пользовательских историй (spec.md)** - ОСНОВНАЯ ОРГАНИЗАЦИЯ:
   - Каждая пользовательская история (P1, P2, P3...) получает свою собственную фазу
   - Сопоставить все связанные компоненты с их историей:
     - Модели, необходимые для этой истории
     - Сервисы, необходимые для этой истории
     - Конечные точки/интерфейсы, необходимые для этой истории
     - Если тесты запрошены: Тесты, специфичные для этой истории
   - Отметить зависимости историй (большинство историй должны быть независимыми)

2. **Из контрактов**:
   - Сопоставить каждый контракт/конечную точку → с пользовательской историей, которую он обслуживает
   - Если тесты запрошены: Каждый контракт → задача тестирования контракта [P] перед реализацией в фазе этой истории

3. **Из модели данных**:
   - Сопоставить каждую сущность с пользовательской(-ими) историей(-ями), которая в ней нуждается
   - Если сущность обслуживает несколько историй: Поместить в самую раннюю историю или фазу настройки
   - Отношения → задачи уровня сервиса в соответствующей фазе истории

4. **Из настройки/инфраструктуры**:
   - Общая инфраструктура → Фаза настройки (Фаза 1)
   - Фундаментальные/блокирующие задачи → Фундаментальная фаза (Фаза 2)
   - Настройка, специфичная для истории → внутри фазы этой истории

### Структура фазы

- **Фаза 0**: Планирование (назначение исполнителя, создание агента, решение исследований)
- **Фаза 1**: Настройка (инициализация проекта)
- **Фаза 2**: Фундаментальная (блокирующие предварительные условия - ДОЛЖНА быть завершена до пользовательских историй)
- **Фаза 3+**: Пользовательские истории в порядке приоритета (P1, P2, P3...)
  - Внутри каждой истории: Тесты (если запрошены) → Модели → Сервисы → Конечные точки → Интеграция
  - Каждая фаза должна быть завершенным, независимо тестируемым приращением
- **Финальная фаза**: Полировка и сквозные вопросы
