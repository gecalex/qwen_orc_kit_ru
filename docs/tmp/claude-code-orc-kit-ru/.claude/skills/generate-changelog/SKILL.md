# Генерация журнала изменений

Автоматическая генерация записей в журнале изменений в соответствии с форматом Keep a Changelog.

## Когда использовать

- Рабочие процессы релизов версий
- Создание разделов CHANGELOG.md
- Документирование изменений между версиями
- Генерация примечаний к релизам
- Резюмирование выполненной работы

## Инструкции

### Шаг 1: Сбор данных об изменениях

Принять информацию об изменениях из различных источников.

**Ожидаемый ввод**:
- `version`: Строка (например, "0.8.0")
- `date`: Строка (ISO-8601 или YYYY-MM-DD)
- `source`: Строка (git|plan-file|manual)
- `sourceData`: Объект (диапазон git, путь к файлу плана или ручные записи)

**Типы источников**:
- **git**: Разбор лога git между двумя ссылками
- **plan-file**: Извлечение из метаданных файла плана
- **manual**: Предоставленные пользователем записи

### Шаг 2: Разбор изменений

Извлечь и категоризировать изменения.

**Категории** (стандарт Keep a Changelog):
- **Added**: Новые функции
- **Changed**: Изменения в существующем функционале
- **Deprecated**: Функции, которые будут удалены
- **Removed**: Удаленные функции
- **Fixed**: Исправления ошибок
- **Security**: Исправления уязвимостей безопасности

**Разбор коммитов Git**:
- Разбор формата conventional commit (feat:, fix: и т.д.)
- Извлечение области и описания
- Сопоставление типа коммита с категорией журнала изменений

### Шаг 3: Группировка и удаление дубликатов

Группировать изменения по категориям и удалять дубликаты.

**Правила группировки**:
- feat → Added
- fix → Fixed
- security → Security
- refactor, perf → Changed
- docs → Changed (если значительные)
- chore → Пропустить (если не критическое изменение)

### Шаг 4: Форматирование раздела журнала изменений

Создать отформатированный раздел журнала изменений.

**Формат**:
```markdown
## [Версия] - Дата

### Added
- Новая функция 1
- Новая функция 2

### Fixed
- Исправление ошибки 1
- Исправление ошибки 2

### Changed
- Изменение 1
```

### Шаг 5: Возврат отформатированного журнала изменений

Вернуть полный раздел журнала изменений, готовый для вставки.

**Ожидаемый вывод**:
```markdown
## [0.8.0] - 2025-10-17

### Added
- Переключатель темного режима для пользовательских предпочтений
- Поддержка аутентификации OAuth2

### Fixed
- Утечка памяти в пуле соединений
- Ошибка конфигурации CORS

### Changed
- Миграция на v2 API аутентификации
- Оптимизация производительности запросов
```

## Обработка ошибок

- **Изменения не найдены**: Вернуть пустой журнал изменений с примечанием
- **Неверный диапазон Git**: Вернуть ошибку с выводом команды git
- **Неверный файл плана**: Вернуть ошибку с описанием проблемы
- **Неразборчивые коммиты**: Включить как "Changed" с исходным сообщением

## Примеры

### Пример 1: Генерация из диапазона Git

**Ввод**:
```json
{
  "version": "0.8.0",
  "date": "2025-10-17",
  "source": "git",
  "sourceData": {
    "fromRef": "v0.7.0",
    "toRef": "HEAD"
  }
}
```

**Коммиты Git**:
```
feat(auth): add OAuth2 support
fix(api): resolve memory leak
docs: update API documentation
chore(deps): bump dependencies
```

**Вывод**:
```markdown
## [0.8.0] - 2025-10-17

### Added
- Add OAuth2 support (auth)

### Fixed
- Resolve memory leak (api)

### Changed
- Update API documentation
- Bump dependencies (deps)
```

### Пример 2: Генерация из файла плана

**Ввод**:
```json
{
  "version": "0.8.0",
  "date": "2025-10-17",
  "source": "plan-file",
  "sourceData": {
    "planFile": ".version-update-plan.json"
  }
}
```

**Содержимое файла плана**:
```json
{
  "changes": {
    "added": ["Переключатель темного режима", "Профили пользователей"],
    "fixed": ["Ошибка перенаправления при входе", "Крах панели управления"],
    "changed": ["Улучшена производительность"]
  }
}
```

**Вывод**:
```markdown
## [0.8.0] - 2025-10-17

### Added
- Переключатель темного режима
- Профили пользователей

### Fixed
- Ошибка перенаправления при входе
- Крах панели управления

### Changed
- Улучшена производительность
```

### Пример 3: Ручные записи

**Ввод**:
```json
{
  "version": "0.8.0",
  "date": "2025-10-17",
  "source": "manual",
  "sourceData": {
    "added": ["Новая интеграция платежного шлюза"],
    "fixed": ["Ошибка расчета корзины"],
    "security": ["Исправлена уязвимость XSS в поиске"]
  }
}
```

**Вывод**:
```markdown
## [0.8.0] - 2025-10-17

### Added
- Новая интеграция платежного шлюза

### Fixed
- Ошибка расчета корзины

### Security
- Исправлена уязвимость XSS в поиске
```

### Пример 4: Пустой журнал изменений

**Ввод**:
```json
{
  "version": "0.8.0",
  "date": "2025-10-17",
  "source": "git",
  "sourceData": {
    "fromRef": "v0.7.0",
    "toRef": "v0.7.0"
  }
}
```

**Вывод**:
```markdown
## [0.8.0] - 2025-10-17

Нет изменений в этом релизе.
```

## Валидация

- [ ] Правильно разбирает conventional commits
- [ ] Сопоставляет типы коммитов с категориями журнала изменений
- [ ] Группирует изменения по категориям
- [ ] Форматирует в соответствии с Keep a Changelog
- [ ] Обрабатывает пустые журналы изменений
- [ ] Удаляет дубликаты записей
- [ ] Включает область в записи при наличии
- [ ] Обрабатывает ручные записи
