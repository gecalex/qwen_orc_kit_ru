# Разбор статуса Git

Разбор вывода команды git status в структурированный JSON для программного анализа.

## Когда использовать

- Предварительные проверки перед выполнением рабочего процесса
- Проверка чистоты рабочего каталога
- Перечисление измененных файлов для коммита
- Проверка наличия несохраненных изменений

## Инструкции

### Шаг 1: Получение вывода статуса Git

Принять необработанный вывод git status в качестве входных данных.

**Ожидаемый ввод**:
- `gitStatusOutput`: Строка (необработанный вывод от `git status --porcelain` или обычного `git status`)

### Шаг 2: Разбор информации о ветке

Извлечь текущую ветку и информацию отслеживания.

**Шаблоны**:
- `## branch-name`: Текущая ветка
- `## branch-name...origin/branch-name`: Отслеживаемая ветка
- `[ahead N]` или `[behind N]`: Опережение/отставание коммитов

### Шаг 3: Классификация файлов

Разбор индикаторов статуса файлов и классификация.

**Индикаторы статуса** (формат porcelain):
- `M `: Изменен (подготовлен)
- ` M`: Изменен (неподготовлен)
- `A `: Добавлен (подготовлен)
- `D `: Удален (подготовлен)
- `R `: Переименован (подготовлен)
- `??`: Непроиндексированный
- `!!`: Игнорируется

### Шаг 4: Возврат структурированных данных

Вернуть разобранные данные в виде объекта JSON.

**Ожидаемый вывод**:
```json
{
  "branch": "main",
  "tracking": "origin/main",
  "ahead": 0,
  "behind": 0,
  "staged": ["file1.ts", "file2.ts"],
  "modified": ["file3.ts"],
  "deleted": [],
  "renamed": [],
  "untracked": ["file4.ts"],
  "clean": false
}
```

## Обработка ошибок

- **Неверный вывод Git**: Вернуть ошибку с описанием проблемы формата
- **Не репозиторий Git**: Вернуть ошибку, указывающую на отсутствие git-репозитория
- **Пустой вывод**: Вернуть чистый статус с пустыми массивами

## Примеры

### Пример 1: Чистый рабочий каталог

**Ввод**:
```
On branch main
Your branch is up to date with 'origin/main'.

nothing to commit, working tree clean
```

**Вывод**:
```json
{
  "branch": "main",
  "tracking": "origin/main",
  "ahead": 0,
  "behind": 0,
  "staged": [],
  "modified": [],
  "deleted": [],
  "renamed": [],
  "untracked": [],
  "clean": true
}
```

### Пример 2: Измененные файлы

**Ввод** (формат porcelain):
```
## main...origin/main [ahead 2]
M  src/utils.ts
 M src/types.ts
A  src/new-feature.ts
?? temp-file.js
```

**Вывод**:
```json
{
  "branch": "main",
  "tracking": "origin/main",
  "ahead": 2,
  "behind": 0,
  "staged": ["src/utils.ts", "src/new-feature.ts"],
  "modified": ["src/types.ts"],
  "deleted": [],
  "renamed": [],
  "untracked": ["temp-file.js"],
  "clean": false
}
```

### Пример 3: Отсоединенный HEAD

**Ввод**:
```
## HEAD (no branch)
 M README.md
```

**Вывод**:
```json
{
  "branch": "HEAD (detached)",
  "tracking": null,
  "ahead": 0,
  "behind": 0,
  "staged": [],
  "modified": ["README.md"],
  "deleted": [],
  "renamed": [],
  "untracked": [],
  "clean": false
}
```

## Валидация

- [ ] Правильно разбирает информацию о ветке
- [ ] Классифицирует файлы по статусу
- [ ] Обрабатывает пустой/чистый статус
- [ ] Разбирает индикаторы опережения/отставания
- [ ] Обрабатывает состояние отсоединенного HEAD
- [ ] Возвращает clean:true только когда это уместно

## Вспомогательные файлы

Не требуются - чистая логика разбора.
