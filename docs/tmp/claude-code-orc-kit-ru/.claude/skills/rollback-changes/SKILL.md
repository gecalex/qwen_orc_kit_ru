# Откат изменений

Автоматический откат изменений из неудавшихся фаз рабочих процессов путем чтения файлов журнала изменений и отмены всех отслеживаемых изменений.

## Когда использовать

- Фаза рабочего процесса завершается с ошибкой и требуется восстановление состояния
- Автоматический откат после неудавшихся исправлений ошибок
- Восстановление предыдущего состояния после неудавшихся патчей безопасности
- Очистка после неудавшихся операций рефакторинга
- Восстановление после ошибок в агентах-рабочих
- Ошибки контрольных точек качества, требующие отката

## Инструкции

### Шаг 1: Чтение файла журнала изменений

Использовать инструмент Read для загрузки файла журнала изменений.

**Ожидаемый ввод**:
- `changes_log_path`: Строка (путь к JSON-файлу изменений, например, `.bug-changes.json`)
- `phase`: Строка (название фазы рабочего процесса, например, "bug-fixing")
- `confirmation_required`: Boolean (по умолчанию: true, спрашивать пользователя перед откатом)

**Используемые инструменты**: Read

### Шаг 2: Разбор журнала изменений

Разбор содержимого JSON и проверка структуры.

**Обязательные поля**:
- `phase`: Строка (фаза рабочего процесса, в которой были сделаны изменения)
- `timestamp`: Строка (отметка времени ISO-8601)
- `files_modified`: Массив объектов с `{path, backup}` (файлы с резервными копиями)
- `files_created`: Массив строк (новые созданные файлы)
- `commands_executed`: Массив строк (выполненные команды)
- `git_commits`: Массив строк (SHA коммитов, если есть)

**Необязательные поля**:
- `artifacts`: Массив строк (временные файлы для удаления)
- `plan_files`: Массив строк (файлы плана для удаления)
- `metadata`: Объект (дополнительный контекст)

### Шаг 3: Запрос подтверждения (если требуется)

Если `confirmation_required` равно true, запросить подтверждение у пользователя.

**Запрос подтверждения**:
```
Откатить изменения из фазы "{phase}"?

Изменения для отката:
- Измененные файлы: {count}
- Созданные файлы: {count}
- Выполненные команды: {count}
- Коммиты Git: {count}

Это действие:
1. Восстановит {count} файлов из резервных копий
2. Удалит {count} созданных файлов
3. Откатит {count} команд
4. Откатит {count} коммитов Git

Продолжить откат? (yes/no)
```

**Если пользователь говорит "no"**: Вернуть без выполнения действий (результат тестового запуска)

**Если `confirmation_required` равно false**: Пропустить подтверждение и продолжить

### Шаг 4: Восстановление измененных файлов

Для каждого файла в `files_modified`, восстановить из резервной копии.

**Процесс восстановления**:
```bash
# Для каждого {path, backup} в files_modified:
if [ -f "{backup}" ]; then
  cp "{backup}" "{path}"
  echo "✓ Восстановлен {path} из {backup}"
else
  echo "⚠ Резервная копия не найдена: {backup} (пропуск)"
fi
```

**Обработка ошибок**:
- Отсутствующий файл резервной копии: Зарегистрировать предупреждение, продолжить с другими файлами
- Ошибка копирования: Зарегистрировать ошибку, продолжить с другими файлами
- Проблемы с правами доступа: Зарегистрировать ошибку, продолжить с другими файлами

**Используемые инструменты**: Bash

### Шаг 5: Удаление созданных файлов

Для каждого файла в `files_created`, удалить, если он существует.

**Процесс удаления**:
```bash
# Для каждого файла в files_created:
if [ -f "{file}" ]; then
  rm "{file}"
  echo "✓ Удален созданный файл: {file}"
elif [ -d "{file}" ]; then
  rm -rf "{file}"
  echo "✓ Удалена созданная директория: {file}"
else
  echo "⚠ Файл не найден (уже удален?): {file}"
fi
```

**Обработка ошибок**:
- Файл не найден: Зарегистрировать предупреждение (может быть уже удален)
- Проблемы с правами доступа: Зарегистрировать ошибку, продолжить с другими файлами
- Директория не пуста: Использовать `rm -rf` с осторожностью

**Используемые инструменты**: Bash

### Шаг 6: Откат команд

Для каждой команды в `commands_executed`, попытаться откатить.

**Сопоставление отката**:
- `pnpm install` → `pnpm install` (повторный запуск для восстановления lock-файла)
- `git add {files}` → `git restore --staged {files}`
- `git commit` → Обрабатывается на Шаге 7
- `pnpm build` → `rm -rf dist/` (удаление артефактов сборки)
- Пользовательские команды → Только регистрировать (нельзя откатить автоматически)

**Процесс отката**:
```bash
# Для каждой команды:
case "{command}" in
  "pnpm install")
    pnpm install
    echo "✓ Повторно запущен pnpm install для восстановления зависимостей"
    ;;
  "git add "*)
    git restore --staged .
    echo "✓ Отменена подготовка всех файлов"
    ;;
  "pnpm build")
    rm -rf dist/
    echo "✓ Удалены артефакты сборки"
    ;;
  *)
    echo "⚠ Невозможно авто-откатить: {command} (требуется ручное вмешательство)"
    ;;
esac
```

**Обработка ошибок**:
- Ошибка команды: Зарегистрировать ошибку, продолжить с другими командами
- Неизвестная команда: Зарегистрировать предупреждение, невозможно откатить
- Уже откатано: Зарегистрировать информацию, продолжить

**Используемые инструменты**: Bash

### Шаг 7: Откат коммитов Git

Для каждого SHA коммита в `git_commits`, откатить коммит.

**Процесс отката**:
```bash
# Для каждого SHA в git_commits (в обратном порядке):
git revert --no-edit {sha}
if [ $? -eq 0 ]; then
  echo "✓ Откат коммита {sha}"
else
  echo "❌ Не удалось откатить коммит {sha} (могут быть конфликты)"
fi
```

**Обработка ошибок**:
- Конфликты отката: Зарегистрировать ошибку, предоставить инструкции для ручного решения
- Коммит не найден: Зарегистрировать ошибку, пропустить коммит
- Отсоединенный HEAD: Зарегистрировать ошибку, пропустить операции git

**Используемые инструменты**: Bash

### Шаг 8: Очистка артефактов

Удалить временные файлы и файлы плана.

**Процесс очистки**:
```bash
# Удалить файлы плана
rm -f .{workflow}-plan.json

# Удалить другие артефакты, если указаны
for artifact in {artifacts}; do
  rm -f "{artifact}"
  echo "✓ Удален артефакт: {artifact}"
done

# Удалить резервные копии отката (необязательно, только если все прошло успешно)
if [ {all_succeeded} == true ]; then
  rm -rf .rollback/
  echo "✓ Удалены резервные копии отката"
fi
```

**Используемые инструменты**: Bash

### Шаг 9: Генерация отчета об откате

Создать структурированный отчет обо всех выполненных действиях.

**Структура отчета**:
```json
{
  "success": true|false,
  "phase": "bug-fixing",
  "actions_taken": [
    "Восстановлено 3 файлов из резервных копий",
    "Удалено 2 созданных файла",
    "Откат 1 коммита git",
    "Очищено 2 артефакта"
  ],
  "files_restored": 3,
  "files_deleted": 2,
  "commands_reverted": 1,
  "git_commits_reverted": 1,
  "artifacts_cleaned": 2,
  "errors": [],
  "warnings": [
    "Резервная копия не найдена: .rollback/file.backup (пропущена)"
  ],
  "timestamp": "2025-10-18T14:45:00Z",
  "duration_ms": 1234
}
```

**Используемые инструменты**: Write

## Обработка ошибок

- **Отсутствует журнал изменений**: Вернуть ошибку "Changes log not found at {path}"
- **Неверный JSON**: Вернуть ошибку "Invalid JSON in changes log: {details}"
- **Отсутствуют обязательные поля**: Вернуть ошибку "Changes log missing required field: {field}"
- **Резервная копия не найдена**: Зарегистрировать предупреждение, продолжить с другими файлами (частичный откат)
- **Ошибка удаления файла**: Зарегистрировать ошибку, продолжить с другими файлами
- **Ошибка отката Git**: Зарегистрировать ошибку с деталями конфликта, продолжить с другими операциями
- **Пользователь отказался**: Вернуть результат тестового запуска с `success: false, user_declined: true`
- **Частичный откат**: Вернуть `success: true` с массивом предупреждений, документирующим проблемы

## Особенности безопасности

### Подтверждение по умолчанию
- Всегда запрашивать подтверждение, если явно не отключено
- Показывать четкое резюме того, что будет откачено
- Позволить пользователю отменить перед любыми изменениями

### Частичный откат приемлем
- Не завершать работу полностью, если некоторые резервные копии отсутствуют
- Продолжать с другими операциями, когда одна не удается
- Документировать все проблемы в массиве предупреждений

### Нет скрытых ошибок
- Явно регистрировать все ошибки и предупреждения
- Возвращать подробный массив actions_taken
- Предоставлять контекст устранения неполадок в ошибках

### Проверка резервных копий
- Никогда не удалять исходные файлы без проверки существования резервной копии
- Проверять существование файла резервной копии перед восстановлением
- Сохранять резервные копии до тех пор, пока полный откат не будет успешным

### Журнал аудита
- Регистрировать все действия отката для отладки
- Включать временные метки и продолжительность
- Возвращать полный отчет для документирования

## Интеграция с рабочими процессами

Рабочие процессы должны создавать файлы журнала изменений во время операций:

```markdown
## Шаг 2: Отслеживание изменений

Перед внесением каких-либо изменений:
1. Создать журнал изменений: `.{domain}-changes.json`
2. Для каждого измененного файла:
   - Создать резервную копию в `.rollback/{path}.backup`
   - Добавить в массив `files_modified`
3. Для каждого созданного файла:
   - Добавить в массив `files_created`
4. Для каждой выполненной команды:
   - Добавить в массив `commands_executed`
5. Для каждого коммита git:
   - Добавить SHA в массив `git_commits`
6. Записать журнал изменений после каждого изменения

## Шаг 5: Откат при сбое

Если контрольная точка качества не проходит или возникает ошибка:
1. Использовать навык rollback-changes с:
   - changes_log_path: ".{domain}-changes.json"
   - phase: "{current-phase}"
   - confirmation_required: false (автоматический откат)
2. Если откат проходит успешно, сообщить о чистом состоянии
3. Если откат не удается, сообщить о частичном состоянии с предупреждениями
```

## Примечания

- Операции отката выполняются наилучшим образом (могут быть не на 100% обратимыми)
- Операции отката Git могут потребовать ручного разрешения конфликтов
- Некоторые команды (пользовательские скрипты) не могут быть откачены автоматически
- Резервные копии должны храниться в директории `.rollback/` (в gitignore)
- Файлы журнала изменений должны быть временными (удаляются после успешного завершения)
- Частичный откат приемлем и документируется в предупреждениях
- Всегда проверять существование резервной копии перед операциями удаления
