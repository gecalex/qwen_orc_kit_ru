# Выполнение контрольной точки качества

Выполнение команд проверки как контрольных точек качества с настраиваемым поведением блокировки/неблокировки и структурированным отчетом об ошибках.

## Когда использовать

- Проверка типов в предварительной проверке или контрольных точках качества
- Проверка сборки перед релизами
- Выполнение тестов как контрольной точки качества
- Проверка линтинга для качества кода
- Пользовательские команды проверки
- Проверка фазы оркестратора
- Самопроверка рабочего процесса

## Инструкции

### Шаг 1: Получение конфигурации контрольной точки

Принять конфигурацию контрольной точки в качестве входных данных.

**Ожидаемый ввод**:
```json
{
  "gate": "type-check|build|tests|lint|custom",
  "blocking": true,
  "custom_command": "pnpm custom-validate"
}
```

**Параметры**:
- `gate`: Тип контрольной точки качества для запуска (обязательно)
- `blocking`: Должна ли ошибка остановить рабочий процесс (по умолчанию: true)
- `custom_command`: Команда для запуска при gate="custom" (обязательна для пользовательских контрольных точек)

### Шаг 2: Сопоставление контрольной точки с командой

Определить команду для выполнения на основе типа контрольной точки.

**Команды контрольных точек**:
- `type-check` → `pnpm type-check`
- `build` → `pnpm build`
- `tests` → `pnpm test`
- `lint` → `pnpm lint`
- `custom` → Использовать параметр `custom_command`

**Проверка**:
- Если gate="custom", должен быть предоставлен `custom_command`
- Команда должна быть действительной командой оболочки

### Шаг 3: Выполнение команды

Запустить команду через инструмент Bash с таймаутом.

**Параметры выполнения**:
- Таймаут: 300000мс (5 минут)
- Захват stdout и stderr
- Запись кода выхода
- Отслеживание продолжительности выполнения

**Используемые инструменты**: Bash

### Шаг 4: Разбор кода выхода и вывода

Определить, прошла ли контрольная точка на основе кода выхода.

**Логика прохождения/провала**:
- Код выхода 0 → Пройдена
- Код выхода не ноль → Провалена

**Извлечение ошибок**:
Искать шаблоны ошибок в выводе:
- Строки, содержащие "error"
- Строки, содержащие "failed"
- Строки, содержащие "✗"
- Коды ошибок TypeScript (TS####)
- Стек трейсы

### Шаг 5: Определение действия

Рассчитать действие на основе результата и флага блокировки.

**Логика действий**:
```
IF exit_code == 0:
  action = "continue"
  passed = true
ELSE:
  IF blocking == true:
    action = "stop"
  ELSE:
    action = "warn"
  passed = false
```

### Шаг 6: Возврат структурированного результата

Вернуть полный результат контрольной точки качества.

**Ожидаемый вывод**:
```json
{
  "gate": "type-check",
  "passed": true,
  "blocking": true,
  "action": "continue",
  "errors": [],
  "exit_code": 0,
  "duration_ms": 2345,
  "command": "pnpm type-check",
  "timestamp": "2025-10-18T14:30:00Z"
}
```

**Поля вывода**:
- `gate`: Тип контрольной точки, которая была запущена
- `passed`: Пройдена ли контрольная точка (boolean)
- `blocking`: Была ли контрольная точка блокирующей
- `action`: Действие для выполнения (continue|stop|warn)
- `errors`: Массив извлеченных сообщений об ошибках
- `exit_code`: Код выхода команды
- `duration_ms`: Время выполнения в миллисекундах
- `command`: Фактически выполненная команда
- `timestamp`: Отметка времени ISO-8601 выполнения

## Обработка ошибок

- **Таймаут (5 минут)**: Вернуть ошибку с сообщением о таймауте
- **Отсутствует custom_command**: Вернуть ошибку с запросом custom_command
- **Неверный тип контрольной точки**: Вернуть ошибку с перечислением допустимых контрольных точек
- **Команда не найдена**: Вернуть ошибку с сообщением о том, что команда не найдена
- **Пустой вывод, но ненулевой выход**: Вернуть ошибку с общим сообщением

## Интеграция с агентами

### Использование оркестратора

```markdown
## Контрольная точка качества: Проверка типов

Использовать навык run-quality-gate с gate="type-check" и blocking=true.

Если action="stop", остановить рабочий процесс и сообщить об ошибке.
Если action="continue", перейти к следующей фазе.
```

### Самопроверка рабочего процесса

```markdown
## Шаг 5: Самопроверка

Использовать навык run-quality-gate для проверки изменений:
1. Запустить проверку типов (blocking=true)
2. Запустить сборку (blocking=true)
3. Запустить тесты (blocking=false)

Если любая блокирующая контрольная точка возвращает action="stop", откатить изменения.
```

### Оркестратор контрольных точек качества

```markdown
## Фаза 2: Выполнение контрольных точек качества

Для каждой контрольной точки в [type-check, build, tests, lint]:
  result = run-quality-gate(gate, blocking=true)

  if result.action == "stop":
    ОСТАНОВИТЬ и сообщить об ошибке

  if result.action == "warn":
    ЗАРЕГИСТРИРОВАТЬ предупреждение и продолжить
```

## Примечания

- Таймаут фиксирован на 5 минутах, чтобы предотвратить бесконечные зависания
- Извлечение ошибок выполняется наилучшим образом (может не захватить все ошибки)
- Пользовательские команды должны быть действительными командами оболочки
- Код выхода 0 всегда означает успех, независимо от вывода
- Ненулевой код выхода всегда означает ошибку
- Флаг блокировки влияет только на действие, а не на статус прохождения
