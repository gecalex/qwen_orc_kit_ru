# Конституция {{PROJECT_NAME}}

> **Полномочия**: Эта конституция имеет приоритет над всеми другими практиками разработки. Руководство по выполнению в `CLAUDE.md` реализует эти принципы на операционном уровне.

## Основные принципы

### I. Разработка с приоритетом контекста (НЕПРЕДМЕТ ДЛЯ ОБСУЖДЕНИЯ)

Перед любой реализацией или делегированием соберите полный контекст:
- Прочитайте существующий код в связанных файлах
- Поищите похожие паттерны и реализации
- Просмотрите спецификации, ADR и недавние коммиты
- Понимайте зависимости и точки интеграции

**Обоснование**: Предотвращает дублирование работы, обеспечивает согласованность, избегает конфликтующих подходов.

### II. Единый источник истины

Типы, константы, перечисления, схемы и общий логический код ДОЛЖНЫ быть определены в назначенных центральных местах (например, `{{SHARED_TYPES_PATH}}`). Дублирование запрещено — потребители должны импортировать из источника.

**Обоснование**: Устраняет расхождения между дублирующимися определениями, упрощает рефакторинг.

### III. Подход с библиотек

Перед реализацией пользовательских решений (>20 строк):
1. Поищите существующие библиотеки (npm, PyPI и т.д.)
2. Оцените: статус сопровождения, безопасность, размер сборки, поддержка TypeScript
3. Используйте библиотеку, если она охватывает >70% требований

**Правило Context7**: Перед написанием кода, использующего ЛЮБУЮ библиотеку, НЕОБХОДИМО получить актуальную документацию через `mcp__context7__get-library-docs`. Это гарантирует правильное использование API и избегает устаревших паттернов.

**Обоснование**: Снижает нагрузку на сопровождение, использует стандарты сообщества и исправления безопасности.

### IV. Повторное использование кода и DRY

Перед созданием новых компонентов, утилит или логики:
1. Поищите в существующей кодовой базе повторно используемые реализации
2. Предпочитайте адаптацию и расширение дублированию
3. Документируйте, почему повторное использование было невозможно при создании нового

**Обоснование**: Снижает размер кодовой базы, обеспечивает согласованное поведение, снижает когнитивную нагрузку.

### V. Строгая типовая безопасность (НЕПРЕДМЕТ ДЛЯ ОБСУЖДЕНИЯ)

- Режим TypeScript `strict` принудительно включен
- `any` запрещен — используйте `unknown` или правильные типы
- Явные возвращаемые типы для экспортируемых функций
- Проверка типов должна пройти перед коммитом

**Обоснование**: Обнаруживает ошибки во время компиляции, позволяет безопасный рефакторинг, улучшает поддержку IDE.

### VI. Атомарное выполнение задач

Каждая задача должна быть независимо выполнимой, тестируемой и пригодной для коммита:
- Отметьте задачу как `in_progress` перед началом
- Проверьте реализацию (прочитайте файлы + запустите проверки)
- Отметьте как `completed` только после валидации
- Коммит после КАЖДОЙ задачи, не пачками

**Правило атомарной делегации**: Один вызов агента = одна задача. Никогда не объединяйте несколько задач в один вызов агента.
- Параллельная работа: Запустите N агентов в одном сообщении, каждый с ровно одной задачей
- Последовательная работа: Завершите один вызов агента, затем начните следующий
- Один и тот же тип агента может работать несколько раз параллельно — каждый экземпляр обрабатывает одну задачу

**Обоснование**: Обеспечивает легкий откат, четкое отслеживание прогресса, лучший код-ревью. Атомарная делегация гарантирует предсказуемое поведение агента и упрощает обработку ошибок.

### VII. Контрольные точки качества (НЕПРЕДМЕТ ДЛЯ ОБСУЖДЕНИЯ)

Перед любым коммитом:
- [ ] Проверка типов пройдена
- [ ] Сборка прошла успешно
- [ ] Нет захардкоженных учетных данных
- [ ] Нет `TODO` без ссылки на задачу

**Обоснование**: Предотвращает сломанный код в основной ветке, поддерживает здоровье кодовой базы.

### VIII. Прогрессивная спецификация

Функции проходят обязательные фазы:
1. **Spec** → Пользовательские истории и требования
2. **Plan** → Технический подход и решения
3. **Tasks** → Атомарные, упорядоченные шаги реализации
4. **Implement** → Выполнение с валидацией

Ни одна фаза не может быть пропущена. Каждый вывод проверяется перед продолжением.

**Обоснование**: Снижает необходимость переделки, проверяет подход до дорогостоящей реализации.

## Операционное совершенство

### IX. Обработка ошибок

- Все ошибки должны быть типизированы (пользовательские классы Error или объединенные типы)
- Ошибки для пользователя: локализованные, с действиями, без трассировок стека
- Внутренние ошибки: структурированное логирование с контекстом
- Никогда не подавляйте ошибки молча

### X. Наблюдаемость

- ID корреляции для распределенных операций
