# Задачи: [НАЗВАНИЕ ФУНКЦИИ]

**Входные данные**: Документы проектирования из `/specs/[###-feature-name]/`
**Предварительные условия**: plan.md (обязательно), spec.md (обязательно для пользовательских историй), research.md, data-model.md, contracts/

**Тесты**: Примеры ниже включают задачи тестирования. Тесты НЕОБЯЗАТЕЛЬНЫ - включайте их только если они явно запрошены в спецификации функции.

**Организация**: Задачи сгруппированы по пользовательским историям для обеспечения независимой реализации и тестирования каждой истории.

## Формат: `[ID] [P?] [Story] Описание`

- **[P]**: Можно выполнять параллельно (разные файлы, без зависимостей)
- **[Story]**: К какой пользовательской истории относится задача (например, US1, US2, US3)
- Включайте точные пути к файлам в описаниях

## Соглашения об именах путей

- **Один проект**: `src/`, `tests/` в корне репозитория
- **Веб-приложение**: `backend/src/`, `frontend/src/`
- **Мобильное приложение**: `api/src/`, `ios/src/` или `android/src/`
- Пути, показанные ниже, предполагают один проект - корректируйте в зависимости от структуры plan.md

## Фаза 0: Планирование (Назначение исполнителей)

**Назначение**: Подготовка к реализации путем анализа требований, создания необходимых агентов и назначения исполнителей.

- [ ] P001 Проанализировать все задачи и определить требуемые типы агентов и возможности
- [ ] P002 Создать отсутствующих агентов с использованием meta-agent-v3 (запустить N вызовов в одном сообщении, по 1 на агента), затем попросить пользователя перезапустить
- [ ] P003 Назначить исполнителей для всех задач: MAIN (только для тривиальных), существующие агенты (100% совпадение), или конкретные имена агентов
- [ ] P004 Решить задачи исследования: простые (решить с помощью инструментов сейчас), сложные (создать запросы в research/)

**Правила**:
- **MAIN исполнитель**: ТОЛЬКО для тривиальных задач (исправления в 1-2 строки, простые импорты, одиночный npm install)
- **Существующие агенты**: ТОЛЬКО при 100% совпадении возможностей после тщательного изучения
- **Создание агентов**: Запустить все вызовы meta-agent-v3 в одном сообщении для параллельного выполнения
- **После P002**: Должны перезапустить claude-code перед переходом к P003

**Артефакты**:
- Обновленный tasks.md с аннотациями [EXECUTOR: имя], [SEQUENTIAL]/[PARALLEL-GROUP-X]
- .claude/agents/{domain}/{type}/{name}.md (если созданы новые агенты)
- research/*.md (если идентифицировано сложное исследование)

## Зависимости и порядок выполнения

### Зависимости фаз

- **Настройка (Фаза 1)**: Без зависимостей - можно начать сразу
- **Фундаментальная (Фаза 2)**: Зависит от завершения Настройки - БЛОКИРУЕТ все пользовательские истории
- **Пользовательские истории (Фаза 3+)**: Все зависят от завершения фундаментальной фазы
  - Пользовательские истории могут затем продолжаться параллельно (если есть ресурсы)
  - Или последовательно в порядке приоритета (P1 → P2 → P3)
- **Доработка (Финальная фаза)**: Зависит от завершения всех желаемых пользовательских историй

### Зависимости пользовательских историй

- **Пользовательская история 1 (P1)**: Может начаться после фундаментальной (Фаза 2) - Без зависимостей от других историй
- **Пользовательская история 2 (P2)**: Может начаться после фундаментальной (Фаза 2) - Может интегрироваться с US1, но должна быть независимо тестируемой
- **Пользовательская история 3 (P3)**: Может начаться после фундаментальной (Фаза 2) - Может интегрироваться с US1/US2, но должна быть независимо тестируемой

### Внутри каждой пользовательской истории

- Тесты (если включены) ДОЛЖНЫ быть написаны и НЕ ПРОХОДИТЬ до реализации
- Модели до сервисов
- Сервисы до конечных точек
- Основная реализация до интеграции
- История завершена до перехода к следующему приоритету

### Возможности параллельного выполнения

- Все задачи Настройки, помеченные [P], могут выполняться параллельно
- Все задачи фундаментальной фазы, помеченные [P], могут выполняться параллельно (внутри Фазы 2)
- После завершения фундаментальной фазы все пользовательские истории могут начаться параллельно (если позволяет емкость команды)
- Все тесты для пользовательской истории, помеченные [P], могут выполняться параллельно
- Модели внутри истории, помеченные [P], могут выполняться параллельно
- Разные пользовательские истории могут обрабатываться параллельно разными членами команды

## Стратегия реализации

### MVP в первую очередь (Только пользовательская история 1)

1. Завершить фазу 1: Настройка
2. Завершить фазу 2: Фундаментальная (КРИТИЧНО - блокирует все истории)
3. Завершить фазу 3: Пользовательская история 1
4. **ОСТАНОВИТЬСЯ и ПРОВЕРИТЬ**: Протестировать пользовательскую историю 1 независимо
5. Развернуть/демонстрация при готовности

### Инкрементальная доставка

1. Завершить Настройку + Фундаментальную → Фундамент готов
2. Добавить пользовательскую историю 1 → Протестировать независимо → Развернуть/Демонстрация (MVP!)
3. Добавить пользовательскую историю 2 → Протестировать независимо → Развернуть/Демонстрация
4. Добавить пользовательскую историю 3 → Протестировать независимо → Развернуть/Демонстрация
5. Каждая история добавляет ценность, не ломая предыдущие истории

### Стратегия параллельной команды

С несколькими разработчиками:

1. Команда завершает Настройку + Фундаментальную вместе
2. После завершения фундаментальной:
   - Разработчик A: Пользовательская история 1
   - Разработчик B: Пользовательская история 2
   - Разработчик C: Пользовательская история 3
3. Истории завершаются и интегрируются независимо

## Примечания

- [P] задачи = разные файлы, без зависимостей
- [Story] метка связывает задачу с конкретной пользовательской историей для отслеживаемости
- Каждая пользовательская история должна быть независимо выполнимой и тестируемой
- Проверьте, что тесты не проходят до реализации
- Коммит после каждой задачи или логической группы
- Остановитесь на любой контрольной точке, чтобы проверить историю независимо
- Избегайте: нечетких задач, конфликтов в одних и тех же файлах, межисторических зависимостей, нарушающих независимость
