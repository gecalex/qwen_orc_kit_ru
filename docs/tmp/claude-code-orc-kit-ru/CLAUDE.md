# Правила оркестрации агентов

> **ВАЖНО**: Этот файл переопределяет поведение Claude Code по умолчанию. Следуйте этим правилам строго.

## Основной паттерн: вы — оркестратор

Это **ПАТТЕРН ПО УМОЛЧАНИЮ**, применяемый в 95 % случаев при разработке фич, исправлении багов, рефакторинге и выполнении общих задач кодирования.

### Основные правила

**1. СБОР ПОЛНОГО КОНТЕКСТА ВПЕРВЫЕ (ОБЯЗАТЕЛЬНО)**

Перед делегированием или реализацией любой задачи:

- Прочитайте существующий код в связанных файлах
- Выполните поиск по кодовой базе на предмет похожих паттернов
- Изучите релевантную документацию (спецификации, дизайн-документы, ADR)
- Проверьте последние коммиты в связанных модулях
- Поймите зависимости и точки интеграции

НИКОГДА не делегируйте и не реализовывайте «вслепую».

**2. ДЕЛЕГИРОВАНИЕ ПОДАГЕНТАМ**

Перед делегированием:

- Предоставьте полный контекст (фрагменты кода, пути к файлам, паттерны, документация)
- Чётко укажите ожидаемый результат и критерии валидации

После делегирования (КРИТИЧЕСКИ ВАЖНО):

- ВСЕГДА верифицируйте результат (прочитайте изменённые файлы, запустите type-check)
- НИКОГДА не пропускайте верификацию
- Если результат неверен: повторно делегируйте с исправлениями и описанием ошибок
- Если ошибки TypeScript: повторно делегируйте тому же агенту **ИЛИ** `typescript-types-specialist`

**3. ПРЯМАЯ РЕАЛИЗАЦИЯ (ТОЛЬКО ДЛЯ ТРИВИАЛЬНЫХ СЛУЧАЕВ)**

Прямая реализация допустима только для:

- Установки одного пакета
- Исправлений в одну строку (опечатки, очевидные баги)
- Простых импортов
- Минимальных изменений конфигурации

Всё остальное — делегируйте.

**4. ОТСЛЕЖИВАНИЕ ПРОГРЕССА**

- Создавайте todo при старте задачи
- Помечайте как `in_progress` **ДО** начала работы
- Помечайте как `completed` **ТОЛЬКО ПОСЛЕ** верификации

**5. СТРАТЕГИЯ КОММИТОВ**

Выполняйте `/push patch` **ПОСЛЕ КАЖДОЙ** завершённой задачи:

- Отмечайте задачу как `[X]` в `tasks.md`
- Добавляйте артефакты: `→ Artifacts: [file1](path), [file2](path)`
- Обновляйте TodoWrite до статуса `completed`
- Затем выполняйте `/push patch`

**6. ПАТТЕРН ВЫПОЛНЕНИЯ**

```
ДЛЯ КАЖДОЙ ЗАДАЧИ:
1. Прочитайте описание задачи
2. СОБЕРИТЕ ПОЛНЫЙ КОНТЕКСТ (код + документация + паттерны + история)
3. Делегируйте подагенту ИЛИ реализуйте напрямую (только если тривиально)
4. ВЕРИФИЦИРУЙТЕ результат (прочитайте файлы + запустите type-check) — НИКОГДА не пропускайте
5. Цикл accept/reject (повторная делегация при необходимости)
6. Обновите TodoWrite до статуса completed
7. Отметьте задачу как [X] в tasks.md + добавьте артефакты
8. Выполните /push patch
9. Перейдите к следующей задаче
```

**7. ОБРАБОТКА ПРОТИВОРЕЧИЙ**

Если возникают противоречия:

- Соберите контекст, проанализируйте паттерны проекта
- Если действительно неоднозначно: задайте уточняющий вопрос пользователю с конкретными вариантами
- Задавайте вопрос **только** если невозможно определить лучшую практику (редко, ~10%)

**8. ПОДХОД «БИБЛИОТЕКА ВПЕРВЫЕ» (ОБЯЗАТЕЛЬНО)**

Перед написанием нового кода (>20 строк) ВСЕГДА ищите существующие библиотеки:

- WebSearch: `"npm {функциональность} library 2024"` или `"python {функциональность} package"`
- Context7: документация по кандидатным библиотекам
- Проверяйте: недельные загрузки >1000, коммиты за последние 6 месяцев, поддержка TypeScript/типов

**Используйте библиотеку, если**:

- Покрывает >70% требуемой функциональности
- Активно поддерживается, нет критических уязвимостей
- Разумный размер бандла (проверьте на bundlephobia.com)

**Пишите кастомный код, если**:

- <20 строк простой логики
- Все библиотеки заброшены или небезопасны
- Ядро бизнес-логики, требующее полного контроля

### Фаза планирования (ВСЕГДА сначала)

Перед реализацией задач:

- Проанализируйте модель выполнения (параллельная/последовательная)
- Назначьте исполнителей: `MAIN` — для тривиальных, `existing` — при 100% совпадении, `FUTURE` — иначе
- Создайте `FUTURE`-агентов: запустите N вызовов `meta-agent-v3` в **одном сообщении**, запросите перезапуск
- Разрешите исследования (простые: решайте сейчас, сложные: промпт `deepsresearch`)
- Атомарность: 1 задача = 1 вызов агента
- Параллелизм: запускайте N вызовов в одном сообщении (не последовательно)

См. `speckit.implement.md` для деталей.

## Соглашения проекта

**Организация файлов**:

- Агенты: `.claude/agents/{domain}/{orchestrators|workers}/`
- Команды: `.claude/commands/`
- Навыки: `.claude/skills/{skill-name}/SKILL.md`
- Временные файлы: `.tmp/current/` (игнорируется Git)
- Отчёты: `docs/reports/{domain}/{YYYY-MM}/`

**Стандарты кода**:

- Type-check **должен проходить** перед коммитом
- Сборка **должна проходить** перед коммитом
- Запрещены жёстко заданные учётные данные

**Выбор агентов**:

- Worker: файл плана указывает `nextAgent` (только для health-workflow)
- Skill: переиспользуемая утилита, без состояния, <100 строк

**Операции с Supabase**:

- Используйте Supabase MCP, если `.mcp.json` включает сервер supabase

**Конфигурация MCP**:

- BASE (`.mcp.base.json`): context7 + sequential-thinking (~600 токенов)
- FULL (`.mcp.full.json`): + supabase + playwright + n8n + shadcn (~5000 токенов)
- Переключение: `./switch-mcp.sh`
