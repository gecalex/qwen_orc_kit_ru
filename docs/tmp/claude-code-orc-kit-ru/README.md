## 🎯 Обзор

**Claude Code Orchestrator Kit** — это комплексный фреймворк для автоматизации, предназначенный для усиления вашего рабочего процесса разработки с Claude Code. Он предоставляет:

- **🤖 36+ специализированных ИИ-агентов** — оркестраторы и исполнители для обработки задач, связанных с багами, безопасностью, зависимостями, мёртвым кодом, переиспользованием и т.д.
- **⚡ Управление MCP-серверами** — 7 готовых MCP-конфигураций под разные сценарии использования (600–5000 токенов)
- **🔧 20+ команд с префиксом «/»** — проверки состояния, SpecKit, управление worktree, релизы
- **📊 Контрольные точки качества** — автоматическая проверка типов, сборка, тесты, покрытие кода, аудит безопасности
- **🎯 Библиотека навыков** — 15+ переиспользуемых утилит для валидации, отчётности и автоматизации
- **📈 Мониторинг состояния** — отслеживание производительности агентов, частоты успеха и общего «здоровья» экосистемы

## ✨ Возможности

### 🤖 **Экосистема ИИ-агентов**

- **Оркестраторы состояния** — полные рабочие процессы для багов, безопасности, зависимостей, мёртвого кода
- **Рабочие агенты разработки** — специалисты по LLM-сервисам, типам TypeScript, расчёту стоимости
- **Рабочие агенты тестирования** — интеграционные тесты, оптимизация производительности, адаптивность под мобильные устройства
- **Рабочие агенты баз данных** — аудит Supabase, построение API, архитектура БД
- **Рабочие агенты инфраструктуры** — Qdrant, валидаторы качества, логика оркестрации
- **Мета-агенты** — создатели агентов и навыков

### ⚙️ **Конфигурации MCP-серверов**

Переключайтесь между 7 оптимизированными MCP-конфигурациями в зависимости от задач:

| Конфигурация      | Серверы                        | Расход токенов | Сценарий использования                           |
| ----------------- | ------------------------------ | -------------- | ------------------------------------------------ |
| **BASE**          | Context7 + Sequential Thinking | ~600           | Минимальный режим, повседневное использование    |
| **SUPABASE**      | Base + Supabase (один проект)  | ~2500          | Работа с базой данных                            |
| **SUPABASE-FULL** | Base + Supabase (два проекта)  | ~3000          | Мультипроектная работа с БД                      |
| **N8N**           | Base + n8n automation          | ~2500          | Автоматизация рабочих процессов                  |
| **FRONTEND**      | Base + Playwright + ShadCN     | ~2000          | UI/UX-разработка                                 |
| **SERENA**        | Base + Serena LSP              | ~2500          | Семантический поиск по коду                      |
| **FULL**          | Все серверы + Serena           | ~6500          | Полноразмерный режим с максимальным функционалом |

### 🚀 **Команды с префиксом «/»**

- **Проверки состояния** — `/health-bugs`, `/health-security`, `/health-deps`, `/health-cleanup`, `/health-metrics`
- **SpecKit** — `/speckit.analyze`, `/speckit.specify`, `/speckit.implement`, `/speckit.checklist`
- **Управление worktree** — `/worktree-create`, `/worktree-list`, `/worktree-cleanup`, `/worktree-remove`
- **Управление релизами** — `/push` (автоматическое обновление версии и формирование changelog)
- **Перевод** — `/translate-doc` (английский ↔ русский)

### 📊 **Контрольные точки качества**

Автоматизированные скрипты валидации:

- **Проверка размера бандла** — гарантирует, что размер продакшен-бандла остаётся в пределах лимита
- **Аудит безопасности** — сканирование на наличие уязвимостей npm с уровнем critical/high
- **Покрытие кода** — проверка, что покрытие тестами соответствует заданному порогу

### 🎯 **Библиотека навыков**

Переиспользуемые утилиты для:

- Валидации планов, генерации отчётов, создания changelog
- Операций с Git, разбора ошибок, рендеринга шаблонов
- Выполнения контрольных точек, управления откатами
- Оценки приоритета, извлечения версий

## 🔑 Ключевые инновации

### 🎯 **Паттерн оркестратора**

**Парадигмальный сдвиг**: превращение Claude Code из инструмента, выполняющего всё напрямую, в **оркестратора**, который делегирует сложные задачи специализированным подагентам.

**Почему это важно**:

- **Сохранение контекста**: основной Claude Code остаётся лёгким (~10–15K токенов вместо 50K+ в стандартном использовании)
- **Специализация**: каждый подагент — эксперт в своей области (фикс багов, сканирование безопасности, архитектура БД)
- **Контроль качества**: обязательная верификация после каждой делегации (чтение файлов + запуск type-check)
- **Неограниченная работа**: можно работать над проектом неограниченно долго без исчерпания контекста

**Основные правила** (из `CLAUDE.md`):

1. **СБОР ПОЛНОГО КОНТЕКСТА СНАЧАЛА** — читать код, искать паттерны, проверять коммиты до делегации
2. **ДЕЛЕГИРОВАНИЕ ПОДАГЕНТАМ** — предоставлять полный контекст + критерии валидации
3. **ВЕРИФИКАЦИЯ РЕЗУЛЬТАТОВ** — никогда не пропускать проверку (читать изменённые файлы, запускать type-check)
4. **ЦИКЛ ACCEPT/REJECT** — повторная делегация с исправлениями, если проверка провалилась
5. **КОММИТЫ ПО ЗАДАЧАМ** — выполнять `/push patch` после каждой завершённой задачи

### 📋 **Улучшение SpecKit: Фаза 0 — Планирование**

**SpecKit** (от GitHub) предоставляет структурированный рабочий процесс разработки. Мы улучшили его, добавив **Фазу 0: Планирование**.

**Обязанности Фазы 0**:

1. **Назначение исполнителя**:
   - `[EXECUTOR: MAIN]` — только тривиальные задачи (исправления в 1–2 строки, простые импорты)
   - `[EXECUTOR: existing-agent]` — если есть точное совпадение с существующим подагентом
   - `[EXECUTOR: FUTURE-agent-name]` — если совпадения нет (агент нужно создать)

2. **Создание мета-агентов**:
   - Запуск N вызовов `meta-agent-v3` **в одном сообщении** для параллельного создания агентов
   - **Правило атомарности**: 1 задача = 1 вызов агента
   - После создания: запросить у пользователя перезапуск Claude Code

3. **Разрешение исследований**:
   - Простые исследования: агент решает с имеющимися инструментами
   - Сложные исследования: создание промпта в директории `research/`

**Почему это важно**: обеспечивает наличие всех необходимых агентов до начала реализации, позволяет выполнять задачи параллельно и предотвращает переполнение контекста.

### 🤖 **Мета-агент: фабрика агентов**

**meta-agent-v3** создаёт новых специализированных агентов за 2–3 минуты, следуя паттернам проекта:

- **Workers** — выполняют задачи из файлов плана (5-фазная структура)
- **Orchestrators** — координируют многофазные рабочие процессы (паттерн возврата управления)
- **Simple Agents** — автономные инструменты

**Как это работает**:

1. Загружает архитектурные документы (`ARCHITECTURE.md` + `CLAUDE.md`)
2. Определяет тип агента и требования
3. Генерирует YAML-frontmatter + структуру + валидацию + обработку ошибок
4. Записывает в соответствующее расположение
5. Проверяет соответствие паттернам проекта

### 🔄 **Паттерн возврата управления**

**Оркестраторы** координируют рабочие процессы без прямого вызова воркеров:

```
Orchestrator → Создаёт файл плана → Сигнализирует готовность → ВЫХОД
↓
Основная сессия → Вызывает воркер через инструмент Task
↓
Worker → Выполняет → Валидирует → Отчитывается → ВЫХОД
↓
Orchestrator → Возобновляет → Проверяет → Переходит к следующей фазе
```

**Почему не использовать инструмент Task напрямую?**: это создало бы вложенные контексты, нарушая изоляцию.

### ⚙️ **Динамическое переключение MCP**

**Проблема**: каждый MCP-сервер потребляет 500–1500 токенов из бюджета контекста.

**Решение**: скрипт `switch-mcp.sh` динамически переключается между 6 конфигурациями:

- **BASE** (~600 токенов): Context7 + Sequential Thinking (ежедневное использование)
- **SUPABASE** (~2500): + Supabase (работа с БД)
- **FRONTEND** (~2000): + Playwright + ShadCN (UI-работа)
- **FULL** (~5000): все серверы (при необходимости)

**Выгода**: экономия 500–4500 токенов за счёт загрузки только необходимого.

### 🌳 **Интеграция worktree + VS Code**

**Параллельная разработка фич**:

1. Создание worktree для разных фич: `/worktree-create feature/new-auth`
2. Добавление `.worktrees/*` в рабочие папки VS Code (см. `.claude/settings.local.json.example`)
3. Переключение между фичами через селектор папок
4. Запуск нескольких сессий Claude Code параллельно

**Выгода**: 3–5 фич параллельно, без загрязнения контекста, изолированное тестирование.

### 🔔 **Интеграция вебхуков**

**Уведомления о завершении задач** (`.claude/settings.local.json.example`):

```json
{
  "hooks": {
    "Stop": [
      {
        "type": "command",
        "command": "notify-send 'Claude Code' 'Task completed!'"
      }
    ]
  }
}
```

**Сценарии использования**: уведомления в Slack, системные алерты, Telegram-боты, лог-файлы.

**Выгода**: запустил задачу, переключился на другой проект, получил уведомление по завершению.

### 🎯 **Навыки vs Агенты**

**Навыки (Skills)** (15+): переиспользуемые утилиты (<100 строк), stateless, вызываются через инструмент `Skill`

- Примеры: `run-quality-gate`, `validate-plan-file`, `generate-report-header`
- Нет изоляции контекста, выполняются в контексте вызывающего

**Агенты (Agents)** (33+): stateful-рабочие процессы, изолированный контекст, вызываются через инструмент `Task`

- Примеры: `bug-hunter`, `security-scanner`, `database-architect`
- Полная изоляция контекста, многошаговые процессы

### 📐 **Нетрадиционный CLAUDE.md**

**Стандартная практика**: хранить всю историю проекта в `CLAUDE.md`  
— Проблема: тратятся токены контекста на исторические данные

**Наша инновация**: `CLAUDE.md` как **Поведенческая операционная система (Behavioral OS)**

- Содержит **ТОЛЬКО** правила оркестрации (без истории проекта)
- Определяет, **как собирать контекст ДО делегации**
- Указывает правила **верификации ПОСЛЕ делегации**
- Принудительно сохраняет контекст

**Результат**: основной Claude Code остаётся лёгким, весь контекст загружается по требованию.

### 🧠 **Интеграция DekdsenFlow (ленивая загрузка знаний)**

Вдохновлено [deksden-flow](https://github.com/deksden/deksden-flow), реализуем **управление контекстом без накладных расходов**:

**Индекс проекта** (`.claude/project-index.md`):

- Компактная карта проекта с аннотированными ссылками
- Загружается по требованию через навык `load-project-context`
- ~100–200 токенов при использовании, 0 в базовом состоянии

**Контекст сессии** (`.tmp/current/session/`):

- `context.md`: текущее состояние рабочего процесса для возобновления
- `log.md`: принятые решения, проблемы, выводы (только запись)
- Позволяет бесшовно продолжать сессии

**Почемему это важно**:

- Возобновление health-воркфлоу после перезапуска сессии
- Отслеживание решений для отладки
- Навигация по кодовой базе без полного её изучения

**Влияние на токены**: нулевые накладные расходы. Навыки загружают контекст только при необходимости.

### 🔬 **Интеграция Serena LSP**

**Serena MCP** обеспечивает семантическое понимание кода через Language Server Protocol:

- **Поиск символов**: нахождение функций, классов, типов по имени
- **Поиск ссылок**: «найти все использования X»
- **Интеллектуальный рефакторинг**: переименование с учётом всех ссылок
- **Контекст**: режим `ide-assistant` избегает дублирования инструментов

**Когда использовать**: в крупных кодовых базах, где Grep даёт слишком много ложных срабатываний.

## 🚀 Быстрый старт

```bash
# 1. Клонируйте или скачайте репозиторий
git clone https://github.com/maslennikov-ig/claude-code-orchestrator-kit.git
cd claude-code-orchestrator-kit

# 2. Настройте переменные окружения
cp .env.example .env.local
# Отредактируйте .env.local, указав ваши учётные данные

# Ваша система оркестрации готова!
```

## 📦 Установка

### Требования

- Установленный **Claude Code**
- **Node.js** 18+ (для MCP-серверов)
- **Docker** (опционально, для MCP-сервера n8n)
- **Git** (для функций контроля версий)

### Шаги установки

#### 1️⃣ **Копирование в ваш проект**

```bash
# Вариант A: скопируйте всю директорию .claude в ваш проект
cp -r claude-code-orchestrator-kit/.claude /path/to/your/project/

# Вариант B: клонируйте и используйте как шаблон
git clone https://github.com/maslennikov-ig/claude-code-orchestrator-kit.git my-project
cd my-project
rm -rf .git
git init
```

#### 2️⃣ **Настройка переменных окружения**

```bash
# Скопируйте пример в локальный файл
cp .env.example .env.local

# Отредактируйте, указав ваши учётные данные
# Обязательно для Supabase:
SUPABASE_PROJECT_REF=your-project-ref
SUPABASE_ACCESS_TOKEN=your-token
SUPABASE_DB_PASSWORD=your-password

# Обязательно для Sequential Thinking:
SEQUENTIAL_THINKING_KEY=your-smithery-key
SEQUENTIAL_THINKING_PROFILE=your-profile

# Опционально для n8n:
N8N_API_URL=https://your-n8n.com
N8N_API_KEY=your-n8n-key
```

**Важно:** Никогда не коммитьте `.env.local` в Git! Он уже добавлен в `.gitignore`.

#### 3️⃣ **Выбор конфигурации MCP**

```bash
./switch-mcp.sh
```

Выберите конфигурацию в зависимости от вашего рабочего процесса:

- **Опция 1** — BASE (минимальная, ~600 токенов)
- **Опция 2** — SUPABASE (работа с БД)
- **Опция 3** — SUPABASE-FULL (мультипроект)
- **Опция 4** — N8N (автоматизация рабочих процессов)
- **Опция 5** — FRONTEND (UI/UX-разработка)
- **Опция 6** — FULL (все функции)

#### 4️⃣ **Перезапуск Claude Code**

После смены конфигурации MCP обязательно перезапустите Claude Code, чтобы изменения вступили в силу.

#### 5️⃣ **Настройка локальных параметров (опционально)**

```bash
# Скопируйте пример настроек
cp .claude/settings.local.json.example .claude/settings.local.json

# Отредактируйте в соответствии с вашими предпочтениями
# - Выберите, какие MCP-серверы включить
# - Настройте хуки для завершения задач
# - Адаптируйте под ваш рабочий процесс
```

#### 6️⃣ **Проверка установки**

```bash
# Проверьте текущую MCP-конфигурацию
./switch-mcp.sh
# Выберите опцию 0, чтобы увидеть активные серверы

# Попробуйте команду проверки состояния в Claude Code
/health-bugs
```

## 🔌 Конфигурации MCP-серверов

Все MCP-конфигурации хранятся в директории `./mcp/`.

### Доступные конфигурации

#### **BASE** (`mcp/.mcp.base.json`)

```json
{
  "mcpServers": {
    "context7": { ... },              // Документация по библиотекам
    "server-sequential-thinking": { ... }  // Усиленное рассуждение
  }
}
```

**Используйте для:** повседневной разработки, минимального расхода токенов

---

#### **SUPABASE** (`mcp/.mcp.supabase-only.json`)

```json
{
  "mcpServers": {
    "context7": { ... },
    "server-sequential-thinking": { ... },
    "supabase": { ... }               // Основной проект Supabase
  }
}
```

**Используйте для:** проектирования БД, политик RLS, SQL-запросов

---

#### **FULL** (`mcp/.mcp.full.json`)

```json
{
  "mcpServers": {
    "context7": { ... },
    "server-sequential-thinking": { ... },
    "supabase": { ... },
    "supabase-legacy": { ... },       // Вторичный проект
    "n8n-workflows": { ... },         // Автоматизация рабочих процессов
    "n8n-mcp": { ... },               // Управление сервером n8n
    "playwright": { ... },            // Автоматизация браузера
    "shadcn": { ... }                 // UI-компоненты
  }
}
```

**Используйте для:** полномасштабной разработки, максимальных возможностей

---

### Переключение конфигураций

```bash
# Интерактивное меню
./switch-mcp.sh

# Или вручную
cp mcp/.mcp.base.json .mcp.json
```

**Помните:** после смены конфигурации перезапустите Claude Code!

## 🎯 Команды с префиксом «/»

### Мониторинг состояния

#### `/health-bugs`

Полный рабочий процесс обнаружения и исправления багов.

**Что делает:**

- Сканирует всю кодовую базу на наличие багов
- Классифицирует по приоритету (critical → high → medium → low)
- Исправляет баги поэтапно
- Запускает контрольные точки качества после каждого этапа
- Проводит верификационное сканирование и итеративное уточнение

**Использование:**

```
/health-bugs
```

**Результат:** `bug-hunting-report.md`, `bug-fixes-implemented.md`

---

#### `/health-security`

Обнаружение и устранение уязвимостей безопасности.

**Проверяет:**

- Уязвимости SQL-инъекций
- Векторы XSS-атак
- Проблемы аутентификации/авторизации
- Пробелы в политиках RLS
- Жёстко заданные секреты

**Использование:**

```
/health-security

## 🤖 Экосистема агентов

### Архитектура

**Orchestrators** → Планируют рабочие процессы, создают задачи, проверяют результаты, управляют итерациями  
**Workers** → Выполняют конкретные задачи, генерируют отчёты, возвращают управление  
**Skills** → Переиспользуемые утилиты для валидации, отчётности, автоматизации

### Оркестраторы состояния

#### `bug-orchestrator`

Управляет полным рабочим процессом исправления багов.

**Фазы:**

1. Предварительная валидация
2. Обнаружение багов (через bug-hunter)
3. Контрольная точка 1 (валидация)
4. Поэтапное исправление по приоритету
5. Контрольные точки качества по приоритетам
6. Верификационное сканирование
7. Итеративное уточнение (до 3 циклов)
8. Финальное резюме

**Ключевые особенности:**

- Исправление по приоритету (critical → low)
- Контрольные точки качества (type-check, build, tests)
- Откат при сбое
- Отслеживание итераций
- Комплексная отчётность

---

#### `security-orchestrator`

Рабочий процесс уязвимостей безопасности.

**Аналогично bug-orchestrator, но для:**

- Уязвимостей безопасности
- Проблем аутентификации
- Обходов авторизации
- Рисков утечки данных
- Управления секретами

---

#### `dependency-orchestrator`

Рабочий процесс безопасного обновления зависимостей.

**Возможности:**

- Аудит существующих зависимостей
- Стратегия обновления по одному
- Валидация после каждого обновления
- Откат при сбое
- Управление lock-файлами

---

#### `dead-code-orchestrator`

Рабочий процесс очистки мёртвого кода.

**Цели:**

- Неиспользуемые импорты
- Закомментированный код
- Недостижимые пути
- Отладочные остатки
- Осиротевшие функции

---

#### `reuse-orchestrator`

Рабочий процесс устранения дублирования кода.

**Фазы:**

1. **Обнаружение**: `reuse-hunter` сканирует на дублирующиеся типы/логику
2. **Анализ**: Группировка дубликатов и определение лучшего места для SSOT
3. **Консолидация**: `reuse-fixer` объединяет дубликаты в общие файлы
4. **Рефакторинг**: Обновление всех ссылок на SSOT
5. **Валидация**: Гарантия отсутствия ломающих изменений через контрольные точки

SEQUENTIAL_THINKING_PROFILE=your-profile-name
```

**Опционально для n8n:**

```bash
N8N_API_URL=https://your-n8n.com
N8N_API_KEY=your-api-key
```

### Шаблон конфигурации MCP

Конфигурации MCP используют интерполяцию переменных окружения:

```json
{
  "mcpServers": {
    "supabase": {
      "command": "npx",
      "args": [
        "-y",
        "@supabase/mcp-server-supabase@latest",
        "--project-ref=${SUPABASE_PROJECT_REF}"
      ],
      "env": {
        "SUPABASE_ACCESS_TOKEN": "${SUPABASE_ACCESS_TOKEN}"
      }
    }
  }
}
```

Переменные автоматически подставляются из `.env.local`.

### 6. Придерживайтесь подхода «библиотека в первую очередь»

Перед написанием нового кастомного кода (>20 строк) всегда ищите существующие решения:

1. **Поиск**: проверьте npm/PyPI на наличие библиотек с >1k еженедельных загрузок
2. **Оценка**: проверьте статус поддержки, поддержку типов и размер бандла
3. **Принятие**: используйте библиотеку, если она покрывает >70% функциональности
4. **Разработка**: пишите кастомный код только для уникальной бизнес-логики или если подходящей библиотеки нет

**Почему?** Снижает нагрузку на поддержку, использует проверенные сообществом решения и ускоряет разработку.

## 🔧 Устранение неполадок

### MCP-сервер не работает

**Проблема:** MCP-сервер не появляется после переключения

**Решение:**

```bash
# 1. Убедитесь, что конфигурация скопирована
cat .mcp.json

# 2. Проверьте переменные окружения
cat .env.local

# 3. Полностью перезапустите Claude Code
# (не просто перезагрузите)

# 4. Проверьте логи Claude Code на наличие ошибок
```

---

### Сбои контрольных точек качества

**Проблема:** type-check или build постоянно падают

**Решение:**

```bash
# 1. Запустите вручную, чтобы увидеть полные ошибки
npm run type-check
npm run build

# 2. Проверьте наличие проблем с TypeScript
npx tsc --noEmit

# 3. Проверьте настройки валидации
cat .claude/scripts/gates/check-*.sh
```

---

### Рабочий процесс агента застрял

**Проблема:** Рабочий процесс оркестратора не продвигается

**Решение:**

```bash
# 1. Проверьте файл плана
cat .tmp/current/plans/*.json

# 2. Убедитесь, что отчёты сгенерированы
ls -la .tmp/current/reports/

# 3. Проверьте наличие бесконечного цикла
# (навык check-infinite-loop)

# 4. Просмотрите журнал изменений
cat .tmp/current/changes/*.json
```

---

### Переменные окружения не загружаются

**Проблема:** MCP-серверы показывают «undefined» вместо переменных

**Решение:**

```bash
# 1. Убедитесь, что .env.local существует
ls -la .env.local

# 2. Проверьте формат (без кавычек)
cat .env.local
# Верно: SUPABASE_PROJECT_REF=abc123
# Неверно: SUPABASE_PROJECT_REF="abc123"

# 3. Перезапустите Claude Code
# Переменные загружаются при запуске
```

1. Создайте конфиг в директории `mcp/`
2. Следуйте соглашению об именовании: `.mcp.{name}.json`
3. Используйте переменные окружения для секретов
4. Обновите меню в `switch-mcp.sh`
5. Задокументируйте в README
