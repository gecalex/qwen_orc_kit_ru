# Шаблон оркестрации на основе агентов

> **КОНТЕКСТ**: Этот шаблон используется в ~5% случаев для автоматизированных рабочих процессов проверки состояния.
> Для ручной разработки (95% случаев) см. CLAUDE.md.

## Когда использовать

Этот шаблон применяется ТОЛЬКО к автоматизированным рабочим процессам проверки состояния:
- `/health-bugs` - Обнаружение и исправление ошибок
- `/health-security` - Сканирование уязвимостей безопасности
- `/health-cleanup` - Удаление мертвого кода
- `/health-deps` - Управление зависимостями

## Почему отличается от основного шаблона?

| Аспект | Основная сессия (95%) | На основе агентов (5%) |
|--------|-------------------|------------------|
| Оркестратор | Сама основная сессия | Команда с косой чертой + подоркестраторы |
| Коммуникация | Прямое делегирование инструментом Task | Файлы плана |
| Контекст | Полная история разговора | Изолированный контекст агента |
| Доступ к инструментам | Основная сессия свободно использует инструмент Task | Агенты-оркестраторы не могут использовать инструмент Task |
| Шаблон | Вы делегируете и проверяете | Команда координирует многофазный рабочий процесс |

## Архитектура

```
Пользователь запускает /health-bugs
  ↓
Команда с косой чертой (главный оркестратор)
  ↓
Инструмент Task → bug-orchestrator (подоркестратор)
  ↓
Создает файл плана: .tmp/current/plans/bug-detection.json
Возвращает управление
  ↓
Команда читает файл плана
  ↓
Инструмент Task → bug-hunter (работник)
  ↓
Работник выполняет, генерирует отчет, возвращает управление
  ↓
Команда → инструмент Task → bug-orchestrator (проверка)
  ↓
Оркестратор проверяет на контрольной точке качества
Создает следующий файл плана
Возвращает управление
  ↓
...цикл продолжается
```

## Ключевые компоненты

### 1. Команды с косой чертой (Главные оркестраторы)

**Расположение**: `.claude/commands/health-*.md`

**Роль**:
- Координировать полный цикл рабочего процесса
- Вызывать подоркестраторы через инструмент Task
- Читать файлы плана между фазами
- Вызывать работников через инструмент Task
- Отображать результаты пользователю

**Ответственность**:
- Координация фаз
- Чтение файлов плана для определения следующих шагов
- Явный вызов агентов с инструментом Task
- Обработка взаимодействия с пользователем

### 2. Агенты-оркестраторы (Подоркестраторы)

**Расположение**: `.claude/agents/health/orchestrators/`

**Примеры**: bug-orchestrator, security-orchestrator, dead-code-orchestrator

**Роль**:
- Создавать файлы плана для каждой фазы
- Проверять результаты работы работников на контрольных точках качества
- Возвращать управление основной сессии после каждой фазы

**Критические правила**:
- НЕ использовать инструмент Task (использовать файлы плана вместо этого)
- НЕ выполнять работу реализации (только координация)
- НЕ пропускать проверки контрольных точек качества
- ДОЛЖЕН создать файлы плана перед возвратом управления
- ДОЛЖЕН проверить результаты работы
- ДОЛЖЕН вернуть управление после каждой фазы

**Почему без инструмента Task?**
Агенты-оркестраторы работают в изолированном контексте без доступа к инструменту Task. Они координируются через файловую коммуникацию (файлы плана).

### 3. Работники

**Расположение**: `.claude/agents/health/workers/`

**Примеры**: bug-hunter, bug-fixer, security-scanner, vulnerability-fixer

**Роль**:
- Сначала прочитать файл плана
- Выполнить работу, специфичную для области (обнаружение, исправление, сканирование)
- Проверить работу внутри себя
- Сгенерировать структурированный отчет
- Вернуться в основную сессию

**Критические правила**:
- ДОЛЖЕН сначала прочитать файл плана
- НЕ вызывать других агентов
- НЕ пропускать генерацию отчета
- ДОЛЖЕН записывать изменения для возможности отката
- ДОЛЖЕН проверить себя перед сообщением об успехе

## Файлы плана

**Расположение**: `.tmp/current/plans/{workflow}-{phase}.json`

**Назначение**: Коммуникация на основе файлов между оркестраторами и работниками

**Формат**:
```json
{
  "workflow": "bug-management",
  "phase": "detection",
  "config": {
    "priority": "all",
    "severity": "critical"
  },
  "validation": {
    "required": ["report-exists", "type-check", "build"],
    "optional": ["tests-pass"]
  },
  "nextAgent": "bug-hunter"
}
```

**Примеры**:
- `.bug-detection-plan.json`
- `.bug-fixing-critical-plan.json`
- `.security-scan-plan.json`
- `.dependency-audit-plan.json`

**Жизненный цикл**:
1. Оркестратор создает файл плана
2. Оркестратор возвращает управление основной сессии
3. Основная сессия (команда) читает файл плана
4. Основная сессия вызывает работника, указанного в `nextAgent`
5. Работник читает файл плана, выполняет работу
6. Работник возвращает управление
7. Основная сессия вызывает оркестратора для проверки

## Файлы отчетов

**Расположение**:
- Временные: `.tmp/current/reports/{report-name}.md`
- Постоянные: `docs/reports/{domain}/{YYYY-MM}/{date}-{report-name}.md`

**Обязательные разделы**:
1. Заголовок: Тип отчета, отметка времени, статус
2. Итоговая сводка: Ключевые метрики, статус проверки
3. Детальные находки: Изменения, проблемы, действия
4. Результаты проверки: ПРОШЕЛ/НЕ ПРОШЕЛ для каждой точки
5. Следующие шаги: Рекомендации

**Формат**: Следовать `docs/Agents Ecosystem/REPORT-TEMPLATE-STANDARD.md`

## Контрольные точки качества

**Блокирующие точки** (должны пройти):
- Файл отчета существует и правильно оформлен
- Статус проверки ПРОШЕЛ
- Проверка типов прошла
- Сборка прошла
- Нет критических ошибок

**Неблокирующие точки** (только предупреждения):
- Тесты прошли (рекомендуется)
- Линтер прошел
- Бенчмарки производительности
- Документация завершена

**При блокирующем сбое**:
1. ОСТАНОВИТЬ рабочий процесс
2. Сообщить о сбоях пользователю
3. Предоставить корректирующие действия
4. Спросить пользователя: "Исправить проблемы или пропустить проверку? (fix/skip)"

**Инструменты проверки**:
- Использовать навык `run-quality-gate` для стандартизированной проверки
- Использовать навык `validate-plan-file` для проверки файла плана
- Использовать навык `validate-report-file` для проверки отчета

## Фаза планирования (Рабочий процесс Speckit)

**Контекст**: Это применяется к разработке функций на основе speckit, НЕ к рабочим процессам проверки состояния.

**Позиция**: Всегда первая фаза перед реализацией в `/speckit.implement`

**Ответственность**:
1. **Анализ задач**:
   - Просмотреть все задачи из tasks.md
   - Классифицировать каждую как выполнение ПАРАЛЛЕЛЬНО или ПОСЛЕДОВАТЕЛЬНО
   - Определить зависимости между задачами

2. **Назначение подагентов**:
   - Сопоставить каждую задачу соответствующему подагенту из существующей базы
   - Документировать исполнителя в аннотации задачи: `[EXECUTOR: subagent-name]` или `[EXECUTOR: MAIN]`
   - Главный агент обрабатывает только тривиальные задачи (исправления 1-2 строк, простые импорты, установка одной зависимости)

3. **Отсутствующие подагенты**:
   - Если для задачи нет подходящего подагента, создать задачу мета-агента
   - Добавить к фазе планирования: `Create [subagent-name] using meta-agent-v3`
   - Выполнить задачи мета-агента параллельно во время планирования

4. **Правило атомарности** (КРИТИЧНО):
   - **1 Задача = 1 вызов агента**
   - Никогда не давать несколько задач одному агенту за один запуск
   - Пример: 3 параллельные задачи для мета-агента → 3 отдельных запуска мета-агента (одновременно)
   - Пример: 5 параллельных задач реализации → 5 отдельных запусков подагента (одновременно)
   - Последовательные задачи: 1 запуск агента, ждать завершения, затем следующий агент

**Выход**: Обновленный tasks.md с:
- Аннотациями `[EXECUTOR: name]`
- Маркерами `[SEQUENTIAL]` или `[PARALLEL-GROUP-X]`
- Задачами создания мета-агента при необходимости

## Рабочий процесс исследовательской задачи (Рабочий процесс Speckit)

**Контекст**: Это применяется к фазе планирования speckit, НЕ к рабочим процессам проверки состояния.

**Когда**: Фаза планирования, после назначения подагента

**Классификация**:
- **Простое исследование**: Вопросы с четким путем ответа, решаемые с помощью агента + существующих инструментов
- **Сложное исследование**: Вопросы без очевидных ответов, требующие глубокого исследования

**Рабочий процесс простого исследования**:
1. Агент определяет вопрос
2. Агент использует доступные инструменты (Grep, Read, WebSearch, Context7, документация Supabase)
3. Агент формулирует ответ
4. Агент документирует находки в заметках планирования
5. Продолжить к следующей задаче

**Рабочий процесс сложного исследования**:
1. Агент определяет сложный вопрос без четкого ответа
2. Агент создает подробный запрос на английском для deepresearch:
   - **Файл**: `{FEATURE_DIR}/research/{topic-slug}.md`
   - **Формат**: Вопрос исследования + контекст + ограничения + ожидаемый формат вывода
   - **Язык**: Английский (для совместимости с deepresearch)
3. Агент добавляет TODO: "Wait for deepresearch results on {topic}"
4. Агент ОСТАНАВЛИВАЕТСЯ и сообщает пользователю: "Complex research required: {file-path}"
5. **Действие пользователя**: Запустить deepresearch с запросом, сохранить результаты
6. Агент читает результаты deepresearch
7. Агент включает находки в планирование
8. Продолжить к следующей задаче

**Выход**:
- Исследовательские находки документированы
- Запросы сложного исследования в `{FEATURE_DIR}/research/` при необходимости
- Обновленный tasks.md с решениями, основанными на исследовании

## Шаблоны рабочих процессов

### Шаблон 1: Итеративный цикл (Ошибки, Безопасность, Мертвый код)

```
Фаза 1: Обнаружение
- Оркестратор создает план обнаружения
- Охотник/Сканер выполняет, генерирует категоризованный отчет
- Оркестратор проверяет отчет

Фаза 2: Исправление (по приоритетам)
- Оркестратор создает план исправления (приоритет=критический)
- Исправитель выполняет критические исправления
- Оркестратор проверяет (контрольная точка: проверка типов, сборка)
- Если проверка проходит, перейти к следующему приоритету
- Повторить для высоких, средних, низких приоритетов

Фаза 3: Проверка
- Оркестратор создает план проверки
- Охотник/Сканер повторно сканирует для проверки исправлений
- Оркестратор проверяет проверку

Фаза 4: Решение об итерации
- Если найдены новые проблемы: вернуться к фазе 2
- Если достигнут максимум итераций (обычно 3): остановить
- Если нет проблем: итоговая сводка

Фаза 5: Итоговая сводка
- Оркестратор генерирует комплексный отчет
```

### Шаблон 2: Последовательное обновление (Зависимости)

```
Фаза 1: Аудит
- Оркестратор создает план аудита
- Аудитор сканирует и категоризует зависимости
- Оркестратор проверяет отчет

Фаза 2: Обновление (по одному)
- Оркестратор создает план обновления (категория=безопасность, серьезность=критическая)
- Обновлятор обновляет ОДНУ зависимость
- Оркестратор проверяет (контрольная точка: действительный файл блокировки, сборка, тесты)
- Если проверка проходит: обновить следующую зависимость
- Если проверка не проходит: откат, отметить проблемную, продолжить

Фаза 3: Проверка
- Оркестратор создает план проверки
- Аудитор повторно сканирует
- Оркестратор генерирует итоговый отчет
```

## Структура временных файлов

**Расположение**: `.tmp/current/`

```
.tmp/
├── current/              # Активный запуск оркестрации
│   ├── plans/           # Файлы плана для работников
│   │   ├── .bug-detection-plan.json
│   │   ├── .bug-fixing-critical-plan.json
│   │   └── .security-scan-plan.json
│   ├── reports/         # Временные отчеты от работников
│   │   ├── bug-hunting-report.md
│   │   └── security-audit.md
│   ├── changes/         # Журналы изменений для отката
│   │   ├── .bug-changes.json
│   │   └── .security-changes.json
│   ├── backups/         # Резервные копии файлов перед модификацией
│   └── locks/           # Файлы блокировки для предотвращения конфликта
└── archive/             # Исторические запуски (автоочистка > 7 дней)
    └── YYYY-MM-DD-HHMMSS/
        ├── plans/
        ├── changes/
        └── reports/
```

**Жизненный цикл**:
1. Команда создает каталоги `.tmp/current/` при начале рабочего процесса
2. Работники читают/пишут в `.tmp/current/` во время выполнения
3. После завершения рабочего процесса команда архивирует `.tmp/current/` в `.tmp/archive/{timestamp}/`
4. Автоочистка удаляет архивы старше 7 дней

## Ведение журнала изменений и откат

**Расположение журнала изменений**: `.tmp/current/changes/.{domain}-changes.json`

**Формат**:
```json
{
  "phase": "bug-fixing",
  "timestamp": "2025-01-10T14:30:00Z",
  "files_modified": [
    {
      "path": "src/components/Button.tsx",
      "backup": ".tmp/current/backups/Button.tsx.backup"
    }
  ],
  "files_created": ["src/utils/newHelper.ts"],
  "commands_executed": ["npm install lodash"]
}
```

**Процедура отката**:
Использовать навык `rollback-changes`:
```
Input: .tmp/current/changes/.bug-changes.json

Actions:
1. Restore modified files from backups
2. Delete created files
3. Revert commands (git checkout, npm install)
4. Remove artifacts (plan files, lock files)
5. Generate rollback report
```

## Шаблон возврата управления

**Зачем нужно?**
Claude Code НЕ поддерживает автоматический вызов агентов. Основная сессия должна явно вызывать всех агентов с помощью инструмента Task.

**Шаблон**:
1. Оркестратор создает файл плана с полем `nextAgent`
2. Оркестратор сообщает о готовности и выходит
3. Основная сессия (команда) читает файл плана
4. Основная сессия вызывает работника с помощью инструмента Task с `subagent_type` из плана
5. Работник выполняет, возвращает управление
6. Основная сессия вызывает оркестратора для проверки
7. Повторить цикл

**Протокол сигнала готовности**:
Оркестраторы должны выводить:
```
Phase preparation complete.

Plan: .tmp/current/plans/.bug-detection-plan.json
Next Agent: bug-hunter

Returning control to main session.
```

## Лучшие практики

**Для оркестрации основной сессии (Speckit)**:
- Применить правило атомарности: 1 задача = 1 вызов агента
- Собрать полный контекст перед делегированием (код, документы, шаблоны, история, исследования)
- Делегировать подагентам с полным контекстом (фрагменты кода, пути к файлам, критерии проверки)
- Тщательно проверять результаты (читать файлы, запускать проверку типов, проверять регрессии)
- Цикл принять/отклонить: повторно делегировать с исправлениями, если проверка не пройдет
- Фиксировать после каждой задачи: `/push patch`
- Отслеживать прогресс с помощью TodoWrite (in_progress → completed immediately)

**Для агентов-оркестраторов (Рабочие процессы проверки состояния)**:
- Всегда проверять файлы плана после создания с помощью навыка `validate-plan-file`
- Отслеживать прогресс с помощью TodoWrite (отметить фазы in_progress → completed immediately)
- Обеспечить контрольные точки качества, никогда не пропускать проверки
- Ограничить итерации (максимум 3 цикла) для предотвращения бесконечных циклов
- Генерировать комплексные отчеты с резюме всех фаз
- Грациозно обрабатывать ошибки с инструкциями отката

**Для работников**:
- Всегда читать файл плана первым, никогда не предполагать конфигурацию
- Записывать все изменения в журнал изменений для возможности отката
- Проверять себя перед сообщением об успехе
- Генерировать структурированные отчеты, следуя стандартному формату
- Использовать серверы MCP когда указано в политике
- Возвращать управление после завершения работы

**Для команд**:
- Читать файлы плана для определения следующих шагов
- Явно вызывать агентов с помощью инструмента Task
- Обрабатывать результаты проверки оркестратора
- Отображать прогресс и результаты пользователю
- Управлять жизненным циклом рабочего процесса (начало → фазы → завершение)

## Примеры атомарности (Speckit)

**Пример 1: Создание мета-агента**

❌ **НЕПРАВИЛЬНО** (группировка нескольких задач):
```
Task tool → meta-agent-v3
Prompt: "Create 3 subagents: user-service-specialist, payment-specialist, notification-specialist"
```

✅ **ПРАВИЛЬНО** (1 задача = 1 запуск агента):
```
Task tool → meta-agent-v3
Prompt: "Create user-service-specialist subagent"

Task tool → meta-agent-v3
Prompt: "Create payment-specialist subagent"

Task tool → meta-agent-v3
Prompt: "Create notification-specialist subagent"
```

**Пример 2: Параллельная реализация**

❌ **НЕПРАВИЛЬНО** (группировка параллельных задач):
```
Task tool → fullstack-nextjs-specialist
Prompt: "Implement these 5 parallel tasks: T001, T002, T003, T004, T005"
```

✅ **ПРАВИЛЬНО** (1 задача = 1 запуск агента, все параллельно):
```
# Запустить все 5 одновременно в одном сообщении с 5 вызовами инструмента Task
```

## Распространенные подводные камни

**Группировка нескольких задач одному агенту (Speckit)**:
- Проблема: Основная сессия дает несколько задач одному агенту за один запуск
- Решение: Применить правило атомарности - 1 задача = 1 вызов агента
- Параллельные задачи: Запустить N агентов одновременно (одно сообщение, N вызовов Task)
- Последовательные задачи: Запустить 1 агент, ждать, проверять, затем следующий

**Оркестратор использует инструмент Task (Рабочие процессы проверки состояния)**:
- Проблема: Оркестратор пытается вызвать подагентов напрямую
- Решение: Удалить использование инструмента Task, создать файлы плана, вернуть управление

**Пропуск проверки плана**:
- Проблема: Недействительный план вызывает сбой работника
- Решение: Всегда использовать навык `validate-plan-file` после создания плана

**Отсутствие ведения журнала изменений**:
- Проблема: Невозможно откатить при сбое проверки
- Решение: Работники должны записывать все модификации файлов в журнал изменений

**Бесконечные циклы итераций**:
- Проблема: Оркестратор продолжает повторять без завершения
- Решение: Установить максимальное количество итераций (обычно 3) и отслеживать прогресс

**Блокировка без запроса пользователя**:
- Проблема: Оркестратор блокируется при сбое без взаимодействия с пользователем
- Решение: Сообщить о сбое, предоставить варианты (исправить/пропустить), ждать решения пользователя

## Использование сервера MCP

**Требования работника**:

**bug-hunter**:
- ДОЛЖЕН использовать Context7 для проверки шаблонов перед флагом ошибок
- Использовать CLI `gh` через Bash для проблем GitHub

**security-scanner**:
- ДОЛЖЕН использовать Context7 для лучших практик безопасности
- Использовать MCP Supabase для проверки безопасности базы данных (когда доступно)

**dependency-auditor**:
- Использовать npm audit (стандартный инструмент)
- Использовать CLI `gh` через Bash для проверки состояния пакета

**database-architect, infrastructure-specialist, integration-tester**:
- Использовать MCP Supabase для всех операций с базой данных
- Ссылка на проект: из среды или файла плана
- Миграции: путь, специфичный для проекта (например, `supabase/migrations/`)

**Стратегия резервного варианта**:
Если MCP недоступен:
1. Записать предупреждение в отчет
2. Продолжить с уменьшенной функциональностью
3. Отметить находки как "требует проверки MCP"
