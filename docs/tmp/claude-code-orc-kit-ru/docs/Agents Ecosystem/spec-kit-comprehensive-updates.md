# Комплексные обновления Spec-Kit

> **Назначение**: Этот документ определяет все требуемые обновления для интеграции улучшений оркестрации, фазы планирования, исследовательских задач и правил атомарности по всему рабочему процессу spec-kit.

---

## Приоритеты и источники

1. **ОСНОВНОЙ**: Правила оркестрации из CLAUDE.md (сбор контекста, проверка, делегирование)
2. **ВТОРИЧНЫЙ**: Новые требования из спецификации пользователя (фаза планирования, исследовательские задачи, атомарность)
3. **ТРЕТИЧНЫЙ**: Существующие шаблоны (сохранять совместимость, где возможно)

---

## Основные шаблоны для реализации

### A. ФАЗА ПЛАНИРОВАНИЯ (Новое - Первый шаг в задачах)

**Позиция**: Всегда первая фаза при выполнении tasks.md

**Ответственность**:
1. **Анализ задач**:
   - Просмотреть все задачи из tasks.md
   - Классифицировать каждую как ПАРАЛЛЕЛЬНОЕ или ПОСЛЕДОВАТЕЛЬНОЕ выполнение
   - Определить зависимости между задачами

2. **Назначение подагента**:
   - `[EXECUTOR: MAIN]` - ТОЛЬКО для тривиальных задач (исправления 1-2 строк, простые импорты, установка одной зависимости)
   - Для сложных задач: тщательно изучить существующих подагентов, назначить только при 100% совпадении
   - Если нет 100% совпадения: назначить имя БУДУЩЕГО агента (будет создан) - `[EXECUTOR: future-agent-name]`

3. **Отсутствующие подагенты**:
   - После всех назначений: если существуют БУДУЩИЕ агенты, запустить N вызовов meta-agent-v3 в одном сообщении
   - Каждый БУДУЩИЙ агент = 1 отдельный вызов meta-agent-v3 (правило атомарности)
   - После создания агента: попросить пользователя перезапустить claude-code

4. **Правило атомарности** (КРИТИЧНО):
   - **1 Задача = 1 вызов агента**
   - Никогда не давать несколько задач одному агенту за один запуск
   - **Параллельное выполнение**: Запустить N вызовов агента в одном сообщении (не последовательно)
   - Пример: 3 параллельные задачи для мета-агента → 3 вызова мета-агента в одном сообщении
   - Пример: 5 параллельных задач для fullstack → 5 вызовов fullstack в одном сообщении
   - Последовательные задачи: 1 запуск агента, ждать завершения, затем следующий запуск агента

**Выход**: Обновленный tasks.md с:
- аннотациями `[EXECUTOR: name]`
- маркерами `[SEQUENTIAL]` или `[PARALLEL-GROUP-X]`
- задачами создания мета-агента при необходимости

### B. ОБРАБОТКА ИССЛЕДОВАТЕЛЬСКИХ ЗАДАЧ (Новое - Во время планирования)

**Когда**: Фаза планирования, после назначения подагента

**Классификация**:
- **Простое исследование**: Вопросы с четким путем ответа, решаемые с помощью агента + существующих инструментов
- **Сложное исследование**: Вопросы без очевидных ответов, требующие глубокого изучения

**Рабочий процесс простого исследования**:
1. Агент идентифицирует вопрос
2. Агент использует доступные инструменты (Grep, Read, WebSearch, Context7, документация Supabase)
3. Агент формулирует ответ
4. Агент документирует находки в заметках планирования
5. Продолжить к следующей задаче

**Рабочий процесс сложного исследования**:
1. Агент идентифицирует сложный вопрос без четкого ответа
2. Агент создает подробный запрос на английском для deepresearch:
   - **Файл**: `{FEATURE_DIR}/research/{topic-slug}.md`
   - **Формат**: Вопрос исследования + контекст + ограничения + ожидаемый формат вывода
   - **Язык**: Английский (для совместимости с deepresearch)
3. Агент добавляет TODO: "Ждать результатов deepresearch по {topic}"
4. Агент ОСТАНАВЛИВАЕТСЯ и сообщает пользователю: "Требуется сложное исследование: {file-path}"
5. **Действие пользователя**: Запустить deepresearch с запросом, сохранить результаты
6. Агент читает результаты deepresearch
7. Агент включает находки в планирование
8. Продолжить к следующей задаче

**Выход**:
- Исследовательские находки документированы
- Запросы сложного исследования в `{FEATURE_DIR}/research/` при необходимости
- Обновленный tasks.md с решениями, основанными на исследованиях

### C. ПРАВИЛА ОРКЕСТРАЦИИ (Улучшенные)

**Роль главного агента**: Только оркестратор
- Выполнять тривиальные задачи напрямую (исправления 1-2 строк, импорты, установка одной npm зависимости)
- Делегировать ВСЕ сложные задачи подагентам
- НИКОГДА не реализовывать изменения в нескольких файлах напрямую
- НИКОГДА не реализовывать изменения логики напрямую

**Перед делегированием**:
1. **Собрать полный контекст** (ОБЯЗАТЕЛЬНО):
   - Прочитать существующий код в целевых файлах
   - Искать по базе кода похожие шаблоны (Grep, Glob)
   - Просмотреть соответствующую документацию (спецификации, ADR, исследования)
   - Проверить недавние коммиты в связанных областях (`git log -- path`)
   - Понять зависимости и точки интеграции
2. **Подготовить полный контекст** для подагента:
   - Фрагменты кода из связанных файлов
   - Пути к файлам (абсолютные)
   - Ожидаемый формат вывода
   - Критерии проверки (проверка типов, тесты, сборка)
   - Шаблоны для следования (примеры из базы кода)

**После делегирования**:
1. **Детальная проверка**:
   - Прочитать ВСЕ измененные файлы
   - Запустить `pnpm type-check` (или соответствующую проверку)
   - Запустить тесты, если применимо
   - Проверить регрессии в связанном коде
2. **Цикл принятия/отклонения**:
   - **Принять**: Если все проверки пройдены → отметить задачу как завершенную
   - **Отклонить**: Если проверки не пройдены → повторно делегировать с:
     - Сообщениями об ошибках (полный вывод проверки типов)
     - Конкретными необходимыми исправлениями
     - Дополнительным контекстом, если отсутствует
   - Цикл до принятия (без ограничений, качество важнее скорости)

**Стратегия коммитов** (РЕШЕНИЕ: Коммиты по задачам):
- **Когда**: После ОТМЕТКИ каждой задачи как завершенной
- **Команда**: `/push patch`
- **Перед коммитом**:
  1. Отметить задачу `[X]` в tasks.md
  2. Добавить артефакты к задаче: `→ Артефакты: [file1](path), [file2](path)`
  3. Обновить TodoWrite на завершено
  4. Затем запустить `/push patch`
- **Обоснование**:
  - Подробная история git (1 коммит = 1 задача)
  - Легкий откат к конкретной задаче
  - Лучше для ревью и отладки
  - Атомарные изменения в системе контроля версий

### D. РЕАЛИЗАЦИЯ МОДЕЛИ ВЫПОЛНЕНИЯ КОМАНД (Улучшенная)

**Обнаружение задач**:
- Найти ПЕРВУЮ незавершенную задачу в tasks.md (не последнюю)
- Уважать порядок фаз: Настройка → Тесты → Основное → Интеграция → Доработка
- Внутри фазы: уважать зависимости и последовательные маркеры

**Цикл выполнения** (На задачу):
```
1. ОБНОВИТЬ TODO: Отметить задачу `in_progress` в TodoWrite
2. ПРОВЕРИТЬ ИСПОЛНИТЕЛЯ:
   - [EXECUTOR: MAIN]? → Выполнить напрямую, если тривиально, иначе делегировать
   - [EXECUTOR: subagent-name]? → Делегировать указанному подагенту
3. СОБРАТЬ КОНТЕКСТ: (см. раздел Правила оркестрации B)
4. ВЫПОЛНИТЬ:
   - Напрямую: Использовать инструменты Edit/Write
   - Делегировано: Запустить инструмент Task с полным контекстом
5. ПРОВЕРИТЬ: (см. раздел Правила оркестрации C)
6. ПРИНЯТЬ/ОТКЛОНИТЬ:
   - Принять? → Продолжить до шага 7
   - Отклонить? → Повторно делегировать с исправлениями, перейти к шагу 4
7. ОБНОВИТЬ TODO: Отметить задачу `completed` в TodoWrite
8. ОБНОВИТЬ tasks.md: Отметить задачу `[X]`, добавить артефакты
9. КОММИТ: Запустить `/push patch`
10. СЛЕДУЮЩАЯ ЗАДАЧА: Перейти к следующей незавершенной задаче, перейти к шагу 1
```

**Критические правила**:
- НИКОГДА не пропускать проверку
- НИКОГДА не продолжать, если задача не удалась
- НИКОГДА не группировать коммиты (1 задача = 1 коммит)
- ОДНА задача in_progress одновременно (атомарное выполнение)

## Ключевые принципы (Сводка)

1. **Планирование первым**: Всегда выполнять фазу планирования перед реализацией
2. **Атомарность**: 1 задача = 1 вызов агента; параллельные задачи = N вызовов агента в одном сообщении
3. **Оркестрация**: Главный агент делегирует сложное, тщательно проверяет, повторно делегирует при сбое
4. **Исследование**: Классифицировать простое против сложного, использовать глубокое исследование для сложных неизвестностей
5. **Коммиты**: 1 задача = 1 коммит (после проверки, с артефактами, через /push patch)
6. **Контекст**: Собирать полный контекст перед каждым делегированием (код, документы, шаблоны, история, исследования)
7. **Проверка**: Прочитать файлы + проверка типов + тесты перед принятием завершения задачи
