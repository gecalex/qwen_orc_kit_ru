## Общие вопросы

### Что такое Claude Code Orchestrator Kit?

Claude Code Orchestrator Kit — это комплексная система автоматизации для Claude Code, которая предоставляет:
- 33+ специализированных AI-агентов для различных задач разработки
- 19+ команд с косой чертой для общих рабочих процессов
- 15+ повторно используемых навыков (утилитарных функций)
- 6 оптимизированных конфигураций серверов MCP
- Контрольные точки качества и мониторинг состояния

Он превращает Claude Code из разговорного помощника в мощную систему оркестровки для профессиональных рабочих процессов разработки.

---

### Для кого это?

Этот комплект предназначен для:
- **Профессиональных разработчиков**, создающих production-приложения с помощью Claude Code
- **Команд**, которым нужны стандартизированные рабочие процессы и контрольные точки качества
- **Проектов**, требующих автоматизированного тестирования, проверки безопасности и управления зависимостями
- **Всех**, кто хочет усилить свой опыт работы с Claude Code

---

### Нужно ли мне уметь программировать, чтобы использовать это?

**Базовое использование**: Для использования существующих агентов и команд с косой чертой не требуется программирование.

**Расширенное использование**: Знание TypeScript, Node.js и Git помогут для:
- Создания пользовательских агентов
- Изменения конфигураций MCP
- Расширения системы

---

### Совместимо ли это с существующими проектами Claude Code?

**Да!** Комплект разработан так, чтобы его можно было добавить к существующим проектам:

1. Скопируйте каталог `.claude/` в свой проект
2. Объедините с существующими агентами (если есть)
3. Настройте серверы MCP
4. Начните использовать команды с косой чертой

См. [MIGRATION-GUIDE.md](./MIGRATION-GUIDE.md) для получения подробных инструкций.

## Конфигурация MCP

### Когда следует использовать BASE или FULL конфигурацию MCP?

**Используйте BASE (~600 токенов)** для:
- Повседневной работы по разработке
- Проверки и рефакторинга кода
- Чтения и понимания кода
- Простых исправлений ошибок
- Работы с ограниченным окном контекста

**Используйте FULL (~5000 токенов)** для:
- Сложных рабочих процессов, требующих нескольких интеграций
- Работы с базой данных + UI + автоматизация в одной сессии
- Интенсивных сессий разработки (2+ часа)
- Работы с несколькими внешними сервисами

**Правило на пальец**: Начинайте с BASE (80% случаев использования), переходите к специализированным конфигурациям только при необходимости.

---

### Как мне понять, какую конфигурацию MCP использовать?

Используйте это дерево решений:

```
Нужна работа с базой данных?
├─ Да → SUPABASE или SUPABASE-FULL
└─ Нет → Продолжить

Нужна автоматизация рабочих процессов (n8n)?
├─ Да → N8N
└─ Нет → Продолжить

Нужна работа с UI/UX (Playwright, ShadCN)?
├─ Да → FRONTEND
└─ Нет → Продолжить

Нужно несколько интеграций одновременно?
├─ Да → FULL
└─ Нет → BASE
```

См. [PERFORMANCE-OPTIMIZATION.md](./PERFORMANCE-OPTIMIZATION.md) для сравнения использования токенов.

### Как переключаться между конфигурациями MCP?

```bash
# Интерактивное меню
./switch-mcp.sh

# Выберите опцию 1-6 в зависимости от ваших потребностей
# Затем перезапустите Claude Code
```

Скрипт копирует выбранную конфигурацию в `.mcp.json` и автоматически перезагружает ее.

---

### Что делать, если мне нужны пользовательские серверы MCP?

1. Создайте новый файл конфигурации: `mcp/.mcp.custom.json`
2. Добавьте свои серверы MCP
3. Обновите `switch-mcp.sh`, чтобы включить вашу конфигурацию
4. Используйте переменные окружения для секретов (`.env.local`)

Пример:
```json
{
  "mcpServers": {
    "my-custom-server": {
      "command": "npx",
      "args": ["-y", "my-mcp-server"],
      "env": {
        "API_KEY": "${MY_CUSTOM_API_KEY}"
      }
    }
  }
}
```

---

### Почему не загружаются переменные окружения?

**Распространенные проблемы:**

1. **Файл не существует**: Убедитесь, что `.env.local` присутствует
   ```bash
   cp .env.example .env.local
   ```

2. **Неправильный формат**: Не используйте кавычки вокруг значений
   ```bash
   # ✅ Правильно
   SUPABASE_PROJECT_REF=abc123xyz

   # ❌ Неправильно
   SUPABASE_PROJECT_REF="abc123xyz"
   ```

3. **Не перезапущен**: Переменные окружения загружаются при запуске Claude Code
   - Полностью закройте и перезапустите Claude Code (не только перезагрузите)

4. **Специальные символы**: Экранируйте специальные символы в значениях
   ```bash
   PASSWORD=my\$ecret\!pass
   ```

## Агенты и рабочие процессы

### В чем разница между Агентом и Навыком?

| Аспект | Агент | Навык |
|--------|-------|-------|
| **Вызов** | Через инструмент Task | Через инструмент Skill |
| **Контекст** | Изолированное окно контекста | Контекст вызывающего |
| **Назначение** | Многоступенчатые рабочие процессы | Одна утилитарная функция |
| **Размер** | Любой размер | <100 строк логики |
| **Состояние** | Состоятельный, отслеживает прогресс | Без состояния, чистая функция |
| **Отчеты** | Генерирует отчеты | Возвращает данные напрямую |
| **Примеры** | bug-hunter, security-scanner | parse-git-status, validate-plan-file |

**Правило на пальец**:
- Агент = автономный рабочий процесс, требующий координации
- Навык = повторно используемая утилита, которую можно вызвать отовсюду

---

### В чем разница между Оркестратором и Работником?

**Оркестратор**:
- Координирует многофазные рабочие процессы
- Создает файлы планов для работников
- Проверяет вывод работников на контрольных точках качества
- Управляет итерациями (например, макс. 3 цикла)
- НЕ вызывает работников напрямую (нарушает PD-1)
- Примеры: bug-orchestrator, security-orchestrator

**Работник**:
- Выполняет конкретные задачи из файлов планов
- Генерирует структурированные отчеты
- Возвращает управление оркестратору
- Выполняет проверку перед сообщением об успехе
- Примеры: bug-hunter, bug-fixer, security-scanner

**Шаблон**: Оркестратор планирует → Работник выполняет → Оркестратор проверяет → Повторить или завершить

---

### Что такое "Шаблон возврата управления"?

**Проблема**: Если оркестраторы вызывают работников через инструмент Task, контекст становится вложенным и сложным.

**Решение**: Шаблон возврата управления

```
1. Оркестратор создает файл плана (.tmp/current/plans/.workflow-plan.json)
2. Оркестратор сигнализирует "Готов к {worker-name}" и выходит
3. Главная сессия вызывает работника вручную
4. Работник читает план, выполняет, генерирует отчет, выходит
5. Главная сессия вызывает оркестратора для возобновления
6. Оркестратор проверяет отчет и продолжает
```

**Почему?** Сохраняет чистоту контекста, позволяет откат, предотвращает бесконечные циклы.

См. [ARCHITECTURE.md](./ARCHITECTURE.md) для визуальной диаграммы.

---

### Как работают контрольные точки качества?

Контрольные точки качества — это автоматизированные контрольные точки проверки, которые обеспечивают качество кода перед продолжением.

**Типы**:
- **Блокирующие**: ДОЛЖНЫ пройти, иначе рабочий процесс останавливается (например, проверка типов, сборка)
- **Неблокирующие**: Необязательные проверки (например, тесты, линтинг)

**Использование в файлах планов**:
```json
{
  "validation": {
    "required": ["type-check", "build"],
    "optional": ["tests"]
  }
}
```

**Поведение**:
- Если блокирующая точка не проходит → ОСТАНОВИТЬ рабочий процесс, сообщить об ошибке пользователю, спросить исправить/пропустить
- Если неблокирующая точка не проходит → Записать предупреждение, продолжить
- Все точки проходят → Перейти к следующей фазе

**Навык**: `run-quality-gate` выполняет команды проверки.

### Могу ли я использовать агентов в конвейерах CI/CD?

**Да!** Пример рабочего процесса GitHub Actions:

```yaml
name: Проверка состояния

on:
  schedule:
    - cron: '0 0 * * 1'  # Еженедельно в понедельник
  workflow_dispatch:

jobs:
  health-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Запустить сканирование ошибок
        run: |
          claude-code /health-bugs

      - name: Загрузить отчет
        uses: actions/upload-artifact@v3
        with:
          name: bug-report
          path: bug-hunting-report.md
```

См. `.github/workflows/` для примеров.

### Что такое CLAUDE.md и почему это важно?

**CLAUDE.md** — это "Поведенческая операционная система" для всех агентов. Она определяет:

- **Основные директивы (PDs)**: Непреложные правила (например, шаблон возврата управления, контрольные точки качества)
- **Поведенческие контракты**: Что обещают сделать оркестраторы/работники/MCP
- **Стратегии отката**: Как справиться с недоступностью Context7, сбоями контрольных точек качества и т.д.
- **Аварийные протоколы**: Обнаружение бесконечных циклов, повреждение файлов, исчерпание токенов
- **Операционные процедуры**: Структура файлов, выбор агентов, формат файла плана

**Все агенты ДОЛЖНЫ соблюдать** CLAUDE.md. Это обеспечивает согласованность, предотвращает анти-паттерны и позволяет самодиагностику.

Думайте об этом как о "конституции" для экосистемы агентов.

## Разработка

### Как добавить новый сервер MCP к существующей конфигурации?

1. Отредактируйте файл конфигурации MCP (например, `mcp/.mcp.base.json`):
   ```json
   {
     "mcpServers": {
       "existing-server": { ... },
       "new-server": {
         "command": "npx",
         "args": ["-y", "new-mcp-package"],
         "env": {
           "API_KEY": "${NEW_SERVER_API_KEY}"
         }
       }
     }
   }
   ```

2. Добавьте переменную окружения в `.env.local`:
   ```bash
   NEW_SERVER_API_KEY=your-key-here
   ```

3. Перезапустите Claude Code

## Устранение неполадок

### Контрольные точки качества продолжают не проходить даже после исправлений

**Диагностика**:
```bash
# Запустите контрольные точки вручную, чтобы увидеть полные ошибки
npm run type-check
npm run build
npm run test

# Проверьте наличие проблем с конфигурацией
cat .claude/scripts/gates/check-*.sh
```

**Распространенные исправления**:
- Обновите строгие настройки TypeScript
- Исправьте конфигурацию теста (jest.config.js)
- Проверьте наличие циклических зависимостей
- Убедитесь, что каталог вывода сборки существует

---

### Рабочий процесс оркестратора застрял или не продвигается

**Проверьте артефакты прогресса**:
```bash
# 1. Проверьте существование и действительность файла плана
cat .tmp/current/plans/*.json

# 2. Проверьте, сгенерировал ли работник отчет
ls -la .tmp/current/reports/

# 3. Проверьте журнал изменений
cat .tmp/current/changes/*.json

# 4. Проверьте наличие бесконечного цикла
# (оркестратор должен использовать навык check-infinite-loop)
```

**Распространенные проблемы**:
- Работник молча не сработал (отчет не сгенерирован)
- Файл плана не содержит поля `nextAgent`
- Контрольная точка блокирует без запроса пользователя
- Обнаружен бесконечный цикл (один и тот же агент вызывается 3+ раза)

**Решение**: Проверьте журналы оркестратора, вручную вызовите следующий шаг или перезапустите рабочий процесс.

---

### Сервер MCP Context7 не работает

**Симптомы**: Предупреждения "Context7 недоступен" в отчетах

**Диагностика**:
```bash
# Проверьте, есть ли Context7 в активной конфигурации MCP
cat .mcp.json | grep context7

# Протестируйте Context7 напрямую
# (инструменты MCP Claude Code должны показывать инструменты mcp__context7__*)
```

**Исправления**:
1. Убедитесь, что `context7` есть в вашей конфигурации MCP
2. Проверьте подключение к интернету (Context7 работает в облаке)
3. Убедитесь, что нет брандмауэра, блокирующего исходящие запросы
4. Переключитесь на BASE или более высокую конфигурацию (Context7 включена)

**Откат**: Работники будут продолжать работу с уменьшенной уверенностью, если Context7 недоступен.

### Отчеты неполные или отсутствуют разделы

**Диагностика**: Используйте навык `validate-report-file`

```
Спросите Claude Code: "Use validate-report-file skill to check {report-path}"
```

**Распространенные отсутствующие разделы**:
- Итоговое резюме
- Внесенные изменения
- Результаты проверки
- Следующие шаги

**Исправление**: Работники ДОЛЖНЫ следовать формату REPORT-TEMPLATE-STANDARD.md. Обновите агента-работника, если он не соответствует требованиям.

---

### Агент говорит "violates PD-1" или аналогичное

**Объяснение**: Агент обнаружил, что вот-вот нарушит основную директиву (PD) из CLAUDE.md.

**Распространенные PDs**:
- **PD-1**: Оркестраторы не могут вызывать работников через инструмент Task
- **PD-2**: Нельзя пропускать блокирующие контрольные точки без подтверждения пользователя
- **PD-3**: Должны регистрировать все изменения файлов
- **PD-4**: Должны проверяться с помощью Context7, когда он доступен
- **PD-6**: Работники должны генерировать структурированные отчеты

**Действие**: Это ПРАВИЛЬНОЕ поведение. Агент предотвращает анти-паттерн. Настройте рабочий процесс в соответствии с PD.

---

### Бюджет токенов исчерпан до завершения рабочего процесса

**На 180k токенов**:
- Агенты автоматически упрощаются (минимальные отчеты, только основной Context7)

**На 195k токенов**:
- Рабочий процесс ОСТАНАВЛИВАЕТСЯ
- Генерируется аварийное резюме
- Пользователь должен начать новую сессию

**Стратегии предотвращения**:
1. Используйте конфигурацию BASE для ежедневной работы (экономит 90% токенов)
2. Разбивайте большие рабочие процессы на более мелкие части
3. Используйте специализированные конфигурации только при необходимости
4. Периодически очищайте контекст (команда `/clear`)

См. [PERFORMANCE-OPTIMIZATION.md](./PERFORMANCE-OPTIMIZATION.md) для получения подробной информации.

---

### Как откатить изменения после неудачного рабочего процесса?

**Автоматический откат**: Работники регистрируют изменения в `.tmp/current/changes/{workflow}-changes.json`

**Ручной откат**:
```
Спросите Claude Code: "Use rollback-changes skill to restore files from last workflow"

Навык:
1. Прочитает журнал изменений
2. Восстановит резервные копии файлов из .tmp/current/backups/
3. Откатит изменения файлов
4. Сообщит восстановленные файлы
```

**Предотвращение**: Контрольные точки качества должны отлавливать сбои до фиксации изменений.
