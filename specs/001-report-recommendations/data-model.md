# Модель данных: Реализовать рекомендации из отчета сравнения архитектур

## Обзор

В рамках реализации рекомендаций из отчета сравнения CLAUDE.md и QWEN.md были идентифицированы следующие ключевые сущности, связанные с архитектурными документами и процессами:

## Сущности

### 1. Архитектурный документ (Architecture Document)

**Поля**:
- ID: уникальный идентификатор документа
- Название: имя документа (например, QWEN.md, CLAUDE.md)
- Версия: номер версии документа
- Статус: черновик, утвержден, устарел
- Дата создания: дата создания документа
- Дата обновления: дата последнего обновления
- Авторы: список авторов документа
- Описание: краткое описание содержания документа

**Отношения**:
- Может содержать много разделов (Section)
- Может быть связан с несколькими отчетами (Report)

**Правила проверки**:
- Название должно быть уникальным в пределах проекта
- Версия должна соответствовать формату семантического версионирования
- Статус должен быть одним из: черновик, утвержден, устарел

### 2. Раздел документа (Document Section)

**Поля**:
- ID: уникальный идентификатор раздела
- Название: название раздела
- Содержание: текстовое содержание раздела
- Тип: функциональный, нефункциональный, технический, организационный
- Статус: активный, временный, устаревший
- Ссылки: список внутренних и внешних ссылок

**Отношения**:
- Принадлежит одному архитектурному документу (Architecture Document)
- Может содержать много подразделов (Subsection)

**Правила проверки**:
- Название должно быть уникальным в пределах родительского документа
- Содержание не должно быть пустым

### 3. Рекомендация (Recommendation)

**Поля**:
- ID: уникальный идентификатор рекомендации
- Заголовок: краткое описание рекомендации
- Описание: полное описание рекомендации
- Тип: паттерн, подход, фаза, соглашение
- Приоритет: высокий, средний, низкий
- Статус: не реализована, в процессе, реализована
- Дата создания: дата создания рекомендации
- Дата реализации: дата завершения реализации

**Отношения**:
- Принадлежит отчету (Report)
- Может быть применена к нескольким архитектурным документам (Architecture Document)

**Правила проверки**:
- Приоритет должен быть одним из: высокий, средний, низкий
- Статус должен соответствовать жизненному циклу реализации

### 4. Отчет (Report)

**Поля**:
- ID: уникальный идентификатор отчета
- Название: название отчета
- Тип: сравнительный, аналитический, технический
- Автор: автор отчета
- Дата создания: дата создания отчета
- Статус: черновик, утвержден, архивный
- Содержание: текстовое содержание отчета

**Отношения**:
- Может содержать много рекомендаций (Recommendation)
- Может сравнивать несколько архитектурных документов (Architecture Document)

**Правила проверки**:
- Тип должен быть одним из: сравнительный, аналитический, технический
- Статус должен быть одним из: черновик, утвержден, архивный

### 5. Паттерн (Pattern)

**Поля**:
- ID: уникальный идентификатор паттерна
- Название: название паттерна
- Описание: описание применения паттерна
- Категория: архитектурный, проектирования, организационный
- Примеры: примеры использования
- Статус: предложен, утвержден, отклонен

**Отношения**:
- Может быть частью архитектурного документа (Architecture Document)
- Может быть связан с несколькими рекомендациями (Recommendation)

**Правила проверки**:
- Категория должна быть одной из: архитектурный, проектирования, организационный
- Статус должен быть одним из: предложен, утвержден, отклонен

### 6. Подход (Approach)

**Поля**:
- ID: уникальный идентификатор подхода
- Название: название подхода
- Описание: описание сути подхода
- Область применения: где применяется подход
- Преимущества: преимущества подхода
- Недостатки: недостатки подхода
- Статус: предложен, утвержден, отклонен

**Отношения**:
- Может быть частью архитектурного документа (Architecture Document)
- Может быть связан с несколькими рекомендациями (Recommendation)

**Правила проверки**:
- Статус должен быть одним из: предложен, утвержден, отклонен
