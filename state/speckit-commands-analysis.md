## 1. СТАНДАРТЫ QWEN CODE CLI ДЛЯ КОМАНД

### 1.1. Структура файла команды

**Стандартный формат:**
```markdown
---
description: Краткое описание команды (одна строка)
handoffs:
  - label: Название handoff
    agent: agent-name
    prompt: Текст промпта для handoff
    send: true|false
---

## Ввод пользователя

```text
$ARGUMENTS
```

Вы **ДОЛЖНЫ** учитывать ввод пользователя перед продолжением (если не пустой).

## Описание команды

Подробное описание функциональности.

## Процесс выполнения

1. Этапы выполнения
2. Какие агенты задействованы
3. Обработка ошибок

## Возвращаемое значение

Описание результата
```

### 1.2. Требования к YAML заголовку

**Обязательные поля:**
- ✅ `description` — краткое описание (одна строка)
- ⚠️ `handoffs` — опционально, но рекомендуется для команд с передачей задач

**Формат handoffs:**
```yaml
handoffs:
  - label: Название
    agent: agent-name
    prompt: Промпт
    send: true  # true для отправки контекста
```

### 1.3. Требования к содержимому

**Обязательные разделы:**
1. ✅ "## Ввод пользователя" — с `$ARGUMENTS`
2. ✅ "## Описание команды" — или аналогичный
3. ✅ "## Процесс выполнения" — или "## Структура"
4. ⚠️ "## Возвращаемое значение" — рекомендуется

**Стиль написания:**
- Использовать **жирный текст** для важных элементов
- Использовать `код` для путей, команд, переменных
- Нумерованные списки для последовательных шагов
- Маркированные списки для опций

## Описание команды

Создает новую спецификацию функции на основе описания пользователя. Анализирует требования, определяет пользовательские сценарии и генерирует функциональные требования.

## Процесс выполнения

### 1. Создание имени ветки

Создайте краткое имя (2-4 слова) для ветки:
- Проанализируйте описание функции
- Извлеките ключевые слова
- Используйте формат действие-существительное

**Примеры:**
- "Аутентификация пользователя" → "user-auth"
- "Интеграция OAuth2 для API" → "oauth2-api-integration"

### 2. Проверка существующих веток

```bash
git fetch --all --prune
```

Найдите самый высокий номер функции:
- Удаленные ветки: `git ls-remote --heads origin`
- Локальные ветки: `git branch`
- Каталоги спецификаций: `.qwen/specify/specs/`

### 3. Создание новой ветки

Запустите скрипт:
```bash
.qwen/specify/scripts/create-new-feature.sh --json "$ARGUMENTS" --number N+1 --short-name "name"
```

### 4. Генерация спецификации

Загрузите шаблон `.qwen/specify/templates/spec-template.md` и заполните разделы:
1. Бизнес-контекст
2. Пользовательские потребности
3. Критерии успеха
4. Функциональные требования
5. Нефункциональные требования

### 5. Проверка качества

Создайте контрольный список в `FEATURE_DIR/checklists/requirements.md`:
- Проверьте полноту разделов
- Проверьте тестируемость требований
- Проверьте критерии успеха

### 6. Возврат результата

Сообщите:
- Имя ветки
- Путь к спецификации
- Результаты проверки
- Готовность к следующему этапу

## Возвращаемое значение

- ✅ УСПЕХ: Спецификация создана в `.qwen/specify/specs/{ID}-{name}/spec.md`
- ❌ ОШИБКА: "Не предоставлено описание функции" или "Невозможно определить пользовательские сценарии"
```
